# BK-Lite Mobile Design

> BK-Lite Mobile 的移动端界面、交互、响应式、文案和基础开发约束。具体业务功能由产品规格定义；本文只约定开发时必须遵守的体验边界。

## 1. 定位

Mobile 是 BK-Lite 的移动 / 桌面壳入口，服务于查看、搜索、通知、确认、协同和轻量处置。它不是 Web 控制台的缩小版。

复杂配置、建模、编排、权限管理和大屏编辑默认回到 Web。移动端页面应围绕单个用户任务组织信息，避免把 Web 的表格、抽屉、多列表单和复杂筛选栏直接搬到小屏。

## 2. 设计基线

Mobile 视觉基线参考 `web/DESIGN.md`：

- 主色、状态色、字体、圆角、间距尽量与 Web 同源。
- 新样式优先使用全局 token 或语义变量，不在组件里硬编码品牌色和状态色。
- 风格保持克制、清晰、可扫读，避免营销感、大卡片堆叠、厚重阴影和装饰性渐变。
- 状态色必须配文字或图标，不只靠颜色表达含义。

## 3. 移动端交互原则

- 手机端按单列任务流设计，不直接复刻 Web 后台布局。
- 一级导航控制在 4-5 个以内；更多入口放到“全部应用”或个人中心。
- 页面深度尽量不超过 3 层：列表 / 详情 / 操作确认。
- 每个页面最多一个主操作；危险操作必须隔离展示并二次确认。
- 所有异步操作必须有 loading、disabled 或进度反馈，避免重复提交。

## 4. 布局与响应式

- 以 375-430px 手机宽度为主要设计基准，最低兼容 320px。
- 页面不得出现整页横向滚动；宽内容只能在局部区域滚动。
- 主要点击热区不小于 44px x 44px。
- 顶部栏、底部导航、底部操作区、输入框必须适配 safe area，不遮挡内容。
- 固定定位元素必须同时兼容 H5 浏览器和 Tauri WebView。
- 平板或宽屏壳可以使用列表 + 详情分栏，手机端必须保持单列。

## 5. 组件使用

- 移动端优先使用 `antd-mobile` 或项目移动端已有组件。
- Ant Design Web 组件不能直接搬到手机主流程；表格应转为列表卡片，复杂筛选应转为筛选面板。
- 表单默认纵向 label；长表单应拆步骤、分组，或跳转 Web 完成。
- 轻确认用 Dialog，复杂选择用 Bottom Sheet / Action Sheet，全屏任务用独立页面。
- 图标使用统一图标库，禁止 emoji 或多套风格混用。

## 6. 文案与国际化

- 所有用户可见文案必须走项目 i18n，不允许在组件中硬编码中文或英文。
- 新增文案必须同步维护 `zh.json` 和 `en.json`，key 保持对齐。
- 按钮、Toast、Dialog、Empty、Error、Form validation、Tooltip、占位文案都属于用户可见文案。
- 日期、时间、数字、单位、文件大小应使用本地化格式，不手写拼接。
- 英文通常更长，按钮、标签、Tab、卡片标题、筛选项必须能换行、省略或进入更多菜单。
- 技术对象名、资源名、主机名、IP、ID、路径、命令不翻译，只翻译周边说明。

## 7. 状态与反馈

- 页面必须覆盖 loading、empty、error、offline、unauthorized、no permission 等基础状态。
- 空状态要说明为什么为空，以及用户下一步可以做什么。
- 错误状态要显示可行动信息：重试、检查配置、联系管理员、返回 Web 等。
- 高风险或执行类状态不能只有成功 / 失败，应区分执行中、部分成功、取消、失败、完成。
- Toast 只承载轻反馈；复杂失败应进入详情或结果页。

## 8. 安全与高危操作

- Mobile 不提供任意脚本、任意 SQL、任意命令编辑入口。
- 执行、删除、关闭、重试、批量操作必须有明确对象、后果说明和二次确认。
- 权限必须以后端校验为准，前端隐藏按钮不能作为安全边界。
- 操作结果必须可追踪、可审计；失败原因不能只显示“操作失败”。

## 9. 工程兼容

- Mobile 页面必须兼容项目既有 H5 与 Tauri 运行形态，不在页面组件中写死部署路径、后端地址或运行环境假设。

## 10. 提交前检查

- `pnpm lint` 通过。
- `pnpm type-check` 通过。
- zh/en locale key 对齐。
- 无新增硬编码用户文案。
- 320、375、390、412、768 宽度下无整页横向滚动。
- 中文、英文、长资源名、空态、错误态、加载态均可读可操作。
- 底部操作区、输入框、导航不被 safe area 或键盘遮挡。
