# BK-Lite Mobile Design

> 本文约束 BK-Lite Mobile 的视觉、交互、响应式、内容与可访问性体验。业务能力及领域规则以 `PRODUCT.md`、`CONTEXT.md` 和 `specs/capabilities/` 为准；代码规范与验证命令以 `specs/capabilities/frontend-engineering.md`、`specs/capabilities/engineering-quality.md` 和 `DEVELOP.md` 为准。

## 0. 文档职责与用语

本文描述用户可观察的结果和跨页面保持一致的设计取舍，不规定某个业务流程的路由算法，也不绑定具体组件名、文件结构或浏览器实现方式。

- **必须 / 不得**：适用范围内不可省略的体验、安全或可访问性要求。
- **应 / 默认**：推荐方案；存在明确任务需求时可以偏离，但应保证同等体验并在交付说明中说明原因。
- **可以 / 例如**：候选方案，不构成唯一实现。

## 1. 定位

Mobile 是 BK-Lite 面向移动场景和桌面壳场景的产品入口，重点支持查看、搜索、通知、确认、协同和轻量处置。它不是 Web 控制台的等比例缩小版。

页面应围绕单个用户任务组织信息。复杂配置、建模、编排、权限管理和大屏编辑默认由更适合的界面承载；具体能力是否在 Mobile 提供，由产品规格决定。

## 2. 视觉基线

Mobile 与 `web/DESIGN.md` 共享品牌和语义体系，Mobile 运行时颜色值以 `src/styles/variables.css` 为准。

- 相同语义应在 Web 与 Mobile 间保持一致；因触摸、尺寸或可读性需要产生差异时，应使用明确的 Mobile 语义变量表达。
- 新增主题相关语义变量时，应同时提供亮色与暗色值。
- 组件应使用语义 token，不得在组件内硬编码品牌色、状态色或只适用于单一主题的中性色。
- 状态不得只依靠颜色表达，应同时使用文案、图标、形状或其他可感知信息。
- 默认风格保持克制、清晰、可扫读，避免把营销页、大屏或装饰性视觉模式直接带入日常操作界面。

## 3. 信息架构与交互

- 手机端默认采用单列任务流，不直接复刻 Web 的表格、抽屉和复杂筛选布局。
- 一级导航通常不超过 5 个；具体层级和入口位置由当前任务的信息架构决定。
- 页面层级应避免不必要的跳转，常用任务应能在较短路径内完成。
- 页面和操作区域应有清晰的主次关系；同一区域默认只突出一个主操作。
- 用户主动触发且持续可感知的异步操作，应提供与等待时间和风险相称的反馈；提交类和高风险操作还应防止重复触发。
- 不可逆、影响范围较大或后果高风险的操作，必须说明对象和后果，并提供与风险相称的确认机制。
- 具有明确上一级的二级页面必须保留可见返回入口，并在运行环境支持时响应平台标准返回手势或系统返回操作。返回应优先关闭最上层临时界面，再恢复上一页及其状态；缺少可信的应用内历史时，应返回预先定义的安全父页面。一级页面应作为应用内返回栈边界，不应继续通过滑动穿透到更早的业务页面，也不应从该边界重新进入已退出的子页面。
- 不得使用全页面自定义横向手势覆盖系统边缘返回，也不得让手势成为唯一返回方式。页面内已有明确横向业务手势时，应规定手势区域和优先级；例如对话详情的右滑用于打开会话侧栏，不同时承担页面返回。
- 对以远程数据为主、允许安全重复读取且用户有时效性预期的列表，应提供下拉刷新或等效刷新能力。下拉刷新仅在主要纵向滚动区域位于顶部时触发，不得与横向滑动、系统导航或业务操作手势冲突。
- 列表刷新期间应保留已有内容和滚动上下文，提供非阻塞进度反馈；刷新失败时不得清空已有数据，并应提供可行动的错误反馈。搜索结果、表单、实时或自动刷新页面，以及高成本或不可安全重复执行的请求，不得机械启用下拉刷新。手势不得作为关键刷新能力的唯一入口。

## 4. 布局、响应式与输入

- 以 375–430px 手机宽度为主要设计区间，最低支持 320px；宽屏布局应根据可用空间渐进增强。
- 页面不得产生非预期的整页横向滚动。命令、日志、宽表格等确需横向浏览的内容，应限制在明确的局部区域内。
- 主要触摸目标不小于 44px × 44px；相邻操作应保留足够间距，避免误触。
- 顶部栏、底部导航、底部操作区和持续输入区域不得被系统安全区、浏览器工具栏或软键盘遮挡。
- 每个安全区边缘应有明确且唯一的布局责任方，避免同一边缘被多层重复叠加。
- 手机端默认保持单列；只有当任务确实受益且各区域仍具备可用宽度时，才采用分栏或并列布局。
- 会话、终端等持续输入页面应保持输入操作可用，并以一个主要纵向滚动区域承载内容，避免页面与内容区形成同方向的嵌套滚动。
- 软键盘展开和收起时，当前输入与相关内容应保持可见，页面不应出现不可恢复的跳动、遮挡或滚动锁死。实现可以使用动态视口、可视视口或平台提供的等效能力。
- 针对特定浏览器或 WebView 的防护不得全局破坏缩放、滚动、焦点或其他页面的输入行为。

## 5. 组件与排版

- 默认优先使用 `antd-mobile` 和 Mobile 已有模块。使用 Ant Design Web 组件前，应确认其触摸、尺寸、可访问性和目标 WebView 兼容性适合当前场景。
- 表格、筛选、表单和选择器应根据任务的信息密度与操作方式适配小屏；卡片、列表、局部滚动、筛选面板和独立页面都是可选方案，不规定唯一形态。
- 轻量确认通常使用 Dialog；需要保留上下文的选择可以使用 Bottom Sheet 或 Action Sheet；信息量较大或步骤较长的任务可以使用独立页面。
- 操作图标和状态图标应来自统一图标体系；用户内容、资源名称和智能体输出不受此限制。
- 页面标题默认使用 `20px / 1.5`，分组标题默认使用 `16px / 1.5`，正文默认使用 `14px / 1.5`，辅助信息通常不低于 `12px / 1.5`。可根据信息层级调整，但不得为局部页面随意创建缺乏语义的平行字号。
- 新建共享抽象前，应先确认已有模块不能覆盖，并以稳定的共同交互和真实消费方为依据；通用 primitive 可以基于明确的复用价值和独立契约提前建立。

## 6. 内容、国际化与可访问性

- 系统生成的用户可见界面文案应通过项目 i18n 管理；新增 key 应同步维护支持的语言。
- 日期、时间、数字和单位等与地区相关的展示，应使用本地化能力，不手写特定语言格式。
- 用户输入、资源标识符、主机名、IP、ID、路径和命令应保留原值；其周边说明和系统定义的类型名称可以本地化。
- 中文、英文和长资源名应能安全换行、省略或按需展示完整内容，不得依赖某一种语言长度设计布局。
- 可点击元素应使用正确的交互语义并支持键盘或辅助技术；图标按钮必须有可感知名称，焦点状态必须可见。
- 动画和过渡不应成为理解状态或完成操作的前提，并应尊重系统的减少动态效果偏好。

## 7. 状态与反馈

- 页面应覆盖与其能力相关的 loading、empty、error、offline、unauthorized、no permission、disabled 等状态，不要求无关页面机械实现全部状态。
- 包含用户或权限范围数据的客户端缓存必须按账号和适用的团队范围隔离；登出、认证失效和权限范围切换时必须失效相关缓存并终止未完成请求。异步结果只能写回发起请求时的同一作用域。
- 当用户可以采取下一步时，空状态和错误状态应提供简明原因与可行动选项；纯展示型空状态可以保持简洁。
- 不得把业务上有意义的执行中、部分成功、取消、失败和完成等状态折叠成误导性的单一结果；仅展示该能力实际支持的状态。
- Toast 适合短暂、无需进一步处理的反馈；需要用户理解原因、范围或补救步骤的结果，应使用持久提示、详情或结果页面。
- 前端显示和隐藏只能改善体验，不能替代后端权限校验。

## 8. 运行形态与兼容性

- Mobile 应支持项目声明的 H5 与 Tauri 运行形态；平台专属能力只需在其适用范围内满足相应约束。
- 关键交互应根据本次改动的影响范围考虑 Android System WebView、iOS WKWebView 和移动端浏览器差异，不要求与改动无关的平台重复验证。
- 使用较新的 CSS 属性或 Web API 前，应确认目标运行环境的支持情况；关键路径需要能力检测、可用降级或明确的最低版本约束。
- 兼容性验证应覆盖受影响交互的状态变化，例如初始、打开、操作、关闭、跳转和恢复，而不只检查静态布局。
- 平台特例应尽量封装在稳定的模块内，页面只依赖其结果，避免在多个调用方分散平台判断。

## 9. 设计验收

工程门禁和运行命令不在本文重复，按 `DEVELOP.md` 与工程质量规格执行。涉及界面体验的改动至少检查：

- 最低支持宽度、主要手机宽度和受影响宽屏断点下，无非预期遮挡或整页横向滚动。
- 亮色、暗色、中文、英文、长文本及适用的 loading、empty、error、disabled 状态可读可操作。
- 主要触摸目标、键盘焦点、图标名称、对比度和动态效果满足可访问性要求。
- 受影响的顶部栏、底部导航、弹层和输入区域不被安全区或软键盘遮挡。
- 持续输入页面在键盘展开、收起和视口变化时保持输入与主要内容可用。
- 按改动影响范围验证相关运行形态；未覆盖的平台或设备风险应在交付说明中明确。
