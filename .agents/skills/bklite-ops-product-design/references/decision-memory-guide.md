# 产品决策记忆维护规则

## 读取顺序

1. 读取 `global-product-principles.md`。
2. 按模块读取对应文档，例如 `patch-management.md`。
3. 以最新有效的“已确认决策”为准；规格与记忆冲突时必须指出，不得自行择一。

## 记录标准

只记录未来可能再次影响判断的内容：

- 产品定位和版本边界；
- 已确认保留或后置的能力；
- 架构/扩展机制选择及原因；
- 状态语义、安全边界和用户明确偏好；
- 已推翻或被替代的旧判断。

不要记录临时措辞、普通实现细节或完整讨论过程。

## 文档结构

每份模块记忆使用：

```markdown
# <模块>产品决策记忆

- 最近更新：YYYY-MM-DD
- 当前规格：<相对路径>

## 产品定位
## 已确认范围
## 已确认设计决策
## 明确后置
## 仍待确认
## 已替代决策
## 决策来源
```

决策条目尽量写成“结论 + 原因/边界”，并引用规格或提交。没有待确认事项时写“无”，不要留下 `TODO`。

## 状态规则

- **已确认**：用户明确批准，后续默认继承。
- **当前假设**：为继续设计暂用，必须在规格落地前确认。
- **待确认**：会显著改变范围、数据模型或执行安全。
- **已替代**：保留旧结论和替代日期，防止旧讨论再次复活。

## 更新纪律

- 用户改变已确认决策时，同步更新规格和记忆。
- 新证据只触发受影响决策的复审，不重开无关范围。
- 不把外部厂商的当前生命周期、价格或功能状态固化为永久事实；需要时重新查官方来源。
