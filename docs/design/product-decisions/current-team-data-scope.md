# current_team 数据权限产品决策记忆

- 最近更新：2026-07-20
- 当前规格：`docs/superpowers/specs/2026-07-20-node-monitor-log-current-team-permission-design.md`

## 产品定位

`current_team` 是用户当前工作上下文，也是已接入组织数据权限的业务对象硬边界。超级管理员拥有当前组织内的全部功能和动作，但不拥有跨当前组织查看业务数据的隐式能力。

## 已确认范围

- 首期整改覆盖 Node Management、Monitor、Log 三个 App；
- 覆盖已接入组织级或对象级数据权限的用户 HTTP 接口及其同步 NATS/RPC；
- 当前没有数据权限设计的公共列表和平台目录不新增权限；
- 后台自发任务权限模型暂不重构，用户入口仍需在提交任务前校验目标。

## 已确认设计决策

1. 超级管理员只绕过菜单、按钮、模块和动作授权，不绕过 current_team 数据边界。
2. 默认只访问 current_team；仅显式 `include_children=1` 时包含当前组织下的授权子组织，超级管理员遵守相同开关。
3. 缺少、非法或空 current_team 时 fail-closed；`None` 和空列表不能表示全平台组织。
4. 对象数据范围必须先按 current_team 收窄，再与组织级/实例级对象权限求交，实例 ID 授权不能跨组织穿透。
5. current_team 限制当前可操作的源对象；目标 `organizations` 使用独立的组织管理/分配授权。
6. 超级管理员可以把当前有权源对象分配给任意合法组织；普通用户只能分配给自己有权管理的组织。
7. 组织共享和转移属于正常业务能力，不因目标组织不在 current_team 而禁止。
8. 批量请求必须先完成全部源对象和目标组织校验，任一越权时零副作用。
9. 权限整改必须同时验证拒绝路径和合法创建、共享、转移、子组织查看、批量操作等真实业务闭环。
10. 已接数据权限但自身没有组织字段的数据，必须声明唯一权限归属根：直接归属按自身组织，继承归属沿固定父链，外部数据只能从已授权本地入口对象查询。
11. 监控和日志的告警、事件、原始数据、快照等固定继承所属策略；采集配置和指标数据固定继承所属实例；VictoriaLogs 原始数据固定继承日志分组。
12. 节点安装状态、任务节点和动作记录固定继承 Node；任务主记录只投影当前可见任务节点，不能通过主任务 ID 返回或操作其他组织节点。
13. 根对象共享、转移或调整组织后，历史子数据动态跟随根对象当前组织；父链缺失、父对象不存在或多父标识不一致时 fail-closed。
14. 子数据不重复持久化组织权限；列表、详情、统计、导出和自定义 action 必须复用同一个权限归属根解析器。
15. 多节点共享配置的读取以当前组织可见绑定节点为入口；修改、删除或解绑必须额外覆盖全部受影响组织授权，并保留授权范围内拆分/复制的可用路径。

## 明确后置

- 全项目其他 App 的 current_team 权限统一；
- 后台定时任务、Sidecar 回传和系统初始化的统一 system actor 模型；
- 全局权限中间件、自动过滤 Manager 或统一授权 DSL；
- 当前无组织权限的公共目录是否需要租户化。

## 仍待确认

无。

## 已替代决策

- 2026-07-07《监控策略与条件对象级权限围栏设计》中的“超管保持现有全量能力”已被替代。新语义是“超管保持当前组织内全量功能和动作能力”。
- “目标 organizations 必须是 current_team 范围的子集”已被替代。目标组织应按用户独立的组织管理/分配授权校验，以保证共享和转移可用。

## 决策来源

- 用户于 2026-07-20 明确确认超级管理员也必须受当前所选组织限制；
- 用户确认只整改已经涉及数据权限的路径，不处理当前无权限的公共列表；
- 用户确认后台自发任务权限暂不扩张；
- 用户确认采用“统一契约 + 三个 App 模块内适配”方案；
- 用户确认组织分配的业务语义是“把源对象分配给自己有权限的组织”；
- 用户要求以真实使用角度设计，不能以权限收紧破坏正常功能闭环。
- 用户确认间接权限对象采用统一权限归属根，历史子数据动态跟随父对象当前组织，异常父链一律 fail-closed。
