# Incident IM 协作产品决策记忆

- 最近更新：2026-07-21
- 当前规格：`docs/superpowers/specs/2026-07-21-incident-feishu-im-group-design.md`

## 产品定位

Incident IM 协作服务于事故处置阶段的快速协同。首期是可在真实飞书租户验证的完整闭环，而不是仅演示一次建群调用。

## 已确认范围

- 首期只实现飞书完整流程，企业微信后置。
- Incident 协作区提供一键建群、状态查看、成员详情、重试、暂停/恢复及解绑。
- 群成员来自 `operator + collaborators` 去重集合，不从 Incident 组织/团队自动展开。
- 支持创建群、首批拉人、Incident 摘要消息和后续新增人员同步。

## 已确认设计决策

- 任意 `Incident.operator` 均可创建和管理群；`collaborators` 只能作为成员和查看者。
- 成员持续同步只增不减；人员从 Incident 移除后不自动踢出外部群。
- 至少一名负责人已映射即可建群。未映射成员不阻塞，开启持续同步后映射补齐会自动补拉。
- Incident 关闭暂停同步，重新打开时按关闭前配置恢复；手工暂停不自动恢复。
- 一个 Incident 同时最多一个待创建或有效绑定；解绑不删除外部群，切换渠道需解绑重建。
- 复用系统管理的飞书集成实例、通知渠道和用户映射；新增独立 `im_group` Provider 能力，Alerts 不保存平台凭据。
- Incident 成员不外键依赖 `IMNotificationUserMapping` 行，按用户名解析并保存外部 ID 快照，因为渠道同步会删除并重建映射行。
- 建群、消息和增员通过 Alerts Outbox 异步、幂等执行；获得外部 `chat_id` 后先落库，后续失败不得重复建群。
- 渠道团队只决定渠道可见/可用范围，不决定 Incident 群成员或建立额外组织绑定。
- 前端采用方案 A：在现有协作页 220px 右侧栏中，把紧凑群状态卡放在协作者列表上方；创建使用 Modal，成员和异常详情使用 Drawer，不新增页面或二级标签。
- 右侧状态卡最多展示“查看详情、当前优先动作、更多”三个入口；暂停/恢复、持续同步设置、复制群 ID 和解绑按状态收进优先动作或更多菜单。
- 关闭持续同步、手工暂停和 Incident 关闭暂停是三种不同语义；只有手工暂停可直接恢复，Incident 关闭必须重新打开 Incident 后按原配置恢复。
- 新增/移除协作人沿用现有入口；新增后刷新群同步状态，移除前明确提示不会自动从飞书群退群。
- 创建中前端使用短轮询并在详情 Drawer 展示服务端确认的阶段进度；稳定状态复用协作页刷新节奏，页面切换时必须清理请求和定时器。
- 建群请求收到 `202` 后，在 `pending_create/creating` 阶段不允许取消或解绑；必须先让外部调用收敛，避免产生归属不明的飞书群。

## 明确后置

- 企业微信 Provider、用户同步和 `userid` 映射。
- 自动踢人、解散群、多个有效群、跨租户用户、群消息回写和群配置双向同步。
- 用户主动退群后的自动循环拉回。

## 仍待确认

无。
