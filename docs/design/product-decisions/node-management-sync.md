# 节点管理同步产品决策记忆

- 最近更新：2026-07-16
- 当前规格：`specs/changes/legacy-2026-07-16-node-management-sync-reconciliation/spec.md`

## 产品定位

节点管理同步是 CMDB 自动发现的系统级资产同步与采集闭环。页面开关必须表达真实运行能力，而不是仅保存配置值；同步、采集、失败和恢复状态必须能指导运维处置。

## 已确认范围

- 覆盖首次打开、关闭、重新开启、重复操作和运行态漂移。
- 覆盖 django-celery-beat 调度、Node Management 取数、CMDB host 差异写入、区域隐藏采集任务、节点配置下发、父子采集状态和页面健康展示。
- 覆盖系统/用户主体、组织边界、错误脱敏、资源预算和 TDD 验收。

## 已确认设计决策

- 采用期望状态对账：`NodeMgmtSyncConfig` 是期望状态，Beat 周期任务和区域节点采集配置是实际状态，幂等 Reconciler 负责收敛。
- 保留首次创建时自动同步、自动采集默认开启，但首次读取必须建立真实调度；配置开启而实际任务缺失时必须显示降级并恢复。
- 后端强制自动采集依赖自动同步；关闭同步同时关闭采集，重新开启同步不会暗中重新开启采集。
- Beat 使用显式可信系统主体；HTTP 全局执行只允许平台管理员，普通用户展示按组织裁剪。
- 采集等待有效同步，子任务投递状态为 `submitted`，只有实际完成才是 `success` 并推进最近采集时间。
- 合法空源、权限/协议错误、无接入点、同步失败、部分成功和超时使用不同状态及稳定 reason code。
- Node Management 节点配置 delete→push 必须按区域和 generation 可重试，不能依赖数据库事务回滚外部副作用。
- 首次打开和关闭/重开是独立 TDD 场景；核心状态、授权和 RPC 契约覆盖率目标不低于 90%。
