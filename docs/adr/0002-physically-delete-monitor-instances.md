# ADR 0002：监控实例采用物理删除

Status: accepted

## Context

`MonitorInstance.id` 是全局主键，现有逻辑删除会留下不可见但持续占用 ID 的记录，导致删除后用相同 ID 创建实例触发唯一约束。当前删除已经清理采集配置和多类当前态引用，却没有恢复入口、删除时间或删除原因，因此墓碑没有形成可用的恢复能力。

## Decision

`MonitorInstance` 只表示当前受平台纳管的实例。用户删除时先幂等清理 NodeMgmt 配置和本地当前态引用，再物理删除实例；不保留永久软删除墓碑，也不为此引入退役状态机或异步操作表。

创建实例时按全局主键判断占用：有效实例冲突返回业务错误，历史 `is_deleted=True` 墓碑先按相同清理路径物理删除，再创建新实例。`is_deleted` 仅作为历史兼容字段，待墓碑清理完成后移除。自动发现失联继续由 `is_active` 表达，永久排除若有需求另行建模。

## Consequences

- 删除会释放全局实例 ID，消除逻辑删除墓碑导致的重建冲突。
- NodeMgmt 不受数据库事务控制；远端清理成功而本地事务失败时，实例会暂时存在但采集配置已缺失，需要用户重复删除使操作收敛。
- 不引入异步任务意味着失败不会后台自动重试，换取更小的实现和运维复杂度。
- 告警、事件和快照使用字符串实例 ID，物理删除后继续保留；同 ID 重建后历史记录不会自动区分实例代际。
