# Core Beliefs —— 工程信条

> 全局设计与协作信条。模块级细化:[algorithms/DESIGN_GUIDE.md](../../algorithms/DESIGN_GUIDE.md)、[web/DESIGN.md](../../web/DESIGN.md)。
> 这些是「为什么」；「怎么做」在 [AGENTS.md](../../AGENTS.md)，质量底线在 [质量门禁](../governance/quality.md)。

## 1. 最小变更(Minimal Diff)
只改与任务相关的文件和代码块。禁止顺手重构、禁止全仓格式化、禁止「路过优化」。需要的重构单独立项，按 [Agent 工作流](../agents/workflow.md) 进入 change spec。

## 2. 证据先于结论(Evidence over Memory)
先读代码、提取事实,再下结论;不凭经验补步骤。无法确认的写 `TODO:` 并附「确认位置(具体路径 + 关键词)」。AGENTS.md 本身就是这条信条的产物(每条都标了证据出处)。

## 3. 渐进式设计,拒绝过早抽象
源自 `algorithms/DESIGN_GUIDE.md`,上提为全局信条:
- 先让一个具体场景跑通,再在出现**第二个**真实用例时才抽象。
- 基类只定义已被两处以上需要的接口契约,不预设未来。
- 标准项目结构优先于「聪明」的目录创新。

**写每行代码前的决策梯**(懒惰高级开发者思维):
1. 这段代码**必需**吗?不需要就不写(YAGNI)。
2. 代码库里**已有**吗?复用,别重写。
3. **标准库**能做吗?用标准库。
4. **原生平台/框架**能做吗?用原生能力。
5. **已装的依赖**能做吗?用现有依赖,别加新依赖。
6. 以上都不行,才写**满足需求的最小实现**。

> "最好的代码是你永远不用写的代码。" 新增依赖必须按 [质量门禁](../governance/quality.md) 附理由。

## 4. 不做向后兼容的影子设计
发现旧入口与现状冲突时,直接更新到当前真实入口,不保留「以防万一」的兼容分支。历史在 git 里,不在 if-else 里。

## 5. 单一真相源(Single Source of Truth)
- 长期规格 = `specs/capabilities/`，跨会话变更 = `specs/changes/`，**不**在别处复制需求。
- 设计 token = `web/DESIGN.md`,前端实现不得硬编码颜色/圆角。
- 运行手册 = `AGENTS.md`。
- 顶层 `*.md`(本目录及根目录)只做**原则 + 指针**,绝不复制实体内容,避免文档漂移。

## 6. 安全默认关闭(Fail-Closed)
密钥只走 `*.example` / `*.template` 与环境注入;`.env` / keystore 不入库;日志不记敏感信息。详见 [安全红线](../governance/security.md)。

## 7. 改完必验
任何变更后至少跑对应模块的最小门禁(见 [质量门禁](../governance/quality.md))。「我觉得没问题」不算验证,命令退出码才算。

## 8. 中文优先
所有交互产物(回答、注释、commit、PR、文档)使用中文。

## 9. 测试是行为契约
先写测试再写实现(TDD 红-绿-重构):测试定义「正确行为长什么样」,实现去满足它。测**行为**不测实现——镜像实现的测试只会锁死重构、给不出信心。覆盖率是结果不是目的:**为凑数写的无效测试比没有更糟**,它把绿灯变成谎言。改动代码 ≥75% 覆盖,关键路径(鉴权/下发/事务)必覆盖。细则见 [质量门禁 §4](../governance/quality.md)。

## 10. 下发不伤宿主
本平台会把插件/作业下发到**客户的真实机器**。那里没有「重来一次」——一次打满内存、一次误删、一次不可逆操作,代价是别人的生产环境。所以下发路径默认**保守**:有资源边界、幂等可回滚、不可逆动作先预检,且这些保障**必须有测试背书**。与 [§6 Fail-Closed](#6-安全默认关闭fail-closed) 同源,但宿主安全是更硬的底线。详见 [可靠性红线 §2.5](../governance/reliability.md)。

> 新增一条信条前,先确认它不是某条已有信条的特例。信条应当稀少而锋利。
