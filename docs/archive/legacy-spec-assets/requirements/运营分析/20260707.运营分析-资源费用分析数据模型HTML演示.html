降级策略（规划中）

对应设计文档 §5.2。Transaction_Log 写入量或单次查询耗时达到阈值时，service 层切到预聚合 rollup 表。

  1. 触发条件：任一 widget 查询耗时 > 3s 或 Transaction_Log 行数 > 500 万（临时阈值，可调）。
  2. Service 路由：get_query_source() 检测后返回 rollup 表 queryset，正常仍走 CMDB 实例表。
  3. 刷新任务：Celery beat 每日 02:00 跑增量刷新，覆盖 rollup_date >= yesterday。
  4. 失败处理：失败重试 3 次，告警到运维；rollup 当日未生成时返回 503「数据准备中，请稍后再试」。
  5. 全量重建：运维介入手动触发；UI「刷新」按钮走增量同步（不阻塞用户）。
  6. 数据延迟：报表 T+1，UI 标注「数据更新至 YYYY-MM-DD」。
  7. 依赖：CMDB 自定义上报需支持 Total_Cost / object_type 等字段索引；不支持时启用 rollup。
# service 层路由
def get_query_source():
    tx_count = TransactionLog.objects.filter(
        organization=organization,
    ).count()
    if tx_count > ROLLUP_TRIGGER_THRESHOLD:
        return CloudResourceCostDailyRollup.objects
    return TransactionLog.objects  # 走 CMDB 实例服务

# rollup 刷新任务（Celery beat 02:00）
@shared_task(bind=True, max_retries=3)
def refresh_rollup(self):
    yesterday = date.today() - timedelta(days=1)
    CloudResourceCostDailyRollup.objects.filter(
        rollup_date__gte=yesterday,
    ).delete()
    aggregate_into_rollup(
        since=yesterday,
        source=TransactionLog,
        group_by=["object_type", "USER_DEPARTMENT", "applicant"],
    )
