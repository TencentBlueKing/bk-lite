# Wiki 决策记录 Storybook 视觉设计

## 状态

- 日期：2026-07-16
- 方案：A（独立全宽决策记录模式）
- 用户确认：已确认

## 目标

简化 Wiki“检查审核”的主要任务流：默认页面只展示真正需要用户介入的待决策项，不再把“已处理”作为并列页签。同时保留人工决策历史、自动回放说明和规则撤销能力，并通过 Storybook 提供可交互的视觉伴随。

## 核心设计

### 待决策主页

- 页面标题继续展示“待决策”和待处理数量。
- 移除左侧“待决策 / 已处理”分段页签，左侧直接展示待决策列表。
- 标题栏右侧增加弱化的“决策记录”按钮，不显示历史数量，避免“已处理 0”占据视觉焦点。
- 保留当前白色侧栏背景、知识对比卡片、决策原因、影响范围和底部决策动作。

### 决策记录模式

- 点击“决策记录”后，在同一工作区切换到独立的全宽历史模式，不使用抽屉、弹窗或并列页签。
- 页面标题变为“决策记录”，副标题说明这里只记录人工决策及其后续自动复用情况。
- 标题栏右侧提供“返回待决策”。
- 左侧列表展示决策类型、语义化结果、处理时间和规则状态，不再使用含义模糊的“已处理”标签。
- 右侧继续复用当前知识与新知识的双栏快照，并在底部展示操作人、处理时间、规则状态、回放次数、最后回放时间和失效原因。
- 有效规则保留“停止后续自动复用”入口；操作只撤销未来回放，不回滚已生效知识。

### 空状态

- 待决策为空时，仍保留“决策记录”入口，方便追溯历史。
- 决策记录为空时，展示“暂无决策记录”，并提供“返回待决策”。

## 数据与交互

- 继续使用现有 `DecisionListView` 的 `pending` 与 `processed` 数据契约。
- “决策记录”与“返回待决策”分别调用现有 `onViewChange('processed')` 和 `onViewChange('pending')`。
- 继续复用 `CheckTab` 的请求失效保护、分页和计数逻辑，不新增后端接口。
- 保留 `CheckItem`、决策规则和 `processed` 查询能力；本次只调整展示入口层级。
- 自动维护、删除、重建和健康诊断仍属于构建记录或系统审计，不进入决策记录列表。

## 视觉方向

- 延续 OpsPilot 控制台的克制型运维界面：白色主背景、细边框、蓝色强调和清晰的信息密度。
- 避免大面积灰色底、重复状态标签和无意义的零计数。
- 历史模式通过标题、返回动作和语义化结果建立明确层级，不依赖颜色作为唯一状态表达。
- 保持现有左右对比图标和知识快照风格，不增加新的装饰图标体系。

## Storybook 场景

1. `KnowledgeConflict`：待决策主页，展示知识冲突与“决策记录”入口。
2. `PageIdentityMerge`：待决策主页，展示页面身份合并决策。
3. `DecisionHistory`：同时展示有效自动回放规则与已撤销规则，可在列表中切换详情。
4. `EmptyPendingWithHistory`：待决策为空但历史可访问。
5. `EmptyDecisionHistory`：决策记录为空并可返回待决策。

视觉伴随故事应使用受控状态或 Storybook `useArgs`，使“决策记录 / 返回待决策”可以真实切换，而不是只记录点击事件。

## 非目标

- 不修改后端审批、规则回放或自动维护逻辑。
- 不删除已处理数据或决策规则。
- 不建设独立规则管理后台。
- 不修改构建记录页面。
- 不改动与 Wiki 决策中心无关的工作区文件。
