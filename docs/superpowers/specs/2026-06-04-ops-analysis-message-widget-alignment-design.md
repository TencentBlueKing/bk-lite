# Ops Analysis Message 组件对齐日志 MsgTable 设计

日期: 2026-06-04
范围: 运营分析仪表盘 Message 组件（一期）
目标: 在不改动日志现网页面的前提下，将运营分析 Message 组件的交互和信息结构对齐日志 MsgTable，并为后续两端共用组件铺设边界。

## 1. 背景与问题

当前运营分析 Message 组件是轻量事件列表，能够展示基础字段，但与日志模块 MsgTable 在以下方面存在差距：

1. 缺少日志流列表 + 展开详情的排障视图。
2. 缺少 Log line / Meta / Fields 分层信息结构。
3. 缺少后续共用组件的明确接口边界。

同时，运营分析与日志模块虽然参数壳子不同（统一筛选、命名空间、固定参数等），但在“拿到日志数据后的展示处理逻辑”上应保持一致。

## 2. 目标与非目标

### 2.1 目标

1. Message 组件对齐日志 MsgTable 的核心体验。
2. 使用双区展开详情：Log line + Meta + Fields。
3. 保持现有仪表盘协议能力（loading/empty/onReady、统一筛选联动）。
4. 抽取可复用展示内核接口，降低后续日志侧接入成本。
5. 增加 Storybook 视觉样例，支持联调与回归。

### 2.2 非目标

1. 本轮不替换日志模块现网 MsgTable 组件。
2. 本轮不改动后端 NATS 查询链路。
3. 本轮不引入超出当前需求的复杂交互（如批量展开、复杂检索、字段拖拽排序）。

## 3. 方案对比与选择

### 方案 A: 快速仿制（仅 ops-analysis 内实现）

优点: 交付快，改动小。
缺点: 形成重复逻辑，后续统一成本高。

### 方案 B: 可演进折中（推荐）

先在 ops-analysis 落地能力对齐，同时抽取“展示内核接口 + 适配层”边界，日志模块后续平滑接入。

优点: 风险可控，当前需求可交付，后续可复用。
缺点: 比方案 A 多一点抽象设计工作。

### 方案 C: 一步到位共用（同时改日志与运营分析）

优点: 一次统一，长期最干净。
缺点: 变更面大，回归压力高，超出当前迭代风险预算。

最终选择: 方案 B。

## 4. 架构与组件边界

### 4.1 逻辑分层

1. 参数壳子层（各自保留）
- 运营分析: 统一筛选、时间范围、命名空间、数据源参数组装。
- 日志模块: 现有日志页参数/查询上下文。

2. 展示内核层（本次统一目标）
- 行摘要展示。
- 单行展开详情。
- Log line / Meta / Fields 渲染。
- 级别颜色与长文本策略。

### 4.2 组件职责

1. Message 外壳组件（ops-analysis 现有入口）
- 保留 loading/empty/onReady 等约定。
- 调用归一化适配层。

2. 归一化适配层（ops-analysis）
- 将 rawData（数组或 items 包裹）转换为统一展示模型。

3. 展示内核组件（可复用）
- 接收统一模型并渲染日志流+展开详情。

## 5. 交互设计

### 5.1 列表区

1. 默认按时间倒序。
2. 行内容展示: 级别、时间、来源、消息摘要。
3. 长文本主列表截断，悬浮可查看。

### 5.2 展开区（双区版）

1. Log line: 完整日志文本（优先 message，必要时回退拼接）。
2. Meta: time / level / source / namespace（存在则展示）。
3. Fields: 其余键值对列表，支持滚动。

### 5.3 展开行为

1. 本轮默认单行展开（single expand）。
2. 数据刷新、参数变化后重置展开态。

## 6. 数据契约

### 6.1 统一展示模型

每条记录归一化为:

1. id: 稳定唯一键。
2. timestamp: 原始时间。
3. displayTime: 格式化时间。
4. level: 级别。
5. source: 来源。
6. message: 消息文本。
7. meta: 结构化元信息。
8. fields: 扩展字段数组（key/value）。
9. raw: 原始对象。

### 6.2 字段映射与容错

1. 主字段采用候选键优先级映射（time/level/source/message）。
2. 关键字段缺失按行级降级，不让整表失败。
3. 未命中主字段的内容进入 fields，保证信息不丢。

说明: 同源接口下 data 字段集合可能随查询参数变化，统一展示模型用于吸收动态字段差异，不改变后端返回语义。

## 7. 错误处理

1. 数据结构不符提示由 WidgetWrapper 统一处理。
2. Message 内核只处理 loading/empty/list 三种展示状态。
3. 行级异常（缺字段、空字段）局部降级，不中断渲染。

## 8. 测试与验收

### 8.1 测试范围

1. 归一化函数测试
- 数组输入。
- items 包裹输入。
- 字段别名映射。
- 缺失字段降级。

2. 组件行为测试或 Storybook 交互校验
- 默认列表。
- 单行展开。
- 空态。
- 长文本。

3. 联动回归
- 统一筛选、时间范围、命名空间变化后刷新正常。
- YAML 导入导出后组件可正常渲染。

### 8.2 验收标准

1. 视觉和交互达到“日志 MsgTable 核心体验一致”。
2. 同源同参时行级语义一致（时间/级别/来源/消息/扩展字段可见）。
3. Storybook 至少覆盖: 正常、缺字段、长文本、空数据。
4. 本轮不引入日志现网回归。

## 9. 实施计划（两阶段）

### 第一阶段（本迭代）

1. 完成 ops-analysis Message 对齐。
2. 增加归一化适配层与 Storybook 样例。
3. 不改日志页现网组件。

### 第二阶段（后续迭代）

1. 抽共享消息流组件至公共层。
2. 日志 MsgTable 切换到共享组件 + 适配器。
3. 清理重复逻辑。

## 10. 风险与缓解

1. 字段映射遗漏
- 缓解: 固化候选键优先级表 + Storybook 异常样例。

2. 展开区性能
- 缓解: 滚动容器、按需渲染、文本截断策略。

3. 样式偏差
- 缓解: 以日志 MsgTable 为视觉基线做对照验收。

4. 后续复用成本上升
- 缓解: 本轮先冻结展示内核输入契约，避免逻辑散落。
