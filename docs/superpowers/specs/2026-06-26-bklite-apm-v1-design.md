# BK-Lite APM v1 规格设计

- 日期：2026-06-26
- 状态：已确认，待进入实施计划
- 目标：定义 BK-Lite APM v1 的产品定位、接入方式、分析能力、拓扑能力、状态语义和 MVP 边界，用于后续详细实施规划与分阶段落地。

## 1. 产品定位

### 1.1 一句话定位

BK-Lite APM v1 是一个面向运维 / SRE 的应用排障工作台，通过标准化 OTLP 接入引导、按业务系统分组的服务分析、服务级依赖拓扑调查、版本对比、轻量错误聚合和 Trace 查询，帮助用户完成“接入 -> 发现异常 -> 定界 -> 归因”的闭环。

### 1.2 目标用户

- 一线运维
- 值班 SRE
- 平台 SRE
- 应用负责人（次级用户）

### 1.3 v1 核心目标

- 让用户以统一 OTLP 方式把应用链路数据接入 BK-Lite。
- 让用户按 `service.namespace` 快速进入业务系统视角排障。
- 让用户围绕服务、资源、Trace、版本和拓扑完成告警后定界与慢请求归因。
- 让用户理解“为什么当前某些能力不可用”，而不是只看到空白页。

### 1.4 v1 非目标

- 不做页面远程注入 / 页面远程安装 Agent。
- 不做复杂 issue 生命周期和高级错误治理。
- 不做 RUM、DBM、Profiler 的深度联动。
- 不做复杂采样治理中心。
- 不做资源级主拓扑。
- 不做 CMDB / 网络物理拓扑融合。
- 不做全局大屏式架构展示优先体验。

## 2. 核心产品判断

### 2.1 协议与存储

- 接入协议：`OTLP`
- 存储底座：`VT`
- v1 围绕 OTLP 建立统一接入心智，不引入多协议并行复杂度。

### 2.2 业务系统分组语义

- `service.namespace` 定义为“业务系统 / 应用域”。
- 例如：`cmdb`、`monitor`、`billing`。
- 它是 BK-Lite APM 的一级组织维度，不只是筛选条件。

### 2.3 版本能力定位

- v1 支持版本对比。
- 版本来源于 OT 原生字段 `service.version`。
- 版本能力服务于排障和回归判断，不扩展为发布管理系统。

### 2.4 错误能力定位

- v1 只做轻量错误聚合。
- 以错误类型、服务、资源、版本维度做聚合视图。
- 不做 Datadog 式复杂 issue intelligence。

### 2.5 拓扑能力定位

- v1 拓扑是“服务级依赖调查能力”。
- 首要支持：
  - 告警后定界
  - 慢请求归因
- 默认只看一跳上下游。
- 资源级归因通过下钻资源和 Trace 完成。

### 2.6 接入中心定位

- 接入中心是 BK-Lite APM v1 的确定模块，不再讨论是否需要。
- 接入中心负责：
  - 帮用户选择接入路径
  - 生成和展示最小可用接入材料
  - 判断接入是否成功
  - 解释为什么数据虽然进来了但能力不完整

## 3. 用户任务与核心场景

### 3.1 接入场景

- 我想让 Java / Python / Go / Node.js 服务把链路数据接进 BK-Lite。
- 我想知道 OTel Collector、Kubernetes 注解注入、eBPF 哪条路径更适合我。
- 我接完了，但为什么还没看到服务出现在分析页。

### 3.2 告警后定界

- 某服务错误率升高，我要判断影响范围。
- 我要知道是这个服务自己坏了，还是某个共享下游坏了。
- 我要知道下一步应该联系哪个团队、查哪个服务。

### 3.3 慢请求归因

- 某服务 P95 上升，我要知道慢在自己还是慢在下游。
- 我想看哪条依赖边最可疑。
- 我需要继续跳到对应资源和 Trace。

### 3.4 版本回归判断

- 我怀疑当前版本引入了问题。
- 我想比较版本前后请求量、错误率、延迟变化。
- 我想确认问题是否集中在某几个资源。

### 3.5 接入质量排查

- 我已经收到数据，但为什么服务分组不对。
- 为什么服务出现在未归类分组中。
- 为什么版本对比和拓扑仍然不可用。

## 4. 信息架构

### 4.1 一级结构

1. `首页`
2. `服务目录`
3. `Trace 查询`
4. `接入中心`

### 4.2 下钻关系

- 首页 -> 服务目录 / 服务详情 / 资源详情
- 服务目录 -> 服务详情
- 服务详情 -> 资源详情 / Trace 查询
- 接入中心 -> 集成目录 -> 接入详情

### 4.3 结构原则

- 主导航少，下钻强。
- 接入与分析是一套闭环，不拆成两个产品。
- `服务详情` 和 `资源详情` 都不是一级菜单，而是典型下钻页。
- 拓扑能力不单独做一级菜单，而是作为服务分析视图的一部分。

## 5. 核心对象模型

### 5.1 业务系统

- 来源：`service.namespace`
- 用途：首页聚合、服务目录分组、拓扑域边界、值班排障主入口

### 5.2 服务

- 身份键：`service.namespace + service.name`
- 是排障主对象

### 5.3 版本

- 来源：`service.version`
- 用于筛选、版本对比和回归判断

### 5.4 资源

- 服务下的 endpoint / operation / route
- 用于细化问题定位

### 5.5 Trace

- 单次请求链路
- 是证据层对象，不是首页主对象

### 5.6 错误聚合项

- 按错误类型 / 服务 / 资源 / 版本聚合
- 非独立 issue 系统

### 5.7 拓扑对象

- 服务节点：承载服务身份和健康摘要
- 调用边：承载排障证据，是拓扑分析的主对象

## 6. 首页：值班调查台

### 6.1 产品定位

首页不是普通概览页，而是值班调查台。它应直接帮助值班 SRE 回答：

- 现在最异常的是哪个业务系统？
- 现在最值得先查的是哪个服务？
- 最近 1 小时是否有疑似版本回归？
- 下一步应该从服务、资源还是拓扑进入？

### 6.2 首页建议模块

- 当前异常业务系统
- 当前最可疑服务
- 新增异常版本提示
- 高风险资源榜
- 快速调查入口

### 6.3 设计原则

- 首页优先分流，不优先展示。
- 首页不做大而全看板，不承载复杂配置与管理操作。

## 7. 服务目录与服务分析

### 7.1 服务目录定位

服务目录是 BK-Lite APM v1 的主入口之一，帮助用户按业务系统进入服务排障主路径。

### 7.2 服务目录能力

- 按 `service.namespace` 分组展示服务。
- 支持列表视图 / 全局服务地图视图切换。
- 支持状态、版本、活跃度、错误率、延迟筛选与排序。

### 7.3 值班排序规则

默认排序优先级：

1. 异常服务优先
2. 最近新异常优先
3. 高影响服务优先
4. 近期版本变更优先

### 7.4 未归类服务

服务目录必须显式存在 `未归类服务` 分组。凡是缺失 `service.namespace` 或 namespace 不合法的服务，均进入该分组，并附修复提示。

### 7.5 服务详情定位

服务详情是服务目录的下钻页，用于判断：

- 是服务整体问题还是资源问题
- 是服务自身问题还是下游依赖问题
- 是否存在版本回归

### 7.6 服务详情内容

- 当前异常摘要
- 请求 / 错误 / 延迟趋势
- 版本对比
- 局部拓扑
- 高风险资源列表
- 轻量错误聚合
- Trace 入口

### 7.7 资源详情定位

资源详情是服务详情的下钻页，重点回答：

- 当前具体哪个接口 / 操作异常
- 是否只在某个版本异常
- 是否存在明确的错误样本和慢请求样本

### 7.8 资源详情内容

- 当前异常摘要
- 版本差异
- 错误分布
- 示例 Trace
- 常规趋势

## 8. Trace 查询与 Trace 详情

### 8.1 产品定位

Trace 查询用于在明确调查任务下深挖证据。

### 8.2 查询维度

- 时间
- `service.namespace`
- `service.name`
- `service.version`
- 资源
- 是否错误
- 延迟区间
- Trace ID

### 8.3 结果形态

- Trace 列表
- 单条 Trace 详情

### 8.4 设计原则

- Trace 是证据层，不是首页层。
- Trace 查询必须支持从服务详情、资源详情和拓扑异常边带上下文跳转进入。

## 9. 接入中心规格

### 9.1 产品定位

接入中心是 BK-Lite APM 的标准化 OTLP 接入入口，负责：

- 帮用户选择接入方式
- 让用户拿到可直接使用的接入材料
- 判断接入是否成功
- 解释为什么数据能力不完整

### 9.2 v1 信息架构

- `集成目录`
- `接入详情`

不单独做：

- 复杂凭证管理中心
- 独立采集端管理页

### 9.3 集成目录

#### 首批覆盖

- Node.js
- Java
- Python
- .NET
- Go
- OTel Collector
- Kubernetes 注解注入
- eBPF 自动注入入口

#### 目录职责

集成目录的目标不是展示支持列表，而是帮助用户选择适合的接入路径。

#### 目录卡片信息

- 名称
- 简短适用说明
- 接入类型标签
- 当前状态摘要
- 推荐场景说明

推荐场景示例：

- Java：适合传统 JVM 服务，优先推荐
- OTel Collector：适合已有采集网关
- Kubernetes 注解注入：适合集群内批量接入
- eBPF：适合低侵入试点

### 9.4 接入详情

#### 固定区块

- 顶部摘要
- 接入自检
- 接入凭证
- 上报端点
- 接入配置
- 字段规范提示
- 最近发现的服务映射结果

#### 最近发现的服务映射结果

这是 v1 接入体验的关键补充，用来回答：

- 平台最近识别到了哪些 `service.namespace`
- 最近识别到了哪些 `service.name`
- 最近识别到了哪些 `service.version`
- 哪些服务进入了 `未归类服务`
- 哪些关键字段被判定为异常或缺失

#### 接入凭证能力边界

v1 只做最小可用能力：

- 生成 token
- 备注
- 首次展示明文
- 查看创建时间
- 失效 / 停用

不扩展为重型凭证管理系统。

### 9.5 接入中心与分析模块联动

- 接入成功后，相关服务应自动出现在服务目录。
- 接入详情页应提示最近发现的服务和业务系统。
- 若字段不完整，应明确提示为什么服务未正常进入服务目录、版本对比或拓扑分析。

## 10. 拓扑能力规格

### 10.1 产品定义

BK-Lite APM v1 的拓扑是“服务级依赖调查能力”：围绕异常服务和异常时间窗，帮助运维 / SRE 判断影响范围，并识别最可能的下游性能瓶颈。

### 10.2 首要支持任务

- 告警后定界
- 慢请求归因

### 10.3 用户角色与价值

#### 一线运维 / 值班 SRE

- 判断异常范围
- 判断问题是否正在扩散
- 找到下一步最值得继续查的下游服务

#### 平台 SRE

- 校验依赖关系是否合理
- 辅助排查 OT 数据接入质量问题

#### 应用负责人

- 识别服务依赖方向
- 判断性能退化是否由下游依赖拖慢

#### 事故指挥 / 跨团队协调者

- 快速形成故障影响范围共识
- 判断问题源头更可能位于哪个业务域

### 10.4 核心体验原则

- 默认围绕一个中心服务展开
- 默认只看一跳上下游
- 默认先突出异常，再补全正常关系
- 边比点更重要
- 时间窗是一等公民
- 必须能跳到下一步证据
- 必须容忍 OT 数据不完美
- namespace 是拓扑组织语义的一部分

### 10.5 两层拓扑

#### 全局服务地图

- 位于服务目录视图切换中
- 用于异常范围观察与业务系统级关系判断
- 不是默认首页

#### 服务详情内局部拓扑

- 是 v1 更核心的拓扑能力
- 用于真正的定界与归因
- 默认围绕当前服务、当前时间窗、一跳上下游加载

### 10.6 拓扑的自动上下文

- 从首页异常服务进入服务详情时，拓扑自动带当前异常时间窗和当前服务上下文
- 从版本回归提示进入服务详情时，拓扑自动带版本上下文，突出当前版本下异常边

### 10.7 v1 不做

- 资源级主拓扑
- 物理网络拓扑
- 历史拓扑回放
- 自定义手工编排依赖关系
- 复杂根因自动推理

## 11. 字段规范与数据质量规则

### 11.1 核心字段

- `service.namespace`
- `service.name`
- `service.version`

### 11.2 字段语义

- `service.namespace`：业务系统 / 应用域
- `service.name`：服务名
- `service.version`：版本号或版本标识

### 11.3 接入成功与分析可用的区分

v1 必须区分：

- 接入链路通了
- 数据可以被服务建模
- 版本对比可用
- 拓扑可用

接入成功不等于数据可分析。

### 11.4 数据能力分层

- 已接入
- 可分析
- 可对比
- 可拓扑

### 11.5 缺失字段影响

- 缺 `service.namespace`
  - 无法正确做业务系统分组
  - 服务进入 `未归类服务`
- 缺 `service.name`
  - 无法稳定建模服务
- 缺 `service.version`
  - 版本对比不可用
- 调用关系信息不足
  - 无法形成稳定拓扑
- 错误字段不足
  - 轻量错误聚合降级

### 11.6 资源和错误的降级原则

- 若资源标识不稳定，只能降级为 Trace 级查看，不应错误聚合资源。
- 若错误字段不足，仍可展示错误 Trace 样本，但错误聚合能力应降级。

### 11.7 设计原则

BK-Lite APM v1 不追求“只要有 Trace 就一切都能看”，而是追求“字段规范后，分析结果可解释、可行动”。

## 12. 状态机与降级路径

### 12.1 接入状态

- 未开始
- 配置中
- 已接入
- 接入异常
- 数据不完整

### 12.2 服务状态

- 正常
- 关注
- 异常
- 无数据

### 12.3 资源状态

- 正常
- 关注
- 异常
- 无数据

### 12.4 拓扑边状态

- 正常
- 关注
- 异常
- 数据不足
- 无活跃

### 12.5 版本状态

- 当前活跃
- 近期活跃
- 疑似回归
- 不可对比

### 12.6 关键降级路径

- 有 Trace 但无版本
  - 可排障，不可版本对比
- 有服务数据但无稳定依赖关系
  - 可看服务与资源，不可拓扑
- 有错误样本但字段不足
  - 可看样本，不可错误聚合
- 有接入数据但 namespace 缺失
  - 数据可接收，但进入未归类服务

## 13. 关键闭环

### 13.1 接入闭环

1. 用户进入接入中心
2. 选择接入方式
3. 生成凭证
4. 复制配置并完成外部接入
5. 平台收到数据
6. 接入详情页显示状态与服务发现结果
7. 服务自动进入服务目录
8. 用户开始排障分析

### 13.2 告警后定界闭环

1. 用户在首页或服务目录看到异常服务
2. 进入服务详情
3. 查看核心趋势和局部拓扑
4. 判断是本服务问题还是下游问题
5. 点击异常边或异常资源
6. 跳到 Trace 查询或资源详情
7. 查看具体样本证据

### 13.3 版本回归闭环

1. 用户发现服务异常
2. 在服务详情看到当前版本和历史版本对比
3. 如指标恶化，标记为疑似回归
4. 下钻到异常资源和 Trace
5. 确认问题是否集中在当前版本

## 14. v1 分析优先级与不做项

### 14.1 P0

- 接入中心
- 集成目录
- 接入详情
- 接入自检
- 服务目录
- 服务详情
- 资源详情
- Trace 查询
- 单条 Trace 详情
- 轻量错误聚合
- 版本对比
- 服务级局部拓扑
- `service.namespace` 分组

### 14.2 P1

- 首页值班调查台
- 全局服务地图视图
- 异常边优先调查
- 未归类服务提示
- 数据质量降级提示
- 最近发现的服务映射结果

### 14.3 明确不做

- RUM / DBM / Profiler 深度联动
- 复杂 issue 生命周期
- 自动根因分析
- 页面远程安装 / 注入 Agent
- 复杂采样治理后台
- 资源级拓扑主视图
- CMDB / 网络物理拓扑融合
- 全局架构展示优先体验

## 15. 与后续版本的演进边界

### 15.1 v1 确定做

- 引导式接入
- 接入自检
- 服务 / 资源 / Trace 排障
- 轻量错误聚合
- 版本对比
- 服务级拓扑

### 15.2 后续版本预留

- 基于 node 模块和 NATS 的页面化接入
- 页面下发 / 安装 / 注入能力
- 更强采集目标管理
- 更复杂错误 issue 能力
- 更深跨产品联动

### 15.3 演进原则

v1 不把未来接入模型写死成纯文档模式，但也不提前引入远程执行复杂度。

## 16. BK-Lite APM v1 的产品个性

### 16.1 业务系统视角优先

相比通用 observability 平台，BK-Lite APM 更强调按 `service.namespace` 组织服务，让运维先按业务系统排障。

### 16.2 接入和分析一体闭环

不是“接入文档区 + 分析台”两张皮，而是“接入好以后立刻进入调查路径”。

### 16.3 拓扑服务排障而不是服务展示

拓扑的核心不是漂亮，而是帮助用户缩小范围、找到下一步证据。

### 16.4 接受 OT 数据不完美并给出可解释降级

BK-Lite APM 不假设 OT 数据永远完整，而是明确告诉用户当前能查什么、不能查什么、为什么。

### 16.5 版本对比是调查能力

版本能力围绕回归判断展开，不扩展为完整发布治理系统。

## 17. MVP 验收场景

### 场景 1：Java 服务接入

- 用户进入接入中心
- 选择 Java
- 生成凭证
- 复制 OTLP 配置
- 服务出现在服务目录

### 场景 2：按业务系统排障

- 用户在首页发现 `monitor` 业务系统异常
- 进入服务目录并定位到异常服务
- 进入服务详情并查看局部拓扑
- 判断下游依赖异常
- 跳转 Trace 查询确认

### 场景 3：版本回归判断

- 服务详情显示当前版本错误率升高
- 用户对比前一个版本
- 下钻资源和错误 Trace
- 确认当前版本问题集中

### 场景 4：字段不完整降级

- 接入成功但缺少 `service.version`
- 服务可见
- 版本对比不可用
- 页面明确提示原因和修复方向

### 场景 5：拓扑不可用降级

- 服务有 Trace
- 但调用关系字段不足
- 服务详情可用
- 拓扑区提示“数据不足，暂无法形成稳定依赖关系”

### 场景 6：namespace 缺失

- 平台收到 Trace 数据
- 服务进入 `未归类服务`
- 接入详情页显示最近发现的服务映射结果和字段问题
- 用户补齐字段后服务进入正确业务系统分组
