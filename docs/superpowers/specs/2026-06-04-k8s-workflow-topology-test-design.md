# K8s 工作流与拓扑专题联合测试设计

## 背景

当前本地已经存在一条可执行的 `k8s配置能力检测` 工作流，能够在真实 Kubernetes 环境中完成基础分流、配置检查和修复建议链路。此前后端重放已经验证：

- 模糊问题会进入 `通用` 分支
- 明确的配置检查问题会进入 `k8s配置检查` 分支
- K8s 检查分支能够真实调用本地集群工具并返回检查结果

但这还不足以回答两个更关键的问题：

1. 当前工作流在更贴近真实运维语境的多类提问下是否仍然稳定可用。
2. 当前能力距离 `docs/superpowers/specs/2026-06-01-opspilot-k8s-agentic-topology-analysis-design.md` 中定义的“全量检查 + 关系洞察 + AI 主导分析”目标还有多大差距。

本设计因此采用一套 **联合测试方案**：既验证当前工作流能不能稳定承接真实用户提问，也验证它在“全量巡检、共享依赖、链式影响、跨 workload 共因”场景下会如何表现。

## 目标

- 为当前 `k8s配置能力检测` 工作流建立一套更强的端到端测试矩阵。
- 使用本地 minikube 构造一套 **中等规模、关系可控** 的测试资源，覆盖共享依赖与链式关系。
- 覆盖 4 类典型用户意图：模糊求助、明确配置检查、全量巡检、修复追问。
- 将 **当前工作流可用性** 与 **拓扑专题目标差距** 分层记录，避免混淆“现状验证”和“未来能力评估”。
- 将根目录 `k8s-config-check-prompt.txt` 作为测试语义基线，使 case 更贴近当前 Z 智能体实际提示词约束。

## 非目标

- 不在本设计中直接实现新的拓扑分析引擎或 Map-Reduce 执行链。
- 不要求当前工作流已经完全达到拓扑专题设计目标。
- 不构造超大规模压测集群；本轮重点是关系覆盖和语义验证，不是极限性能测试。
- 不改造前端交互或工作台画布结构。

## 核心设计结论

### 1. 测试分为“两层”

本轮测试必须严格拆成两层：

1. **当前工作流层**
   - 关注现有 `web_chat -> intent_classification -> k8s配置检查/通用` 链路是否正确。
   - 结论是“当前是否可用”。

2. **拓扑专题目标层**
   - 关注当前能力在“全量检查、关系保留、共因识别、链式影响”场景下是否失真。
   - 结论是“距离专题目标还差什么”。

只有这样，测试结果才不会把“工作流能跑”误写成“已经具备拓扑专题能力”。

### 2. 集群资源不追求大，而追求关系清晰

本轮不做无目的地堆 workload 数量，而是构造一套 **中等规模、依赖显式、故障点可预测** 的测试资源。判断标准不是“对象够多”，而是是否能稳定覆盖：

- `ingress -> service -> deployment -> pod` 主链
- 共享 `ConfigMap`
- 共享 `Secret` 或 `PVC`
- selector 不匹配
- 同一配置来源导致多个 workload 同时受影响

这样既能支撑当前工作流检查，也能暴露“当前是否能识别全局共因”。

### 3. 用户提问覆盖按“意图差异”组织

本轮优先覆盖以下 4 类用户意图：

- 模糊求助
- 明确配置检查
- 全量巡检
- 修复追问

每一类都同时补两种变体：

- **短句版**：更容易暴露误判和分流问题
- **带上下文版**：更贴近真实值班/排障语境

### 4. 结果记录分为“硬标准”和“观察项”

每个测试 case 统一按 4 个维度记录：

1. **路由是否正确**
2. **是否真实调用到 K8s 能力**
3. **输出是否覆盖用户意图**
4. **是否保留关系/全局语义**

其中：

- 前 3 项属于当前工作流层的 **硬标准**
- 第 4 项属于拓扑专题目标层的 **观察项**

## 本地测试资源设计

### 命名空间

- `opspilot-check-a`
- `opspilot-check-b`

这两个命名空间分别承担“主链路验证”和“跨 workload / 共享依赖验证”。

### 资源拓扑

建议至少包含以下资源：

#### A 组：主链路资源

- 2~3 个 Deployment
- 2~3 个 Service
- 1 个 Ingress

用于验证：

- 明确配置检查是否能定位指定目标
- service / ingress / workload 主链是否会被保留在结果中

#### B 组：共享依赖资源

- 2~3 个 Deployment
- 至少 2 个 Deployment 共享同一个 ConfigMap
- 至少 2 个 Deployment 共享同一个 Secret 或 PVC

用于验证：

- 共享配置问题能否体现为多 workload 共同受影响
- 输出是否仍然只把问题切成孤立 deployment 级问题

### 故障注入点

本轮故障点不追求多，而追求可归因：

1. 一个 Deployment 缺少 probe
2. 一个 Deployment 缺少 resources.requests/limits
3. 一个 Deployment 的 Service selector 不匹配
4. 一个共享 ConfigMap 被错误配置，导致两个 Deployment 同时带出问题
5. 可选：一个 workload 绑定共享 PVC，用于观察是否出现跨 workload 存储关系线索

### 资源规模原则

- 总 Deployment 数量控制在 4~6 个
- 总关系边足够覆盖共享配置、服务暴露、依赖链
- 不故意造大于 50 workload 的分页压力

本轮目标是验证“关系覆盖”和“语义退化点”，不是验证大规模分页策略。

## 用户意图测试矩阵

### 1. 模糊求助

示例：

- `帮我看一下`
- `看看这个集群`
- `感觉有点不对劲`

验证重点：

- 应命中 `通用` 或引导型响应
- 不应直接编造问题结论
- 应引导补充动作、范围或目标对象

观察重点：

- 是否能最少轮次澄清，而不是泛泛教学

### 2. 明确配置检查

示例：

- `检查 opspilot-check-a 里 payment deployment 配置`
- `看下 default namespace 的 deployment 配置是否有问题`

验证重点：

- 应命中 `k8s配置检查`
- 应真实调用 K8s 工具
- 应返回问题、影响对象、报告或修复后续路径

观察重点：

- 输出是否与 `k8s-config-check-prompt.txt` 中“按意图执行、不要擅自扩大范围”的约束一致

### 3. 全量巡检

示例：

- `帮我做一次全集群配置巡检`
- `检查这个集群整体有哪些配置风险`

验证重点：

- 不能被误判到 `通用`
- 不能退化成只看单 workload

观察重点：

- 当前实现是否还能保留“全量检查”语义
- 结果是否表现为局部问题堆叠，而缺乏共因/链式关系
- 如果当前能力退化，要明确记录是在哪一步退化：范围、关系还是总结方式

### 4. 修复追问

示例：

- `按问题类别给我修复方案`
- `先看 payment 怎么修`
- `全部一次性展示`

验证重点：

- 应延续上一轮检查上下文
- 应进入修复建议 / 命令输出链路
- 不应退化成重新执行一轮普通问答

观察重点：

- 默认选项超时分支是否可接受
- 显式选择与默认回落的结果差异是否符合预期

## 通过标准

### 当前工作流层：必须通过

以下结论必须成立，才能说明当前工作流“可用”：

- 模糊求助进入通用/引导路径
- 明确配置检查进入 K8s 检查路径
- 全量巡检类请求不误入通用
- 修复追问能够承接上一轮检查上下文
- 至少一组检查请求真实命中本地 K8s 工具

### 拓扑专题目标层：记录差距，不强行判失败

以下维度作为专题目标观察项：

- 是否能识别共享依赖导致的多 workload 共因
- 是否能表达 service / ingress / configmap / pvc 等关系
- 是否保留“全集群检查”语义
- 是否随着资源数量增加而明显退化为局部报告拼接

这些项本轮允许“不满足”，但必须记录差距和触发条件。

## 执行顺序

### 阶段 1：环境准备

- 检查 minikube 当前资源
- 创建 `opspilot-check-a` / `opspilot-check-b`
- 应用测试 workload、service、ingress、configmap、secret/pvc
- 校验故障注入是否生效

### 阶段 2：当前工作流验证

- 逐条重放 4 类用户意图
- 记录路由节点、工具调用、最终结果
- 确认哪些路径在后端已跑通，哪些仍需 UI 交互补证

### 阶段 3：拓扑专题目标观察

- 用“全量巡检”与“共享依赖问题”场景重放
- 记录结果是否出现全局视角、共因识别、链式影响
- 对照 `2026-06-01-opspilot-k8s-agentic-topology-analysis-design.md` 标记差距

### 阶段 4：测试结论输出

最终输出分成两部分：

1. **当前工作流结论**
   - 哪些用户意图已经可以稳定承接
   - 哪些交互路径仍依赖默认选项或 UI 实际点选

2. **拓扑专题差距结论**
   - 当前在哪些场景还能保持全量语义
   - 哪些场景已经暴露出关系丢失或全局洞察不足
   - 后续优先该补什么能力

## 风险与边界

- 当前 `request_user_choice` 在后端重放时可能因无前端点选而走默认值，这会影响“真实交互表现”，但不影响“是否进入修复链路”的判断。
- 当前工作流可能具备“基础全量检查”能力，但这不等价于已经满足拓扑专题中的关系图驱动分析目标。
- 如果本地集群状态与测试设计冲突，应优先保证关系链和故障点可复现，而不是迁就现有资源残留状态。

## 验证方法

本设计落地后，执行结果应至少包含：

- 每个 case 的输入语句
- 命中的工作流节点链
- 真实工具调用证据
- 最终输出摘要
- 对“关系/全局语义”的观察结论

这样既能验证当前工作流，也能为后续拓扑专题实现提供一组稳定的基线场景。
