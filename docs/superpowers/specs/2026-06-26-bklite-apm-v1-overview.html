BK-Lite APM v1 全貌

面向运维 / SRE 的应用排障工作台。统一使用 OTLP 接入，围绕业务系统（service.namespace）组织服务， 通过服务目录、版本对比、服务级拓扑、资源分析与 Trace 证据，完成“接入 -> 发现异常 -> 定界 -> 归因”的闭环。

OTLP 接入 VT 存储 service.namespace 分组 版本对比 服务级拓扑 轻量错误聚合

主导航少、下钻强。服务详情和资源详情都不是一级菜单，而是围绕调查任务逐步深入。

  • 最可疑服务

定界 / 归因

两条核心闭环

v1 不追求大而全平台，而是把“接入闭环”和“排障闭环”做完整。

1

接入开始

用户进入接入中心，选择 Java / Python / Go / Node.js / OTel Collector / K8s 注解 / eBPF 路径。

接入中心的作用是帮用户选路径，而不是只展示文档。

2

接入确认

生成凭证，复制 OTLP endpoint 与配置模板。

平台同时展示最近发现的 namespace / service / version，确认“平台认出了什么”。

3

发现异常

服务进入服务目录，首页值班调查台开始高亮异常业务系统、异常服务、疑似版本回归。

未归类服务显式暴露，避免数据进来但用户找不到对象。

4

定界与归因

从服务详情进入局部拓扑，默认一跳上下游。

再下钻到资源和 Trace，判断是服务自身问题、版本回归，还是下游依赖拖慢。

接入中心全貌

接入中心是标准化 OTLP 接入入口，不做远程控制台，但必须让用户知道“怎么接、接好了没有、平台看到了什么”。

集成目录

按推荐场景而不是纯技术名词帮助用户做选择。

  • 主流语言卡片
  • Collector 入口
  • K8s 注解注入
  • eBPF 快速试点

接入自检

判断“卡在哪一步”，而不是简单红绿灯。

  • 接入凭证
  • 传输与准入
  • 数据接收
  • 字段规范
  • 分析可用性

服务发现映射

把“平台识别到了什么”前置给用户。

  • 最近发现的 namespace
  • 最近发现的 service
  • 最近发现的 version
  • 未归类服务提醒

最小可用凭证

有凭证能力，但不做重型凭证管理中心。

  • 生成 token
  • 首次展示明文
  • 备注
  • 失效 / 停用

拓扑能力定位

先不谈展示形态，先定义它为什么存在、解决什么问题、必须满足哪些调查原则。

拓扑首要支持的调查任务

  • 告警后定界：判断影响范围、识别共享异常下游、缩小排查面。
  • 慢请求归因：判断慢在本服务还是慢在下游，识别最可疑的调用边。
  • 逐步下钻：默认一跳，沿节点推进，而不是一上来两跳三跳全展开。

拓扑不是这些东西

  • 不是 CMDB 物理拓扑
  • 不是炫技大屏
  • 不是资源级主图
  • 不是自动根因系统
  • 不是只看关系、不看健康的依赖图

异常优先

默认先突出异常边、异常邻接服务和最近退化关系，正常关系只做补充。

边比点更重要

节点承载身份和摘要，边承载请求量、错误率、P95、最近活跃等排障证据。

时间窗是一等公民

所有拓扑关系都绑定当前时间窗，避免把历史残影误当成当前故障路径。

状态与降级语义

BK-Lite APM v1 的关键不是“有没有数据”，而是“当前能查到哪一层、为什么不能再往下”。

接入状态

未开始 / 配置中

说明还没有形成可用接入闭环，用户应停留在接入中心继续配置。

服务状态

正常 / 关注 / 异常

服务状态只表达当前时间窗的调查健康度，不混入接入配置语义。

版本状态

可对比 / 不可对比

缺少 service.version 时，服务仍可分析，但版本回归判断必须降级。

拓扑边状态

数据不足 / 无活跃

允许 OT 数据不完美，显式告诉用户是数据不足还是当前无活跃依赖。

未归类服务

字段不完整

Trace 进来了但 service.namespace 缺失时，不隐藏对象，而是进入未归类分组并提示修复。
