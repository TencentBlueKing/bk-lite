# K8s 配置检查报告美化设计

## 背景

当前 K8s 配置检查结果虽然已经能输出结构化结论，但在聊天区里的最终观感仍然偏“长段 Markdown 文本”：

- 信息层级不够清晰
- 问题项经常堆成一大段
- 表格不像真正的表格
- 与用户期望的“正式运维检查报告”存在明显落差

已有代码路径表明，这类结果不是普通聊天文本，而是通过 `config_analysis_report` 事件进入前端渲染。因此这次改造不应只靠 prompt 优化文案，而应明确区分：

- 普通聊天消息：继续走现有 Markdown 渲染
- 配置检查报告：走更稳定、更轻量的专用报告渲染

## 目标

- 保留当前聊天区的轻量风格，不做深色大屏化改造。
- 让 K8s 配置检查报告具备明确的信息层级、摘要区和真正的表格展示。
- 仅增强 `config_analysis_report` 的显示效果，不影响普通聊天消息渲染。
- 为后续扩展排序、折叠、复制、导出等能力保留结构化基础。

## 非目标

- 不重做整个聊天 UI。
- 不把所有 Markdown 消息都升级成专用组件。
- 不在这一版里引入复杂 dashboard、图表、拖拽、筛选器。
- 不修改配置检查规则本身，只改报告承载和展示方式。

## 现状

### 相关文件

- `server/apps/opspilot/metis/llm/chain/node.py`
- `web/src/app/opspilot/components/custom-chat-sse/index.tsx`

### 当前行为

- 后端会在配置检查阶段生成 `config_analysis_report` 相关内容。
- 聊天区最终仍以通用 markdown 渲染为主。
- 报告内容虽然有标题、分级和结论，但结构仍偏向文本拼接。
- 前端缺少专门面向“配置检查报告”的轻量组件。

## 核心设计结论

### 1. 采用“两者结合”模式

普通回答继续使用现有 Markdown 渲染路径，仅对 `config_analysis_report` 这一类结构化结果启用专用组件。

这样可以同时满足两个要求：

- 聊天气质不变，界面仍然轻量
- 配置检查报告获得独立的信息架构和表格渲染能力

### 2. 报告视觉保持“聊天区轻样式”

报告视觉不走深色运维大屏路线，而是延续聊天区的浅色、低压、留白型风格：

- 浅底卡片
- 轻边框
- 小体量摘要块
- 明确但不过重的分区标题
- 真表格而不是 bullet list 伪装表格

目标不是“做成 dashboard”，而是“在聊天气泡里看起来像一份清爽的正式报告”。

### 3. 前端不再从 Markdown 猜结构

这次改造的关键边界是：前端不能继续依赖 markdown 文本去猜测报告结构。推荐改为：

- 后端输出稳定的结构化报告 payload
- 前端识别到该事件后直接渲染报告组件
- 若新字段缺失，则自动降级回现有 markdown 展示

这样可避免样式和内容长期被 prompt 文案波动牵着走。

## 组件设计

### A. `ConfigAnalysisReportCard`

作为配置检查报告的最外层容器，负责承接：

- 报告标题
- 实例名 / 资源范围
- 生成时间或扫描范围信息
- 顶部摘要区
- 问题分区
- 修复建议区

该容器只服务于结构化配置检查结果，不承担普通消息渲染职责。

### B. `ReportSummaryCards`

位于报告顶部，采用 3~4 个轻量摘要卡展示核心结果，建议字段为：

- 扫描资源数
- 问题总数
- 高优先级问题数
- 当前最优先建议

该区域的目标是让用户不读完整报告，也能先抓住总体结论。

### C. `IssueTableSection`

每个严重级别（如 High / Medium / Low）单独一个 section。每个 section 内使用真正的 HTML 表格展示问题项，而不是继续使用长文本列表。

推荐列结构：

| 列名 | 含义 |
|---|---|
| 问题项 | 问题名称，如“未配置存活探针” |
| 影响范围 | 受影响的工作负载数量或对象范围 |
| 典型对象 | 代表性 workload / resource 示例 |
| 风险说明 | 对当前问题的运维影响说明 |

样式要求：

- 固定表头
- 行分隔明显
- 轻度斑马纹或隔行底色
- 列宽可控
- 超长内容截断并支持补充查看

### D. `RecommendationTable`

报告末尾单独展示修复建议表，避免建议内容和问题结果混在一起。

推荐列结构：

| 列名 | 含义 |
|---|---|
| 优先级 | P1 / P2 / P3 |
| 修复动作 | 建议执行的修复措施 |
| 目标对象 | 推荐优先处理的对象 |
| 预期收益 | 修复完成后带来的收益 |

## 数据契约设计

### 后端职责

后端继续负责生成配置检查结论，但输出应从“偏文本拼接”收敛为“结构化报告对象”。建议至少包含以下字段：

- `title`
- `instance_name`
- `scope`
- `summary`
- `severity_sections`
- `recommendations`
- `fallback_markdown`

其中：

- `summary` 用于顶部摘要卡
- `severity_sections` 用于问题表格分区
- `recommendations` 用于建议表
- `fallback_markdown` 用于前端降级显示

### 前端职责

前端在 SSE 消息流中识别配置检查报告事件：

1. 若结构化字段完整，则渲染专用报告组件
2. 若结构化字段不完整，则回退使用 `fallback_markdown`
3. 普通聊天消息继续走现有 Markdown 路径

这样可以把“报告结构”和“普通对话文本”从渲染层面彻底分开。

## 交互与展示规则

### 1. 有问题场景

- 展示摘要卡
- 按 severity 分组展示问题表
- 报告末尾展示修复建议表

### 2. 无问题场景

不渲染空表，直接展示一个清晰的“未发现明显配置问题”结果卡，并保留扫描范围与资源数信息。

### 3. 超长内容场景

- 单元格默认截断 1~2 行
- 保持报告整体高度可控
- 完整内容通过 tooltip、展开或二级明细承接

### 4. 历史兼容场景

旧消息历史不需要迁移。仅当检测到新结构化配置检查报告时，才启用新组件。

## 错误处理

- 结构化字段缺失：前端自动降级为 Markdown，不白屏、不报错占位。
- 单个 severity 分区无条目：不渲染该分区。
- 建议区为空：允许省略，不强行输出空表。
- 后端仍输出旧格式：前端按现有逻辑兜底。

## 测试与验证

### 前端验证

- `web` 下运行 `pnpm lint`
- `web` 下运行 `pnpm type-check`

### 手工验证重点

1. 有问题报告能以浅色专用组件渲染。
2. 问题区以真实表格展示，而不是长文本堆叠。
3. 普通聊天消息仍按原有 markdown 展示。
4. 无问题场景不出现空表。
5. 缺少结构化字段时可自动降级。
6. 超长单元格内容不会把聊天区撑坏。

## 风险与控制

### 风险

- 若后端结构化字段定义不稳，前端组件容易被迫频繁跟着调整。
- 若表格信息过多，聊天区可能出现高度过长或视觉压迫。
- 若直接替换现有 markdown 路径，可能误伤普通消息展示。

### 控制措施

- 仅对 `config_analysis_report` 启用专用组件。
- 保留 `fallback_markdown` 作为兼容兜底。
- 初版控制字段数量和表格列数，不做复杂交互。
- 样式上保持轻量，不引入与当前聊天区冲突的视觉系统。

## 预期结果

完成后，K8s 配置检查报告将从“聊天区里的一大段 markdown 文本”升级为“聊天区里的轻量正式报告”：

- 一眼能看懂摘要
- 问题项有真正的表格感
- 修复建议和问题结果分区明确
- 普通聊天体验保持不变

这会显著提升配置检查结果的可读性和可信度，同时为后续继续增强报告能力打下稳定结构基础。
