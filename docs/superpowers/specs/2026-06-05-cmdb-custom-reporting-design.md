# CMDB 自定义上报设计

日期: 2026-06-05
范围: `server/apps/cmdb` 与 `web/src/app/cmdb`，覆盖自定义上报任务、快速模型、批次记录、接入文档与关系/清理链路
目标: 在不开发新采集插件、不接管客户脚本调度的前提下，给 CMDB 增加一条“客户主动推送”的统一数据接入通道，并与现有自动发现共用合并、关系和变更记录语义。

## 1. 背景与问题

当前 CMDB 的自动发现链路主要面向平台内置插件覆盖到的标准对象：主机、数据库、中间件、K8s、云资源、SNMP 网络设备等。这条链路默认假设：

1. 平台知道怎么连目标源端；
2. 平台负责调度采集任务；
3. 平台能用内置插件把数据采成统一格式。

但真实客户环境里始终存在长尾对象：

1. 私有业务系统；
2. 自研中间件；
3. 特殊设备或私有协议；
4. 客户内部平台与脚本生成的数据。

这些对象往往等不到平台提供官方插件，也未必有标准协议可接入。如果只能走“自动发现”，客户就必须等待插件开发或改平台代码，无法自助补盲。

监控系统已经有“自建插件 + 自定义上报”的接入心智，但 CMDB 的接入比监控更重，因为它不仅要接指标，还要处理：

1. 模型字段与字段分组；
2. 联合唯一规则；
3. 实例关系；
4. 自动关联规则；
5. 数据清理策略；
6. 变更记录归属。

因此本次问题不是“再开一个上传接口”，而是要补一条**比自动发现更宽松、但进入 CMDB 后仍与现有数据治理语义完全一致**的接入通道。

## 2. 目标与非目标

### 2.1 目标

1. 在 CMDB 中新增与“自动发现”并列的“自定义上报”业务模块。
2. 支持任务的新建、编辑、删除、列表、详情。
3. 任务支持绑定已有完整模型，或在创建过程中内联创建快速模型。
4. 自定义上报进入 CMDB 后，实例合并、关系建立、正式模型自动关联、变更记录表现与自动发现保持一致。
5. 支持在一次上报中同时提交实例数据与关系数据。
6. 支持本批互引、引用现有实例、引用尚未落地实例三种关系场景。
7. 支持不删除、过期清理、快照清理三档策略，并为快照清理提供删除比例阈值保护。
8. 为每批上报生成可追溯的批次记录，并为每个任务生成可复制的接入文档。
9. 自定义上报触发的变更记录归入新的“自定义上报”场景。
10. 任务具备独立凭证，支持轮换和作废，且互不影响。

### 2.2 非目标

1. 本期不承接客户脚本的调度、运行、重试与失败补偿。
2. 本期不提供客户脚本模板库或脚本托管能力。
3. 本期不为自定义上报单独开发新的资产拓扑或数据可视化能力。
4. 本期不提供“快速模型转正式模型”的升级通路。
5. 本期不让快速模型参与模型管理导入导出。
6. 本期不让快速模型主动声明新的关系类型。
7. 本期不让自动关联规则在快速模型作为源模型时生效。
8. 本期不新增测试文件，只在现有测试承载文件中补充覆盖。

## 3. 方案对比与选择

### 方案 A: 新增独立自定义上报模块，复用现有合并引擎（推荐）

新增独立的“自定义上报任务”领域对象、任务凭证、批次记录、接入文档和快照审核流程；实例合并、关系建立、正式模型自动关联、变更记录写入复用现有 CMDB 能力。

优点:

1. 与“自动发现”形成清晰的产品心智边界：一个是平台主动拉，一个是客户主动推。
2. 权限、凭证、批次、审核、接入文档都能独立建模，不污染现有自动发现任务结构。
3. 底层仍复用现有 CMDB 引擎，避免出现第二套合并语义。
4. 后续可以平滑扩展更多接入形态而不影响自动发现。

缺点:

1. 后端和前端都需要增加新模块，改动面中等。

### 方案 B: 直接扩展现有自动发现任务模型

在 `CollectModels` 或自动发现任务结构上增加 `push` 模式，让现有任务同时承载“平台主动拉”和“客户主动推”。

优点:

1. 数据表复用最多，短期开发量最小。

缺点:

1. 自动发现现有任务天然绑定调度、接入点、凭据池、插件、执行状态等“主动采集”语义。
2. 强行混入 push 模式会让 UI、权限、任务状态与接入文档语义都变得别扭。
3. 后续维护时很容易继续向 `CollectModels` 堆叠与 push 无关的兼容逻辑。

### 方案 C: 只提供通用上报 API，不建立任务模块

只做一个 API，按组织级或模型级 token 允许客户脚本写入实例与关系，不建立任务对象、批次记录与接入文档。

优点:

1. 最快能打通“数据可入库”的最小闭环。

缺点:

1. 不满足任务 CRUD、独立凭证、批次追踪、快照审核、接入文档等核心验收项。
2. 无法清晰承接清理策略和权限隔离，产品形态不成立。

最终选择: 方案 A。

## 4. 总体架构与职责边界

### 4.1 模块边界

“自定义上报”作为 CMDB 管理域下的独立模块，与“自动发现”并列，但底层不重复实现合并逻辑。

职责划分如下：

1. **自定义上报模块**
   - 管理任务、任务凭证、批次、接入文档、快照审核。
   - 提供接收 HTTP Push 数据的接口。
2. **现有 CMDB 模型管理**
   - 继续承载正式模型定义、字段、字段分组、联合唯一规则、关系类型、自动关联规则。
   - 为快速模型提供同构存储容器。
3. **现有实例/图数据库写入链路**
   - 负责实例新增、更新、删除、关系创建与关系回填。
4. **现有变更记录链路**
   - 负责把本次写入统一记录为“自定义上报”场景。

### 4.2 平台职责边界

平台职责收敛为：

1. 接收任务级上报请求；
2. 做基础鉴权和契约校验；
3. 调用 CMDB 合并/关系/清理引擎完成写入；
4. 留下批次、变更和审核记录。

平台**不负责**：

1. 调度客户脚本；
2. 管理客户脚本运行状态；
3. 帮客户做源端重试或失败恢复。

### 4.3 接入方式选择

本期默认采用**任务级 HTTP Push API**：

1. 最符合“客户主动推”的产品心智；
2. 最容易生成接入文档；
3. 最适合一任务一凭证的安全模型；
4. 最容易复用现有 Django API、权限与日志能力。

本期不引入消息总线直连或文件导入作为主接入形态。

## 5. 核心领域模型设计

### 5.1 自定义上报任务

新增独立任务对象，至少具备以下字段：

1. `name`: 任务名称，同一组织内唯一。
2. `team`: 组织/团队隔离信息。
3. `model_mode`: `standard` / `quick`，表示绑定完整模型还是快速模型。
4. `model_id`: 目标模型标识。
5. `identity_keys`: 身份键列表。
6. `cleanup_strategy`: 清理策略。
7. `expire_days`: 过期清理天数。
8. `snapshot_delete_ratio_threshold`: 快照清理删除比例阈值。
9. `credential_status`: 凭证状态。
10. `last_reported_at`: 最近接入时间。
11. `doc_version`: 接入文档版本。

该任务对象不能复用现有 `CollectModels`，因为它不需要：

1. 插件类型；
2. 调度周期；
3. 接入点；
4. 平台凭据池；
5. 主动执行状态。

### 5.2 任务凭证

任务级凭证采用“一任务一凭证”模型：

1. 创建任务时自动签发。
2. 支持轮换，新 token 生效后旧 token 立即失效。
3. 支持作废，作废后任务立即停止接收上报。
4. 库内只保存凭证摘要、状态和使用时间，不保存长期明文。

凭证对象至少记录：

1. `task_id`
2. `token_hash`
3. `rotated_at`
4. `revoked_at`
5. `last_used_at`

### 5.3 快速模型

快速模型与正式模型底层同构存储，不新建第二套图数据库表达，只增加标志位与治理元数据：

1. `is_quick_model`
2. `identity_keys`
3. `schema_state`
4. `source_task_id`

快速模型创建时只强制要求：

1. 模型名
2. 身份键
3. 清理策略

字段行为如下：

1. 首次上报出现的字段自动登记。
2. 后续新增字段自动追加。
3. 默认按字符串类型落地。
4. 自动追加字段标记为“未定型字段”。

快速模型不参与：

1. 导入导出；
2. 升级为正式模型；
3. 主动声明关系类型。

### 5.4 上报批次

每次上报创建批次记录，至少包含：

1. `task_id`
2. `requested_at`
3. `status`
4. `summary_counts`
5. `error_count`
6. `pending_relation_count`
7. `review_status`
8. `cleanup_decision`
9. `payload_digest`
10. `operator_hint` 或来源标识

批次状态建议统一为：

1. `processing`
2. `success`
3. `partial_success`
4. `failed`
5. `pending_review`

### 5.5 待关联关系

为支持跨批引用和乱序上报，需要独立表示“待关联关系”：

1. 源实例已存在，但目标实例尚未落地；
2. 关系类型已知；
3. 目标模型与目标身份键已知；
4. 后续目标实例落地后可自动补齐。

## 6. 模型与治理策略

### 6.1 完整模型路径

任务绑定现有正式模型时：

1. 上报数据按现有模型 schema 严格校验；
2. 联合唯一规则沿用现有模型定义；
3. 关系类型沿用现有模型管理配置；
4. 自动关联规则按现有正式模型能力执行。

### 6.2 快速模型路径

任务在创建过程中可内联创建快速模型：

1. 向导只要求填写模型名、身份键、清理策略。
2. 字段细节不在任务创建期强制声明。
3. 首批数据进入后补足字段元数据。

设计取向是：

1. 把“命名、身份、清理”提前做强决策；
2. 把“字段细节”延迟到数据进入后再治理。

### 6.3 自动关联规则边界

自动关联规则只在目标为正式模型时生效：

1. 快速模型实例可以被其他实例引用。
2. 快速模型本身不主动声明关系类型。
3. 快速模型作为源模型时，不触发自动关联规则。

这样可以保持快速模型接入宽松，但不让治理复杂度失控。

## 7. 上报数据流设计

### 7.1 接收入口

新增任务级 HTTP API，例如：

1. `POST /cmdb/api/custom-reporting/tasks/{task_id}/ingest/`

请求包含：

1. `instances`
2. `relations`
3. `batch_metadata`

Header 携带任务级 token。

### 7.2 鉴权与基础校验

接收到请求后按以下顺序处理：

1. 根据任务 ID 和 token 鉴权。
2. 校验任务是否被作废或停用。
3. 更新 `last_used_at`。
4. 创建批次记录，初始状态为 `processing`。
5. 对 payload 做基础结构校验。

校验规则：

1. 正式模型走严格 schema 校验。
2. 快速模型只强校验身份键和基础 JSON 结构。
3. 无效请求可直接拒绝，批次记为失败或错误批次。

### 7.3 快速模型字段登记

若任务绑定快速模型，则在本批处理前先扫描字段：

1. 已登记字段直接复用。
2. 新字段自动追加到模型元数据。
3. 新字段默认字符串类型，状态为“未定型”。

本期不在主链路做复杂类型推断，避免出现不可预测的 schema 漂移。

### 7.4 实例合并

实例合并沿用现有 CMDB 语义：

1. 按模型 + 身份键查找已有实例。
2. 命中则更新，未命中则新增。
3. 所有写入继续走现有 FalkorDB / CMDB 实例写回链路。
4. 变更记录场景使用新的“自定义上报”枚举值。

### 7.5 关系处理

关系支持三种场景：

1. **本批互引**
   - 在批次内先构建实例身份索引，再解析关系。
2. **引用 CMDB 已有实例**
   - 按目标模型 + 身份键即时绑定。
3. **引用尚未落地目标**
   - 关系记为待关联，后续目标实例落地后补齐。

该设计允许最终一致，不要求客户脚本保证全局拓扑顺序。

## 8. 清理策略与审核设计

### 8.1 不删除

只做新增和更新，不删除历史实例。

### 8.2 过期清理

为任务作用域内的实例维护“最近被该任务覆盖时间”：

1. 上报命中的实例更新覆盖时间。
2. 超过 `expire_days` 未被覆盖的实例进入清理。

### 8.3 快照清理

本次上报视为该任务的全量真相：

1. 找出该任务作用域内、本批未出现的实例。
2. 将这些实例作为候删集合。
3. 计算候删比例。

若候删比例超过阈值：

1. 不直接删除。
2. 批次状态记为 `pending_review`。
3. 等待人工审核后再执行删除。

该设计与自动发现现有清理概念保持同源，降低用户学习成本。

## 9. 变更记录、审计与接入文档

### 9.1 变更记录归属

在 `ChangeRecord` 场景中新增“自定义上报”：

1. 与“自动采集”区分；
2. 可独立检索；
3. 仍复用现有变更记录展示能力。

### 9.2 批次记录展示

任务详情页展示：

1. 最近批次趋势；
2. 新增/更新/删除/错误统计；
3. 最近一次接入时间；
4. 待审核批次状态。

视觉上对齐现有自动发现任务概览。

### 9.3 接入文档

每个任务自动生成可复制的接入文档，至少包含：

1. 端点地址；
2. Header 与 token 使用方式；
3. JSON payload 示例；
4. 字段含义；
5. 关系上报示例；
6. 常见错误码；
7. 快速模型场景下的字段自动登记说明。

正式模型文档展示固定 schema；快速模型文档展示当前已登记字段与身份键约束。

## 10. 前端设计

### 10.1 菜单与路由

在 `CMDB > 管理` 下新增“自定义上报”，与“自动发现”平级。

建议页面包括：

1. 任务列表页
2. 任务创建/编辑向导
3. 任务详情页
4. 批次详情或批次抽屉
5. 快照审核入口

### 10.2 创建向导

向导分两条路径：

1. 绑定完整模型
2. 内联创建快速模型

快速模型路径只暴露：

1. 模型名
2. 身份键
3. 清理策略

避免把字段细节、关系细节提前塞进创建流程。

### 10.3 详情页

详情页聚合：

1. 基础信息
2. 批次记录
3. 最近趋势
4. 接入文档
5. 凭证轮换/作废操作

### 10.4 模型管理页联动

模型管理中需要识别快速模型，但只做必要能力暴露：

1. 支持用标识区分快速模型与正式模型。
2. 快速模型不展示导入导出操作。
3. 快速模型仍可被查看属性和基础元数据。

## 12. 验收标准映射

本设计对应验收点如下：

1. 客户脚本可通过 HTTP Push API 持续把任意对象数据上报到 CMDB。
2. CMDB 内可独立完成自定义上报任务的新建、编辑、删除、列表、详情查看。
3. 任务既可绑定完整模型，也可内联创建快速模型。
4. 上报后的实例合并、关系建立、正式模型自动关联与自动发现保持一致。
5. 关系支持本批互引、引用现有实例、引用尚未上报实例并最终补齐。
6. 三档清理策略按定义生效，快照删除超阈值进入审核。
7. 每批上报均留下可追溯批次记录。
8. 每个任务均有接入文档。
9. 变更记录中可独立检索“自定义上报”场景。
10. 任务凭证可轮换、可作废，且互不影响。

## 13. 风险与缓解

1. **把 push 语义强塞进现有自动发现任务结构**
   - 缓解: 保持任务层独立，避免复用 `CollectModels`。

2. **快速模型字段无限扩张，治理失控**
   - 缓解: 只放宽字段登记，不放宽身份键、清理策略和正式模型自动关联边界。

3. **快照清理误删过多实例**
   - 缓解: 强制删除比例阈值和人工审核状态。

4. **跨批关系长期悬挂**
   - 缓解: 为待关联关系建立明确记录，并在目标实例落地时收口补齐。

5. **前端把自定义上报与自动发现混在一起，心智不清**
   - 缓解: 菜单、页面、文案保持并列而非嵌套复用。
