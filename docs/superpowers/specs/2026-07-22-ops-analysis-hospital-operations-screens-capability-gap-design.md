# 医院运维监控大屏能力评估与数据源补齐设计

## 1. 背景与目标

本设计用于评估三张医院运维监控大屏示意图在运营分析模块中的可搭建性，并定义首期需要补齐的数据源能力。

首期以数据内容和统计口径正确为优先，视觉效果允许使用现有大屏组件近似实现，不新增专用视觉组件。

本次交付只补齐缺失的后端能力及运营分析数据源注册。具体大屏的创建、组件摆放、样式配置和页面验收由使用方自行完成，不属于本次实施范围。

## 2. 结论摘要

现有运营分析组件足以近似完成三张大屏的布局和图表表达。资源总览、院区资源概览、主机性能 Top10 已有可复用数据源；本次只需补齐以下两个数据源：

1. 网络设备性能 Top10：CPU、内存、当前总流量三个排行。
2. 告警来源分布：按 `Alert.source_name` 聚合权限和时间范围内的告警数量。

以下能力虽已识别，但明确不纳入本次处理：

- `get_alert_trend_data` 的图表声明和默认参数调整。
- `get_active_alert_top` 的运营分析数据源注册。
- “最近产生的活跃告警”排序口径。
- 新增容器、分组框、标题条等专用视觉组件。
- 通用指标聚合框架。

## 4. 方案选择

采用“定向补齐数据源”方案：复用运营分析现有数据源注册机制和现有图表组件，仅新增两个面向明确业务口径的数据源。

未选择纯配置方案，因为缺少网络设备排行和按告警接入来源聚合的数据；未选择通用聚合框架，因为首期只有两个明确缺口，引入通用 DSL、指标映射和权限抽象会明显扩大范围。

## 5. 网络设备性能 Top10

### 5.1 数据源定义

- 标识：`monitor/get_network_device_resource_top`
- 支持图表：`topN`、`table`
- 参数：
  - `metric_type`：必填，取值为 `cpu`、`memory`、`traffic`
  - `limit`：可选，默认 `10`
- 输出：按指标值降序的设备列表，每台设备只保留一行。

建议统一输出字段：

| 字段 | 含义 |
| --- | --- |
| `instance_id` | 归一化后的监控实例标识 |
| `name` | 设备显示名称，缺失时回退到实例标识 |
| `value` | 排序和图表使用的数值 |
| `unit` | `percent` 或 `byteps` |
| `device_type` | 设备监控对象类型 |
| `updated_at` | 指标样本时间 |

### 5.2 首期设备范围

首期只纳入项目当前已经使用统一网络设备身份适配的监控对象类型：

- `Switch`
- `Router`
- `Firewall`
- `Loadbalance`

设备还必须同时满足：

1. 当前用户具有该监控对象的查看权限。
2. 对应实例有效且可归一化为唯一 `instance_id`。
3. 目标指标存在。
4. 最新样本仍在有效时间窗内。

无线、传输、接入、语音网关、控制台服务器等类型虽可归入广义网络设备，但当前项目尚未证明它们共享一致的实例身份、指标语义和采集完整性，因此不进入首期排行。后续完成身份和指标契约统一后再扩展。

### 5.3 指标口径

#### CPU

按设备聚合 `device_cpu_usage` 的当前有效样本，存在多条样本时取设备维度平均值，单位为百分比。

#### 内存

优先使用 `device_memory_usage`。缺少直接使用率时，沿用现有专业监控大盘的回退规则，根据 total/free 或 used/total 指标计算使用率。无法可靠计算时跳过该设备，不以 `0` 代替缺失值。

#### 总流量

每台设备的排行值定义为：

`当前入方向总流量 + 当前出方向总流量`

优先读取设备级聚合指标 `device_total_incoming_traffic` 与 `device_total_outgoing_traffic`。这两个指标应等价于该设备所有接口当前速率的汇总；若由接口计数器计算，则使用现有 5 分钟速率窗口，并按 `instance_id` 汇总所有接口后相加。单位为 `byteps`，前端可格式化为 KB/s、MB/s 或 GB/s。

不得把累计字节计数直接相加作为当前流量，也不得只选择单个接口代表整台设备。

### 5.4 新鲜度与异常处理

- 有效时间窗采用实例采集周期的 2 倍；实例未提供采集周期时回退为 5 分钟。
- 样本过期、指标缺失或无法关联到唯一设备时跳过该设备。
- 没有权限范围内的数据时返回成功的空列表。
- `metric_type` 非法时返回参数错误。
- 指标存储查询失败时返回统一的数据源执行错误，不返回部分结果冒充完整排行。

## 6. 告警来源分布

### 6.1 数据源定义

- 标识：`alert/get_alert_source_distribution`
- 支持图表：`pie`
- 统计对象：`Alert`
- 聚合字段：`Alert.source_name`
- 时间字段：`Alert.created_at`
- 时间范围：不接收时间参数，统计全量告警
- 展示：返回全部明确来源，空来源单独归为“未知来源”

返回结构沿用饼图数据约定：

```json
[
  {"name": "Zabbix", "value": 42},
  {"name": "Prometheus", "value": 28},
  {"name": "未知来源", "value": 6}
]
```

### 6.2 统计规则

1. 必须复用统一告警权限查询范围，不能绕过业务组、组织或用户权限限制。
2. `source_name` 为 `null`、空字符串或仅空白字符时统一归为“未知来源”。
3. 全部明确来源按告警数降序排列；数量相同时使用来源名称稳定排序。
4. 不做 TopN 截断，不合并“其他”；“未知来源”单独作为一个来源返回。
5. 各扇区数量之和必须等于权限范围内的告警总数。
6. 数据源统计告警而不是事件。现有事件来源统计不能复用为该口径。

需要注意：若一个聚合告警由多个不同接入来源的事件组成，当前告警构建逻辑可能无法给出唯一 `source_name`，该告警将进入“未知来源”。这是现有告警数据模型的语义，不在本次数据源中推断或拆分来源。

## 7. 数据流与注册

两个新增能力均沿用现有调用链：

`大屏组件配置 → 运营分析数据源注册 → 对应业务 Handler → 权限过滤与指标/数据库查询 → 标准图表数据 → 现有组件渲染`

在 `source_api.json` 中新增两个数据源声明：

- 网络设备性能 Top10：声明 `topN`、`table`，暴露 `metric_type` 和 `limit`。
- 告警来源分布：声明 `pie`，不暴露参数，直接返回全量来源分布。

不修改现有 `get_alert_trend_data` 和 `get_active_alert_top` 的实现或注册。

## 8. 测试与验收

### 8.1 网络设备性能 Top10

- 四类首期设备均可参与排行，其他类型不会被误纳入。
- 无权限设备不会出现在结果中。
- 同一设备存在多个接口或多条指标序列时只输出一行。
- CPU、内存回退计算和流量入出汇总口径正确。
- 排序、默认 10 条和自定义 `limit` 正确。
- 过期、缺失、无法识别的指标不会以零值污染排行。
- 无数据、非法参数和指标存储异常分别符合约定。

### 8.2 告警来源分布

- 只统计权限范围和时间范围内的 `Alert`。
- 相同 `source_name` 正确合并，空值正确进入“未知来源”。
- 全量来源、“未知来源”的边界正确且总量守恒。
- 无数据时返回可被饼图组件安全处理的空数组。

### 8.3 能力集成验收

- 新增的两个 NATS Handler 可通过运营分析数据源调用，并能被现有 `topN`、`table`、`pie` 组件消费。
- 网络设备排行的数据口径和单位明确，网络流量为整机所有接口当前双向流量之和。
- 告警来源分布按告警 `source_name` 展示。
- 不要求在本次交付中创建或配置任何大屏页面。
- 告警趋势与活跃告警不在本次能力补齐范围内。

## 9. 风险与后续事项

- CMDB 分类和院区标签质量会直接影响第一张大屏，需在上线前核验真实数据覆盖率。
- 网络设备类型扩展依赖统一实例身份与指标契约，不能仅凭导航分组或示意图名称扩容。
- 若设备级总流量指标未覆盖全部接口，需先修正采集或聚合规则，否则排行无法保证内容一致。
- `source_name` 的填充质量决定来源分布的解释力，应监控“未知来源”占比。
- 后续可单独设计告警趋势的单/多序列选择，以及“最近产生的活跃告警”与“持续时间最长告警”的可配置排序。
