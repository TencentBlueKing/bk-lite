# CMDB 生产级 Review 缺陷分阶段修复设计

## 1. 背景与目标

本设计用于修复 CMDB 生产级 Review 中确认的权限、异步状态、跨存储一致性和规模化问题。修复拆成四个可独立交付的阶段，每个阶段必须完成独立的 RED→GREEN→回归闭环后，才能进入下一阶段。

目标是：

- 敏感操作默认拒绝，权限与组织范围在副作用发生前完成校验。
- 异步事件不因锁冲突、重试或迟到回调而丢失或覆盖新状态。
- FalkorDB、关系库、MinIO 和 NATS 之间的非原子边界具备操作日志、幂等与补偿能力。
- API 的分页和批量语义在底层查询层同样成立，不把全量数据加载到进程内后再切片。

## 2. 范围与非目标

### 2.1 本次范围

- `InstanceViewSet.ipam_reconcile` 权限与异步触发边界。
- `K8sSetupViewSet` 安装令牌、安装命令和上报验证权限。
- 订阅通知的可靠投递和幂等消费。
- 采集任务超时、执行代次和迟到结果状态转换。
- 配置文件回调唯一约束、MinIO 内容生命周期和关系库一致性。
- IPAM 全量对账的单执行者、任务记录和可观测状态。
- 实例主数据写入与审计/后置动作的操作日志和补偿。
- K8s 资源视图、实例唯一校验和批量审计的规模治理。

### 2.2 非目标

- 不调整前端交互和页面布局。
- 不改变 FalkorDB、Django ORM、MinIO、Celery、NATS 的基础技术选型。
- 不处理当前工作区缺失的企业版 `custom_reporting` 实现；该能力继续保持社区版 fail closed，并单独等待企业源码审查。
- 不在本次修复中做无关的 ViewSet、Service 或模型重构。

## 3. 全局设计原则

1. **权限先于副作用**：令牌生成、命令生成、全局对账派发之前必须完成权限检查。
2. **操作有身份**：长任务和跨存储操作使用稳定的 `execution_id` 或 `operation_id`，所有回调必须说明自己属于哪次执行。
3. **终态单向**：同一执行的 `SUCCESS/ERROR/TIME_OUT/FORCE_STOP` 为终态；迟到结果只记录，不回写当前终态。
4. **事件先持久化后消费**：业务事件进入异步发送前持久化，锁只保护消费者，不决定事件是否存在。
5. **数据库约束兜底幂等**：应用层预查询用于友好提示，唯一约束负责最终并发正确性。
6. **外部存储显式补偿**：MinIO 和 NATS 不被假设参与数据库事务；使用状态、outbox 和清理任务实现最终一致。
7. **底层真实分页**：分页、聚合和批量查询下推到 GraphClient/ORM，不在 Service 中先构建全量快照。

## 4. 阶段一：P0 权限边界

### 4.1 IPAM 手动对账

- `ipam_reconcile` 增加 `HasPermission("asset_info-Edit")`。
- 接口不再同步执行全量对账，统一派发阶段三定义的 IPAM 后台作业；在阶段三完成前，阶段一先保留当前调用方式，但确保无权限请求无法进入 Service。
- 超级用户沿用 `HasPermission` 的现有放行规则。
- 权限失败必须发生在 `run_reconciliation`、Celery 派发和任何图写入之前。

### 4.2 K8s 引导式接入

- `install_token`、`install_command` 使用 `HasPermission("auto_collection-Execute")`。
- `verify` 使用 `HasPermission("auto_collection-View")`。
- Open API 的 `render` 继续只依赖一次性安装 token，不改为登录态接口。
- 任何拒绝路径不得创建 token、读取云区域环境变量或查询 VictoriaMetrics。

### 4.3 验收标准

- 普通已认证但无对应权限的用户收到 403。
- 有权限用户保持现有成功响应契约。
- P0 接口具有 ViewSet 级正向、负向和“无副作用”测试。

## 5. 阶段二：异步状态、投递与回调幂等

### 5.1 订阅通知可靠投递

- 新增关系库通知投递记录，保存稳定 `dedupe_key`、规则、触发类型、事件内容、渠道、状态、尝试次数和最后错误。
- 规则检查在派发 Celery 前通过 ORM 幂等写入 `PENDING` 记录。
- Worker 通过条件更新抢占 `PENDING/RETRY` 记录；全局锁冲突只能触发重试，不能删除或完成记录。
- 渠道发送成功后标记 `SENT`；失败进入带上限和退避的 `RETRY`，超过上限标记 `FAILED` 并保留诊断信息。
- 幂等键至少由规则、触发类型、实例、业务变更标识和渠道组成，避免 Celery 重投导致重复通知。

### 5.2 采集执行状态机

- 以 `CollectModels.task_id` 作为执行代次 `execution_id`。
- Worker 结束写回必须使用 `id + task_id + RUNNING` 条件更新；条件不匹配代表结果已过期，只记录日志。
- 超时巡检使用任务配置的 deadline；缺少专属配置时使用 Worker `TASK_JOB_TIMEOUT` 的服务端对应默认值，不再固定 5 分钟。
- 超时更新使用 `id + task_id + RUNNING` 条件，将状态写为 `TIME_OUT`，而不是普通 `ERROR`。
- 配置文件回调必须携带或能可靠解析当前 `execution_id`；与当前任务不匹配的回调标记为 stale，不改变任务状态。

### 5.3 配置文件回调数据库幂等

- `ConfigFileVersion` 增加 `(collect_task, instance_id, version)` 唯一约束。
- 并发创建以数据库约束为最终裁决；发生 `IntegrityError` 时读取已存在版本并走幂等返回。
- 相同业务键但内容哈希不同视为协议冲突，不覆盖已落库正文，记录明确错误。
- 数据迁移在增加约束前按稳定规则保留一条记录，并清理重复记录引用的冗余对象。

### 5.4 验收标准

- 两个并发通知任务的事件最终都进入终态，不因锁冲突消失。
- 旧执行的迟到结果不能覆盖新执行或超时终态。
- 两个独立事务并发提交同一配置回调，最终只有一个版本。
- 状态转换和并发测试使用真实 ORM 行为，不只断言 mock 调用。

## 6. 阶段三：跨存储一致性与后台作业

### 6.1 FalkorDB 主数据操作日志

- 新增关系库 CMDB 操作日志表，使用 `operation_id` 和调用方幂等键记录请求摘要、目标、状态和错误。
- HTTP 写入口先创建或读取操作日志，再执行 FalkorDB 变更；相同幂等键不得重复执行主数据写入。
- 图写成功后，关系库审计和后置动作通过 outbox 处理；同步镜像失败不把已经成功的图写包装成 HTTP 失败。
- `PENDING` 或状态不确定的操作由补偿任务重新核对图中目标，再推进审计和后置动作，不能盲目重复创建。

### 6.2 MinIO 配置正文生命周期

- 配置版本引入 `PENDING/READY/DELETE_PENDING/ERROR` 内容生命周期。
- 上传先写唯一临时对象，保存元数据后在事务提交回调中发布为正式对象；失败保留可清理状态。
- 删除先把关系库记录标记为 `DELETE_PENDING`，事务提交后删除 MinIO 对象，再删除或归档元数据。
- 周期清理任务处理超时临时对象、孤儿对象和失败删除，并保证重复执行安全。

### 6.3 IPAM 作业化

- 手动和周期入口统一调用一个 Celery 作业。
- 使用带 owner token 的分布式锁保证单执行者；只有锁持有者可以释放锁。
- 新增或复用任务执行记录，保存 `RUNNING/SUCCESS/ERROR`、统计、开始结束时间和错误摘要。
- 手动重复触发返回当前运行任务，不创建并发全量对账。
- 对账内部按来源和实例分批处理，避免单次加载全部资产；每批具有稳定游标或明确的重跑语义。

### 6.4 验收标准

- 审计或 NATS 短暂不可用时，已成功主数据写入不会导致客户端重复写；恢复后审计最终补齐。
- DB 回滚不会遗留正式 MinIO 对象；DB 删除失败不会先丢失正文。
- 手动和周期 IPAM 同时触发时只有一个作业执行，并能查询最终统计。

## 7. 阶段四：规模治理

### 7.1 K8s 资源视图

- 集群概览使用批量关联查询和聚合计数，不加载 Pod 实体。
- Namespace、Workload、Node 和 Pod 的搜索、排序、分页下推到图查询。
- Pod→Node 关联按当前页 Pod ID 批量查询，禁止逐 Pod RPC/图查询。
- Service 保留权限裁剪，但只处理当前页候选和聚合结果。

### 7.2 实例唯一规则

- 从模型唯一规则提取实际参与字段和值，生成定向候选查询。
- 只有可能冲突的实例进入现有字段校验逻辑，不再按 `model_id` 全量加载。
- 并发最终正确性由图存储原子约束或等价的原子写路径保证；若底层暂不支持，则写入口必须串行化相同唯一键。

### 7.3 批量审计

- `ChangeRecord` 继续使用 ORM `bulk_create`。
- 平台操作日志镜像改为 outbox 批量消费或批量 RPC，每批调用次数有固定上限。
- 下游不可用时只积压 outbox，不阻塞采集主任务完成。

### 7.4 验收标准

- 默认 K8s 概览不查询 Pod 实体，查询次数不随 Workload 数线性增长。
- 百万级模型的单实例写入不产生全模型查询。
- 一万条采集变更不会产生一万次同步 NATS RPC。

## 8. TDD 与验证策略

每个独立缺陷执行以下固定流程：

1. 在现有测试模块或新的聚焦测试模块中写一个行为级失败测试。
2. 单独运行该测试，确认因当前缺陷而失败，而不是测试配置、导入或 mock 错误。
3. 只实现让该测试通过的最小生产代码。
4. 运行聚焦测试、相关模块测试和 CMDB 总体门禁。
5. 涉及迁移时执行 `makemigrations --check`，并验证 SQLite 以外数据库不依赖原生 SQL。
6. 核心状态机、权限和幂等代码覆盖率达到 90%，本次新增/修改后端代码整体达到 80%。

阶段检查点：

- 阶段一完成后验证所有敏感入口的拒绝路径无副作用。
- 阶段二完成后验证并发、重投、超时和迟到回调。
- 阶段三完成后验证跨存储故障注入与恢复。
- 阶段四完成后验证查询预算和大数据量性能契约。

## 9. 发布与回滚

- 每个阶段独立提交，不把四个阶段合并为一个不可回滚变更。
- 新表和新字段先以兼容方式发布，再切换读写路径；删除旧路径安排在后续清理变更。
- 状态机、outbox 和作业化切换必须提供可观测指标：待处理数、失败数、重试次数、最长积压时间和迟到回调数。
- 回滚应用版本时保留新增表和字段，避免旧版本运行期间破坏已经持久化的操作与事件。

## 10. 完成定义

- 四阶段列出的验收标准全部由自动化测试覆盖。
- 所有新迁移可在项目支持的数据库方言上通过 Django ORM 执行。
- CMDB 对应聚焦测试和服务端质量门禁通过。
- 项目记忆中的 #0034、#0020/#0007、#0035—#0044 在各自验证完成后逐项关闭。
- 企业版 `custom_reporting` 缺失实现仍作为独立阻塞项保留，不因本次修复被误标为完成。
