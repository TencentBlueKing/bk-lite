# 运营分析“多值”图表设计

## 背景与目标

运营分析当前支持折线图、柱状图、饼图、单值等图表类型，但缺少在一个组件内紧凑展示多组名称和值的方式。本次新增 `multiValue` 图表类型，用于展示由数据源返回的多条 `label`/`value` 数据。

首版支持普通仪表盘、大屏和拓扑图表节点，三种承载面复用同一渲染组件。

## 范围

### 包含

- 在数据源支持的图表类型中增加“多值”。
- 在普通仪表盘、大屏和拓扑图表节点中选择并渲染 `multiValue`。
- 每行左侧展示 `label`，右侧展示 `value`。
- 内容超过组件可视高度时，在组件内部纵向滚动。
- 提供数据校验、空状态、错误状态以及长文本处理。

### 不包含

- label/value 字段映射配置。
- 单位、精度、阈值颜色、值映射等高级数值格式配置。
- 排序、搜索、分页或单项交互。
- 新增独立的拓扑节点类型。

## 数据契约

数据源返回 JSON 数组，每项必须是包含 `label` 和 `value` 的对象：

```json
[
  { "label": "CPU", "value": 80 },
  { "label": "内存", "value": 65 }
]
```

规则如下：

- 顶层必须是数组。
- 每一项必须是普通对象，并且同时拥有 `label` 和 `value` 字段。
- `label` 接受字符串或数字，渲染时转换为文本。
- `value` 接受字符串或数字，按接口原值展示，不做隐式单位或精度处理。
- `label` 或 `value` 为 `null`、`undefined` 或空字符串时显示 `--`。
- 对象、数组、函数等复杂值视为无效数据，整个数据集进入现有图表错误状态，避免显示 `[object Object]`。
- 空数组是合法数据，显示统一空状态。
- 允许 label 重复；渲染键使用索引和规范化后的 label 组合。

## 架构与组件

### 图表注册

图表标识使用 `multiValue`。在公共图表类型列表中加入 `dataSource.multiValue`，并在共享 widget registry 中注册独立的 `ComMultiValue` 组件。

采用独立组件，不复用表格或扩展单值组件，原因是：

- 多值列表没有表格的列配置、分页和筛选语义。
- 单值和多值具有不同的数据契约与布局职责。
- 独立组件能让三种承载面共用明确、可单测的接口，也便于未来在不影响单值图表的情况下扩展。

### 渲染组件

`ComMultiValue` 只负责展示已校验的数据：

- 根容器占满可用宽高并控制纵向滚动。
- 每条记录为一行，label 左对齐，value 右对齐并使用更突出的字重或颜色。
- 行之间使用符合现有主题的轻量分隔或间距。
- label 与 value 均限制为单行；溢出时显示省略号，并通过现有 Tooltip/省略提示组件显示完整文本。
- 颜色、背景和文字层级使用运营分析现有主题变量，不写死亮色或暗色主题值。
- 大屏缩放沿用现有 widget renderer 传入的缩放与容器机制。

### 数据校验

在现有图表数据校验入口增加 `multiValue` 分支，使普通仪表盘、大屏和拓扑共用同一校验规则。校验结果继续走现有的加载、空数据和错误提示流程，不在组件内部复制请求或错误处理逻辑。

## 各承载面接入

### 普通仪表盘

- 数据源声明 `multiValue` 后，图表配置下拉框显示“多值”。
- 沿用现有数据源参数、统一筛选和自动刷新链路。
- 不显示单值、仪表盘或表格专属设置区。
- 保存时沿用现有 `valueConfig.chartType`，值为 `multiValue`。

### 大屏

- `ScreenWidgetChartType` 增加 `multiValue`。
- 大屏组件定义中增加名称、描述和合理的默认宽高；建议初始尺寸为宽 360、高 260，以容纳多行内容。
- 创建、编辑、保存和恢复继续复用现有大屏 widget 配置流程。

### 拓扑

- `multiValue` 作为现有“图表”节点的一种 chart type，不新增节点类型。
- 图表节点继续通过共享 widget renderer 获取 `ComMultiValue`。
- 配置、序列化、恢复和运行时刷新沿用 `valueConfig.chartType` 及现有图表节点数据链路。

## 国际化

至少补充中英文翻译：

- `dataSource.multiValue`：多值 / Multi-value
- `opsAnalysis.screen.widgets.multiValue`：多值 / Multi-value
- `opsAnalysis.screen.widgetDescriptions.multiValue`：说明该组件逐行展示 label/value 数据。

若项目的其他语言文件要求键集合一致，应同步补齐对应语言，避免回退为原始键名。

## 错误与边界行为

- 请求中：沿用现有 loading 状态。
- 请求失败：沿用现有数据请求错误状态。
- 空数组：显示统一空状态。
- 非数组、缺少必需字段或包含复杂字段值：显示图表数据格式错误。
- 单行文本过长：省略显示，悬浮可查看全文，不撑破布局。
- 数据量过多：组件内部滚动，外部仪表盘、大屏或拓扑容器不随内容增长。
- 未识别的图表类型：保持现有 unknown component 行为，不影响其他组件。

## 测试与验收

### 自动化测试

- 数据校验：覆盖合法数组、空数组、非数组、缺字段、空值、字符串/数字值和复杂值。
- 组件渲染：覆盖多行 label/value、`--` 占位、重复 label、长文本提示和滚动容器。
- 注册：确认 `getWidgetComponent('multiValue')` 返回新组件。
- 普通仪表盘：确认支持该类型的数据源可选择并保存 `multiValue`。
- 大屏：确认类型定义、组件目录项、默认尺寸以及创建/恢复流程支持 `multiValue`。
- 拓扑：确认图表节点可保存、恢复并渲染 `multiValue`。

### 验收标准

1. 数据源设置可勾选“多值”。
2. 普通仪表盘、大屏和拓扑图表节点均可选择该类型。
3. 合法数据按一行一个 label/value 展示，label 左对齐、value 右对齐。
4. 超出高度时仅组件内容区域纵向滚动。
5. 空数据、错误数据、加载失败均有明确且一致的状态。
6. 其他图表类型及已保存视图不受影响。
7. 运营分析相关测试、TypeScript 类型检查和 lint 通过。

## 实施约束

- 遵循现有 widget registry、配置保存和数据请求模式，保持改动聚焦。
- 实施阶段按项目要求采用 TDD：先添加失败测试，再完成最小实现，最后执行验证。
- 不修改后端数据协议；数据源提供方负责返回约定的 `label`/`value` 数组。
