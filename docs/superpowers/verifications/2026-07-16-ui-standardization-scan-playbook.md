# UI 样式与交互规范化扫描手册

> 适用范围：运营分析、告警、CMDB，以及后续需要做同类“页面样式/交互/中英文展示/自适应”统一的前端模块。
> 目标：下次扫描时能按同一套标准判断问题、区分例外、落地修复，并能用自动化命令验证到可提交状态。

## 1. 扫描前置条件

1. 新开 worktree 和独立分支，避免污染主工作区。
2. 先跑模块级基线门禁，记录阻塞项，不要把历史问题混进本次规范化结论。
3. 先确认三类基线：
   - i18n key 是否对齐。
   - TypeScript 是否能通过目标模块门禁。
   - 页面本身是否存在未登录、后端代理、依赖缺失等环境限制。
4. 扫描时以“页面场景是否合理”为准，不为了统一而统一。画布、大屏、拓扑、宽表、复杂表单可以保留局部特化，但必须说明原因和边界。

## 2. 必扫页面类型

每个模块至少覆盖以下页面类型：

1. 管理列表页：表格、筛选区、批量操作、分页、空态。
2. 设置页：配置卡片、开关、说明文案、保存按钮、局部滚动。
3. 表单弹窗/抽屉：字段 label、说明、校验错误、英文长文案、底部按钮。
4. 详情页：标题、状态、属性项、Tab、长字段展示。
5. 画布/大屏/拓扑：工具栏、侧边栏、浮层、缩放后内容可达性。
6. 宽表/日志页：横向滚动边界、分页可见性、列内容省略。
7. 搜索/筛选页：输入框、下拉、多选 tag、日期范围、按钮换行。

## 3. 统一判断标准

统一不是把所有页面套成同一个外壳，而是让用户在相同任务中遇到相同的表达和交互。扫描时先判断页面任务，再判断是否应该统一：

1. 同类管理页、设置页、列表页、表单页应尽量统一标题、说明、筛选、表格、分页、空态、操作按钮和反馈方式。
2. 画布、大屏、拓扑、向导、编辑器、宽表、强风险操作页允许保留特化布局，但必须保留可达性、可读性和中英文自适应。
3. 新增说明、按钮、筛选项、状态栏后，必须重新检查剩余高度、表格滚动、分页可见性和抽屉/弹窗底部按钮可达性。
4. 判断“是否需要统一”时不要只看视觉相似度，要看用户任务是否相同、信息密度是否相近、页面是否有宽度或高度硬约束。

### 3.1 页面头部和说明

优先使用统一页头组件，例如 `PageIntro` 或模块已有 introduction 的统一封装。

需要检查：

1. 标题在英文下不撑破容器。
2. 管理页原则上应有标题和一句说明，除非页面已经在导航、Tab、面包屑或主体内容中明确表达用途，且新增说明会挤压核心操作。
3. 说明文案不占用过高首屏空间；新增头部说明后必须同步调整表格或内容区高度，不能让分页、底部操作、主内容被截断。
4. 长说明优先使用 Tooltip、Popover 或折叠说明，不在页面上铺满长段文字。
5. 统计信息、状态标签、辅助描述必须能省略或换行。
6. 不把已有业务风险提示隐藏到 Tooltip 里；规则影响范围、破坏性动作、不可逆说明必须显式展示。
7. 已经在标题下显示了说明时，不要再在标题旁放重复的 info 图标；Tooltip 只用于补充非核心细节。
8. 同一模块内相似管理页的说明卡片应统一位置、间距、字号和背景，但允许按页面高度压力减少文案长度。

不建议统一的情况：

1. 大屏、画布、拓扑的顶部工具栏需要贴合编辑场景。
2. 页面首屏高度非常紧张，统一头部会挤压核心操作。
3. 原页面的说明本身是业务必读风险提示。
4. 页面是单一表单向导或沉浸式编辑器，独立标题区会打断任务流。

例外说明必须写清楚：

1. 为什么不能加统一头部或说明卡片。
2. 页面靠什么替代说明，例如 Tab 标题、空态、步骤标题或表单分组标题。
3. 是否已经验证中英文、低高度和内容增多后的可达性。

### 3.1.1 管理页说明卡片判定

管理页、配置页、日志页、库表页默认应检查是否需要说明卡片。判断顺序：

1. 页面是否承担“让用户理解当前资源、规则或记录用途”的任务。是，则优先加说明。
2. 页面首屏是否已经有清晰标题和描述。没有，则补齐。
3. 新增说明后是否会压缩表格高度。会，则必须让主体区域形成 `min-h-0` 加局部滚动，或重新计算表格 `scroll.y`。
4. 是否属于高频操作或窄宽页面。是，则说明应短、轻、可省略，不应做大块卡片。
5. 是否有强风险提示。是，则风险提示保留显式展示，普通解释才收进 Tooltip 或短说明。

经验规则：

1. 类似 SOI 特征库、采集工具、扫描特征、全局配置、操作日志这类管理页，若缺少页面用途说明，应补齐说明卡片。
2. 已有说明但重复标题、重复 info 图标、长段说明挤压主体时，应优先精简和统一，而不是叠加更多提示。
3. 新增说明卡片后必须用至少一个低高度桌面视口验证表格主体、分页和操作按钮仍可见。

### 3.2 操作区和工具栏

优先使用统一操作区，例如 `ResponsiveActionBar`。

需要检查：

1. 按钮英文变长后是否换行或进入更多菜单。
2. 主按钮、危险按钮、加载态、禁用态、Tooltip 是否保留。
3. 输入框、Select、DatePicker 不写死不可收缩宽度。
4. 小屏下筛选项能纵向排列，按钮不会被裁切。
5. 工具栏换行后，表格区域仍有局部滚动，分页仍可见。

允许特化的情况：

1. 画布工具栏需要固定贴边或悬浮。
2. 大屏编辑器需要保持紧凑图标按钮。
3. 宽表页需要保留横向滚动，而不是强行压缩列。

### 3.3 文本展示

长文本默认按场景选择：

1. 表格单元格：单行省略 + Tooltip。
2. 卡片标题：单行或两行省略 + Tooltip。
3. 说明文案：Tooltip、Popover、Alert 或折叠块。
4. 代码、ID、路径：等宽字体，可复制，允许横向滚动或省略。
5. 标签/状态：短文本固定样式，长文本必须收缩。

禁止模式：

1. 容器 `white-space: nowrap` 但没有 `min-width: 0` 或 overflow 兜底。
2. 英文 locale 文案直接硬塞到按钮、标签、卡片标题里。
3. 多语言说明只在中文下看起来正常，英文下遮挡按钮或表单项。

### 3.3.1 说明文案类型规范

说明文案先按信息重要性分层，不按实现方便程度选择组件。

| 类型 | 展示方式 | 规则 |
| --- | --- | --- |
| 页面用途说明 | 页头说明卡片或 Introduction | 一句话为主，允许英文两行，不能重复标题 |
| 字段解释 | Label Tooltip 或 Form `extra` | 普通解释优先 Tooltip；影响填写判断的内容可用 `extra` |
| 规则范围/风险 | Alert、显式提示、表单 `extra` | 不隐藏到 Tooltip，必须直接可见 |
| 操作后果 | Popconfirm、Modal、Alert | 危险/不可逆动作必须二次确认 |
| 表格长内容 | 单行省略 + Tooltip | Tooltip 内容应完整，不改变行高 |
| 空态引导 | Empty description 或轻提示 | 不写成长段说明，不替代表单校验 |
| 校验错误 | Form error/help | 直接显示，不放 Tooltip |

同一页面内避免混用多种解释方式。一个表单内如果多数普通字段解释用 Tooltip，后续普通字段也优先 Tooltip；只有风险、范围、校验、不可逆行为才显式展示。

发现某一类说明文案不一致时，不能只修当前截图或当前文件，必须横向复扫三个模块的同类实现：

1. `Form.Item` 的 `extra`、`help`、`tooltip`。
2. 表单字段附近的灰色小字、`Text type="secondary"`、`text-xs/text-sm text-gray-*`。
3. Drawer/Modal header 的 `extra` 说明。
4. 表格列说明、图标说明、按钮旁说明。

复扫结论必须区分两类：普通解释已统一为 Tooltip/Label Tooltip；规则范围、风险、校验、不可逆影响等显式保留，并写明保留原因。

Tooltip 使用边界：

1. Tooltip 适合补充“看不看都能继续操作”的解释，不适合承载必须阅读的风险。
2. Tooltip 图标不要和已展示的说明文案重复。
3. Tooltip 内容需要支持中英文长句换行，不能固定过窄导致英文一列竖排。
4. 表格 Tooltip 不应触发行高变化，不应挡住当前行主要操作。
5. 字段解释、规则解释等长说明 Tooltip 应显式放宽，推荐最大宽度 480-520px，并保留小屏视口兜底；短按钮 Tooltip 不需要放宽。

### 3.3.2 中英文长度风险

中文正常不代表英文安全。扫描时必须主动判断这些位置：

1. 按钮：英文动词短语比中文长，按钮组必须可换行、可收缩或进入更多菜单。
2. 表单 label：普通表单默认上下两行，不能只按中文固定 label 宽度设计；横向 label/control 只作为紧凑筛选、表格行内编辑、条件表达式等局部例外。
3. Select 多选 tag：英文 tag、模型名、策略名可能撑破筛选栏，必须可换行、省略或限制容器宽度。
4. 表格列名和单元格：英文表头和长 ID、路径需要 `ellipsis`、`scroll.x` 或合理列宽。
5. 页头说明：英文说明允许两行，但不能把主体内容挤到不可操作。
6. 状态标签：短状态可以固定，长状态或动态名称必须 `max-width` 加省略。

判断方法：

1. 不只读 locale 文件，要看文案放在哪种容器里。
2. 对 `title`、`button`、`placeholder`、`label`、`extra`、`help`、`columns.title` 做长文本风险扫描。
3. 找到固定宽度、`nowrap`、`overflow-hidden` 时，必须确认是否有 `min-width: 0`、Tooltip、横向滚动或换行兜底。
4. 英文页面至少抽查低宽视口和低高视口，确认没有遮挡、裁切和整页横向滚动。

### 3.4 表单弹窗和抽屉

需要检查：

1. 普通创建、编辑、确认类 Modal 默认居中，不贴顶；长表单、步骤任务、右侧上下文编辑优先用 Drawer。
2. Modal 宽度不要只写死数字，低宽视口应配合 `maxWidth`、`calc(100vw - 48px)` 或等价兜底。
3. Modal/Drawer body 必须有最大高度和内部滚动。
4. 普通表单默认使用上下两行布局，label 在上、控件在下；不要通过放大固定 label 宽度挤压输入控件。
5. 嵌套表单、配置块、动态字段也按上下两行检查；不能只改外层 `Form`，还要查手写 `label + field`、`rowStyle`、固定 label 宽度和负 margin。
6. `extra`、`help`、校验错误不能覆盖下一个字段。
7. 底部按钮始终可达，不被内容撑出视口。
8. 动态增删行、嵌套表单、规则组必须能在窄屏换行。
9. 横向布局只允许在紧凑筛选器、表格行内编辑、条件表达式 builder、`Space.Compact` 等局部密集场景中保留；保留时必须说明理由并验证英文、125% 缩放和低宽视口。
10. Modal/Drawer 内部出现长表单、卡片选择器、动态表格、字段列表时，必须让 header/footer 固定可见，body 使用 `maxHeight: calc(100vh - Xpx)` 加 `overflowY: auto` 或等价样式；不能靠页面滚动把底部按钮“滚出来”。
11. 普通表单改成上下两行后，必须复扫旧横向表单留下的对齐补丁，例如 `label=" "`、`colon={false}`、`labelCol/wrapperCol`、`margin-left`、`padding-left`、`ml-*`、`pl-*`、固定 label 宽、负 margin。只有树、步骤、规则组、时间线等真实层级缩进可以保留。

三模块弹窗/抽屉细则：

1. 运营分析：数据源、命名空间、组件配置、拓扑节点/连线配置等配置面板按普通表单处理；画布本体和浮层工具栏可以保留特化布局。
2. 告警：动作规则、分派规则、屏蔽策略、全局配置等设置表单默认上下两行；匹配条件、升级链、字段映射这类规则编辑器可保留局部紧凑布局，但必须可换行。
3. CMDB：订阅规则、采集任务、模型/属性/关联关系/唯一规则等表单默认上下两行；资产筛选器和表格行内编辑可保留横向紧凑布局。
4. 卡片式选择器不要固定三列硬塞；优先响应式 grid/flex wrap，标题和描述允许换行。
5. footer 按钮组右对齐且允许换行，不靠单个按钮 `marginLeft` 拼间距；主按钮、次按钮、取消按钮顺序在同类页面保持一致。

说明文案规则：

1. 普通解释性文案优先统一到 Tooltip/LabelWithTooltip。
2. 强风险提示、作用范围、会影响线上行为的说明必须显式展示。
3. 校验错误必须直接显示，不放 Tooltip。
4. 字段 `placeholder` 不承担说明职责；重要说明必须在 label、extra、Tooltip 或 Alert 中出现。
5. 英文 label 特别长时，优先采用上下两行；字段概念需要解释时再用 Tooltip，不要通过放大固定 label 宽度挤压输入控件。

### 3.5 宽高和滚动

需要检查：

1. 父容器是否形成完整高度链路：`h-full`、`min-h-0`、`overflow-hidden`、局部 `overflow-auto`。
2. 表格区域是否有可解析的 `scroll.y`。如果底层组件不能解析 CSS `max()`，不要传 `max(...)`。
3. 宽表使用表格自身横向滚动，不让整页横向滚动。
4. 固定 `height`、`max-height`、`min-width`、`w-[...]` 必须有场景理由。
5. 缩放、浏览器小窗、内容变多后，分页、底部按钮、危险提示仍可见。

新增内容后的高度规则：

1. 加页头说明、筛选行、Tab、统计条后，必须重新计算表格容器剩余高度。
2. 主体容器要形成闭合高度链路：外层 `h-full`，中间 `min-h-0`，内容区 `overflow-hidden`，真正滚动区 `overflow-auto`。
3. 表格页优先让表格区域滚动，不让整页滚动到分页消失。
4. 抽屉和弹窗优先固定 header/footer，body 内部滚动。
5. 低高度视口下，如果核心内容高度不足，应优先缩短说明文案或折叠次要信息，而不是隐藏分页或按钮。

宽度规则：

1. 宽表保留 `scroll.x`，不要强行压缩所有列。
2. 普通管理页避免整页 `min-width` 撑出横向滚动；确需宽表时只让表格内部横向滚动。
3. 筛选区和操作区必须支持换行；换行后仍要给表格留下可用高度。
4. 多选 Select、长按钮、长标签、英文表头必须有最大宽度或省略策略。
5. 不要把 `overflow-hidden` 放在会承载动态内容的父容器上，除非内部已经有明确滚动区。

缩放和内容增多检查：

1. 浏览器缩放到 125% 时，按钮、表单错误、分页不能互相遮挡。
2. 表格数据行变多时，分页仍在可视区域或可通过局部滚动到达。
3. 表单字段增多、动态行展开、校验错误出现后，底部按钮仍可达。
4. 空态、加载态、错误态都要验证，不能只看有数据状态。

### 3.6 组件交互一致性

同类组件交互应统一，但保留业务语义：

1. 表格：筛选在上、操作在右或左按模块习惯统一；批量操作、导出、刷新、创建的按钮顺序保持一致。
2. 搜索：简单关键字用 Search；复杂条件用筛选区；高级筛选不应和普通输入混成难以换行的一行。
3. 删除、禁用、下发、执行：危险操作必须有 Popconfirm 或确认弹窗，文案明确对象和后果。
4. 创建、编辑：同类资源使用同类 Modal/Drawer 密度、底部按钮顺序和保存反馈。
5. Tab：同级内容用 Tab；流程步骤用 Steps；不要为了统一把明显流程页改成普通 Tab。
6. 状态反馈：保存成功、失败、加载、空态统一使用项目已有 Message、Spin、Empty、Alert 模式。
7. 图标按钮必须有 Tooltip 或可识别文本；常用图标使用现有图标库，不手写临时图形。

特殊页面例外：

1. 画布、拓扑、大屏可以使用悬浮工具栏、紧凑图标按钮和局部固定尺寸，但必须支持缩放后可达。
2. 审计日志、操作日志、指标列表等宽表优先保障信息完整性，允许横向滚动。
3. 高风险配置页可以保留显式说明和 Alert，即使和普通说明卡片不完全一致。
4. 低频配置向导可以牺牲列表页密度，换取分步骤理解。

## 4. 自动化扫描命令

在 `web/` 下执行：

```bash
pnpm test:ui-i18n-keys
pnpm test:ui-shell-components
pnpm test:ui-text-risk
NEXTAPI_INSTALL_APP=ops-analysis,alarm,cmdb pnpm type-check
git diff --check
```

如果 `pnpm lint` 失败，必须区分：

1. 本次改动引入的问题：必须修。
2. 历史基线问题：记录文件、错误、与本次改动无关的证据，不要顺手大范围修。

## 5. 文本风险扫描重点

扫描脚本需要覆盖这些模式：

1. 固定宽度：`width`、`min-width`、`w-[...]`。
2. 固定高度：`height`、`max-height`、`h-[...]`。
3. 横向溢出：`overflow-x`、`overflowX`。
4. 禁止换行：`white-space: nowrap`、`whiteSpace: 'nowrap'`。
5. 表格滚动：`scroll={{ x: ... }}`、`scroll={{ y: ... }}`。
6. 未走 i18n 的中英文硬编码。
7. Tooltip/Popover/Alert/Form extra/help 的说明文案使用是否一致。

扫描结果不能只看数量，要按页面场景判断：

1. 宽表的 `scroll.x` 通常是合理风险。
2. 画布、大屏、拓扑存在固定尺寸时，需要确认缩放/滚动/可达性。
3. 表单 label 固定宽度在中文下正常，不代表英文下正常；普通表单应先改为上下两行，再讨论是否有横向例外。
4. `overflow-hidden` 在卡片标题上可能合理，在页面主容器上可能导致内容不可达。

## 6. 人工验证矩阵

每个核心页面至少验证这些视口：

1. `1440 x 900`：常规桌面。
2. `1280 x 720`：低高度桌面。
3. `1024 x 760`：窄宽桌面。
4. `390 x 844`：移动宽度参考；控制台不一定要求完整移动适配，但不能出现不可操作的遮挡。

每个视口检查：

1. 页面是否出现整页横向滚动。
2. 操作按钮是否可见可点击。
3. 表格分页是否可见。
4. 抽屉/弹窗底部按钮是否可达。
5. 切到英文后标题、按钮、说明、表单 label 是否溢出。
6. 内容增加后是否被固定高度隐藏。

每个改动批次至少覆盖这些状态：

1. 中文 + 常规数据。
2. 英文 + 常规数据。
3. 英文 + 长名称、长描述、长 tag、长按钮文案。
4. 空数据、加载中、接口错误。
5. 表单校验错误展开后。
6. 多选项超过一行后。
7. 表格数据超过一屏后。

检查结论要写“能否操作”，不只写“看起来正常”。例如：分页可见、底部保存按钮可点击、横向滚动只在表格内、Tooltip 不遮挡主操作。

## 7. 修复优先级

P0：

1. 英文切换后按钮、表单、表格遮挡核心操作。
2. 弹窗/抽屉内容无法滚动，底部按钮不可达。
3. 页面高度链路错误导致主体内容被隐藏。
4. i18n key 不对齐或英文页面显示 key。

P1：

1. 管理页头部和说明风格明显不一致。
2. 操作区在窄屏不能换行。
3. 长文本没有省略或 Tooltip。
4. 宽表横向滚动泄漏到整页。
5. 新增说明卡片后没有同步调整主体区域高度，低高度视口下可用空间明显不足。
6. 同一类普通说明在 Tooltip、Alert、正文段落之间随意混用。

P2：

1. 局部间距、按钮顺序、卡片圆角、图标使用不一致。
2. 说明文案类型不够统一，但不影响操作。
3. 特殊页面保留了局部旧样式，但有明确场景理由。

不处理项必须满足：

1. 有明确场景理由。
2. 不影响核心操作。
3. 已验证中文、英文、低高度、窄宽和内容增多状态。
4. 在总结或 PR 描述中标明为什么保留。

## 8. 提交前可靠性标准

改动可提交前必须满足：

1. 目标模块 type-check 通过。
2. i18n key 扫描通过。
3. UI shell 组件脚本通过。
4. 文本风险扫描通过；剩余风险有场景解释。
5. `git diff --check` 通过。
6. 至少完成目标模块关键路由浏览器 smoke。
7. 如果全仓 lint 被历史问题阻塞，必须记录阻塞文件和证据。
8. 最终 diff 中不能混入无关格式化、依赖锁污染、缓存、构建产物。

## 9. 已知项目级注意事项

1. Web 运行需要 Node 24；使用系统 Node 18 会导致开发服务或依赖行为异常。
2. 本地浏览器 smoke 可能因未登录跳转到 `/auth/signin`，此时横向溢出可能来自登录页固定壳，不应误判为目标业务页问题。
3. `CustomTable` 对 `scroll.y` 的解析存在限制，传入 CSS `max(...)` 可能不兼容；优先用组件可解析的 `calc(...)`。
4. CMDB locale 曾出现大量 zh/en key 不对齐；多语言扫描必须作为前置基线。
5. 全仓 `pnpm lint` 可能被未触碰历史文件阻塞；目标改动仍需用 type-check、扫描脚本、diff-check 和浏览器 smoke 兜底。

## 10. 推荐执行顺序

1. 建 worktree，记录分支和目标模块。
2. 跑基线：i18n、type-check、文本风险扫描。
3. 按模块列页面清单，不跳过设置页、表单抽屉、详情页、宽表页。
4. 先修共享组件和扫描脚本，再迁移页面。
5. 每迁移一批页面，跑目标模块 type-check。
6. 对高风险页面做窄屏和英文验证。
7. 复跑全部目标门禁。
8. 记录剩余历史阻塞和合理例外。
9. 最终再做 diff 审查，确认只包含本次规范化相关改动。
