# 产品白皮书设计规范（通用版）

> Migrated from `spec/whitepaper/产品白皮书设计规范.md` as legacy capability evidence.

> 本规范基于对阿里云"云监控 2.0"产品介绍/白皮书页面的结构拆解反推而成，提炼出一套可复用、标准化的产品白皮书撰写方法与排版约定，适用于 IT / 云 / SaaS 类产品。

---

## 一、白皮书的本质与目标

产品白皮书不是功能手册，也不是 API 文档。它是一份**面向"评估与决策"的说服性文档**，回答读者心中三个递进的问题：

1. **这是什么？** —— 用一句话讲清产品定位，再用一段话补全边界。
2. **为什么值得选？** —— 提炼差异化优势，给出价值主张。
3. **能解决我的什么问题？** —— 用场景和能力把价值"落到地面"。

阿里云监控 2.0 的页面是这一逻辑的典型范例：开篇定义 → 产品优势 → 基本概念 → 功能特性 → 应用场景 → 应用清单，**从抽象价值逐层下沉到具体能力**。规范的核心就是复刻这条"价值递进线"。

### 双读者原则

一份好的白皮书要同时服务两类读者，且让两类读者都能各取所需：

| 读者 | 关注点 | 对应章节 |
|---|---|---|
| 决策者 / 业务方 | 价值、ROI、适用场景、风险 | 定义段、产品优势、应用场景、客户价值 |
| 技术评估者 / 架构师 | 架构、能力深度、集成、规格 | 功能特性、技术架构、功能参数、开放能力 |

写作时应让"价值类"章节在前、"技术类"章节在后，决策者读完前半部分即可形成判断，技术者可继续深入。

## 二、标准章节结构

下面是反推得到的通用骨架。**带 ★ 为必备章节**，其余按产品复杂度取舍。

### 1. ★ 开篇定义段（What is X）

紧跟标题的无小标题段落，是全篇最重要的部分。遵循"**一句话定位 + 一段话展开 + 一段价值升华**"的三段式：

- **第一句**：用一句话说清"产品是什么 + 核心能力 + 给谁用"。
  - 范例（阿里云）："云监控 2.0 是融合 SLS、CMS、ARMS 后全新升级的一站式可观测平台，可将指标、链路、日志、事件汇集于统一视图。"
- **第二段**：展开核心机制与端到端价值（基于什么技术、实现什么效果、适用什么场景）。
- **第三段**：价值升华，点明产品所处的时代趋势与战略意义（如"AI-native 时代""更智能、更高效、更低成本"）。

> 要点：定义段不堆功能，只讲"定位 + 价值 + 趋势"。功能留到后面。

### 2. 功能体验入口（可选但推荐）

提供 Demo / Playground / 截图，让读者"先看到再相信"。这是 PLG 类产品的关键设计——降低体验门槛。即使白皮书是静态文档，也应放置产品截图或演示链接。

### 3. ★ 产品优势（核心差异化）

**这是白皮书的"灵魂章节"。** 阿里云用了 4 个并列的优势支柱，每个都是"**小标题 + 一段解释**"：

- 一站式融合观测
- 数据统一建模
- AI 驱动智能分析
- 开放兼容，拥抱主流生态

规范约定：

- 优势数量控制在 **3–5 个**，每个是一个"高度概括的名词短语"（不超过 8 字），便于记忆。
- 每个优势配 **1 段 80–150 字**的解释，先讲机制再讲收益。
- 优势之间应正交（不重叠），共同支撑产品定位。
- 优势是**抽象提炼**，不是功能罗列——功能罗列放到"功能特性"章节。

### 4. ★ 基本概念 / 术语表

在展开功能前，先统一语言。用**两列表格（术语 | 说明）**定义产品独有的核心概念。

- 只收录"理解本产品必需"的术语，不做百科。
- 复杂概念（如阿里云的"工作空间 WorkSpace"）可在说明栏内用子条目展开命名规则、约束、特殊情况。
- 术语表既可放在前部（帮助阅读），也可放在末尾（备查）；产品概念多时前置，少时后置。

### 5. ★ 功能特性（能力全景）

与"产品优势"互补：优势是"为什么好"，功能特性是"具体有什么"。阿里云采用**两列表格（功能特性 | 描述）**，并在描述栏内用**分层列表**展开：

```
| 全栈数据采集与监控 | * 基础设施监控
                       + 云资源监控：……
                       + 网络性能监测：……
                     * 应用性能监控（APM）
                       + 分布式链路追踪：……
```

规范约定：

- 按"能力域"分组（采集、分析、可视化、告警、开放、安全、成本……），每组一行。
- 描述栏用"**加粗子能力 + 冒号 + 简述**"的句式，保证扫读性。
- 覆盖面要全，体现产品的"完整性"，这是技术评估者最看重的部分。

### 6. ★ 应用场景（价值落地）

**最具说服力的章节。** 阿里云用**三列表格（场景 | 场景描述 | 方案优势）**，列出 7 个场景。这是"场景驱动"写法的范本：

- **场景名**：用"角色 + 诉求"命名（如"全栈统一监控""智能异常检测与故障预测"）。
- **场景描述**：先讲客户的真实困境（痛点），再讲产品如何介入。
- **方案优势**：用 2–4 个加粗要点说明该场景下产品的独特价值，**尽量量化**（如"运维工作量降低 90%""资源利用率提升 30%"）。

规范约定：场景数量 **4–7 个**，覆盖产品的主要买点；每个场景必须"痛点—方案—收益"闭环。

### 7. 能力/应用清单（规格佐证）

用表格列出可量化的覆盖面（支持的应用、对象、数据源、指标数等）。这类"清单"是产品实力的硬证据，例如阿里云的"可观测应用列表"，或监控类产品的"支持的采集对象与指标数量"。**用数字说话**比形容词更有力。

### 8. 技术架构（技术读者向）

非阿里云该页重点，但完整白皮书应包含：架构分层表、数据流主线、技术栈、核心组件与端口职责。用"分层表格 + 一段数据流文字描述"组合表达。

### 9. 客户价值与 ROI

从**效率、风险、成本**三个维度归纳收益，尽量给出量化或可衡量的表述。

### 10. 安全与合规

数据加密、权限管控（RBAC）、审计、合规认证、高可用。面向有监管诉求的客户，是决策的硬门槛。

### 11. 开放与集成能力

API/SDK、开源生态兼容（如 Prometheus、OpenTelemetry）、第三方对接。体现"无厂商锁定"，降低客户的迁移与选型顾虑。

### 12. 差异化对比与适用边界

客观陈述与竞品/传统方案的差异，并**诚实说明不适用的场景**。适当的边界声明反而提升可信度。

### 13. 导航 / 索引 / 落款

上下篇导航、模块白皮书索引、版本与免责声明（"以最新发布版本为准"）。

## 三、写作原则（从范例中提炼）

1. **价值先行，功能后置。** 先回答"为什么"，再回答"是什么""怎么做"。开篇绝不堆功能清单。

2. **抽象与具体分层。** "产品优势"做抽象提炼（3–5 个支柱），"功能特性"做具体罗列，两者呼应而不重复。

3. **场景驱动说服。** 用"痛点 → 方案 → 收益"的场景三段式代替自卖自夸，让读者对号入座。

4. **量化一切可量化的。** "提升 30%""降低 90%""50+ 数据源""719 个指标"远比"显著提升""丰富"更有力。

5. **统一语言。** 先定义术语再展开，避免读者因概念歧义产生理解成本。

6. **趋势绑定。** 把产品价值绑定到行业趋势（如 AI-native、云原生、可观测），提升战略高度。

7. **诚实的边界。** 说明适用与不适用场景，建立专业可信形象。

8. **可扫读。** 长段落拆成"加粗关键词 + 简述"，重信息用表格，让读者 30 秒能抓住重点。

## 三·五、语气与表达规范（客户视角）★

这是白皮书与 PRD/技术文档最本质的区别，也是最容易写偏的地方。核心一句话：**主语是"您"，谓语是"得到/省去/不必担心"，技术只在能打消顾虑时才出现。**

### 五个核心手法（自阿里云文案反推）

**1. 人称用"您"，从客户处境出发，而非从产品出发。**
关键句的主语应是客户，而不是产品或组件。

- ✅ "您无需再部署和维护多套工具，即可在一个平台上完成全栈观测。"
- ❌ "本平台通过 FalkorDB 承载资产关系与拓扑查询。"

**2. 用"收益动词"，不用"实现动词"。**
多用：降低、提升、保障、消除、节省、快速发现并解决、提前识别、不必担心。
少用（或下沉到附录）：采集、调度、聚合、映射、序列化、扫描。

**3. 开场先共情痛点，再给方案。**
每个场景遵循"客户的难处 → 产品介入 → 量化收益"：
> "企业资产分散、关系不清，故障时难以判断影响范围。XXX 通过一张实时拓扑图……让您几秒钟看清影响面。"

**4. 把技术藏起来，只露收益。** 技术名词（Prometheus、向量检索等）只在"消除客户顾虑"时出现——讲"平滑迁移、无厂商锁定、私有化不外发"，而不是炫耀实现细节。

**5. 营销节奏：排比 + 四字短语 + 数字。**
"更智能、更高效、更低成本""效率提升 30%""接入只需几分钟"。句子有韵律，章节收尾可带价值升华。

### 同一句话的两种写法（务必对照）

| 技术/内部视角（错误示范） | 客户/价值视角（正确示范） |
|---|---|
| 通过 NATS-Executor 在目标节点执行命令，内置高危路径拦截 | 故障来临时，您可以一键批量修复，平台自动拦截高危操作，让您既快又稳，不必担心误删误改 |
| CMDB 使用图数据库承载资产关系，支持多跳查询 | 谁依赖谁，一张拓扑图看清；故障时您能立刻判断影响哪些业务，不再靠人脑回忆 |
| 告警采用三层模型，经聚合引擎收敛 | 告警风暴时，平台帮您把上千条重复告警收敛成几条关键事件，值班同学只看真正要紧的 |

### 技术内容怎么处理

并非完全不写技术，而是**分层安置**：价值章节（定义段、产品优势、应用场景）一律客户视角、零实现细节；技术细节集中到靠后的"技术架构""功能参数"附录，供技术评估者备查。这样既不丢深度，又不让技术淹没价值。

## 四、排版与格式约定

### 标题层级

- `#` 一级：白皮书标题（如"XXX 产品白皮书"或"什么是 XXX"）。
- `##` 二级：主章节（产品优势、功能特性、应用场景……）。
- `###` 三级：章节内并列项（如各个优势支柱、各个场景）。
- 层级不超过四级，保持结构清晰。

### 表格使用

表格是产品白皮书的"主力武器"，下列内容**优先用表格**：

| 内容类型 | 推荐表头 |
|---|---|
| 术语表 | 术语 \| 说明 |
| 功能特性 | 功能特性 \| 描述 |
| 应用场景 | 场景 \| 场景描述 \| 方案优势 |
| 能力清单 | 类型 \| 名称 \| 描述 |
| 客户画像 | 客户类型 \| 典型诉求 \| 匹配点 |
| 架构分层 | 层级 \| 组件 \| 作用 |
| 规格参数 | 项目 \| 要求/参数 |

### 列表与强调

- 表格单元格内用 `*` 一级、`+` 二级缩进，配合**加粗子标题**做分层。
- 正文段落中，关键名词用 `**加粗**`，但全篇加粗比例不宜过高（喧宾夺主）。
- 避免大段无结构纯文字；超过 4 行的说明应拆为要点或表格。

### 视觉与信任元素

- 产品截图 / 架构图 / 拓扑图（提升可信度与可读性）。
- Demo / Playground 体验入口（PLG 关键）。
- 版本号、更新时间、免责声明（专业性与可追溯性）。

## 五、通用白皮书骨架模板（可直接套用）

```markdown
# 什么是 {产品名} / {产品名} 产品白皮书

{一句话定位 —— 产品是什么 + 核心能力 + 面向谁}

{一段话展开 —— 核心机制、端到端价值、适用场景}

{一段价值升华 —— 绑定行业趋势与战略意义}

## 功能体验
{Demo / Playground 链接或产品截图}

## 产品优势
### {优势支柱一}
{机制 + 收益，80–150 字}
### {优势支柱二}
…（共 3–5 个）

## 基本概念
| 术语 | 说明 |
|---|---|
| … | … |

## 功能特性
| 功能特性 | 描述 |
|---|---|
| {能力域一} | * {子能力}：… |

## 应用场景
| 场景 | 场景描述 | 方案优势 |
|---|---|---|
| {场景一} | {痛点 + 介入} | * {量化收益} |

## 能力/应用清单
| 类型 | 名称 | 描述 |
|---|---|---|

## 技术架构
{架构分层表 + 数据流描述 + 技术栈}

## 客户价值与 ROI
{效率 / 风险 / 成本 三维度}

## 安全与合规
{加密 / 权限 / 审计 / 合规}

{API/SDK / 开源兼容 / 第三方对接}

{客观对比 + 不适用场景}

## 索引 / 导航 / 落款
{相关文档索引 + 版本与免责声明}
```

## 六、快速自检清单

发布前对照检查：

- [ ] 开篇是否在 3 句话内讲清"是什么 + 为什么 + 给谁"？
- [ ] 产品优势是否为 3–5 个正交的抽象支柱？
- [ ] 功能特性是否覆盖全面且可扫读？
- [ ] 是否有 4–7 个"痛点—方案—收益"闭环的场景？
- [ ] 收益是否尽量量化？
- [ ] 核心术语是否统一定义？
- [ ] 是否同时照顾了决策者与技术评估者？
- [ ] 是否诚实说明了适用边界？
- [ ] 表格、加粗、截图是否让文档可在 30 秒内抓住重点？
- [ ] 是否标注版本、更新时间与免责声明？
