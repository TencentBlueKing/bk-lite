## 序言

随着云原生架构的大规模普及，分布式系统的日志规模正以指数级速度增长。一次业务请求可能跨越数十个微服务、数百个容器实例，每秒产生的日志条目动辄数十万行。面对如此体量的数据，传统"逐机登录、手工检索"的运维方式已彻底失效——工程师花费在查找日志本身的时间，往往远超实际排障时间。日志的价值不在于数量的积累，而在于从海量噪声中精准提取信号的能力。

BK-Lite 日志系统的研发初衷，源于对这一行业困境的深刻洞察。团队系统性地梳理了运维团队在日志管理场景中的真实痛点，提炼出四个核心命题：日志的采集接入如何做到标准化与零脚本化？日志的检索定位如何做到秒级响应？日志的异常告警如何做到主动发现而非被动响应？日志的处置追溯如何做到从告警到原始记录的无缝闭环？围绕这四个命题，BK-Lite 日志系统构建了一套完整的统一日志可观测技术体系。

本白皮书系统阐述 BK-Lite 统一日志可观测技术体系的设计原则、参考架构与核心技术支柱，旨在为技术决策者和系统架构师提供全面、翔实的技术参考。BK-Lite 日志系统的核心价值主张可以凝练为一句话：**"接入即检索、检索即定位、日志即事件、事件即追溯"**。这十六个字，既是产品的设计纲领，也是衡量日志系统真实价值的基准尺度。

## 1. 日志管理的时代背景与挑战

### 1.1 日志规模爆炸：从可管理到失控

过去十年，企业 IT 架构经历了从单体应用到微服务、从物理机到容器、从数据中心到混合云的深刻变革。这一演变在带来弹性与敏捷性的同时，也引发了日志数据的规模性爆炸。在传统单体架构下，一个应用系统的日志来源通常不超过十台服务器，日志格式相对统一，运维团队尚可通过人工方式进行管理。

然而在云原生环境下，一个中等规模的微服务系统可能包含数十个服务、数百个 Pod 实例，分布于多个 Kubernetes 集群和多个云区域。这些实例每秒产生的日志条目动辄数十万行，涵盖应用日志、容器运行时日志、Kubernetes 系统日志、数据库慢查询日志、网络流量日志、操作系统审计日志等多种类型，格式各异，字段定义五花八门。日志规模的爆炸性增长，使得传统的日志管理方式在技术层面已经走到了尽头。

> **小结**：云原生架构的普及使日志来源由单一走向多元，日志规模从可管理走向失控，传统人工方式已无法应对这一量级的变化。

### 1.2 检索低效：关键时刻的系统性失灵

日志数据的价值高度时效性依赖——在故障发生后的黄金时间窗口内快速定位根因，与在故障平息后进行事后分析，对业务的影响截然不同。然而，绝大多数企业的日志检索效率与这一时效性要求之间存在显著落差。

典型的日志排查流程如下：首先确认与故障相关的主机列表（耗时 5～10 分钟），然后逐台 SSH 登录（耗时 10～20 分钟），再手工执行 grep 命令并比对多机结果（耗时 10～20 分钟）。上述流程合计耗时 30～60 分钟，且还未真正开始解决问题。这种低效不仅仅是操作层面的繁琐，更是一种系统性失灵——在检索效率如此低下的情况下，工程师被迫依赖经验猜测来缩小排查范围，这进一步降低了定位的准确性，形成恶性循环。

即便企业引入了日志集中存储方案，如果检索能力未能同步提升，工程师依然会陷入"猜关键词"和"等结果"的泥潭。缺乏字段智能提示、缺乏字段分布统计、缺乏趋势直方图辅助，都会导致检索效率的实质性下降。

> **小结**：日志检索的低效是一种系统性问题，其根源不仅在于操作流程的繁琐，更在于缺乏将检索结果转化为定位洞见的智能化工具支撑。

### 1.3 告警割裂：故障发现永远滞后于用户感知

在绝大多数运维体系中，监控告警与日志系统是两套相互独立的子系统。监控系统负责指标采集与阈值告警，日志系统负责数据汇聚与被动检索，两者之间缺乏有机联动。这种架构性割裂带来的后果是：故障的发现时间系统性地晚于用户感知时间。

典型场景如下：监控系统触发"API P99 延迟升高"告警，但这条告警本身并不告诉运维工程师"为什么"。工程师需要手动切换到日志系统，重新认证登录，重新设定时间范围，重新构造查询条件，才能开始尝试理解问题的根因。在这个切换过程中，排查的上下文一次次被打断，宝贵的黄金时间在无意义的重复劳动中流逝。

更严重的是：相当一部分故障模式根本不会引发指标层面的异常，只会在日志里留下特定的错误模式或异常字符串。内存泄漏的前兆、数据库连接池的耗尽预警、特定接口的间歇性 500 错误——这些信号都埋藏在日志流中，等待着被发现。在缺乏基于日志内容的主动告警机制的环境中，这些信号永远无法在造成业务影响之前被捕获。

> **小结**：监控告警与日志系统的架构性割裂，导致故障发现时间系统性滞后，且使运维团队对仅能在日志中体现的故障模式完全失明。

### 1.4 多组织治理：规模化使用的安全困境

随着企业数字化转型的深入，日志平台的使用边界从单一运维团队扩展至研发、安全、数据等多个职能团队。不同团队对日志的访问诉求各不相同：生产环境日志应仅对运维团队开放，开发环境日志对研发团队可见，安全审计日志则应严格限制在安全合规团队的访问范围内。

多租户场景下的日志治理是一个被大量企业低估的技术难题。如果没有内生的组织隔离机制，日志平台的规模化使用将不可避免地引发数据越权访问风险。为此专门维护一套独立的权限体系，又会带来配置冗余和管理复杂度的双重问题。

> **小结**：多组织、多角色的日志访问诉求，要求日志平台内生支持细粒度的组织隔离与权限控制，这是企业级日志治理的基础性要求，而非可选项。

## 2. BK-Lite 统一日志可观测技术体系

### 2.1 设计原则

BK-Lite 统一日志可观测技术体系遵循以下四项核心设计原则：

**原则一：接入标准化，消除脚本依赖。** 所有采集类型均提供预置字段定义与配置引导，用户无需编写采集脚本。系统内置 18 种涵盖七大分类的采集类型，覆盖企业 IT 环境中的绝大多数日志来源场景。

**原则二：检索智能化，降低定位门槛。** 日志检索不应依赖工程师事先知道"问题在哪"。通过字段 Top 值分布、趋势直方图、点击追加查询条件等交互机制，将检索过程转化为数据驱动的逐层收敛，而非关键词猜测。

**原则三：告警主动化，先于业务感知。** 日志系统的核心价值不止于存储与检索，更在于对日志模式的持续监控与主动告警。每一条告警均应携带触发它的原始日志片段，使接收人无需二次查询即可理解事件上下文。

**原则四：追溯可靠化，闭环不留空白。** 从告警到原始日志、从事件到处置操作，全链路可追溯。AlertSnapshot 机制确保即使存储层因数据保留期到达而清理了原始日志，触发告警时的关键日志记录仍永久可查。

## 3. 统一接入引擎：UnifiedIngest

### 3.1 技术定位

统一接入引擎（UnifiedIngest）是 BK-Lite 日志可观测体系的最前端，其核心使命是将分散于企业 IT 环境中的多源异构日志，以标准化、可管理、零脚本的方式汇聚至日志存储层。

传统日志接入方案的主要痛点在于：每种日志类型需要单独开发采集脚本，字段定义因人而异，采集配置分散在各台主机上，变更时需要逐机登录操作。UnifiedIngest 的设计彻底改变了这一局面：所有采集类型均在平台界面完成配置，采集参数通过 NATS JetStream 消息总线自动下发至目标节点，无需 SSH 登录。

### 3.3 核心采集器技术特性

**Vector**：采用 Rust 语言编写的高性能日志采集与处理工具，资源占用极低，是 BK-Lite 通用场景的首选采集器。Vector 承担文件日志、Docker 容器日志、Kubernetes Pod 日志、Syslog 四类场景的采集任务。在 Kubernetes 接入场景中，Vector 以 DaemonSet 方式部署，确保集群每个节点运行一个采集副本，自动为每条日志附加 k8s_namespace、k8s_pod_name、k8s_node_name 等容器编排元数据字段，研发人员无需手动维护这些字段即可按 Pod 或命名空间进行精确检索过滤。

**Filebeat/Auditbeat/Winlogbeat/Packetbeat**：Elastic Stack 采集器家族，专为结构化日志场景设计。BK-Lite 利用其成熟的模块化能力，覆盖数据库、中间件、安全审计和网络流量等专业场景，直接复用其预置的字段映射规则，大幅降低接入复杂度。

**Snmptrapd**：专为网络设备 SNMP Trap 事件采集设计，填补了通用采集器在网络设备日志场景下的空白。

### 3.4 配置下发与生命周期管理

采集配置的生命周期管理是 UnifiedIngest 的核心设计挑战之一。BK-Lite 通过 NATS JetStream 消息总线实现采集配置从平台后端到目标节点的可靠下发。当工程师在界面上更新采集参数时，变更指令即时写入 NATS JetStream 并向目标节点推送，无需 SSH 登录修改任何配置文件。

JetStream 的持久化消息机制解决了节点临时离线场景下的配置一致性问题：当目标节点因网络抖动或计划维护而临时离线时，配置变更指令在 JetStream 中持久化保存；节点恢复连接后，指令自动重新投递并执行，确保配置最终一致性。

## 4. 高性能存储与检索：LogStore

### 4.1 技术选型：VictoriaLogs

BK-Lite 选择 VictoriaLogs 作为日志存储与全文检索引擎，这一选型决策基于以下核心技术考量：

相比业界广泛使用的 Elasticsearch，VictoriaLogs 在内存和磁盘资源占用方面具有显著优势。Elasticsearch 集群的运维复杂度较高，需要配置分片、副本、索引模板等大量参数，且对内存资源消耗较大。VictoriaLogs 单节点即可满足中小团队的日常日志存储与检索需求，运维复杂度大幅低于 Elasticsearch 集群，使日志系统本身不再成为需要专人维护的基础设施负担。

VictoriaLogs 运行于端口 9428，与 BK-Lite 指标存储引擎 VictoriaMetrics（端口 8428）共同构成平台可观测数据的存储双柱，两者在技术栈和运维方式上高度统一，降低了整体平台的维护复杂度。

### 4.2 LogsQL 查询语言

VictoriaLogs 原生支持 LogsQL 查询语言，提供以下四类核心查询能力：

- **关键字全文检索**：在日志的全文内容中搜索任意关键字或短语，无需指定字段名。
- **字段精确匹配**：基于特定字段名和字段值进行精确过滤，例如 `response_code:500` 或 `host:web-server-01`。
- **正则匹配**：支持正则表达式模式匹配，适用于字段值格式不完全固定的场景。
- **范围查询**：支持数值型字段的范围条件，例如 `query_time:>1000` 用于检索慢查询。

上述四类查询能力可通过逻辑运算符自由组合，构造复杂的多维度检索条件。BK-Lite 的搜索界面提供字段名和字段值的候选辅助输入，降低了 LogsQL 表达式的构造门槛。

### 4.3 统一搜索上下文

BK-Lite 定义了"统一搜索上下文"这一核心概念，将日志分组、查询表达式、时间范围三个维度组合为一个完整的检索状态基础。统一搜索上下文同时驱动趋势直方图、日志明细列表、字段 Top 值分布、实时 tail 流和分析仪表盘，确保同一上下文下的所有视图保持数据一致性，消除多系统切换带来的上下文丢失问题。

**日志分组**机制是统一搜索上下文的重要组成部分。日志分组通过字段规则（支持 AND/OR 逻辑组合）定义日志的逻辑集合，是搜索范围和策略范围的基本单元。系统提供内置的默认日志分组，代表全量日志口径，适用于跨分组的全局检索场景。

### 4.4 AlertSnapshot：持久化追溯保障

告警触发时，系统自动创建 AlertSnapshot——触发该次告警的原始日志快照。AlertSnapshot 的元数据记录存储于 PostgreSQL，快照实际数据通过 S3JSONField 自动上传至 MinIO/S3 对象存储，并以 gzip 压缩格式持久化，独立于 VictoriaLogs 的数据保留期。AlertSnapshot 的设计解决了一个关键的可靠性问题：当 VictoriaLogs 中的原始日志因数据保留期到达而被清理后，触发告警时的关键日志记录依然可以通过 AlertSnapshot 进行完整追溯。

## 5. 实时追踪与分析：LiveInsight

### 5.1 智能化检索定位

LiveInsight 的核心设计理念是：日志定位不应依赖工程师事先知道"问题在哪"。传统检索模式要求工程师先有假设再验证，而 LiveInsight 通过字段 Top 值分布机制，将检索过程转化为数据驱动的逐层收敛。

**字段 Top 值分布**在当前统一搜索上下文下实时计算每个字段的取值频次分布，并以可视化方式呈现各个取值的命中数量与占比。工程师无需猜测哪个 response_code 异常、哪台主机出了问题——数据分布本身就给出了答案。点击任意分布项即可将该字段条件追加到查询表达式，实现一步步逼近根因的下钻检索。

**趋势直方图**按时间维度展示日志命中量的分布，帮助工程师快速判断异常发生的时间边界——某个时间段的日志量突然攀升，通常是异常事件的直接信号。

### 5.2 实时 tail 模式

实时 tail 模式以流式方式持续展示最新流入的日志，与当前的查询表达式和日志分组保持一致。这一模式的核心价值在于：工程师在执行变更操作（如应用发布、配置变更）的同时，可以在浏览器中实时观察目标服务的日志输出，无需另开终端进行 SSH 登录。这不仅消除了多终端切换的操作负担，更重要的是将观察窗口与操作窗口统一在同一上下文中，变更与响应的关联关系一目了然。

### 5.3 搜索条件保存与团队共享

BK-Lite 提供搜索条件的保存、加载、更新与删除能力。保存的搜索条件记录创建人，并按当前组织范围隔离管理。这一机制将工程师个体的排障经验转化为团队可复用的知识资产——高频使用的排障条件被显式保存后，新成员加载即可复现完整的排查上下文，排障经验不再只存在于少数资深工程师的记忆中。

### 5.4 内置日志分析仪表盘

BK-Lite 为数据库、中间件、网络、容器等典型采集场景提供开箱即用的内置分析仪表盘，按场景预置图表与数据表格，覆盖以下分析维度：

- **数据库场景**：MySQL 慢查询趋势、查询时间分布、慢查询 Top 用户与 Top Host
- **中间件场景**：Nginx/Apache HTTP 状态码分布、请求量趋势、Top URI
- **网络场景**：流量统计、连接数趋势
- **容器场景**：按命名空间/Pod 的日志量分布

内置仪表盘按日志分组、时间范围与刷新频率进行参数化配置，无需额外的看板搭建工作即可立即使用。

## 6. 日志事件治理：LogEvent

### 6.1 双策略体系

LogEvent 围绕主动发现与闭环处置构建，提供关键字告警和聚合告警两类策略类型，覆盖绝大多数生产环境的告警场景：

**关键字告警**：基于日志内容匹配触发，策略命中即触发通知，适用于"特定错误出现即告警"的场景。关键字告警支持展示字段配置——告警通知中直接携带触发本次告警的原始日志片段，接收人打开通知即知事件详情，无需二次查询日志。

**聚合告警**：基于日志数量在时间窗口内的聚合统计触发，当统计值超过阈值时告警，适用于"5 分钟内超过 100 条 HTTP 500 即告警"此类场景。聚合告警支持聚合维度（按 host、service 等字段分组统计）与多条件规则组合，能够精确区分"单台主机异常"与"全局性故障"。

两类策略均支持组织范围、日志分组、查询条件、执行频率、检测周期、告警等级（info/warning/error/critical）、通知渠道等完整参数配置，以及启停控制能力。

### 6.2 补偿回扫：防漏告警机制

告警策略的可靠性不应依赖于调度系统的完美运行。BK-Lite 实现了最多 10 个周期的补偿回扫机制，覆盖约 24 小时内的调度空窗：当平台因维护或故障而停止运行时，告警策略的扫描任务出现空缺；平台恢复后，Celery 调度器自动对历史调度空窗执行补偿扫描，确保空窗期内应触发的告警不会因平台不可用而漏发。

扫描执行时，以 `safe_time = 当前时间 - 60 秒` 为安全截止点，规避因日志写入延迟导致的边界条件漏检问题。

### 6.3 事件生命周期与追溯

每一条告警事件从产生到关闭遵循完整的生命周期管理流程：

- **活跃状态（new）**：策略扫描命中，自动创建，尚未处置。
- **关闭状态（closed）**：人工确认处置完成，系统记录关闭操作人与关闭时间。

事件时间线提供告警从产生到关闭的全程状态变化历史视图，处置操作者和操作时间均有完整记录，满足合规审计对操作留痕的要求。

对于每一条活跃告警，工程师可在同一界面直接查看：触发本次告警的最近原始日志内容（通过 AlertSnapshot 保障持久性）、告警在时间轴上的完整历史、单次事件对应的具体日志记录。从"看告警"到"查日志"到"关闭告警"的全过程在同一界面完成，无需跨系统跳转。

### 6.4 通知渠道集成

LogEvent 的告警通知渠道直接读取系统管理模块的统一渠道配置，支持企业微信、钉钉、飞书机器人及自定义 Webhook 等主流渠道。通知渠道配置一次，全平台所有策略均可直接引用，无需在日志系统中重复维护渠道信息，保证通知配置的全平台一致性。

## 8. 部署与演进路径

### 8.1 部署方式

BK-Lite 采用 Docker Compose 一键部署模式，日志系统所需的全部基础组件（VictoriaLogs、NATS JetStream、PostgreSQL、MinIO、Celery 等）随平台整体部署，无需单独安装：

```bash
curl -sSL https://bklite.ai/install.run | bash -s -
```

对于网络隔离或安全要求较高的生产环境，BK-Lite 支持离线安装模式：在联网环境中制作离线安装包，通过介质拷贝至目标服务器完成安装，目标服务器无需访问外部网络。

### 8.2 推荐演进路径

BK-Lite 日志系统的建设推荐按以下三个阶段有序推进，每个阶段均可独立交付价值：

**第一阶段（第 1～2 周）：基础接入阶段**
完成核心业务服务的日志接入，建立初步的日志分组体系。目标：团队日常排障从登录主机转为在平台界面操作，完成工作习惯的根本性转变。具体动作：从 18 种采集类型中选择目标场景完成接入，按业务线/环境（生产/测试/开发）规划分组，验证数据流入正常。

**第二阶段（第 2～4 周）：精细化运营阶段**
精细化日志分组规则，沉淀高频搜索条件，验证内置分析仪表盘的数据覆盖。目标：将个人排障经验转化为团队可复用的搜索条件资产，通过内置仪表盘建立对关键场景的持续可见性。

**第三阶段（第 1 个月后）：主动运营阶段**
为关键业务配置告警策略，建立从日志模式到告警通知的主动发现机制，培养通过事件中心完成处置闭环的运维工作流。目标：故障发现时间从"用户反馈"提前至"系统主动告警"，日志排查操作流程（登录主机、手工 grep）被平台界面操作取代，减少重复性手工步骤。

## 9. 客户价值

### 9.1 故障定位效率：消除手工 SSH 日志排查流程

**改变前**：典型日志排查流程（确认主机列表 + 逐机 SSH 登录 + 手工 grep 比对）需要多个串行步骤，且每次故障均需重复这一流程，问题尚未解决之前已消耗大量时间。

**改变后**：通过日志分组锁定检索范围、通过字段 Top 值分布发现异常模式、追加条件精确定位根因，整个过程在平台界面完成，无需登录任何主机。对于频繁发生的故障场景，加载已保存的搜索条件可快速复现排查上下文。

上述流程变化将"逐机 SSH 登录 + 手工 grep"这一串行操作替换为平台界面的集中检索，有效降低手工操作步骤数量，使工程师能更快进入实质性排障阶段。

### 9.2 故障发现时机：从被动响应转向主动告警

在缺乏日志告警的传统环境中，日志里的异常模式往往只有在造成明显业务影响后才被发现，此时问题已处于恶化阶段。BK-Lite LogEvent 的策略告警使故障发现时机从"用户反馈"前移至"日志异常模式出现时"。

对于可预判的故障模式，如：数据库慢查询聚积（通过 MySQL 慢查询日志的聚合告警"5 分钟内超过 10 条慢查询即通知 DBA"）、内存泄漏前兆日志、连接池耗尽预警——LogEvent 使平台能够在这些模式形成时即触发通知，而非等待业务指标恶化后再被动发现。实际提前量取决于故障的具体演化周期与策略配置的检测窗口。

### 9.3 团队知识资产化：经验不随人员流动流失

搜索条件保存与共享机制将团队的排障经验以可复用的形式沉淀在平台中。新成员加载团队积累的搜索条件即可快速复现排查上下文，无需依赖口口相传。结合 OpsPilot 知识库，历史故障分析过程还可转化为 AI 辅助排障的知识基础，进一步放大经验的复用价值。

### 9.4 合规审计内生支持，无需额外采购

BK-Lite 日志系统内置以下合规能力：

- **文件完整性监控**：Auditbeat 采集，59 个预置字段，含 hash.md5、hash.sha256，满足关键主机文件变更的哈希审计要求。
- **Windows 安全事件**：Winlogbeat 采集，38 个预置字段，含 event_id、winlog.event_data，覆盖账号管理、登录失败、权限变更等 Windows 安全审计核心事件。
- **操作审计记录**：告警关闭操作记录操作人与操作时间，满足运维操作留痕要求。
- **组织隔离访问控制**：非授权范围内的数据不可见、不可改，满足信息安全合规对访问控制的要求。

上述能力内生于平台，无需额外采购独立的安全审计工具，显著降低企业合规建设成本。

## 10. 技术演进展望

### 10.1 智能异常检测

当前 LogEvent 的告警策略基于预先配置的规则进行匹配。未来的演进方向是引入基于机器学习的日志异常检测能力，通过对历史日志模式的建模，自动识别偏离正常基线的异常模式，减少对人工规则配置的依赖，实现更广泛的故障前兆发现。

### 10.2 链路追踪与日志的深度关联

随着企业可观测性体系的成熟，日志与分布式链路追踪（Trace）的关联联动将成为重要演进方向。通过在日志中自动注入 TraceID/SpanID，实现从一条日志记录直接跳转至对应的完整分布式链路，构建日志-指标-链路三位一体的统一可观测体系。

### 10.3 自适应日志分组与智能路由

随着企业日志规模的持续增长，静态配置的日志分组规则可能难以跟上业务变化的节奏。未来将探索基于日志内容特征的自适应分组机制，结合 AI 能力实现日志的智能分类与路由，进一步降低日志治理的人工维护成本。

### 10.4 流式处理与实时聚合

当前的告警策略基于定时批量扫描模式。未来将引入流式处理能力，支持毫秒级的实时告警触发，进一步压缩从日志写入到告警通知的端到端延迟，满足对时效性要求极高的业务场景。

|---|---|
| 内置采集类型数量 | 18 种 |
| 覆盖采集分类 | 通用、Kubernetes、数据库、中间件、网络、容器、安全，共七大分类 |
| 采集器类型 | Vector、Filebeat、Auditbeat、Winlogbeat、Packetbeat、Snmptrapd |
| 配置下发机制 | NATS JetStream 消息总线，支持节点离线后恢复重投递 |
| Kubernetes 接入方式 | DaemonSet 方式部署 Vector，自动附加容器元数据字段 |
| 预置字段数范围 | 8 字段（snmp_trap）～59 字段（file_integrity） |

### A.2 存储检索规格

| 规格项 | 参数值 |
|---|---|
| 存储引擎 | VictoriaLogs |
| 存储引擎端口 | 9428 |
| 查询语言 | LogsQL（支持关键字全文检索、字段精确匹配、正则匹配、范围查询） |
| 管理数据存储 | PostgreSQL |
| AlertSnapshot | 告警触发时自动创建快照，元数据存于 PostgreSQL，快照数据持久化至 MinIO/S3，独立于 VictoriaLogs 数据保留期 |

### A.3 告警策略规格

| 规格项 | 参数值 |
|---|---|
| 策略类型 | 关键字告警、聚合告警 |
| 告警等级 | info / warning / error / critical |
| 补偿回扫周期数 | 最多 10 个周期（约覆盖 24 小时调度空窗） |
| 扫描安全截止点 | 当前时间 - 60 秒（规避日志写入延迟导致的漏检） |
| 告警状态 | new（活跃）/ closed（已关闭，记录关闭操作人） |
| 通知渠道 | 企业微信、钉钉、飞书机器人、自定义 Webhook（来源于系统管理统一配置） |

### A.4 搜索上下文规格

| 规格项 | 参数值 |
|---|---|
| 时间范围预置选项 | 最近 15 分钟 / 1 小时 / 6 小时 / 24 小时，支持自定义时间段 |
| 日志分组选择 | 单选或多选，多选时各分组条件 OR 合并；选择默认分组时优先以全量口径执行 |
| 搜索条件保存 | 记录创建人，按组织范围隔离，支持 CRUD |
| 字段 Top 值 | 基于当前搜索上下文实时计算，支持点击直接回填查询表达式 |
