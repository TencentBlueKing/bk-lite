## 序言

当代 IT 运维体系正面临前所未有的数据爆炸挑战。监控指标以秒级频率涌入，告警事件在多个系统中并行触发，资产配置数据在 CMDB 中持续演变，日志流水在分布式节点上不断积累。这一切数据理论上足以支撑完整的运营决策，然而现实的困境在于：大量有价值的数据被孤立在各自的系统边界之内，无法被高效地感知、汇聚与利用。

BK-Lite 运营分析（Operation Analysis）模块的研发初衷，正是为了从根本上解决上述矛盾。通过构建以 NATS 消息总线为通信骨干、以命名空间为治理边界的轻量化技术体系，BK-Lite 运营分析将多源异构的运维数据统一纳入可视化分析工作台，使仪表盘、拓扑图、架构图三种载体在同一目录化空间协同运作，为运维团队提供从数值观测到关系定位再到结构表达的完整分析能力。

本白皮书系统阐述 BK-Lite 运营分析模块的技术体系、架构设计及工程实现，面向平台架构师、运维工程师及技术决策者提供权威参考。BK-Lite 运营分析的核心技术理念可凝练为十六字：**数据即汇聚、汇聚即可视、可视即洞察、洞察即决策**。

## 1. 运营可视化的时代背景与挑战

### 1.1 数据孤岛：数据就在那里，全貌却无从把握

现代 IT 运维场景中，监控平台采集主机与应用的性能指标，告警系统负责事件聚合与通知，CMDB 维护资产与依赖关系的权威记录，日志平台汇聚来自各节点的海量流水。这四类数据系统各自独立演进，形成了根深蒂固的数据孤岛格局。

一次典型的故障排查场景可以清晰地揭示这一困境：当监控系统报告某台主机 CPU 利用率达到 90% 时，运维工程师必须先切换至 CMDB 确认该主机承载的应用清单，再跳转告警系统核查下游服务是否级联触发告警，最终进入日志平台定位根因日志。这一信息收集过程通常耗费 10 至 20 分钟，而此时业务已处于降级或中断状态。传统应对方案（购买专业 ETL 平台或通用 BI 工具）引入了高昂的数据仓库维护成本，且与运维场景之间存在根本性的语义鸿沟——通用 BI 工具不理解"告警"与"CMDB 依赖"的运维语义，最终往往制造出另一个新的孤岛。

> **小结：** 数据孤岛的本质不是数据缺失，而是数据汇聚能力与运维语义理解能力的双重缺位。

### 1.2 可视化语义缺失：图表工具与运维场景的结构性错位

市场上的图表工具种类繁多，但其设计出发点是通用的商业数据分析场景，与运维可视化存在结构性错位。一张反映服务健康状态的动态拓扑图，在通用 BI 工具中需要大量定制开发工作；一个依据告警数量实时变换颜色的节点，在现有开源图表库中缺乏开箱即用的运维语义支撑。

更深层的挑战在于可视化成果的生命周期管理。通用工具以"报告"为导向——生成一次、使用一次、归档了事。然而运维场景需要的是随系统架构持续演进的动态知识资产：一张稳定性总览仪表盘、一张核心服务依赖拓扑图，必须能够跟随系统架构的变化而持续更新，并在人员流动时被新成员无缝继承。这种"可维护的运营知识资产"诉求，超出了通用图表工具的设计边界。

> **小结：** 运维可视化的核心需求不是"画图"，而是构建能够承载实时数据、支持持续维护、具备运维语义的动态知识资产体系。

### 1.3 多环境汇聚难：边缘自治与中心可见的内在张力

随着企业业务扩展至多地域、多云环境，每个区域的 BK-Lite 实例在本地独立运行以保障边缘自治能力。然而管理层对"上帝视角"的需求从未消失——在不破坏各区域独立性的前提下，需要通过单一屏幕掌握所有区域的运营状况。

传统的"集中写入式"方案将各区域数据汇聚至中央数据仓库，代价是显著增加的数据传输延迟、中央节点单点故障风险，以及潜在的跨公网数据合规问题。这一方案将架构复杂度从"可见性"转移至"存储治理"，并未从根本上解决问题。真正的需求是"请求时查询"而非"存储时汇聚"——数据保留在各自区域，仅在用户发起查询时按需请求目标区域数据。

> **小结：** 多环境可见性挑战需要"请求时查询"而非"预先同步"的架构范式，数据主权留在边缘，可见性在中心按需获取。

### 1.4 汇报靠人工：分析结论无法自动转化为决策依据

在缺乏统一可视化工作台的环境中，运营分析报告的生产依赖人工汇聚：工程师从多个系统中提取数据、手工制作 PPT，每次管理汇报或技术评审都意味着重复劳动。这一过程不仅消耗工程师大量时间，还引入了数据口径不一致的风险——不同人提取的数据可能来源于不同时间点、采用不同的过滤条件，导致汇报结论与实际状态存在偏差。

> **小结：** 汇报生产的人工依赖是可视化基础设施不完善的直接表征，其本质是从数据到洞察的转化链路断裂。

## 2. BK-Lite 运营分析与可视化技术体系

### 2.1 设计原则

BK-Lite 运营分析技术体系的构建遵循以下五项核心设计原则：

**原则一：轻量接入（Lightweight Integration）**
数据系统无需为运营分析做专项适配，通过 NATS 消息总线注册服务即可接入。不走直连数据库路线，消除对各数据系统内部协议的深度依赖，将新增数据源的接入成本压降至最低。

**原则二：按需查询（On-Demand Query）**
数据不做预先同步，在用户发起查询时才向目标命名空间请求数据。这一设计保障了数据的实时性，同时使各区域数据主权得到保留，规避集中存储带来的合规风险。

**原则三：运维语义内建（Built-in Ops Semantics）**
可视化载体从运维场景需求出发设计，拓扑图节点支持数据绑定与阈值配色配置，架构图内置多云图标体系，仪表盘组件支持告警状态表达。运维语义内嵌于产品设计，而非通过二次开发叠加。

**原则四：资产化治理（Asset-Oriented Governance）**
分析成果以目录化方式组织，通过细粒度权限控制实现可见性与可编辑性的独立管理，通过 YAML 导入导出支持 GitOps 式版本管理，使可视化成果成为可继承的团队知识资产。

**原则五：容错降级（Fault-Tolerant Degradation）**
单个命名空间的连接异常不阻断其他命名空间的数据返回，单个组件的查询失败不影响整个仪表盘的正常展示，系统在局部故障场景下保持整体可用。

## 3. 支柱一：多源数据汇聚（DataAggregate）

### 3.1 技术定位与挑战

多源数据汇聚层是整个运营分析技术体系的数据基础。其核心挑战在于：在不要求各数据系统做专项改造的前提下，以统一的通信协议接入异构数据源，按需向目标命名空间发起查询并获取结果。

### 3.2 REST API over NATS 接入机制

BK-Lite 运营分析采用以 NATS 消息总线为传输骨干的轻量级 RPC 接入机制，而非传统的直连数据库方案。其技术逻辑如下：

各数据系统（CMDB、监控、告警、日志）在 NATS 消息总线上注册为服务提供方，通过标准化的消息主题（Subject）对外暴露 REST API 风格的数据查询接口。运营分析后端以目标命名空间对应的 NATS 连接向指定消息主题发起请求，获取 JSON 格式的查询结果。

这一机制带来三项关键工程收益：

- **轻量接入**：数据系统只需在 NATS 总线上注册服务并暴露标准接口，运营分析侧仅需配置 NATS 连接参数与 API 路径，无需修改运营分析核心代码。
- **低耦合扩展**：新增数据源通过配置实现接入，不影响已有数据源和画布。
- **协议统一**：所有数据系统通过统一的 NATS 通道通信，消除了对各系统私有协议的依赖。

连接协议根据命名空间的 TLS 开关动态选择：关闭时采用 `nats://` 明文连接，开启时自动切换至 `tls://` 加密连接，默认端口为 4222。

### 3.3 按需查询与命名空间路由

数据源可关联多个命名空间，前端在查询时可通过 `namespace_id` 参数指定本次请求路由至哪个命名空间；未指定时默认使用第一个关联命名空间。后端根据路由结果建立对应的 NATS 连接，向目标服务发起请求并返回结果。

若指定的命名空间连接建立失败或数据源未关联该命名空间，后端返回明确的错误信息，不影响其他请求的正常执行。表格组件在多命名空间场景下可通过切换命名空间下拉筛选实现分区域数据查看。

每次数据请求自动携带当前用户与团队的上下文信息（用户名、所属团队、时区、权限集合、团队树结构），透传至数据服务层。各数据服务基于该上下文信息执行权限过滤，确保多租户场景下数据访问的隔离性与合规性。

### 3.4 数据流全链路时序

一次标准的仪表盘查询遵循以下时序：

```
用户调整筛选条件
      │
      ▼
前端携带筛选值 + 组件私有参数 → 后端 API
      │
      ▼
后端解析请求，提取用户/团队上下文，附加至查询参数
      │
      ▼
根据路由规则确定目标命名空间，建立 NATS 连接
      │
      ▼
目标命名空间数据服务基于权限上下文执行查询 ──→ 异常时返回错误信息
      │
      ▼
后端获取查询结果
      │
      ▼
结果返回前端，各组件按类型渲染图表
```

整个流程无状态，每次查询独立执行，不依赖服务端数据缓存，确保数据实时性。查询结果直接来源于目标数据服务，运营分析层不做额外存储。

## 4. 支柱二：可视化载体（VisualCanvas）

### 4.1 三种载体的技术定位

BK-Lite 运营分析提供仪表盘、拓扑图、架构图三种可视化载体，分别覆盖"数值观测—关系定位—结构表达"的完整分析链路，在同一目录化工作台中统一管理与导航。

### 4.2 仪表盘：数值观测与趋势分析

仪表盘基于组件化画布编排，支持多种图表组件类型的自由拖拽布局：

| 组件类型 | 技术标识 | 适用场景 |
|----------|----------|----------|
| 仪表图 | `gauge` | 关键指标速览、健康状态 |
| 折线图 | `line` | 指标趋势、时序变化分析 |
| 柱状图 | `bar` | 分类对比、分布统计 |
| 饼图 | `pie` | 占比分析、构成展示 |
| 散点图 | `scatter` | 多维度数据分布 |
| 地图 | `map` | 地域分布展示 |
| 文本 | `text` | 标注说明、富文本展示 |

**统一筛选联动机制**是仪表盘的核心技术特性。画布级统一筛选项通过参数 key 与数据源中标记为"筛选类型"的参数建立绑定关系。当用户调整筛选值时，所有已声明绑定该筛选项的组件同步发起新的后端查询，实现真正的服务端重查询，而非前端表面过滤。参数生效优先级遵循严格的四级规则：固定参数 > 统一筛选参数 > 组件私有参数 > 数据源默认参数，确保数据质量与用户自由度之间的边界清晰可控。

API 层分页上限为 10,000 条（由平台统一分页器控制）。当数据源关联多个命名空间时，表格可通过切换命名空间下拉筛选查看不同区域数据。

### 4.3 拓扑图：动态关系与故障影响定位

拓扑图是运营分析最具运维语义深度的可视化载体，支持节点与有向/无向边，具备以下核心技术能力：

- **数据绑定**：节点可绑定数据源，通过前端定时刷新机制实现状态更新，拓扑图从静态结构图升级为动态状态观测画布。
- **阈值配色语义**：节点颜色可依据绑定数据的数值范围动态切换，直观表达服务健康状态（例如：未恢复告警数为 0 时显示绿色，≥1 时显示黄色，达到危急阈值时显示红色）。
- **有向边依赖表达**：有向边描述服务间的上下游依赖关系，故障传播路径在拓扑图上可视化呈现，辅助运维工程师快速定位影响范围。

拓扑图支持两大节点类别：单值节点（`single`，用于展示关键指标数值）和图标节点（`icons`，用于表达服务、主机等资产）。具体节点形态由前端画布能力决定。

### 4.4 架构图：结构表达与多云资产可视化

架构图面向系统结构的静态表达场景，提供层次化的图形组织能力。其核心差异化能力在于多云图标体系的内置支持：

- **平台资源图标**：BK-Lite 平台组件、中间件等内置图标
- **CMDB 图标库**：与 CMDB 模块保持一致的资产类型图标
- **AWS 云资源图标**：覆盖计算、存储、网络、数据库等主要服务类型
- **阿里云资源图标**：覆盖 ECS、OSS、RDS 等核心产品图标
- **腾讯云资源图标**：覆盖 CVM、COS、TencentDB 等核心产品图标

在同一画布中混合使用上述图标体系，可完整表达跨云、跨层次的复杂架构，满足技术评审、上云规划、管理汇报等多类场景的图形语言统一诉求。

## 5. 支柱三：命名空间治理（Namespace）

### 5.1 命名空间的技术语义

命名空间（Namespace）是 BK-Lite 运营分析数据接入体系的核心治理单元，代表一个特定的 NATS 消息总线连接上下文，通常对应一个独立的数据环境（集群、区域或逻辑分区）。

每个命名空间维护以下连接信息：连接域名、账号、密码、NATS 逻辑命名空间（消息主题前缀，默认为 `bklite`）以及 TLS 开关状态。一个命名空间可被多个数据源复用，实现连接配置的集中管理与复用。

### 5.2 安全机制

**密码加密存储**：命名空间密码以 AES 加密算法加密后持久化存储，数据库中不保存任何明文形式的认证信息。YAML 导出时，密码字段自动替换为占位符，防止通过配置文件传输导致凭据泄露；导入时必须手动补充密码字段方可提交。

**TLS 传输加密**：命名空间支持 TLS 开关配置。启用时，NATS 连接自动切换至 `tls://` 协议，确保跨数据中心或跨公网传输场景下的通信加密，适应金融、政务等高安全等级行业的合规要求。

**权限上下文透传**：每次通过命名空间发起的数据请求，自动附带当前用户与团队的权限上下文（用户名、所属团队、权限集合、团队树结构），由目标数据服务基于该上下文执行精确的数据过滤，确保多租户隔离在数据服务层得到严格执行。

### 5.3 容错机制

当命名空间连接建立失败（网络不通、认证异常等）或数据服务返回错误时，系统返回明确的错误状态信息，前端可据此提示用户当前区域不可用，并支持切换至其他命名空间继续查看数据。

### 5.4 细粒度权限控制

权限控制体系从两个维度实现精细管理：

- **操作维度**：查看、新增、编辑、删除权限独立控制。查看权限与编辑权限彼此解耦，普通成员可消费分析结果而无需获得编辑授权，避免"可见即可改"的权限扩散风险。
- **组织维度**：分析资产与数据源遵循组织维度可见性策略，不同团队的数据资产在同一平台实例内保持隔离，团队维度的可见范围与权限授权保持一致。

权限粒度覆盖：主视图（查看、目录增改删、图表增改删）与管理页（数据源增改删查、命名空间增改删查）。

## 6. 支柱四：多环境聚合（MultiEnv）

### 6.1 技术原理：请求时聚合而非存储时汇聚

多环境聚合能力是 BK-Lite 运营分析解决边缘自治与中心可见张力的核心技术方案。其技术原理区别于传统的集中存储方案：各区域数据保留在其原始存储环境中，不做预先同步；仅当用户发起查询时，运营分析后端才通过指定命名空间向对应区域请求数据并返回结果。

这一"请求时查询"范式的工程优势体现在：
- **数据实时性**：查询结果直接来自目标数据服务，而非预先同步的历史快照。
- **无中央存储依赖**：不引入中央数据仓库，消除单点故障风险与存储运维成本。
- **数据主权保留**：各区域数据不离开原始环境，满足数据本地化合规要求。

### 6.2 多命名空间场景的工程实现

在多区域场景中，管理员为每个区域创建对应的命名空间，并将这些命名空间关联至同一数据源。前端通过 `namespace_id` 参数指定本次查询路由至哪个命名空间，后端根据路由建立对应的 NATS 连接并获取结果。用户可通过切换命名空间下拉筛选在不同区域间切换查看，实现多区域数据的分区对比。

表格组件在多命名空间场景下提供下拉筛选控件支持聚焦单一区域的数据查看。

### 6.3 目录化管理与资产治理

分析资产的目录化组织是实现规模化运营的基础机制。目录树支持最大 3 层嵌套结构，同一父目录下目录名称保持唯一性。目录支持启用/停用软归档，停用后的目录及其内容从主视图中隐藏，但数据予以保留，便于历史内容的可逆归档。

仪表盘、拓扑图、架构图名称全局唯一，确保分析资产在平台实例内具备清晰的身份标识，防止重名导致的资产混淆。

### 6.4 YAML 导入导出与 GitOps 集成

YAML 导入导出机制是分析资产跨环境迁移与版本管理的核心工程能力。支持导出的对象类型包括：仪表盘（含统一筛选定义、默认值与组件绑定关系）、拓扑图、架构图、数据源、命名空间。

导入流程分为三个阶段：

1. **预检阶段**：系统解析导入 YAML，识别命名冲突（同名对象已存在）、权限冲突（当前账号权限不足以覆盖目标对象）及敏感字段缺失（命名空间密码等），向用户返回预检报告。
2. **策略确认阶段**：用户根据预检报告选择冲突处理策略——覆盖（以导入内容替换现有对象）、重命名（自动为导入对象添加后缀以避免冲突）、跳过（保留现有对象，忽略导入内容）；并补充密码等敏感字段。
3. **执行阶段**：系统按策略逐对象执行导入，返回每个对象的处理结果（创建成功、覆盖成功、跳过、失败及失败原因）。

YAML 格式的配置文件可纳入 Git 版本控制，通过 GitOps 工作流实现分析资产的版本化管理、变更审计与多环境一致性保障。开放 API 的导入导出接口支持 CI/CD 流水线集成，测试环境验证通过的分析资产配置可自动化部署至生产环境。

## 7. 平台协同

BK-Lite 运营分析的价值在与平台各模块的数据协同中持续放大。监控、告警、日志、CMDB 四个模块通过 NATS 消息总线以标准化接口向运营分析提供数据，形成覆盖 IT 运维全链路的统一分析视图。

```
┌─────────────────────────────────────────────────────────────────────┐
│              BK-Lite 平台协同数据汇聚示意图（图2）                      │
├──────────────────┬──────────────────────────────────────────────────┤
│   平台模块        │         在运营分析中的数据汇聚价值                   │
├──────────────────┼──────────────────────────────────────────────────┤
│   监控模块        │  CPU/内存/网络指标趋势图 + 拓扑图节点实时健康状态      │
├──────────────────┼──────────────────────────────────────────────────┤
│   告警模块        │  告警趋势折线图 + 级别分布饼图 + 未恢复告警计数节点    │
├──────────────────┼──────────────────────────────────────────────────┤
│   日志模块        │  错误日志密度趋势 + 关键字日志计数 + 双重信号预警      │
├──────────────────┼──────────────────────────────────────────────────┤
│   CMDB 模块       │  资产总览仪表盘 + 架构图图标体系 + 依赖关系拓扑图     │
└──────────────────┴──────────────────────────────────────────────────┘
```

**与监控模块协同**：监控数据进入运营分析后，在仪表盘中呈现 CPU 趋势、内存分布、网络吞吐量等指标时序图，在拓扑图的单值节点中实时刷新各服务的健康状态。孤立的监控指标仅能告知"某台主机 CPU 达到 90%"，结合拓扑依赖关系后，运维工程师能够看到"该主机承载了哪些服务、下游服务受到何种影响"——这是完整排障上下文的必要组成部分。

**与告警模块协同**：告警数量趋势、告警级别分布、未恢复告警列表是运营大屏中最受关注的核心内容。将告警接口接入运营分析后，拓扑图中的单值节点可绑定特定服务的未恢复告警数，配合阈值配色机制（0条绿色、≥1条黄色、达危急阈值红色），故障传播路径在拓扑图上实现可视化表达，值班人员无需在监控大屏与告警平台之间切换。

**与日志模块协同**：错误日志数量趋势与特定关键字的日志计数可作为"信号密度"指标纳入仪表盘展示。当错误日志数量突增与监控指标波动在同一时间窗口叠加出现时，这一双重信号往往是系统性问题的早期预警，辅助运维团队在事故扩大之前及时介入处置。

**与 CMDB 模块协同**：CMDB 是资产信息的权威来源。在仪表盘中展示资产总览（主机数量、应用数量、业务线分布），在架构图中通过 CMDB 图标库将已录入资产可视化，在拓扑图中将 CMDB 依赖关系转化为可视化节点连线并叠加实时监控状态——"资产关系 + 实时状态"的融合视图，是运营分析与平台协同最具深度的实践场景：在同一工作台完成从"资产盘点"到"健康状态评估"再到"告警根因分析"的完整分析链路。

## 8. 部署与演进路径

### 8.1 部署模型

BK-Lite 运营分析是 BK-Lite 平台的内置模块，采用与平台其他服务共享基础设施的部署模型，无需独立部署额外的服务进程或数据存储组件。DuckDB 以进程内模式运行，不需要独立的数据库实例。

部署前置条件仅需确认：各数据源对应的 NATS 服务对运营分析后端网络可达（默认端口 4222）。多区域场景下，各边缘区域无需做额外配置，确保其 NATS 服务对主区域网络可达（通常通过企业内网或 VPN 实现）即可完成接入。

系统初始化时自动完成以下准备工作，无需人工干预：
- 默认命名空间创建
- 内置数据源标签写入（CMDB、告警、监控、日志、业务、其他）
- 预置数据源模板加载

### 8.2 三阶段演进路径

**第一阶段：即开即用（Day 0–3）**
获得账号权限后，直接在内置数据源基础上创建首张仪表盘。建议按业务域或团队职责规划第一级目录结构（如"电商业务"、"基础平台"），添加资产统计、告警趋势等基础组件，配置时间范围统一筛选。无需管理员专项介入即可完成基础可视化配置。

**第二阶段：能力扩展（Day 3–30）**
基础仪表盘验证可用后，逐步扩充分析资产：管理员补充自定义数据源（接入业务系统 API、自定义指标接口）；运维工程师构建核心服务拓扑图并配置告警阈值配色；架构师将现有架构文档迁移为架构图。建立统一的目录命名规范（如"业务域_场景"格式），为规模化使用打好治理基础。

**第三阶段：规模化运营（Day 30+）**
通过 YAML 导出将成熟的仪表盘模板与数据源配置迁移至其他环境；配置多区域命名空间实现跨区域汇聚分析；将分析资产纳入 GitOps 流程实现版本化管理；为重点分析资产指定维护负责人，推动分析能力的持续沉淀。

## 9. 客户价值

### 9.1 故障响应效率提升

预先构建的核心服务依赖拓扑图与运营大屏使故障信息收集阶段得到简化：告警触发后，依托节点阈值配色与有向边依赖关系，运维工程师可在单一界面内直接获得受影响服务清单与依赖传播路径，减少跨系统切换。相较于依次登录各独立系统逐一核查的传统方式，信息收集效率有所提升，有助于缩短 MTTR（平均修复时间）。实际效果因团队现有流程和分析资产成熟度而异。

### 9.2 日常运营效率提升

运营分析通过统一数据接入将 CMDB、监控、告警、日志数据汇聚于单一工作台，减少运维人员在多个系统间的切换频次。统一筛选联动机制消除了逐一调整各图表查询参数的重复操作——调整一次筛选条件（如时间范围）即可驱动所有已绑定组件同步发起新的后端查询，操作效率优于逐组件手动调整。

### 9.3 分析资产可继承性提升

传统运维环境中，关键人员离职往往伴随着其维护的仪表盘与拓扑图的隐性损失。BK-Lite 运营分析通过目录化组织、权限控制与 YAML 导入导出，将分析成果转化为可继承的团队公共资产。新成员可直接沿用已有分析资产，通过 GitOps 集成实现配置变更的版本追踪，降低知识流失风险。

### 9.4 管理层可见性与决策质量提升

运营全景大屏使 IT 管理层无需具备专业技术背景，即可通过直观图表把握系统整体运营状态，推动技术与业务之间的信息同频。架构图的多云图标体系统一了工程师与管理层在架构评审场景中的图形语言，消除了因表达方式不一致导致的沟通摩擦，技术评审与变更审批的效率相应提升。

### 9.5 零额外采购成本

BK-Lite 运营分析作为平台内置模块，无需额外购买独立 BI 工具、数据可视化平台或数据集成中间件，无需专项实施服务，即可获得覆盖仪表盘、拓扑图、架构图的完整运营分析能力。与市场上独立的运维可视化产品相比，总拥有成本（TCO）具有明显优势。

## 附录 B：术语表

| 术语 | 定义 |
|------|------|
| **命名空间（Namespace）** | 一个独立的 NATS 消息总线连接上下文，代表一个特定的数据环境（集群、区域或逻辑分区），维护连接域名、账号、密码、TLS 配置等信息，是数据接入的基础治理单元。一个命名空间可被多个数据源复用。 |
| **数据源（DataSource）** | 定义了从特定 REST API 获取数据的配置单元，包含 API 路径、参数模板、图表类型约束和命名空间关联。数据源是仪表盘组件的数据来源，可被多个组件复用，是分析资产标准化的基础。 |
| **命名空间路由** | 数据源可关联多个命名空间，查询时通过 `namespace_id` 参数指定本次请求路由至哪个命名空间，未指定则默认使用第一个关联命名空间。 |
| **统一筛选（Unified Filter）** | 仪表盘画布级别的公共筛选机制，允许定义一组筛选项统一控制已绑定的多个组件的查询条件。支持关键字输入与时间范围两种输入形态，触发后端重查询而非前端表面过滤。 |
| **分析资产** | 在运营分析中沉淀的可复用分析内容，包括仪表盘、拓扑图、架构图。通过目录化组织和权限控制，成为团队公共知识库，可通过 YAML 导入导出在环境间迁移。 |
| **拓扑图（Topology）** | 以节点和边表达对象关系的可视化载体，支持节点数据绑定和前端定时刷新，适合表达服务依赖关系、故障影响范围等动态场景。节点支持单值（single）和图标（icons）两类，与关注静态结构的架构图形成互补。 |
| **架构图（Architecture）** | 以层次化图形表达系统资源结构的可视化载体，支持多云图标体系（AWS/阿里云/腾讯云）、平台组件图标和 CMDB 图标，适合技术评审、架构沟通等静态结构表达场景。 |
| **固定参数** | 数据源参数模板中由管理员锁定、在仪表盘画布中不可修改的参数，用于保证数据查询语义的一致性与可管控性。 |
| **筛选参数** | 数据源参数模板中可被仪表盘统一筛选接管的参数类型，允许用户通过统一筛选控件调整时间范围、业务范围等查询条件。 |
| **组件私有参数** | 仅作用于当前仪表盘组件的参数配置，不影响同一仪表盘中的其他组件，适用于特定展示场景下的个性化参数设置。 |
| **NATS 消息总线** | BK-Lite 平台内部各模块间通信的轻量级消息中间件。运营分析通过 NATS 向各数据服务发起请求并获取结果，实现跨模块数据接入，无需直连各系统数据库。 |
| **DuckDB** | BK-Lite 平台中告警聚合模块所使用的进程内 OLAP 引擎（位于 alerts 模块），与运营分析模块无直接关联。 |
| **边缘自治** | 各区域 BK-Lite 实例保持独立运行能力，本地数据在本地处理，不依赖中央节点的运营模式。 |
| **中心可见** | 在边缘自治前提下，主区域通过命名空间机制按需请求各边缘区域数据，在主区域统一展示跨区域运营状态的能力，数据不离开原始区域。 |
| **请求时查询** | 数据不做预先同步存储，仅在用户发起查询时才向目标命名空间请求数据，保障数据实时性并避免集中存储风险的架构设计原则。 |
| **MTTR（Mean Time To Repair）** | 平均修复时间，衡量从故障发生到故障恢复的平均时长，是评估运维响应效率的核心 KPI 之一。 |
| **AES 加密** | 高级加密标准（Advanced Encryption Standard），BK-Lite 运营分析使用 AES 加密算法对命名空间密码进行存储加密，数据库中不存储明文，防止凭据泄露。 |
| **TLS（Transport Layer Security）** | 传输层安全协议，命名空间支持启用 TLS，使 NATS 连接采用 `tls://` 协议，确保数据传输加密，适合跨数据中心或跨公网的连接场景。 |
| **GitOps** | 以 Git 为版本控制核心，通过声明式配置文件管理基础设施与应用状态的运维方法论。BK-Lite 运营分析的 YAML 导入导出支持将分析资产纳入 GitOps 工作流。 |
| **PLG（Product-Led Growth）** | 产品驱动增长，以产品自身价值体验驱动用户激活与留存的增长方法论。BK-Lite 运营分析通过开箱即用的内置能力和快速上手路径践行 PLG 理念。 |
