## 序言

当今企业 IT 运维格局正在经历深刻重构。工具链的持续扩张在带来专业化能力的同时，也在入口层面制造了前所未有的分散与碎片化。工程师每日的工作不再从解决问题开始，而是从找到入口开始；告警响应的黄金时间，往往消耗在认证、导航与状态同步的摩擦之中。这不是某个团队的个性化困境，而是整个行业在工具规模化演进过程中所共同面对的结构性挑战。

BK-Lite 统一门户与工作台技术体系的构建，正是对上述挑战的系统性回应。控制台模块以"入口聚合"为起点，以"个性化布局"为承载，以"消息总线"为神经枢纽，以"跨端同源"为基础设施，从架构层面重新定义了运维平台的人机交互入口。技术选型与设计决策的每一步，均服从于一个核心命题：降低认知负载，提升响应效能。

## 1. 统一工作入口的时代背景与挑战

### 1.1 入口分散：工具扩张的代价

企业 IT 运维工具链的专业化演进，产生了不可回避的入口碎片化问题。资产台账、基础监控、日志分析、作业执行、AI 智能体——每个系统拥有独立的登录地址、独立的认证机制与独立的导航结构。经验数据表明，中型企业的运维工程师每日平均需要在 5 到 8 个独立系统之间来回切换，每次跨系统跳转均需完成重新认证、重新导航与重新建立工作上下文的完整流程。

更深层的问题在于可见性缺失：在缺乏统一入口的环境中，用户无法预判自己是否拥有某个系统的访问权限。误入无权限页面、遭遇 HTTP 403 响应、因此触发安全告警——这一链条在大规模运维团队中每日都在重复发生，既消耗有效工作时间，也在用户心理上积累对系统的不信任感。

> **小结：** 入口分散问题的本质是"寻址成本"的系统性浪费。在工具专业化加速的趋势下，若不在架构层面提供统一聚合层，入口碎片化只会随工具数量线性恶化。

### 1.2 信息淹没：通知渠道越多，信号越稀薄

传统运维环境中，通知渠道的多元化并未带来信息效率的提升，而是制造了相反的结果。告警通知走邮件、权限变更推企业微信、审批请求需主动刷新——当信息分布在多个异构渠道时，关键信号被大量噪声湮没的概率随渠道数量指数级增长。

通知生命周期管理的缺失进一步放大了这一问题。某条告警是否已被处理？哪位工程师正在跟进？在缺乏统一通知收口的架构中，团队协作依赖人工沟通填补状态空白，信息不对称成为响应延迟的系统性根因。与此同时，通知来源缺乏白名单管控，外部注入与重复投递难以被有效过滤，进一步加剧了通知疲劳。

> **小结：** 通知渠道的碎片化与生命周期管理缺失，将运维团队的注意力资源持续耗散在信息分拣而非问题解决上。统一收口不是功能增强，而是协作效率的架构前提。

### 1.3 跨端不一致：配置漂移的隐性风险

现代运维响应的时间窗口远超办公室工作时段。应急响应的深夜、出差途中的机场、非工作时间接到故障电话的瞬间——移动端的可用性已从"锦上添花"演变为"基础设施级"需求。

然而，传统运维平台在多端一致性上存在根本性的架构缺陷：移动端与 Web 端往往维护各自独立的配置状态，时区设置、语言偏好、通知状态彼此隔离，用户在一端完成的操作无法可靠地传播到其他端。密码重置后各端 Token 状态不同步，则进一步构成安全合规层面的显性漏洞。

> **小结：** 多端配置漂移不仅是用户体验问题，更是安全状态管理的合规风险点。消除漂移需要从接口层面实现同源架构，而非依赖定期同步机制。

### 1.4 个性化缺失：统一视图的个性化悖论

传统平台以管理员视角构建统一默认首页，将所有岗位用户引导至同一布局。然而，研发工程师、运维工程师与系统管理员的高频操作对象存在根本性差异：前者的核心关注点是模型训练状态与发布流水线，后者优先处理监控告警与 CMDB 资产变更，系统管理员则以用户权限管理为首要入口。

强制统一的视图布局，意味着每位用户都必须在大量无关信息中筛选出与自身岗位相关的内容。认知筛选成本随工具模块数量线性增长，高频操作的效率因导航摩擦而持续受损。更关键的是，这种模式在用户心理上构建了"系统是为别人设计的"认知，直接影响平台的长期采纳率与用户粘性。

> **小结：** 个性化不是对用户偏好的迁就，而是对专业岗位差异的精准响应。在多角色平台上，个性化布局引擎是提升工具采纳率的关键技术杠杆。

## 2. BK-Lite 统一门户与工作台技术体系

### 2.1 设计原则

BK-Lite 统一门户与工作台技术体系的构建，遵循以下四项核心设计原则：

**原则一：鉴权下沉，入口透明**。应用可见性由后端权限体系实时驱动，前端仅渲染用户有权访问的入口。权限变更在后端即时生效，无需任何前端手动刷新或管理员通知，确保"所见即有权"的强一致性。

**原则二：状态服务端化，配置跨端持久**。用户的工作台布局、语言偏好、时区设置、通知状态等所有个人化配置，均以服务端为唯一真相来源（Single Source of Truth），任何端侧的读写操作均直接作用于服务端状态，从架构层面消除多端数据不一致的可能性。

**原则三：消息总线解耦，通知异步投递**。各业务模块通过 NATS RPC 异步向通知中心投递消息，发送方无需感知通知中心的内部实现，投递操作不阻塞业务主流程，通知创建失败的异常被捕获并记录，不传播至发送方。

**原则四：最小化认知负载，优先快速路径**。所有交互设计以减少用户决策步骤为目标：首登初始化自动触发而非依赖用户发现、批量操作（全部已读）优先于逐条操作、权限卡片自动更新而非依赖用户感知。

## 3. 技术支柱一：统一应用门户（UnifiedPortal）

### 3.1 概述

统一应用门户（UnifiedPortal）是 BK-Lite 控制台模块对外提供的核心能力之一，其技术使命是将分散于各业务模块的访问入口聚合为单一的权限感知型门户界面。UnifiedPortal 的核心设计约束是：**门户所呈现的任何应用入口，必须与用户的实时授权状态保持强一致性**。

### 3.2 鉴权对接机制

控制台模块所有 API 的身份认证委托给平台统一鉴权网关完成。网关对每个入站请求执行 Token 有效性校验，并在校验通过后将用户上下文对象注入至 `request.user`。控制台模块以网关注入的 `request.user` 作为业务操作的唯一身份基础，不在模块侧重复实现认证逻辑，从而保证鉴权策略的一致性与可维护性。

权限可见性的动态特性体现在：当系统管理模块调整用户的应用访问权限后，控制台在下一次渲染首页时即从权限体系读取最新授权集合，无需任何缓存失效或推送操作。应用卡片列表的内容由权限状态实时决定，从架构上消除了越权访问的可能性。

### 3.3 应用卡片引擎

**数据模型：** 用户应用配置以 `UserAppSet` 实体存储，核心字段包括 `username`（用户标识）、`domain`（所属域）与 `app_config_list`（应用配置 JSON 列表）。数据库层面在 `(username, domain)` 二元组上建立唯一约束与复合索引，所有写操作遵循 upsert 语义——记录存在则更新，不存在则创建，确保任意并发写入场景下的数据一致性。

**国际化支持：** 内置应用卡片的显示文案依据用户语言偏好（`locale`）动态切换，文案来源以系统应用中心的最新配置为准，支持 `zh-Hans`（简体中文）与 `en`（英文）双语。

**默认配置降级：** 当用户尚未保存个性化配置（如首次登录场景）时，`current_user_apps` 接口返回空列表，前端负责处理空配置状态下的首页渲染逻辑。

**排除规则：** 首页应用卡片列表中不展示控制台模块自身的入口，避免循环导航。

## 4. 技术支柱二：个性化工作台（Personalize）

### 4.1 概述

个性化工作台（Personalize）引擎解决了"统一平台服务多角色用户"的核心矛盾：在保持平台统一性的同时，为不同岗位用户提供各自专属的工作界面布局。其设计哲学是将布局决策权完全交还给用户，而将布局状态的持久化与跨端同步责任由平台承担。

### 4.2 拖拽排序与布局持久化

用户可通过控制台首页的编辑模式对应用卡片执行以下操作：添加可用应用至个人首页、移除已添加应用、拖拽调整应用卡片的显示顺序。完成调整后执行保存操作，当前布局状态以 JSON 格式序列化持久化至服务端 `UserAppSet.app_config_list` 字段。

**布局状态的跨设备一致性**由服务端持久化机制天然保证：用户从任意终端（PC 浏览器、移动端 APP、新设备首次登录）发起首页加载请求时，系统均从同一服务端数据源读取该用户的最新布局配置，无需任何终端侧同步机制。

### 4.3 首登初始化闭环

首次登录初始化机制专为 PLG（产品驱动增长）场景设计，旨在消除新用户从"裸账号"到"可用工作台"的激活摩擦。

**首登判定逻辑：** 系统基于后端用户组状态执行判定。判定条件为：当前登录用户仅属于 `OpsPilotGuest` 默认组，且所属组数量为 1。满足此条件的用户在完成认证后，系统自动触发初始化向导弹窗，引导用户完成业务组归属设置与基础属性配置。**初始化操作以当前登录用户身份执行，服务端不信任前端传入的用户标识**，防止越权初始化。

**闭环设计：** 初始化完成后，用户的组归属状态发生变化（不再仅属于 `OpsPilotGuest`），下次登录时将不再触发初始化向导，直接进入正常控制台首页。初始化失败时系统向用户提供明确的失败反馈，不导致用户进入不一致状态。

**激活时效：** 首登初始化流程无需管理员介入，实现新用户的全自助平台激活。邮箱验证码（若涉及邮箱绑定操作）有效期为 10 分钟。

## 5. 技术支柱三：全局消息总线（MessageBus）

### 5.1 概述

全局消息总线（MessageBus）是 BK-Lite 控制台模块在平台级消息治理层面的核心技术支柱。其架构目标是将散落于各业务模块的通知投递行为统一收口至控制台通知中心，为运维团队提供单一的消息感知界面，同时通过双表数据模型与异步 RPC 机制保障通知状态的用户级独立性与系统级高可用性。

### 5.2 通知中心功能引擎

**NotificationViewSet** 提供完整的通知生命周期管理能力：

- **列表查询：** 支持 `unread_only`（仅未读）和 `app_module`（来源模块）两个过滤维度，以 `notification_time` 倒序排列，支持分页加载，确保大量历史通知场景下的查询性能。
- **已读管理：** 支持单条标记已读（`mark_as_read`）、一键全部已读（`mark_all_as_read`）与批量标记指定通知已读（`mark_batch_as_read`）三种操作模式，覆盖从精细处理到批量消费的完整需求谱系。
- **通知计数：** `unread_count` 端点提供实时未读数量查询，用于驱动前端铃铛角标的动态渲染。
- **软删除：** 通知的删除操作仅标记当前用户的删除状态（`is_deleted`），不执行物理记录删除，不影响其他用户的通知视图。
- **已读状态不可逆：** 已读状态的流转方向为单向（未读 → 已读），不支持回退，确保消费状态的可审计性。

**关键约束：** 前端创建通知接口被系统级禁用，通知创建的唯一合法路径是通过内部 NATS RPC 调用，从架构层面防止通知内容的外部注入。

### 5.3 双表通知数据模型

通知中心采用"通知主体与用户状态分离"的双表设计，这是支撑多用户独立消费语义的关键架构决策。

**Notification（通知主体表）：** 存储通知的内容与元数据，包括通知时间（`notification_time`）、来源模块（`app_module`）、通知内容（`content`，`TextField` 无数据库层长度约束，RPC 投递层限制为 2000 字符）与来源标识（`source`）。`notification_time` 与 `app_module` 字段均建有单字段索引，支持高效时间排序与模块级过滤查询。

**NotificationRead（用户已读状态表）：** 按"用户-通知"维度记录消费状态，存储已读标记（`is_read`）、已读时间与软删除标记（`is_deleted`）。数据库约束设计如下：

- 在 `(notification, user)` 二元组上建立唯一约束，确保每个用户对每条通知只有一条状态记录；
- 在 `(user, is_read)` 上建立复合索引，优化未读列表查询性能；
- 在 `(user, is_deleted)` 上建立复合索引，优化软删除过滤性能。

双表架构的核心价值在于：同一条通知记录对 N 个用户产生 N 条独立的已读状态记录，任意用户的已读/删除操作均不影响其他用户的消费状态，实现了真正意义上的多用户独立消费语义。

### 5.4 NATS RPC 异步投递机制

通知创建通道基于 NATS 轻量级分布式消息队列实现。`nats_api.py` 中注册了 `create_notification` RPC 函数，提供给平台内各业务模块调用。

**异步解耦：** 业务模块通过 NATS 发布消息触发通知创建，无需等待 RPC 响应即可继续执行业务主流程，通知投递操作的延迟不传播至发送方。

**来源白名单校验：** 通知创建流程内置两层安全校验机制。第一层为来源模块白名单校验：仅已注册的内部业务模块（`monitor`、`cmdb`、`alerts`、`opspilot`、`mlops`、`job_mgmt`、`node_mgmt`、`log`、`system_mgmt`、`console_mgmt`、`operation_analysis` 等）可向通知中心投递消息；第二层为内容长度校验：单条通知 `content` 字段上限为 2000 字符，超限请求被拒绝。

**故障隔离：** 通知创建失败时，`nats_api.py` 捕获异常并写入系统日志，RPC 异常不向 NATS 总线传播，不影响发送方业务模块的执行继续。

## 6. 技术支柱四：端云一致性（CrossDevice）

### 6.1 概述

端云一致性（CrossDevice）技术支柱解决了多端环境下用户信息状态管理的根本性挑战。其核心架构决策是：**Web 端与移动端 APP 共享同一套用户信息读写接口，以 `system_mgmt` 用户模型为唯一数据来源**，从接口层面消除多数据源带来的状态漂移风险。

### 6.2 同源接口体系

**`get_user_info` 接口**同时服务 Web 端与移动端的用户信息读取请求，返回字段包括：显示名、角色集合、所属组织路径、绑定邮箱、语言偏好（`locale`）、时区设置及最近登录时间。所有时间字段在服务端按用户个人时区配置完成 UTC → 本地时间转换后返回，确保不同时区用户在各端看到的时间展示均为本地化时间。

**`update_user_base_info` 接口**同时服务 Web 端与移动端的用户信息更新请求，支持修改显示名、语言偏好与时区。由于写操作直接作用于 `system_mgmt` 用户模型，无论更新操作从哪个端发起，修改结果对所有端即时可见，不存在同步延迟或状态漂移窗口。

### 6.3 账户安全全端一致性

密码安全操作是端云一致性的关键测试场景。BK-Lite 控制台的密码操作安全设计如下：

**密码校验：** `validate_pwd` 接口对当前登录用户的密码正确性进行校验，作为邮箱换绑、密码修改等高权限操作的前置身份核验步骤。

**邮箱验证码双向校验：** 邮箱变更流程采用发送验证码（`send_email_code`）+ 校验验证码（`validate_email_code`）的双步设计，验证码以 6 位随机数字生成，服务端以 `make_password` 对验证码哈希后将哈希值随响应返回前端，服务端不保留验证码明文；校验时前端将哈希值与用户输入一并提交，服务端使用 `check_password` 完成比对。

**密码重置后 Token 处理：** `reset_pwd` 接口将密码重置请求委托至 `system_mgmt` 模块执行，Token 失效的具体行为由 `system_mgmt` 的密码重置逻辑决定。控制台作为调用方，不在本模块侧重复实现 Token 管理逻辑。

**安全策略联动：** 密码修改流程强制执行 `system_mgmt` 模块所维护的平台级安全策略（密码长度下限、复杂度要求），控制台作为策略消费方，不自行定义安全基线，确保全平台安全策略的统一执行。

## 7. 平台协同

BK-Lite 控制台模块在平台整体架构中扮演双重角色：作为**全模块统一入口**与**全平台消息汇聚点**。这两个角色使控制台成为平台横向基础设施的核心节点，而非孤立的功能模块。

                    └────────────┬────────────┘

        │  入口跳转                             │  通知接收（NATS RPC）
        ▼                                       ▼
 ┌────────────────────────────────────────────────────┐
 │         业务模块层（通知来源白名单）                │
 │  monitor │ cmdb │ alerts │ opspilot │ mlops │ ...  │
 └────────────────────────────────────────────────────┘

图2：BK-Lite 控制台平台协同关系图
```

### 7.1 与系统管理鉴权底座的协同

控制台模块与 `system_mgmt`（系统管理模块）之间存在深度的能力依赖关系，具体体现在以下三个维度：

| 协同维度 | 控制台的消费方式 | system_mgmt 的提供方式 |
|---|---|---|
| 用户身份与权限 | 通过网关注入的 `request.user` 获取用户上下文；基于权限体系渲染可见应用卡片 | 维护用户模型、角色体系与菜单权限映射 |
| 安全策略 | `reset_pwd` 接口强制读取并执行密码安全策略 | 统一定义密码长度、复杂度等安全基线 |
| 邮件通道 | 邮箱验证码通过平台统一邮件服务发送 | 维护平台级邮件发送配置与能力 |
| 用户数据 | `get_user_info` 与 `update_user_base_info` 直接操作 system_mgmt 用户模型 | 提供用户信息的持久化存储 |

这种"控制台作为消费方，system_mgmt 作为能力提供方"的分层设计，确保安全策略与权限逻辑在平台层统一定义，所有模块统一消费，不存在安全逻辑的重复实现或碎片化。

### 7.2 作为全模块入口与消息汇聚点

**统一入口角色：** 控制台首页的应用卡片体系覆盖 BK-Lite 平台全部业务模块，包括 CMDB 资产管理、基础监控告警、日志分析检索、作业管理、运营分析、OpsPilot 智能体平台及 MLOps 机器学习平台等。系统管理模块完成应用注册与权限分配后，控制台首页卡片自动更新，无需人工干预。

**消息汇聚点角色：** 已接入通知投递的模块与典型场景如下：

| 模块标识 | 业务模块 | 典型通知场景 |
|---|---|---|
| `monitor` | 基础监控 | 告警触发、告警恢复、阈值超限 |
| `alerts` | 告警中心 | 告警聚合、告警升级 |
| `cmdb` | 配置管理数据库 | 资产变更、自动发现完成 |
| `system_mgmt` | 系统管理 | 权限变更、账号状态变更 |
| `opspilot` | OpsPilot 智能体 | 模型训练完成、任务执行结果 |
| `mlops` | MLOps 平台 | 数据集处理完成、模型部署状态 |
| `node_mgmt` | 节点管理 | 采集器安装完成、节点状态变更 |
| `job_mgmt` | 作业管理 | 作业执行完成、执行失败 |
| `log` | 日志分析 | 日志检索任务完成 |
| `operation_analysis` | 运营分析 | 分析报告生成完成 |

除内置模块外，在系统管理模块中注册的自定义应用同样可通过 NATS RPC 向通知中心投递通知，支持平台能力的横向扩展。

```

NATS 连接配置需在平台环境变量或配置文件中指定 NATS 服务地址，`nats_api.py` 在服务启动时完成 RPC 函数注册。

## 9. 客户价值

### 9.1 效率提升：降低操作摩擦与激活门槛

**入口切换成本消除：** 统一应用门户将运维工程师每日多次的独立系统登录与导航操作整合为单次登录后的卡片直达，降低跨系统认证与重新导航的重复摩擦，有助于减少入口寻址所消耗的无效时间。

**新人激活自助化：** 首登初始化闭环使新员工可在登录后自助完成业务组归属配置，无需等待管理员手工介入，有助于缩短新用户进入可用工作台状态的周期。

**批量通知消费效率：** `mark_all_as_read` 批量操作支持在单次操作中完成对积压通知的全量消费，避免逐条标记的重复操作，有效降低通知积压场景下的处理负担。

### 9.2 安全保障：权限透明与账户闭环

**越权访问风险归零：** 应用卡片基于用户实时权限状态动态渲染，用户界面层不存在超出授权范围的入口。对于金融、政企等等保合规要求严格的客户，"所见即有权"的架构特性天然简化了权限审计工作，管理员无需额外措施防范用户的越权操作。

**多重账户安全机制：** 邮箱变更验证码双向校验、密码修改安全策略强制执行（策略由 `system_mgmt` 统一定义）、密码重置委托 `system_mgmt` 执行，多重机制共同构成账户安全防护体系。

**通知来源可信性保障：** NATS RPC 白名单校验机制确保只有已注册的内部业务模块才能向通知中心投递消息，外部系统与非预期来源无法注入通知内容，通知信息的可信度有架构层面的制度性保障。

### 9.3 协作提升：信息对称与响应加速

**跨端信息完全一致：** 同源接口架构确保 Web 端与移动端的用户信息及通知状态实时一致。出差中的工程师在手机端处理了告警后，PC 端刷新即呈现已处理状态，彻底消除多端状态不同步导致的协作误判。

**个人消费状态独立不干扰：** 双表通知数据模型保证每位团队成员的已读/删除状态完全独立，A 工程师的操作不影响 B 工程师的待办视图，协作边界清晰，责任归属明确。

**时区感知消除协作误读：** 各工程师按所在地时区独立设置，平台所有时间展示均按个人时区完成 UTC 本地化转换，跨国分布式团队的时间协同误差归零。

BK-Lite 统一门户技术体系在当前版本（V1.0）完成四大技术支柱的工程落地后，后续演进方向将沿以下三条技术路径展开：

**实时推送增强：** 当前通知未读数依赖客户端轮询获取。下一版本计划引入 WebSocket 长连接机制，实现未读通知的服务端主动推送，将通知感知延迟从轮询间隔压缩至亚秒级，进一步提升告警响应的时效性。

**智能化个性化：** 在现有用户手动排序的基础上，计划引入基于用户历史访问行为的应用卡片智能排序推荐模型，将工作台布局优化从"用户主动调整"演进为"系统学习推荐"，进一步降低个性化配置的认知成本。

| 通知 `content` 字段最大长度 | 2000 字符 | 超限请求在 RPC 处理层被拒绝 |
| 邮箱验证码位数 | 6 位数字 | 随机生成，服务端以 `make_password` 哈希后将哈希值随响应返回前端；服务端不保存验证码明文，校验时由前端提交哈希值，服务端以 `check_password` 比对 |
| 首登判定条件 | 仅属于 `OpsPilotGuest` 组且组数量 = 1 | 满足则自动触发初始化向导 |
| 用户应用配置唯一键 | `(username, domain)` | upsert 语义写操作，保证并发写入一致性 |
| 通知列表默认排序 | `notification_time` 倒序 | 最新通知始终在列表首位 |
| 已读状态流转方向 | 仅允许 未读 → 已读 | 已读状态不可逆，保证消费状态可审计性 |
| 密码重置后 Token 处理 | 委托 `system_mgmt` 执行 | 控制台调用 `system_mgmt.reset_pwd`，Token 失效行为由 system_mgmt 密码重置逻辑决定 |
| Web/Mobile 接口共享 | `get_user_info` / `update_user_base_info` | 同源接口，指向同一 `system_mgmt` 用户模型 |

## 附录 B：术语表

| 术语 | 定义 |
|---|---|
| 控制台（Console） | BK-Lite 平台统一入口与个人工作台模块，模块代码标识为 `console_mgmt`，是用户登录平台后首先到达的界面层 |
| 统一应用门户（UnifiedPortal） | 控制台的四大技术支柱之一，负责聚合所有业务模块的访问入口，并基于用户实时权限动态渲染应用卡片 |
| 个性化工作台（Personalize） | 控制台的四大技术支柱之一，负责用户级工作台布局的持久化存储与跨端同步，以及首登初始化引导流程 |
| 全局消息总线（MessageBus） | 控制台的四大技术支柱之一，包含通知中心功能引擎、双表数据模型与 NATS RPC 异步投递机制 |
| 端云一致性（CrossDevice） | 控制台的四大技术支柱之一，通过同源接口体系保障 Web 端与移动端 APP 的用户状态完全一致 |
| 应用卡片 | 控制台首页展示的各业务模块入口单元，仅渲染用户有权访问的应用，随权限变更自动更新 |
| 首登初始化 | 新用户首次登录时系统自动触发的自助引导流程，完成业务组归属配置，使账号从"裸账号"进入可用工作台状态 |
| `UserAppSet` | 存储用户应用配置的数据实体，唯一键为 `(username, domain)`，遵循 upsert 写入语义 |
| `Notification` | 通知主体表，存储通知内容与元数据，与 `NotificationRead` 共同构成双表通知数据模型 |
| `NotificationRead` | 用户已读状态表，按"用户-通知"维度独立记录每位用户的消费状态，支撑多用户独立消费语义 |
| NATS RPC | 基于 NATS 轻量级分布式消息队列的远程过程调用机制，BK-Lite 平台模块间通信的基础设施，通知投递通道的技术底座 |
| 白名单校验 | 通知创建时对来源模块标识进行的安全验证，仅已注册的内部模块可投递，防止外部注入 |
| upsert | 数据库操作语义：记录存在则更新，不存在则创建，控制台用于保存用户应用配置，保证并发写入一致性 |
| 软删除 | 通知删除操作仅标记 `is_deleted` 字段，不执行物理记录删除，不影响其他用户的通知视图 |
| 同源接口 | Web 端与移动端 APP 共享同一套后端 API，数据来源指向同一存储，从接口层面消除跨端不一致风险 |
| Token 废弃 | 密码重置操作委托 `system_mgmt` 模块执行，Token 失效行为由该模块的密码重置逻辑决定 |
| 时区转换 | `get_user_info` 接口返回的所有时间字段，均按用户个人时区设置完成 UTC → 本地时间转换后输出 |
| PLG | Product-Led Growth（产品驱动增长），指用户可自助完成平台激活与价值发现，首登初始化闭环是 PLG 设计理念的具体技术实现 |
| `OpsPilotGuest` | 新用户注册时默认所属的临时组，首登状态判定逻辑的依据，完成初始化后用户离开此组 |
| `locale` | 用户语言偏好标识，`zh-Hans` 表示简体中文，`en` 表示英文，控制台界面语言与卡片文案随之动态切换 |
| 单源真相（Single Source of Truth） | 架构原则：用户状态以服务端存储为唯一权威来源，所有端侧直接读写服务端，不维护本地副本 |
