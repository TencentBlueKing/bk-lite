# BK-Lite 功能清单编制规范（SOP）

> Migrated from `spec/fuctionlist/功能清单编制规范（SOP）.md` as legacy capability evidence.

**文档版本：** V1.0
**发布日期：** 2026-06-02
**用途：** 指导 BK-Lite 功能清单的新增、更新与评审，保证多模块、多人/多次编制的口径与格式一致。
**适用对象：** 编写或维护 `功能清单/` 目录文件的人员或自动化助手。

---

## 0. 一句话原则

**以代码为准、以事实说话、可核对、不夸大。** 功能清单只记录"现有版本已实现的能力"，不写规划、演示或营销话术。

---

## 1. 定位与边界

功能清单用于**核对产品交付范围**，回答"这个模块到底有哪些功能、支持哪些对象"。它与其他文档的分工：

| 文档 | 回答的问题 | 语气 |
|---|---|---|
| 白皮书（`白皮书/`） | 为什么做、解决什么问题、价值如何 | 方案/价值阐述 |
| PRD（`prd_bk/`） | 应该做什么、规则与验收 | 需求定义 |
| **功能清单（`功能清单/`）** | **实际有什么、支持哪些对象与规格** | **客观、可核对** |

功能清单**不替代**白皮书，也不照抄白皮书措辞。

---

## 2. 事实来源与优先级（重要）

编制任一功能项前，按以下优先级取证。**当多源冲突时，以代码为准**——代码常常比 PRD 更新。

1. **源代码（最高）**：`bk-lite/server/apps/<模块>/` 与 `bk-lite/web/src/app/<模块>/`。
   - 枚举/状态机/默认值：优先看 `constants/`、`enum.py`、`models/`、`choices.py`、前端 `constants/*.ts`。
   - 对象/采集/指标/接入清单：看 `support-files/`、`migrations/` 初始化数据、`*.json` 种子数据、插件目录。
2. **PRD**：`prd_bk/<模块>/*.md`，作为功能域划分与业务规则的主线参考。
3. **白皮书**：`白皮书/*.md`，仅用于理解背景，**不得**作为功能或数字的唯一依据。

> 凡功能清单中给出的"数量/范围/默认值/状态机取值"，都应能在代码中找到对应出处。编制时随手记录出处文件，便于评审复核。

### 2.1 各类"对象范围"在代码中的典型位置

| 数据类型 | 典型出处 |
|---|---|
| CMDB 采集对象/内置模型 | `cmdb/constants/constants.py`、`cmdb/collection/plugins/` |
| 监控对象与指标 | `monitor/support-files/plugins/**/metrics.json`（每对象一份，`metrics` 数组即指标） |
| 日志采集类型/采集器 | `log/` 下 collect type 定义、`support-files` |
| 告警源/状态机/窗口 | `alerts/constants`、`alerts/models`（`WindowType`、`Alert.status` 等） |
| 节点采集器/控制器/OS 架构 | `node_mgmt/` 采集器注册、`support-files`、`NodeConstants` |
| 作业脚本语言/驱动/状态 | `job_mgmt/constants/choices.py`、`models/scheduled_task.py` |
| 运营分析数据源/图表类型 | `operation_analysis/support-files/*.json`、前端 `ops-analysis/constants` |
| OpsPilot 模型/渠道/工具/节点 | `opspilot/enum.py`、`tools_loader.py`、前端 `constants/chatflow.ts` |

### 2.2 权限作用域术语统一映射（强制）

各模块在描述"可见范围/资源边界"时，代码与 PRD 历史上混用了**组织 / 组 / 团队 / 分组**等词。经核对源码，它们在 BK-Lite 中是**同一个权限实体**，编制功能清单时必须按下表统一口径，不得让读者误以为存在多套并行权限模型。

| 出现过的叫法 | 真实实体 | 说明 |
|---|---|---|
| 组织 / 组 / 组织（组） | `system_mgmt.Group` | 树形结构（`parent_id`），支持虚拟组、子组织角色继承，是平台唯一的组织/权限分组实体。 |
| 团队 / 分组（OpsPilot） | 同上 | OpsPilot 各资源（模型、知识库、机器人、记忆等）的 `team` 字段是 `JSONField`，存储一串 **Group ID 列表**；`DEFAULT_TEAM=[1]` 即根组织。"团队"即组织（组）。 |
| 默认组 / OpsPilotGuest（控制台） | 同上 | 控制台首登初始化绑定的默认组，同样是一个 `Group` 实例。 |

**编制要求：** 正文统一表述为"**组织（组）**"；在 OpsPilot、控制台等沿用"团队/分组/默认组"叫法的地方，首次出现时补一句"（即系统管理中的组织/组）"，其余照旧。数据权限的行级隔离均以该 `Group` 为边界。

- 所有文件位于 `功能清单/`。

- 索引文件：`00-功能清单总览与编制规范.md`（模块索引 + 简要规范，指向本 SOP）。
- 本规范：`功能清单编制规范（SOP）.md`。

当前模块编号：01 CMDB、02 监控、03 告警中心、04 日志、05 作业、06 节点、07 系统管理、08 控制台、09 运营分析、10 OpsPilot、11 MLOps。新增模块顺延编号，并同步更新 `00` 索引表。

## 4. 文档结构（统一模板）

每个模块文件固定如下结构：

```
# <模块名> · 功能清单

**文档版本：** V1.x
**发布日期：** YYYY-MM-DD
**适用范围：** BK-Lite <模块名>模块
**编制依据：** <模块> PRD vX.x（日期）与 server/apps/<app>、web 源代码核对

---

## 一、模块定位
（2–4 句客观描述模块职责与边界；写明"本清单仅列已实现能力"。）

## 二、功能清单
### 1. <一级功能域>
| 功能项 | 功能说明 | 规格 / 约束 | 状态 |
|---|---|---|---|
| ... | ... | ... | GA/Beta |
### 2. <一级功能域>
...

## 三、能力边界与约束
（来自 PRD"关键规则"：产品不做什么、限制条件、前置条件。）

## 四、平台协同
（与其他模块的数据/能力依赖，客观陈述。）

## 五、支持的<对象/采集/数据源…>范围   ← 仅"含外部对象/采集/接入"的模块需要
（按分类列对象清单，给出数量与状态；末尾用一句 `> 说明：` 补充口径。）
```

### 4.1 何时需要"## 五" 对象范围节

- **需要**：模块管理外部对象、采集源、接入源、数据源、模型/渠道/工具等可枚举的支持范围。
  当前含此节：CMDB、监控、告警中心、日志、作业、节点、运营分析、OpsPilot。
- **不需要**：纯平台内能力、无外部对象清单的模块。
  当前无此节：系统管理、控制台、MLOps（其算法场景已在"二、功能清单"中列出）。

## 5. 表格列规范

### 5.1 功能清单表（二级）

固定四列，表头一字不差：

| 列 | 写法要求 |
|---|---|
| **功能项** | 可独立验收的最小功能单元名词，简洁。 |
| **功能说明** | 客观描述"做什么"，不含价值修辞、不用形容词堆砌。 |
| **规格 / 约束** | 可核对的参数、范围、状态机、默认值、前置条件；**无则填 `—`**。 |
| **状态** | `GA`（默认，正式可用）/ `Beta`（试验，能力存在但可能不完整/不稳定）。 |

### 5.2 对象范围表（五级）

列可按对象类型灵活设计，但必须满足：按分类组织、给出**数量**、给出**枚举真实取值**（如 `shell`、`anthropic`）便于核对、并**保留状态标注**。复合对象（含子对象，如 K8s/VMware）要列出子对象及其规格（如每对象指标数）。

**状态标注的两种写法（二选一，不可省略）：**

- **逐行状态列**：当同一表内 GA/Beta 混杂时，必须加"状态"列逐行标注（如 CMDB 采集对象表）。
- **整表状态声明**：当一节内对象状态完全一致时，可不加列，但**必须在该节表格上方或下方用一句话声明**，例如"本节所列对象/指标均为 GA"；若存在个别例外，须紧接着点名列出（如"其中 X、Y 为 Beta / 需模板扩展，非开箱预置"）。

无论哪种写法，读者都应能在对象范围处直接判断每一项的 GA/Beta 边界——这是该表存在的意义。

---

## 6. 状态（GA / Beta）判定

- 默认 `GA`。
- 标 `Beta` 的依据：代码中有显式 Beta/试验标记（如名称含【BETA】、`is_beta`、注释标注试验），或产品已知不稳定/不完整。
- **不得**把"计划中/未实现"的能力写入正文表格。确需说明路线，只能放在文末说明中并明确标注为非当前能力，或干脆不写。

---

## 7. 文风约束（去营销化）

- 禁止：夸大词（"业界领先""极致""彻底解决""毫秒级"等无依据表述）、价值渲染、对比竞品。
- 禁止：把白皮书的叙事句直接搬进功能说明。
- 提倡：动词 + 对象的客观陈述（"支持实例的新增、编辑、删除"）。
- 数字必须有代码出处；拿不准的范围用"约"并在说明中注明统计口径（如"指标数按 `metrics.json` 条目统计"）。

---

## 8. 新增 / 更新工作流程

1. **确定来源**：找到模块的 PRD 与 `server/apps/<app>`、`web/src/app/<x>` 代码路径。
2. **通读 PRD**：梳理功能域（对应"二"的 `### N`）与关键规则（对应"三"）。
3. **核对代码**：逐项验证枚举、状态机、默认值、对象/采集/接入清单；冲突以代码为准，记录差异。
4. **套用模板**：按第 4 节结构成文，表头按第 5 节固定。
5. **补对象范围**：如属第 4.1 节"需要"的模块，补"## 五"。
6. **标注差异**：代码与 PRD 不一致处，在"规格/约束"列或"能力边界"中如实说明（不要悄悄改）。
7. **自检**：对照第 9 节检查清单。
8. **更新索引**：新增模块时同步更新 `00` 索引表与本 SOP 第 3 节编号。
9. **版本与日期**：更新文档头 `文档版本`/`发布日期`/`编制依据`。

### 8.1 版本号约定

- 模块文件 `文档版本` 从 `V1.0` 起；功能实质变更（新增功能域、状态变化、对象范围调整）升次版本 `V1.x`；仅文字订正不必升版，更新日期即可。
- `编制依据` 须写明对照的 PRD 版本与核对日期，便于追溯"这版清单基于哪个时点的代码"。

---

## 9. 提交前检查清单

- [ ] 文档头四项齐全（版本/日期/适用范围/编制依据）。
- [ ] 结构为 一、二、三、四（含对象范围的模块再加 五）。
- [ ] "二"按 `### N. 功能域` 分节，每节一张四列表，表头为 `功能项 | 功能说明 | 规格 / 约束 | 状态`。
- [ ] 每个功能项均对应代码/PRD 已实现能力，无规划项混入。
- [ ] 所有数量/默认值/枚举可在代码中找到出处。
- [ ] Beta 项有依据并已标注；其余为 GA。
- [ ] 规格列无内容处填 `—`，未留空。
- [ ] 无营销夸大用语，无照抄白皮书叙事。
- [ ] 代码与 PRD 的差异已如实标注，未做无依据美化。
- [ ] 对象范围表给出分类、数量、真实枚举值、状态；复合对象列出子对象。
- [ ] `00` 索引与本 SOP 编号已同步（若新增模块）。

## 10. 已知差异登记（示例与维护建议）

编制过程中发现的"代码与 PRD/前端不一致"建议集中登记，便于回推修订源文档。已发现示例：

| 模块 | 差异点 | 处理（以代码为准） |
|---|---|---|
| CMDB | 订阅触发类型：PRD 3 种，代码 5 种 | 取代码 5 种（属性/关联/到期/新增/删除） |
| 告警中心 | Alert 默认状态：代码为 `unassigned`；`WindowType` 定义 3 类但运行时仅用 sliding/session | 按代码标注 |
| 作业 | 高危处理策略标签、SSH 作为连接方式而非独立驱动枚举 | 按代码区分"驱动枚举"与"界面选项" |
| 运营分析 | 后端 `DashboardType` 含 gauge/scatter 等遗留枚举，前端落地 6 类 | 只列已落地的 6 类 |
| OpsPilot | 工具模块键 18 个（cmdb 注释关闭），对应 16 功能类目；无 vLLM 本地推理引用 | 按代码写 18 键/16 类目，不写未实现的本地推理 |
| 节点 | 常量声明 arm64，但内置采集器注册全为 x86_64 | 标注实际仅 x86_64 |

新发现的差异请按此表追加，并视情况反馈给 PRD 维护方。
