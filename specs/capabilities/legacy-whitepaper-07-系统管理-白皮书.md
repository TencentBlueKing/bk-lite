## 序言

数字化转型浪潮席卷各行各业，企业 IT 基础设施从单一系统演进为覆盖运维、监控、资产、智能运营的综合平台集群。然而，规模扩张带来的不仅是业务价值，更是身份管理、权限治理与合规审计的深层挑战。当一家企业的运维平台上同时运行着监控、CMDB、日志分析、告警中心、作业管理与 AI 智能体，账号的碎片化、权限的失控与操作的不可追溯，已成为制约平台可信运营的核心痛点。

BK-Lite 系统管理模块作为整个平台的安全管控底座，以**"身份即统一、授权即生效、敏感即加密、操作即留痕"**为核心设计理念，将用户与组织管理、双层权限模型、认证与安全防护、全维审计日志四大技术支柱整合于统一中台。通过集中配置、全平台即时生效的架构设计，实现了从身份录入到权限分发、从登录防护到操作留痕的端到端安全闭环，为 BK-Lite 平台上的每一个业务模块提供稳定可靠的安全基础设施服务。

本白皮书面向企业 IT 决策者与架构师，深度解析 BK-Lite 安全管控底座的技术架构、核心能力、合规支撑与演进路径。无论企业正处于平台化建设的起步阶段，还是面临多团队治理、监管合规的深水区挑战，本文档旨在提供清晰的技术全景图，帮助决策者理解 BK-Lite 系统管理模块的核心价值与战略意义。

## 1. 安全治理的时代背景与挑战

### 1.1 身份碎片化：账号黑洞的隐患

当企业同时运行多套 IT 系统时，身份数据往往各自为营，缺乏统一的生命周期管理机制。员工入职时，账号在各系统分别开通，耗时费力且容易遗漏；员工离职时，HR 通知了邮件系统，却未能同步到运维平台——这一典型场景在行业中并非个例，而是系统性通病。行业调研显示，相当比例的企业存在僵尸账号（账号已失效但权限仍有效）问题，这类"权限幽灵账号"是内部安全威胁的重要入口。

在缺乏统一身份目录的环境中，安全管理员无法在统一视图下回答"这名员工在所有系统中还持有哪些权限"这一基础问题，账号治理黑洞成为绝大多数企业安全事件的起点之一。

> **小结**：身份数据的碎片化分布，导致账号全生命周期管理失控，离职账号残留、权限遗漏回收等问题长期困扰企业安全团队，形成难以量化的隐性安全风险。

### 1.2 权限失控：粗放管理与繁琐维护的两难困境

在众多运维平台中，权限管理往往走向两个极端。其一是粗放模式：所有人共享管理员账号，一旦发生安全事件，操作责任无法追溯到具体个人；其二是人工脚本模式：运维人员靠人工脚本逐人维护权限清单，每次人员流动都可能产生配置漂移，新入职人员权限开通流程耗时且容易遗漏。

当企业进入"一平台多团队"阶段，数据隔离问题进一步加剧。研发团队不应看到生产环境的监控告警策略，安全团队不应触碰研发团队的 CMDB 资产数据，但粗粒度的功能权限控制无法实现行级数据隔离。权限过度授予（Over-privileged Access）与数据越权访问，构成了内部威胁的双重风险面。

> **小结**：传统权限管理模式要么牺牲精细化控制，要么带来不可承受的运维复杂度，无法在安全性与可运维性之间实现有效平衡，亟需系统化的双层权限架构解决方案。

### 1.3 合规压力：日志分散与举证困境

《网络安全法》《数据安全法》《个人信息保护法》以及等保2.0（GB/T 22239-2019）、ISO 27001:2022等合规框架，对企业的日志审计能力提出了明确要求：必须能够清晰回答"谁在什么时间从哪个IP登录了系统"、"谁修改了哪条关键配置"。

然而，当企业运行着多套异构系统，每个系统各自维护独立的操作日志，审计人员面对监管检查时，不得不跨系统逐一导出日志，手工汇总整理后才能提交审计报告，流程耗时且容易出现遗漏。更严峻的是，分散存储的日志缺乏统一的完整性保护机制，日志本身的可信度难以向监管机构充分举证。

等保2.0三级要求明确规定，安全审计覆盖范围需涵盖每个用户的每次登录操作，且审计记录需防止未授权访问、修改或删除。分散架构无法满足这一要求，集中化审计成为合规的必要路径。

> **小结**：日志分散存储、审计举证效率低下、日志完整性无法保证，三重合规压力推动企业寻求集中化、不可篡改的统一审计解决方案。

### 1.4 多组织治理：边界模糊的协同难题

成长型企业的 IT 平台普遍面临"一平台多团队"的治理挑战。研发、测试、运维、安全各职能团队在同一平台上协同工作，既需要严格的权限边界和数据隔离，又需要在某些场景下实现跨团队协作。为每个团队单独部署一套平台实例，意味着成倍的基础设施成本、运维复杂度与数据孤岛风险；而共享单一平台实例，又面临权限隔离失效、数据越权访问的治理挑战。

如何在单一平台实例上实现多组织的独立治理，同时支持临时性跨团队协作授权，是每个平台化建设项目绕不开的核心命题。

> **小结**：多团队协同治理需求与单一平台实例成本优势之间的矛盾，要求平台具备细粒度的组织级权限隔离能力与灵活的临时授权机制，以兼顾安全边界与协作效率。

## 2. BK-Lite 安全管控底座技术体系

### 2.1 设计原则

BK-Lite 安全管控底座的设计遵循以下核心原则：

**最小权限原则（Least Privilege）**：用户默认以最小权限集合运行，仅在明确授权后方可访问对应功能与数据。角色体系从只读权限出发，按需扩展写权限，杜绝过度授权。

**零信任边界（Zero Trust Boundary）**：不依赖网络边界作为安全边界，每次访问请求均经过身份验证与权限校验，权限变更即时生效，无静默窗口期。

**集中配置，全局生效**：权限规则在系统管理模块统一配置，通过权限缓存引擎自动传播至所有业务模块，消除权限漂移。

**深度防御（Defense in Depth）**：功能访问控制、数据行级隔离、登录安全防护、操作审计留痕四层机制协同工作，单层失效不影响整体安全态势。

**密码合规与敏感加密**：所有敏感凭据（密码、API密钥、渠道配置）均经加密存储，接口层自动脱敏返回，遵循密码应用相关规范要求。

**全量审计，只读留痕**：三类审计日志（登录日志、操作日志、错误日志）以只读方式持久化存储，任何角色（包括超级管理员）均无法通过界面修改或删除，确保审计轨迹的完整性与可信度。

## 3. 第一支柱：统一身份与组织（Identity）

### 3.1 全局用户目录

BK-Lite 系统管理模块建立全局统一的用户目录，作为平台所有身份数据的权威来源（Source of Truth）。用户目录以"用户名+域"作为唯一键，支持从内部手动录入与外部认证源自动同步两条数据通道的协同工作。

用户档案涵盖完整的身份信息维度：基础信息（用户名、显示名、邮箱、电话）、安全信息（OTP密钥、最后登录时间、密码修改时间、密码错误次数、账号锁定截止时间）、组织归属（支持多组织绑定）与角色授权（直接授权与组织继承权限）。用户详情页提供"基础信息、组织、角色、数据规则"四维透视，安全管理员可在单一界面完成对任意用户的全权限核查，无需跨系统逐一排查。

用户管理支持按用户名、姓名、邮箱关键字搜索，支持按组织筛选，支持超级管理员用户独立查询视图。用户禁用操作即时生效，关联模块的访问权限同步失活，彻底消除离职账号残留风险。

### 3.2 树形组织架构

组织管理采用无限层级的父子树形结构，支持"集团→子公司→部门→小组"等多层级企业组织架构的完整映射。组织节点名称在同一父节点下保证唯一性，删除操作需确保该节点及所有子节点下无用户绑定，以防误删导致权限孤岛。

组织架构具备以下关键治理特性：

- **外部ID映射**：支持与外部系统（如认证源中的部门ID）建立映射关系，为认证源同步提供精确的组织对齐锚点。
- **虚拟组织**：支持创建不对应真实部门的逻辑分组，满足临时项目组、跨部门协作等场景的灵活授权需求，项目结束后撤销虚拟组织即可，不影响正式组织架构树。
- **子组织角色继承**：组织绑定角色后，支持配置是否向下级子组织继承传递，实现组织级权限的灵活继承控制。

### 3.3 认证源接入机制

BK-Lite 支持微信开放平台（OAuth扫码）、蓝鲸平台（bk_login）接入，以"认证源"（LoginModule）抽象封装屏蔽各认证通道的对接细节。认证源同步通过 Celery Beat 周期任务异步执行，管理员可配置定时同步策略（如每天凌晨02:00），系统在后台静默完成用户与组织数据的批量导入，不影响前台用户正常操作。

认证源的核心价值在于：对于已接入认证源的企业，员工信息变动（入职、离职、部门调动）在上游系统侧完成后，BK-Lite 自动同步，权限随之更新。认证源的 AppSecret 等敏感凭据通过 EncryptMixin 机制自动加密存储，接口层返回时自动脱敏，确保凭据不以明文形式暴露于任何接口返回中。

支持认证源的启停管理，停用认证源不影响已同步的用户数据，仅暂停后续同步任务。

## 4. 第二支柱：双层权限模型（RBAC + 数据权限）

### 4.1 RBAC 角色与菜单权限

第一层权限模型基于角色的访问控制（Role-Based Access Control，RBAC），解决"谁能做什么操作"的核心问题。权限模型的核心设计如下：

**角色是权限的最小分配单元**。每个角色与一组菜单权限绑定，管理员通过将角色授予用户或组织来完成权限分发。角色名称在同一应用域内保证唯一性，内置关键角色（admin、normal）受保护不可删除，被用户引用的角色不可直接删除，以防误操作导致权限断层。

**组织级角色绑定支持批量授权**。将角色绑定至组织节点后，该组织下的所有用户自动继承对应权限，新入职员工绑定组织后即刻获得岗位所需权限，无需逐人授权。组织角色变更触发批量权限缓存刷新，确保变更即时生效。

**自定义菜单组支持多版本管理**。管理员可基于已有菜单组克隆新版本进行调整，同一应用下同一时刻仅允许一个菜单组处于启用状态，支持在测试环境验证后安全切换至生产，内置菜单组受保护不可编辑或删除。通过为不同角色裁剪专属导航，只读审计人员仅能看到审计相关页面，运维工程师仅能访问运维功能，界面聚焦减少误操作风险。

### 4.2 数据权限：行级数据隔离

第二层权限模型基于组织维度的数据权限规则，解决"能访问哪些数据"的核心问题，是 BK-Lite 多团队治理能力的技术基石。

数据权限规则以"规则名+组织ID+应用"三元组作为唯一键，规则内容以 JSON 格式存储，由各业务模块定义和解析规则语义。用户通过组织绑定关系获得具体的数据权限规则集合，业务模块在执行查询时自动注入规则定义的过滤条件，实现对用户透明的行级数据隔离。

以典型场景为例：为研发团队配置"仅可查看标记为研发环境的服务器实例"的数据权限规则后，研发工程师在 CMDB 中执行资产查询时，系统自动注入该过滤条件，生产环境资产在视图层面完全不可见，无需 CMDB 模块自行实现权限逻辑。

数据权限规则的优势在于两层权限相互解耦：RBAC 层控制功能入口可见性，数据权限层控制数据访问范围，业务模块灵活组合两层能力，无需重复实现权限逻辑。

### 4.3 权限缓存引擎：写时失效策略

权限计算是高频操作，每次 API 请求均需完整评估用户的菜单权限与数据规则。BK-Lite 采用 Redis 作为权限缓存层，用户登录后构建并缓存完整权限集合（菜单列表 + 数据权限规则），后续请求直接命中缓存，无需重复查询数据库。

缓存一致性通过**写时失效（Write-through Invalidation）**策略保证：任何影响权限的数据变更——用户信息变更、角色菜单授权更新、组织角色绑定调整、数据权限规则增删改——触发时，系统精确识别受影响的用户范围，定向清除其权限缓存，强制下次请求时重新加载最新权限。该策略将缓存刷新范围最小化，兼顾权限变更的即时生效性与非受影响用户的访问性能。

用户无需重新登录即可感知权限变更，缓存刷新过程对用户完全透明，实现了"授权即生效"的设计承诺。

## 5. 第三支柱：认证与安全（AuthSec）

### 5.1 多认证源登录

BK-Lite 支持微信开放平台扫码（OAuth）、蓝鲸平台（bk_login）以及 BK-Lite 本地用户域作为认证源，通过统一的 LoginModule 抽象封装屏蔽各认证通道的对接细节，降低认证摩擦的同时简化账号管理。

认证源支持启停管理，管理员可灵活控制各认证通道的生效状态。

### 5.2 密码策略体系

BK-Lite 内置可配置的密码策略引擎，覆盖密码安全的全维度管控：

| 策略维度 | 配置项 | 默认值/规格 |
|---------|-------|------------|
| 密码复杂度 | 最小长度 | 8位（可配置） |
| 密码复杂度 | 最大长度 | 20位（可配置） |
| 密码复杂度 | 必要字符类型 | 大写字母、小写字母、数字、特殊符号（可多选） |
| 账号安全 | 连续失败锁定阈值 | 可配置失败次数 |
| 账号安全 | 锁定时长 | 可配置时长 |
| 会话安全 | 登录有效期 | 可配置 |
| 多因素认证 | OTP动态口令 | 可选开启（TOTP标准，RFC 6238） |

密码策略由系统设置统一控制，在密码创建、修改、重置操作时全局生效，确保策略一致性，不存在策略绕过的配置漏洞。用户密码变更后，系统自动更新密码修改时间并重置错误计数与锁定状态，防止历史锁定记录对合法用户的误伤。

特殊符号集合覆盖 `!@#$%^&*()_+-=[]{};\'"\\|,.<>?/~\`` 等常用特殊字符，满足主流密码合规要求。

### 5.3 OTP 多因素认证

BK-Lite 支持基于 TOTP 标准（RFC 6238）的 OTP 动态口令二次验证。每位用户具有独立的 OTP Secret，以加密方式存储于用户安全档案中。开启 OTP 后，用户登录时需在输入账号密码后，额外提供 Authenticator 应用（Google Authenticator、Microsoft Authenticator 等）生成的6位动态验证码，每30秒更换一次，有效防御密码泄露场景下的账号接管攻击。

OTP 开关支持在系统级全局配置（`enable_otp` 系统设置项），开启后要求所有用户在首次登录时完成 OTP 绑定并进行二次验证，为平台整体提供额外的安全锁。

### 5.4 敏感数据加密（EncryptMixin）

BK-Lite 通过 EncryptMixin 加密混入机制，在模型层统一处理敏感字段的加解密操作，实现对业务逻辑层代码完全透明的加密保护：

- **通知渠道配置**：Webhook URL、企业微信 AppSecret、钉钉机器人 Token 等渠道凭据，在写入数据库时自动加密，接口返回时自动脱敏，业务模块调用时透明解密。
- **认证源凭据**：各认证源的 AppSecret 等敏感字段，遵循同等加密保护策略。
- **用户密码**：采用标准单向哈希算法存储，即使数据库数据泄露，也无法还原明文密码。
- **OTP Secret**：用户 OTP 密钥加密存储，防止密钥直接暴露。

该机制遵循"敏感即加密"的设计原则，符合密码应用相关规范对存储密码安全性的要求，有效降低数据库数据泄露场景下的凭据暴露风险。

### 5.5 系统密钥管理

BK-Lite 提供 API Secret 的统一签发与生命周期管理能力，支持自动化脚本、CI/CD 流水线等机器账号场景的 API 调用鉴权。所有 API 密钥的签发、使用与吊销均通过系统管理模块统一管控，避免凭据无序扩散，自动化操作同样纳入审计体系的追踪范围。

## 6. 第四支柱：全维审计（Audit）

### 6.1 审计设计原则

BK-Lite 审计体系的设计遵循"全量采集、集中存储、只读留痕"三项原则。任何角色（包括超级管理员）均无法通过界面修改或删除已记录的审计日志，审计轨迹的完整性与可信度在架构层面得到保障，而非依赖管理流程约束。三类审计日志（登录日志、操作日志、错误日志）默认按时间倒序展示，支持多维筛选；其中登录日志与操作日志支持 Excel 导出，错误日志仅支持查询，满足等保2.0三级对安全审计的合规要求。

### 6.2 登录日志

登录日志记录平台上每一次登录行为的完整上下文，覆盖以下关键字段：用户名、来源IP地址、登录状态（成功/失败）、失败原因、设备信息（UA字符串、操作系统、浏览器）、地理位置信息、所属域。

登录日志是异常访问检测的核心数据源。安全管理员可通过时间范围、用户名、登录状态等维度快速筛选，识别非工作时间登录、异常地理位置登录、暴力破解尝试等安全事件。支持筛选导出与按选中项导出两种 Excel 导出模式，满足向监管机构提交证明材料的格式要求。

### 6.3 操作日志

操作日志记录平台上所有数据变更操作（创建、更新、删除、执行）的完整审计轨迹，覆盖以下关键字段：用户名、来源IP地址、所属应用、动作类型（create/update/delete/execute）、操作摘要、所属域。

操作日志是合规审计举证的核心数据源，直接回应"谁在什么时间修改了哪个配置"的合规查询。业务模块的高风险操作（如作业脚本执行、CMDB资产变更）均接入统一操作日志体系，避免审计盲区。支持按时间范围、用户名、动作类型、应用四个维度进行精确筛选，一键导出满足合规举证要求的 Excel 报告。

### 6.4 错误日志

错误日志以只读方式呈现系统运行中产生的异常信息，记录时间、用户、所属应用、模块标识、错误信息和调用堆栈等字段，供技术团队快速定位故障根因。错误日志仅支持查询，不支持导出，最大程度降低敏感技术信息的外泄风险。

### 6.5 合规对应关系

| 合规要求 | 等保2.0对应项 | BK-Lite 实现机制 |

| 登录失败处理 | 8.1.3.1 | 账号连续失败锁定机制 |
| 会话管理 | 8.1.3.1 | 登录有效期配置、JWT会话控制 |

| 审计记录保护 | 8.1.3.3 | 任何角色不可修改/删除日志 |

## 8. 部署与演进路径

### 8.1 部署架构

BK-Lite 系统管理模块采用 Django/DRF 三层架构：前端 Next.js 通过 REST API（JWT鉴权）与后端交互，后端业务逻辑由 Django ViewSet 和 Service 层承载，数据持久化到 PostgreSQL，异步任务通过 Celery + Redis 处理。

模块支持容器化部署（Docker/Kubernetes），基础设施依赖为 PostgreSQL 和 Redis，可与 BK-Lite 平台其他模块共享基础设施资源，亦可独立部署以满足资源隔离需求。

### 8.2 分阶段演进路径

BK-Lite 系统管理模块遵循渐进式上线的演进理念，支持企业按照自身成熟度阶段逐步解锁能力，避免一次性投入过高的实施复杂度：

**第一阶段：基础建设（约1个工作日）**

完成组织架构录入、用户账号创建与组织绑定、基础角色配置与权限授予、密码安全策略配置、通知渠道基础配置。该阶段完成后，平台即具备基本的身份管理与权限控制能力，四步操作无需专项实施顾问介入。

**第二阶段：安全加固**

在基础建设基础上，接入外部认证源（微信开放平台或蓝鲸平台），启用定时同步，消除手动维护账号的运维负担；开启 OTP 二次验证，提升平台整体安全防线；配置账号锁定策略，激活多因素认证防护。

**第三阶段：精细化治理**

按照业务团队边界配置组织级数据权限规则，实现多团队数据隔离；为各角色定制菜单组，按最小权限原则裁剪功能入口；配置审计日志导出策略，与企业安全运营中心（SOC）或 SIEM 系统对接。

**第四阶段：平台化扩展**

随业务规模增长，接入更多业务模块，将系统管理的统一鉴权能力扩展至全平台；评估超大规模用户目录（超过10万用户）场景下的数据库分区与缓存策略；基于审计日志数据，建立持续的访问行为基线与异常检测机制。

### 8.3 边界与约束说明

以下能力不在本模块当前交付范围，实施前需知悉：

- **许可管理**（许可码、模块授权、容量与到期提醒）由企业版独立能力承载。
- **业务应用内部工作流**（如审批流程）需在各业务模块自行实现或借助第三方工作流引擎。
- **第三方身份系统管理控制台**（如在 BK-Lite 内直接管理上游认证系统配置）不在本模块范围，BK-Lite 仅作为认证源的消费方。
- **超大规模用户目录**（超过10万用户）场景建议在部署规划阶段评估数据库分区或缓存扩展策略。

## 9. 客户价值

### 9.1 权限管理从多系统分散操作转为集中一次性操作

在传统多系统分散管理模式下，处理一次权限变更（新员工入职、员工离职、岗位调动）需在各系统逐一登录操作，流程繁琐且容易遗漏。BK-Lite 系统管理模块统一管理后，管理员仅需在一处完成操作，变更自动传播至全平台，显著减少重复劳动与人为疏漏风险。

IT管理员从繁琐的重复性权限操作中释放，可将精力投入架构优化、安全建设等更高价值工作。

### 9.2 安全风险从"依赖人工"转变为"系统自动执行"

离职账号未及时注销、权限过度授予、高危操作无留痕，是企业内部安全事件的三大主要来源。BK-Lite 系统管理通过统一账号目录（即时禁用生效）、写时失效权限缓存（变更即时生效）、三类只读审计日志（操作全量留痕）三重机制，将上述风险从依赖人工流程执行转变为系统层面自动执行。账号"幽灵"不再游荡，越权访问无处遁形，安全事件发生概率从根本上降低。

### 9.3 合规审计准备从跨系统手工汇总转为集中一键导出

传统方式下，每次监管审计准备需从多个系统分别导出日志、人工汇总整理，流程耗时且易出错。BK-Lite 系统管理集中存储所有审计日志，支持按时间范围、用户名、操作类型、应用维度灵活筛选，一键导出满足格式要求的 Excel 报告（登录日志与操作日志均支持），让企业在面对监管检查时准备更高效、举证材料更完整。

### 9.4 消除通知渠道重复建设成本

通知渠道基础设施统一管理，消除了各业务模块分别对接企业微信、钉钉、飞书、自定义 Webhook 的重复开发工作。7种渠道类型（邮件、企业微信、企业微信机器人、NATS、飞书机器人、钉钉机器人、自定义 Webhook）统一托管，Webhook 密钥集中加密存储，降低凭据泄露风险，同时避免各模块 API 调用量分散导致的资源浪费。

### 9.5 开箱即用降低实施门槛

系统管理模块遵循 PLG（产品驱动增长）设计理念，内置安全默认值，无需专项实施顾问。按照四步快速上手指引，管理员在一个工作日内即可完成从组织架构搭建到权限分配的完整闭环，初期投入极低，随业务规模扩展再逐步启用外部认证源（微信/蓝鲸平台）、数据权限隔离等进阶能力，按需解锁，不必一次到位。

## 10. 技术演进展望

**自适应身份治理**：随着企业身份数据的持续积累，将探索基于用户行为基线的异常访问检测能力，自动识别偏离正常行为模式的高风险操作，为安全团队提供主动预警，从被动审计向主动防御演进。

**细粒度权限模型扩展**：在现有 RBAC + 数据权限双层模型基础上，研究引入属性基访问控制（ABAC）能力，支持基于资源属性、环境条件的动态权限评估，满足更复杂业务场景下的细粒度访问控制需求。

**审计日志与 SIEM 深度集成**：规划提供标准化的日志推送接口，支持将登录日志与操作日志实时推送至企业 SIEM 系统（如 Splunk、Elastic Security），实现与企业安全运营中心的深度融合。

**合规框架自动化映射**：构建合规框架（等保2.0、ISO 27001、SOC 2等）与系统配置的自动化映射引擎，定期生成合规状态报告，识别配置偏离合规基线的风险项，降低合规维护的人工负担。

**多租户隔离增强**：在现有组织级数据隔离基础上，探索更强的多租户隔离架构，满足 SaaS 化部署场景下不同客户间的数据严格隔离需求，支撑 BK-Lite 向托管服务模式演进。

## 附录 B：术语表

| 术语 | 说明 |
|-----|-----|
| **RBAC** | Role-Based Access Control，基于角色的访问控制。用户通过被授予角色来间接获得权限，角色是权限集合的抽象单元。 |
| **SSO** | Single Sign-On，单点登录。BK-Lite 通过支持企业现有认证渠道（如微信OAuth）完成登录，员工无需为运维平台单独记忆账号密码。 |
| **OTP** | One-Time Password，一次性密码。基于 TOTP 标准，每30秒更换一次的六位动态验证码，用于多因素认证。 |
| **TOTP** | Time-based One-Time Password，基于时间的一次性密码算法（RFC 6238），BK-Lite OTP功能的技术基础。 |
| **MFA** | Multi-Factor Authentication，多因素认证。结合密码（知识因素）与 OTP 动态口令（持有因素）的双重认证机制。 |
| **PLG** | Product-Led Growth，产品驱动增长。以产品本身的易用性和价值驱动用户自助采用，无需重度依赖销售和实施服务。 |
| **EncryptMixin** | BK-Lite 系统管理模块的敏感字段加密混入机制，在模型层统一处理加解密，对业务逻辑层代码透明。 |
| **数据权限** | 在角色权限（控制功能入口）之上的第二层权限控制，限定用户在特定功能内可以访问的数据范围，实现行级数据隔离。 |
| **写时失效** | 权限缓存更新策略：当权限相关数据发生变更时，立即清除受影响用户的缓存，强制下次请求时重新加载最新权限。 |
| **虚拟组织** | 不对应企业真实部门的逻辑分组，用于临时性跨部门合作或特殊权限授予场景，不影响正式的组织架构树。 |
| **认证源** | BK-Lite 对认证通道（微信开放平台OAuth、蓝鲸平台bk_login等）的抽象封装（LoginModule），支持用户和组织数据的同步导入及扫码/OAuth登录认证。 |
| **权限缓存** | 将用户的完整权限集合（菜单列表、数据规则）缓存在 Redis 中，避免每次请求重复计算，任何权限变更触发相关用户缓存失效。 |
| **等保2.0** | 《信息安全技术 网络安全等级保护基本要求》（GB/T 22239-2019），中国国家标准，规定不同安全等级信息系统的安全保护要求。 |
| **超级管理员** | BK-Lite 中权限最高的管理角色，可执行全局系统配置、权限配置与审计管理，建议极少数人持有。 |
| **平台管理员** | 负责组织、用户、角色、应用、菜单、渠道日常配置的管理角色，权限范围限定于业务配置。 |
| **安全管理员** | 负责认证源配置、密码策略管理、登录安全设置及日志审计的管理角色，专注于安全与合规职能。 |
| **Source of Truth** | 权威数据来源，指系统中某类数据的唯一权威存储位置，其他系统从此处同步数据而非各自维护。 |
| **ABAC** | Attribute-Based Access Control，基于属性的访问控制，支持基于资源属性、环境条件的动态权限评估。 |
| **SIEM** | Security Information and Event Management，安全信息与事件管理，企业级安全运营中心的核心技术平台。 |
