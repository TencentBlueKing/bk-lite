# PRD 编写规范

> Migrated from `spec/prd/规范.md` as legacy capability evidence.

本规范根据当前 `spec/prd/` 下各模块 PRD 的实际编写方式总结，用于约束 PRD 的来源、结构、粒度与文风，保证全平台 PRD 一致、可维护。

## 1. 目标

- PRD 只描述产品的稳定最终状态与规则，用于对齐"产品是什么、要做成什么"。
- PRD 以**当前代码为唯一事实标准**忠实反推：写"现在是什么"，不写"从什么改成什么"。
- 只写业务逻辑与关键技术架构选择，不写技术实现细节。

## 2. 目录与文件结构

- 一级：`spec/prd/<模块>/`，每个产品模块一个目录，目录名用中文（与产品模块对应，如 `CMDB`、`监控系统`、`告警中心`）。
- 二级：每个"大的功能菜单"一份 `.md`，文件名用中文（如 `资产.md`、`模型管理.md`）。
- 菜单清单的事实来源是前端导航定义 `web/src/app/<模块>/constants/menu.json`，不照搬历史文档。

### 菜单拆分粒度

- 顶层导航项各自成文。
- 顶层若为纯分组容器（其下是多个独立大功能），则其子项各自成文，容器本身不单独成文。
- 详情页的子标签（tab）、二级小设置项折叠进所属菜单文件，不单独成文。

## 3. 文件骨架（统一 6 节）

```
# <模块> · <菜单>

## 1. 背景与目标
## 2. 范围定义
## 3. 关键能力
## 4. 关键规则
## 5. 关键技术架构选择
## 6. 验收标准
```

各节职责：

- **背景与目标**：该菜单解决什么问题、面向什么场景（简短）。
- **范围定义**：该菜单覆盖的能力清单（结论式罗列）。
- **关键能力**：业务逻辑主体，按子功能分小节描述用户能做什么。
- **关键规则**：约束、边界、权限、状态流转、唯一性、互斥等稳定规则。
- **关键技术架构选择**：只写"选了什么、起什么作用"，不写字段/接口/算法。例如：存储用图数据库 FalkorDB、指标存 VictoriaMetrics、检索基于 pgvector、执行经 NATS + Sidecar、实验跟踪用 MLflow 等。
- **验收标准**：可据以验证该菜单是否达成的结论式条目。

## 4. 文风要求

- 使用结论式短句，避免过程化措辞。
- 已确认的硬约束（含具体数字上限，如"每模型最多 3 条唯一规则"）直接写入正文作为产品规则。
- 不单列"待决问题/批判"清单；评审中的疑点须先与产品确认结论，再回写为稳定规则。

## 5. 禁止项

- 禁止使用"本次调整为/由…改为…"等过程化措辞。
- 禁止记录迁移过程、替换过程、讨论过程（如有变更过程，写入同模块目录下的更新日志）。
- 禁止堆砌技术实现细节：接口字段、代码路径、表结构、算法参数、具体代码常量等。
- 禁止照搬已失效的历史 PRD；以当前代码为准。

## 6. 维护方式

- 代码演进后，按"读该菜单 server + web 代码 → 忠实更新 PRD"的方式做增量校准。
- 事实来源优先级：当前代码 > 产品最新要求确认 > 历史文档（历史文档仅作参考，不作依据）。
