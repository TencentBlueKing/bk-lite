## specs: 2026-06-03-k8s-config-check-workbench-design.md

## 背景

当前已经有一个可用的 K8s 智能体，能够承接查询、配置检查、修复对比和修复命令输出。但如果直接把它裸露给工作台入口，会有两个明显问题：

1. 用户提问方式不稳定，常常只说“帮我看一下”“检查下”“修一下这个”，系统很难直接判断应该走查询、检查还是修复。
2. 工作台如果把“怎么问”常驻展示为说明文档，会破坏入口的自然感，变成一个学习成本较高的表单或说明页，而不是一个可直接提问的助手。

本设计的目标不是重做 K8s 智能体，而是为它设计一个更稳定的工作台流程，让用户可以“随便先问一句”，系统再决定是直接执行，还是最小化地补充必要信息。

## 目标

- 提供一个 **单输入框主入口** 的 K8s 配置检查工作台。
- 用户一句话输入后，系统能自动识别是 **查询 / 配置检查 / 修复 / 问答 / 不清楚**。
- 对于清晰问题，直接进入主智能体执行，不额外打断用户。
- 对于模糊问题，只在必要时触发引导，并且采用最少轮次完成澄清。
- 延续现有 K8s 主智能体能力，不拆散已有的查询、检查、修复能力。
- 把“入口引导”与“主业务执行”职责拆开，便于后续调优和运营。

## 非目标

- 不在这一版中设计新的 K8s 检查规则或新的修复能力。
- 不改造已有 K8s 主智能体内部执行链路。
- 不把工作台做成复杂向导、表单页或配置中心。
- 不常驻展示大段“如何提问”的帮助文案。
- 不在第一版中引入多个业务智能体并行协作。

## 核心设计结论

### 1. 工作台采用“单输入框智能路由型”

工作台首页只保留一个主输入框作为核心交互入口。用户输入自然语言问题后，系统先做意图判断，再决定是直接执行还是先引导。

推荐入口感受应当接近：

- 用户可以直接问：“检查 payment-gateway 配置”
- 也可以直接问：“列出所有 deployment”
- 也可以直接问：“给我修复命令”
- 如果用户只说“帮我看看”“修一下这个”，系统再补充澄清

这意味着工作台的重点不是“提前教育用户怎么问”，而是“在用户问得不完整时，把问题改造成可执行请求”。

### 2. 保留一个 K8s 主智能体，不拆分查询/检查/修复

当前已有智能体已经能覆盖：

- 资源查询
- 配置检查
- 修复对比
- kubectl 修复命令输出

因此工作台不应该为了流程整洁而强行拆成多个业务智能体。拆得过细会带来 prompt 维护成本、路由复杂度和上下文传递成本。

推荐做法是继续保留一个 **K8s 主智能体**，统一负责业务执行；工作台只在入口前增加一个轻量引导层。

### 3. 单独保留一个“引导智能体”

虽然查询、检查、修复不必拆开，但“入口澄清”仍应独立出来。原因是：

- 引导逻辑和业务执行逻辑不是同一种能力
- 引导节点只需要做“把问题问清楚”这件事
- 一旦入口效果不理想，可以单独调整引导策略，不影响 K8s 主智能体

因此推荐的结构是：

**Web 对话 → 意图分类 → 条件分支 →（必要时）引导智能体 → K8s 主智能体**

## 工作台画布设计

### 推荐节点

工作台画布只保留以下 5 类核心节点：

1. **Web 对话**
   作为唯一用户输入入口，承接所有自然语言问题。

2. **意图分类**
   判断当前问题属于：
   - 查询
   - 配置检查
   - 修复
   - 一般问答
   - 不清楚

3. **条件分支**
   判断当前问题是否已经足够明确：
   - 明确：直接进入 K8s 主智能体
   - 不明确：进入引导智能体

4. **引导智能体**
   只负责最小化澄清，把模糊问题改造成明确请求。

5. **K8s 主智能体**
   统一负责查询、配置检查、修复对比、修复命令。

### 不推荐的节点设计

以下设计不建议放进第一版工作台：

- 在入口前增加表单式“先选模式”
- 把查询/检查/修复拆成多个业务智能体
- 让引导智能体直接输出检查结果
- 在条件分支里堆大量业务规则
- 常驻帮助面板解释“怎么问问题”

## 用户主链路

### 链路 A：直接执行

适用于问题已经足够明确的情况，例如：

- “检查 payment-gateway 配置”
- “列出所有 deployment”
- “给我修复命令”
- “什么是 liveness probe”

链路为：

**Web 对话 → 意图分类 → 条件分支 → K8s 主智能体**

### 链路 B：先引导，再执行

适用于问题不完整、动作或范围缺失的情况，例如：

- “帮我看一下”
- “看看有没有问题”
- “修一下这个”
- “看下 k8s”

链路为：

**Web 对话 → 意图分类 → 条件分支 → 引导智能体 → K8s 主智能体**

引导智能体产出的不是最终结果，而是一个可执行的业务请求，再回流给主智能体执行。

## 引导策略设计

### 原则：先补动作，再补范围

这是本工作台入口体验的关键设计。

当用户问得不清楚时，不要一次性追问多个问题，也不要直接展示长说明。推荐固定采用两级引导：

1. **先补动作**
   - 你是想查资源列表？
   - 还是做配置检查？
   - 还是看修复建议？

2. **再补范围**
   - 是看全集群？
   - 还是指定 workload？
   - 还是指定 namespace？

动作优先于范围，因为如果连用户到底想“查 / 检 / 修”都没搞清楚，先问范围没有意义。

### 引导形式

引导智能体应优先使用 **可点击选项**，而不是长段文字或开放式教学。

推荐形式：

- 第一轮补动作：`查资源列表 / 做配置检查 / 看修复建议`
- 第二轮补范围：`检查全集群 / 检查某个 workload / 检查某个 namespace`

如果用户原问题已经明确了动作，只缺范围，则直接进入第二级引导。

### 引导边界

引导智能体必须遵守以下边界：

- 只做澄清，不产出技术结论
- 不直接做配置检查
- 不直接给修复方案
- 不长篇解释“你应该怎么问”
- 每次只补一个关键信息

## K8s 主智能体职责

K8s 主智能体保留现有能力，但在工作台设计层面，需要明确其职责边界：

### 应该负责

- 查询资源列表和基本信息
- 执行配置检查
- 输出配置检查报告
- 基于已有上下文输出修复对比
- 输出 kubectl 修复命令
- 回答 K8s 配置相关概念问题

### 不应该负责

- 在入口阶段教学用户如何提问
- 对模糊问题进行多轮澄清
- 替代引导智能体做交互式补全

## 意图分类建议

工作台意图分类节点建议至少区分 5 类：

| 分类 | 典型输入 | 去向 |
|---|---|---|
| 查询 | 列出所有 deployment、有哪些 service | K8s 主智能体 |
| 配置检查 | 检查配置、扫描问题、看看有没有风险 | K8s 主智能体 |
| 修复 | 给修复建议、输出命令、修一下这个问题 | K8s 主智能体 |
| 问答 | 什么是 liveness probe、requests 怎么配 | K8s 主智能体 |
| 不清楚 | 帮我看看、看一下、修一下这个 | 引导智能体 |

第一版不需要把意图分类做得过细，重点是把“不清楚”识别出来并稳定兜底。

## 结果体验设计

### 查询结果

- 只给列表或资源信息
- 不主动扩展成配置检查

### 配置检查结果

- 输出完整检查报告
- 不默认直接进入修复

### 修复结果

- 输出修复对比
- 输出可执行 kubectl 命令

### 问答结果

- 直接回答概念
- 不触发检查流程

## 异常与兜底

### 1. 用户问题过于模糊

进入引导智能体，优先补动作。

### 2. 用户选择后仍不够明确

继续补范围，但仍然保持一次只问一个问题。

### 3. K8s 主智能体没有足够上下文执行修复

应回退为明确提示，例如：

- 当前缺少检查结果上下文
- 请先执行配置检查或指定目标

不应直接猜测并输出修复命令。

### 4. 用户其实只是想问概念问题

直接由 K8s 主智能体以问答模式处理，不进入检查链路。

## 为什么这是第一版最合适的方案

这套方案的优点是：

- **兼容现有资产**：不需要推翻已有 K8s 主智能体
- **工作台简单**：节点数量少，画布易维护
- **入口体验自然**：用户可直接自由提问
- **对模糊问题有兜底**：不会因为用户不会问而直接失败
- **后续可调**：入口问题只调引导智能体，业务问题只调 K8s 主智能体

相比“全部交给一个智能体自己处理”更稳定；相比“拆成多个业务智能体”更轻量。

## 验收标准

当工作台按本设计搭建后，应满足以下标准：

1. 用户可以通过一个输入框直接进入 K8s 工作流。
2. 明确问题不需要二次提问，直接得到结果。
3. 模糊问题会被最小化引导，而不是收到长篇教学。
4. 引导顺序固定为：**先补动作，再补范围**。
5. 工作台只保留一个 K8s 主智能体，不人为拆散查询、检查、修复。
6. 检查结果默认不自动跳修复，修复意图由用户明确表达后再进入。
7. 第一版工作台画布不依赖复杂右侧帮助面板、表单或向导页。
