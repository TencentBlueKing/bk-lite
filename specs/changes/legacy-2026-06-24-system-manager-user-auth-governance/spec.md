﻿# 系统管理用户与认证治理规则收敛设计

> 说明：本文档聚焦系统管理中“登录认证 + 用户同步”的规则收敛，只覆盖产品规则、接口语义、生命周期约束与分阶段落地方案，不单独展开视觉重构。

## 背景

当前系统管理里的“用户与认证”存在两类混在一起的问题：

- 一类是前端命名、列表文案、弹窗联动等体验问题
- 另一类是登录控制、同步冲突、删除语义、停用语义、根组织占用等后端治理问题

这两类问题优先级不同。本次设计先处理“规则优先”的部分，把高风险、不一致、会造成错误治理预期的问题先收敛。

结合当前仓库实现与既有 spec，当前存在以下关键问题：

1. `bk_lite_builtin` 内建登录实例在系统管理中存在，但它本质上只是平台账号密码登录链路的展示映射对象，当前产品语义仍容易被误解为普通可治理实例。
2. `init_login_settings` 对内建登录对象只补创建、不补修复，现状虽不影响真实账号密码登录，但文档尚未明确“只初始化、不自愈”的约束边界。
3. 用户同步遇到跨同步源用户冲突时，当前会整次失败，和“跳过冲突继续同步”的预期不一致。
4. 停用的用户同步源仍可通过入口触发“立即同步”，但异步执行时又被拒绝，前后端语义不一致。
5. ??????????????????????????????
6. 根组织名称当前只在真实 `Group` 已存在时才会校验冲突，导致“先创建多个源、后同步才冲突”的窗口。

## 设计目标

1. 明确登录认证与用户同步都属于平台级治理对象，不按组织私有资源建模。
2. 明确 `bk_lite_builtin` 是展示映射对象，不是账号密码登录的真实控制开关，也不是普通集成实例。
3. 让用户同步的创建、执行、停用、删除、冲突处理形成一致的生命周期语义。
4. 把“尽可能完成、诚实记账、危险操作显式拦截”作为用户同步的统一原则。
5. 继承现有集成中心 spec，不重复设计已收敛的 `available_instances` 规则。

## 非目标

- 不在本次设计中重做登录页或系统管理页视觉结构。
- 不把账号密码登录立即改造为强依赖 `LoginAuthBinding` 的强绑定架构。
- 不在本次设计中重做组织隔离模型。
- 不在本次设计中设计用户物理删除策略。
- 不在本次设计中为用户同步引入复杂状态机或新工作流引擎。

这套行为与业界主流一致，应明确为正式规则，而不是偶然实现细节。

## 设计原则

### 1. 展示映射对象与真实控制面分离

系统管理中用于表达、排序、文案和初始化的对象，不应被误认为实际运行时的唯一控制开关。尤其是内建登录对象，必须与真实密码登录链路解耦表达，并与普通可治理实例区分开。

### 2. 生命周期语义前后一致

入口允许执行的动作，异步执行阶段也必须允许；入口禁止的动作，异步执行阶段不应再靠兜底失败来表达。不能再出现“入口说成功发起，worker 说对象不可执行”的双重语义。

### 3. 优先软失效，不做高风险物理删除

对于用户、组织这类带审计和归属历史的对象，优先采用禁用、脱钩、清权限等软处理语义，避免同步流程直接破坏历史数据。

### 4. 冲突要可继续执行，但必须诚实记账

用户同步不应因为局部冲突导致整次任务无产出，但也不能把部分成功伪装成完全成功。系统必须提供可审计、可理解的冲突和跳过信息。

## 详细设计

## 一、登录认证规则设计

### 1. 平台账号密码登录继续按弱绑定处理

本阶段明确：

- 账号密码登录是平台基础认证能力
- 它不以 `bk_lite_builtin` 的存在与否作为前置条件
- 它也不以某个 `LoginAuthBinding.enabled` 字段作为唯一控制开关

因此，即使：

- 平台尚未初始化出内建登录实例
- 登录认证页里不存在对应内建 binding
- 内建对象发生数据漂移

账号密码登录链路仍可存在。

### 2. `bk_lite_builtin` 的正式定位

`bk_lite_builtin` 及其内建 `LoginAuthBinding` 定位为展示映射对象：

- 表达“平台账号密码登录”这一内建方式
- 参与系统管理中的展示、排序与初始化口径
- 不作为真实登录控制面的唯一开关
- 不作为普通集成实例或普通登录认证配置项参与运营治理

相应约束继续成立并应强化：

- 集成系统实例列表中隐藏内建实例
- 登录认证列表中保留该记录展示
- 内建 binding 的编辑、删除、启用/禁用操作一律禁止点击
- 内建 binding 默认展示为启用状态，但该状态仅用于说明，不表达真实登录开关
- 若初始化资源缺失，登录认证列表允许暂时不显示该记录，不为前端展示额外兜底

### 3. `available_instances(capability=login_auth)` 继续排除内建实例

登录认证页选择第三方登录方式时，`bk_lite_builtin` 不应作为普通集成系统出现在“可用集成系统”列表中。

原因：

- 它不是第三方登录接入实例
- 它不承担外部 provider 的接入职责
- 它本质上只是平台账号密码登录链路的展示映射
- 把它暴露在业务引用列表中会误导用户把“平台内建登录”理解成普通第三方登录配置

### 4. 初始化命令只负责缺失补齐

`init_login_settings` 在本阶段只做“缺了就补”，不做“漂了就修”。

建议保证以下对象在缺失时被初始化：

- 内建 `IntegrationInstance`
- 内建 `LoginAuthBinding`

目标语义：

- 缺对象时创建
- 已存在对象时不额外回写漂移字段
- 暂不引入自愈语义，避免在当前无实际应用场景时过早设计漂移治理

### 5. “关闭账号密码登录”不复用内建绑定字段

如果未来产品真的需要关闭平台账号密码登录，应设计为单独的平台策略位，而不是复用：

- 内建 instance 的 `enabled`
- 内建 binding 的 `enabled`

否则会继续制造“展示映射字段”和“真实认证开关字段”混用的问题。

## 二、用户同步规则设计

### 1. 根组织名采用“创建时预留”语义

当前问题在于：

- 创建同步源时，只在真实根组织 `Group(parent_id=0, name=...)` 已存在时才会报冲突
- 根组织真实对象又是在首次同步时才创建

这会造成：

- 多个同步源可以先成功创建
- 后续首次同步时才发现名字冲突
- 甚至把同名根组织归属覆盖到后一个同步源

本次建议改为：

- 根组织名在创建 `UserSyncSource` 时就被预留
- 同一时刻只能被一个有效同步源占用
- 预留不等价于提前创建真实 `Group`

规则上表现为：

- 创建时校验预留是否已被其他源占用
- 编辑时根组织名仍不可随意改写
- 删除或显式清理后，预留名才可释放

#### 兼容与落地约束

当前模型中只有：

- `UserSyncSource.root_group_name`
- 同步后真实落地的 `Group(parent_id=0, name=...)`

第一阶段默认不新增预留模型，也不引入新的数据库约束，而是在现有模型上先完成最小约束收敛。

建议落地约束如下：

1. 先以现有 `UserSyncSource.root_group_name` 作为预留语义来源，在创建时做重名校验，而不是先引入独立预留承载。
2. 历史数据扫描时，若发现：
   - 多个 source 已声明同名 `root_group_name`
   - 或同名真实根组织已分别指向不同 source
   - 则这些 source 视为“历史冲突数据”，第一阶段不自动修复、不强行改写。
3. 第一阶段只对“新建 source”与“无历史冲突的存量 source”启用创建期校验。
4. 对历史冲突数据：
   - 保持只读或受限编辑
   - 禁止继续创建新的同名 source
   - 通过后台治理或后续人工处理完成消歧后，再纳入统一规则
5. 只有当现有模型无法稳定承载上述约束时，才考虑引入最小模型变更；在此之前默认零 migration。

这样可以避免第一阶段一上来就把存量脏数据全部打成不可用状态，也避免在当前阶段过早扩大模型改造范围。

### 2. 用户同步执行采用“跳过冲突继续”语义

当同步遇到用户冲突，例如某用户已经归属于其他同步源时：

- 不再整次失败
- 将该用户记录为冲突/跳过项
- 继续处理其他用户和组织
- 最终 run 状态记为 `partial`

该规则适用于：

- `user.sync_source` 已归属于其他源
- 用户当前组织归属指向其他同步源产生的组织

目标语义是：

- 尽可能产出有效同步结果
- 冲突不能悄悄吞掉
- 冲突也不应导致整批同步完全失败

### 3. 停用同步源后，任何同步入口都禁止执行

`enabled=false` 的含义应统一为“该同步源不可执行”。

它同时约束：

- 列表页“立即同步”按钮
- 手动触发接口 `sync_now`
- 异步 worker 实际执行
- 定时任务触发

推荐行为：

- 前端按钮置灰或隐藏，并给出禁用原因
- 后端 `sync_now` 入口直接返回失败，不投递任务
- worker 执行逻辑继续保留最终兜底校验，但不再承担主要语义表达

#### 异步竞争语义

由于当前实现通过 Celery 异步执行，第一阶段必须明确“入队后状态变化”的处理规则：

- 若任务已入队，但执行前 source 被停用：
  - worker 不执行实际同步
  - 生成一条可审计 run，状态记为 `failed`
  - `summary` 明确为“source disabled before execution”或等价用户文案
- 若任务已入队，但执行前 source 被删除：
  - worker 不执行实际同步
  - 不尝试隐式恢复 source
  - 记一条失败审计记录，或在无法建立 run 归属时写稳定日志；实现时优先选择“保留失败 run”而不是静默丢弃
- 若任务已入队，但执行前 source 配置被修改：
  - 本阶段沿用“执行时读取最新 source”语义
  - 不在第一阶段引入配置快照运行机制

核心原则是：不能静默吞掉已被用户触发过的任务。

### 4. 用户离开同步范围时，不物理删除

当某用户曾属于同步源、但新一轮同步中已消失时，正式规则定义为：

- 保留用户主体记录
- 设置 `disabled=True`
- 清空该同步流程赋予的组织归属
- 保留历史操作、审计、任务、告警等历史关联

若用户后续重新出现在同一同步源中：

- 允许自动恢复启用状态
- 按最新同步结果重建组织归属

该语义表示“离开同步范围/离职自动停用”，不表示“删除用户”。

## 三、运行记录与状态语义

### 1. 用户同步 run 状态需要支持 `partial`

建议将 `partial` 视为一等状态，而不是前端临时文案：

- `success`：无冲突、无跳过、执行完成
- `partial`：执行完成，但存在冲突、跳过、部分未处理项
- `failed`：同步无法完成，未形成有效结果

### 2. run 摘要需诚实表达结果

run 记录建议至少体现：

- 同步成功用户数
- 同步成功组织数
- 因离开同步范围而被禁用的用户数
- 因冲突被跳过的用户数
- 冲突摘要或失败原因摘要

目标不是立即做复杂报表，而是让用户能回答：

- 这次到底完成了多少
- 跳过了多少
- 为什么跳过

## 四、分阶段落地方案

### 第一阶段：高风险后端语义收敛

优先处理会造成真实语义错误的问题：

1. 固化登录认证弱绑定口径
2. 收敛内建登录对象为展示映射语义，并补齐前端只读展示约束
3. 保持 `init_login_settings` 为缺失补齐，不引入漂移自愈
4. 统一 `enabled=false` 的同步源不可执行语义
5. 将用户冲突处理改为“跳过并记为 `partial`”
6. 为 run 记录补充冲突/跳过统计，优先复用现有 payload/summary
7. 在现有模型上实现根组织名创建期预留校验
8. 将同步源删除改为受控拒绝，而不是直接删除
9. ????????????????????????

第一阶段原则：

- 不重做大 UI
- 不扩展新的复杂后台流程
- 先修正错误治理语义和高风险数据行为
- 不能在没有最低限度治理出口的前提下，把存量脏数据永久卡死

### 第二阶段：治理承接与体验一致性
在第一阶段规则稳定后，再补剩余前端承接和运维辅助：

1. “立即同步”按钮按 `enabled` 禁用
2. 删除失败时展示明确原因
3. run 摘要文案改成用户可理解表达
4. 补充后台检查工具，查看：
   - 根组织名预留占用
   - 不可删除原因

说明：

- 登录认证列表中的内建项只读启用态、以及集成系统实例列表隐藏内建实例，属于第一阶段的最小前端承接
- 第二阶段补的是更完整的可视化承接与运营工具
- 第一阶段仍需提供最小可执行治理能力，至少让存量脏数据有修复路径

## 测试策略

### 登录认证

- 缺失内建 instance/binding 时，初始化命令可补齐
- 内建对象缺失时，初始化命令可补齐初始化资源
- `available_instances(capability=login_auth)` 继续排除 `bk_lite_builtin`
- 账号密码登录在内建对象缺失时仍可走原有链路

### 用户同步

- 停用同步源时，前端入口不可触发、后端接口直接拒绝、worker 不会收到合法任务
- 两个同步源遇到同一用户时，后执行源记为 `partial`，其他用户仍继续同步
- 同步结果中消失的用户会被禁用并清空组织归属，不会被物理删除
- 创建两个同名根组织的同步源时，第二个在创建阶段即被拒绝
- 有落地数据的同步源删除时直接拒绝，而不是留下孤儿组织和用户归属
- 历史已存在重复 `root_group_name` 的 source 在第一阶段有明确定义：不自动修复，但不会允许继续扩散
- 任务入队后若 source 被停用或删除，不会静默吞掉，必须留下失败审计
- 用户从禁用状态重新出现在同一 source 中时，会被自动恢复启用
- 执行清理后，根组织名预留可被正确释放
- 历史孤儿 `User` / `Group` 记录有最小治理与测试覆盖

## 风险与缓解
### 风险 1：弱绑定口径被误解为“永远不需要强绑定”

本次只是按现状与兼容性选择弱绑定，不代表长期架构终态。

缓解：

- 在文档和实现中明确这是“当前阶段治理口径”
- 后续若产品明确要求，再单独设计强绑定迁移方案

### 风险 2：冲突改为 `partial` 后，调用方误把其当成功

缓解：

- 将 `partial` 作为明确状态，而不是落在普通成功文案里
- 让前端记录页展示冲突/跳过摘要

### 风险 3：删除禁止后，用户感觉“删不掉”

缓解：

- 返回可操作原因
- 在第二阶段补充清理或解除依赖的治理承接

### 风险 4：根组织名预留引入新的占用状态

缓解：

- 预留必须有清晰释放条件
- 提供后台检查或修复入口
- 不把预留直接混入真实 `Group` 生成逻辑


1. `bk_lite_builtin` 被明确视为展示映射对象，而非账号密码登录的真实运行开关。
2. `init_login_settings` 仅负责在内建登录对象缺失时补齐初始化资源，不承担漂移修复。
3. 集成系统实例列表中不展示内建实例；登录认证列表中该内建项默认显示为只读启用态，且编辑、删除、启用/禁用操作均不可点击。
4. 停用的用户同步源不能再通过任何正常入口触发同步。
5. 用户同步遇到冲突用户时，不再整次失败，而是跳过冲突并记为 `partial`。
6. 用户离开同步范围时被禁用并解除同步归属，不会被物理删除。
7. 同名根组织在创建同步源阶段即被拦截，而不是等首次同步才报错。
8. 存在落地数据的同步源不能直接删除，系统会返回明确拒绝原因。

## 结论

本次推荐采用“规则收口 + 分阶段落地”方案：

- 对登录认证，先承认并固化当前弱绑定现实，把内建登录对象定位为展示映射对象，并只保留缺失补齐初始化能力。
- 对用户同步，统一为“创建时预留、执行时诚实、停用时一致、删除时保守”的生命周期规则。

这能在不推翻现有架构的前提下，先修正高风险治理语义，为后续 UI 承接和更强控制面演进建立稳定基础。
