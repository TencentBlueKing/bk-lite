# 监控实例组织模型 — 现实校准后的简化方案

> Status: draft
> 2026-07-22
> 原始需求：组织 A、B 都要监控同一对象，实例是否应按组织分离？
> 经现实校准：真实冲突只发生在无业务归属的拨测类对象上。

---

## 1. 产品判断（简化版）

物理设备总有唯一运维归属方，其他组织的诉求是**看数据**——现有「全局唯一实例 + 组织授权」已覆盖，**不动**。

真问题只在网站拨测这类无归属对象：实例身份被定义成了「拨测目标（URL）」，导致组织间撞车。

**解法**：把拨测类对象的实例身份改为「拨测任务」——一个小尺寸后端变更，而非产品模型重构。

---

## 2. 核心定义：两类监控对象，两种实例语义

| | 资产型对象（主机 / 交换机 / 防火墙…） | 任务型对象（网站拨测 / Ping / TCP） |
|---|---|---|
| 实例代表什么 | 一台真实存在的设备 | 一次「我去探测某目标」的行为 |
| 唯一性归谁 | 物理世界 · 全局唯一 | 发起方 · 组织内唯一，可并存 |
| 组织间关系 | 授权共享同一份数据 | 各建各的任务，互不相干 |
| 重复采集的代价 | SNMP/Agent 双倍负载，撞下发红线 | 多一个 HTTP 请求，无害 |
| 竞品对应 | Zabbix Host · Datadog Host | Datadog Synthetics Test |
| 本方案改动 | 无（现状已覆盖） | 实例身份从「目标」改为「任务」 |

**一句话**：资产型对象的实例是设备，归物理世界；任务型对象的实例是任务，归发起组织。

## 3. 改动清单

| 概览 | |
|---|---|
| 资产型对象改动 | **0**（现状模型不动） |
| 模型改动 | **1**：`MonitorObject` 增加实例身份语义 |
| 泛化功能延后 | **3**：分组层 / 接入向导 / 参数仲裁 |

| 改动 | 内容 | 性质 |
|---|---|---|
| 实例身份语义 | `MonitorObject` 增加标记：资产级（现状）/ 任务级；Web/Ping/TCP 声明为任务级 | 模型字段 + 接入逻辑 |
| 任务级 `instance_id` | 接入时生成带组织/任务维度的 `instance_id`，注入 Telegraf tag；存量实例不迁移 | 接入逻辑，时序链路无改动 |
| 名称唯一性 | 任务级对象的实例名称校验从「对象内全局唯一」收敛为「组织内唯一」 | **唯一真正的产品规则调整** |
| 列表加「拨测目标」列 | 实例名代表任务，URL 独立成列（可选） | 前端展示 |
| 界面术语 | 拨测实例是否改叫「拨测任务」（可选，见决策点） | 文案 |

## 4. 体验说明

### 4.1 场景

组织 A 已拨测 `www.example.com`（60s · 华东节点）。组织 B 也要拨测同一个网站（30s · 华南节点）。

### 4.2 现状行为（实例身份 = 拨测目标，URL 全局唯一）

1. 组织 B 填写 URL、间隔 30s、华南拨测节点，提交接入
2. 系统按 URL 找到组织 A 的实例，**静默复用**（组织 B 无感知、无选择）
3. 追加同类型采集配置撞唯一约束（实例 + collector + collect_type + config_type），接入失败或被合并进组织 A 的配置
4. 即使拿到可见性：实例名是组织 A 起的、间隔是组织 A 定的，组织 B 改配置会反过来影响组织 A

**痛点**：组织 B 对同一网站「按自己的方式拨测」这件事，当前模型做不到——这才是本需求的现实来源，而不是物理设备的归属问题。

### 4.3 新方案（实例身份 = 拨测任务，组织内唯一）

1. 组织 B 填写 URL、间隔 30s、华南拨测节点，提交接入
2. **直接创建**组织 B 自己的拨测实例（`instance_id` 带组织/任务维度），不检测、不复用、不提示
3. 自己的采集配置、自己的告警策略与通知，时序标签跟任务走，与组织 A 完全平行
4. 组织 A 全程无感；多拨一次只是多一个 HTTP 请求，对目标网站无害，无需仲裁

**用户感知**：不是「学会一个新概念」，而是「少踩一个莫名其妙的坑」——接入就是接入，没有撞车环节。

### 4.4 实例列表示意

同一个 `www.example.com` 在两个组织各是一个独立拨测实例；建议列表新增「拨测目标」列。

**组织 A**

| 实例名称 | 拨测目标 | 间隔 | 拨测节点 | 状态 |
|---|---|---|---|---|
| 官网拨测-华东 | www.example.com | 60s | 华东-node01 | 正常 |
| 内部门户可用性 | portal.corp.local | 60s | 华东-node01 | 正常 |

**组织 B**

| 实例名称 | 拨测目标 | 间隔 | 拨测节点 | 状态 |
|---|---|---|---|---|
| 官网监测 | www.example.com | 30s | 华南-node02 | 正常 |
| 上海站点入口拨测 | sh.example.com | 30s | 华南-node02 | 告警中 |

资产型对象（主机、交换机）不受影响：仍是全局唯一实例，其他组织走 `MonitorInstanceOrganization` 授权共享同一份数据。

## 5. 与此前泛化方案的对比（做什么 / 不做什么）

最初曾把需求泛化为「所有对象的组织自治」（通用分组层、接入冲突向导、采集参数仲裁）。经现实校准后收敛如下：

| 此前泛化方案 A 的内容 | 简化版处理 | 原因 |
|---|---|---|
| 通用「实例分组/标签」新实体 | 不做（Later） | 分组诉求未被现实场景证实，等真实需求出现再评估 |
| 接入冲突向导 + 申请可见性流程 | 不做 | 物理设备由归属方接入、他组织走授权，冲突场景现实中不存在 |
| 采集参数仲裁规则 | 不做 | 资产型无冲突场景；任务型各自独立配置，无需仲裁 |
| 拨测类实例身份改为任务级 | **做（唯一改动）** | 这是需求的真实来源，改动小、无红线风险、竞品同型 |

## 6. 变更方案初稿（技术草案 · 待评审）

### 6.1 前提假设

按下方决策点的建议项拟定：

- Web / Ping / TCP 全部标记任务级
- 存量不迁移
- 名称组织内唯一
- 界面术语暂不改

任一决策变化会影响本节细节。代码事实来自 2026-07-22 的链路调研。

### 6.2 目标 / 非目标

**本次要做到**

- 拨测类对象的实例身份改为任务级：接入即创建本组织实例，组织内唯一
- 同一目标（URL/host）可被多组织并行拨测，配置、告警、时序完全独立
- 存量拨测实例行为不变，历史曲线连续

**本次明确不做**

- 不改资产型对象的身份与组织授权模型（Host/网络设备 identity adapter 不动）
- 不引入分组/标签新实体，不做接入向导与参数仲裁
- 不迁移存量数据，不改 VM 查询模板与告警扫描链路

### 6.3 核心机制

`MonitorObject` 新增「实例身份语义」字段；任务级对象在后端创建时派生带组织维度的 ID。

| 环节 | 现状 | 变更后（仅任务级对象） |
|---|---|---|
| 前端生成 | 模板 `web_<url>` / `ping_<host>` / `tcp_<host>_<port>` 做 FNV 哈希（16 位） | 不变（前端零改动） |
| 后端创建 | `str((hash,))` 作为全局主键，同 URL 跨组织撞车 | 任务级对象追加组织维度：`str((hash + '-g<组织ID>',))`，组织内唯一 |
| Telegraf tag | `instance_id = hash` | `instance_id = hash-g<组织ID>`（随派生值下发） |
| VM 时序 | 同 URL 全局一条序列 | 每任务一条序列，天然隔离 |
| 自动发现 SyncInstance | 按 VM 标签重建全局实例 | 按标签重建任务实例，代码无需修改 |

派生逻辑放在后端创建分支一处（与 Host/网络设备 identity adapter 同层、平行的新分支）。主键、Telegraf tag、VM 标签、自动发现全链路自然一致——NATS、告警扫描、仪表盘都按主键/标签工作，均无需修改。

### 6.4 分阶段落地

按仓库红线：先写测试，TDD 红-绿-重构。

| 阶段 | 内容 | 主要触点 |
|---|---|---|
| 1 模型与元数据 | 新增 `instance_identity_scope` 字段（默认 `asset`）+ migration；web/ping/tcp 的 `metrics.json` 声明 `task`；`plugin_init` 更新分支回写新字段 | `models/monitor_object.py` · migrations · `services/plugin.py` · `support-files/plugins/Telegraf/{web,ping,tcp}` |
| 2 创建链路 | 任务级 ID 派生分支；任务级对象名称唯一性校验收敛到组织内（并补齐自动接入路径缺失的名称校验） | `services/node_mgmt.py` · `services/monitor_object.py` |
| 3 前端展示 | 拨测实例列表增加「拨测目标」列（可选项，按决策点） | `web/src/app/monitor/(pages)/integration/asset` |
| 4 收口 | 规格决策沉淀 + 全量测试 + 模块门禁（`cd server && make test`） | 本 change spec · `server/apps/monitor/tests` |

### 6.5 兼容与回滚

- 字段默认 `asset`，所有对象行为不变
- `task` 标记只影响新接入，存量拨测实例保持旧 ID 与历史曲线
- **回滚** = 把对象标记改回 `asset`：已创建的任务实例保留且继续正常采集查询，无数据损失

### 7.3 风险矩阵

| 级别 | 风险 | 缓解 |
|---|---|---|
| **高** | 现状隐患：复用实例时组织关联被整体清空重写（`services/node_mgmt.py` 复用分支），今天组织 B 接入同 URL 会「抢走」组织 A 的实例归属 | 任务化后拨测不再跨组织复用，该路径自然规避；但此逻辑对资产型对象仍生效，本次必须加回归测试圈住行为 |
| 中 | 新旧并存：存量拨测实例（旧全局 ID）与新任务实例同时监控同一 URL，出现两条实例、两份序列 | 属预期语义（各组织自己的任务）；列表「拨测目标」列澄清；存量实例仍归原组织 |
| 中 | 编辑/重存流程：前端编辑已接入拨测时会重算 hash，若更新路径按重算值定位可能产生第二实例 | 更新必须按已存 PK 定位（不重派生）；针对编辑流程补集成测试 |
| 中 | SyncInstance 会按 VM 标签复活软删实例，且拨测 Telegraf tag 无 `organization_id`，同步重建的实例无组织关联 | 现状已如此，非本次引入；可后续在 tag 注入组织维度作为独立改进 |
| 低 | 名称收敛组织内唯一后，跨组织重名（两个组织都叫「官网拨测」） | 列表本就按组织过滤，展示歧义低；全局视角（管理员）加组织列即可 |

## 8. 待讨论决策点

| 决策点 | 背景 | 选项 | 建议 |
|---|---|---|---|
| 任务级身份适用的对象范围 | 无归属的拨测类对象都有同样问题；建议在 `MonitorObject` 上加身份语义标记，按对象声明 | ① Web / Ping / TCP 全部标记为任务级 · ② 仅 Web 先行，Ping/TCP 观察 · ③ 由插件包自行声明 | **①** |
| 界面术语 | 实例名本来就是用户自定义的，多数用户并未意识到「实例 = URL」的内部约束 | ① 保持「实例」不改，仅列表加「拨测目标」列 · ② 拨测类对象界面改叫「拨测任务」 | **①** |
| 存量拨测实例 | 已有实例的 `instance_id` 已写入时序标签，迁移会造成历史数据断裂 | ① 不迁移，新语义仅对新接入生效 · ② 提供一次性迁移工具（历史曲线断裂需告知） | **①** |
| 任务级对象的名称唯一性范围 | 当前名称是同对象下全局唯一（应用层校验），两个组织都叫「官网拨测」会冲突 | ① 改为组织内唯一（与任务级语义一致） · ② 保持全局唯一，冲突时自动加后缀 | **①** |

## 附录 A · 讨论历程：此前的泛化分析（已被现实校准取代）

最初把需求泛化为「所有对象的组织自治」，比较了三个方案；后经现实校准（只有拨测类对象真实发生冲突）收敛为上文简化版。保留此段供追溯。

### A.1 竞品模型对照

| 产品 | 物理对象唯一性 | 分组机制 | 两个团队监控同一设备时 |
|---|---|---|---|
| Zabbix | Host 全局唯一，采集一份 | Host Group + User Group 授权 | 同一 Host 挂两个 Host Group，不复制 |
| Datadog | Org 内 Host 唯一（Org 是租户边界） | Tag / Team + Restriction Queries | 同一 Host 打两个团队 Tag，各看各的视图 |
| BK-Lite | `MonitorInstance` 主键 = 物理身份 | `MonitorInstanceOrganization` 授权（分组与权限耦合） | 实例授权给两个组织，接入静默复用 |

### A.2 「按组织复制实例」被否决的四个 Blocker

（对**资产型**对象依然成立）

| Blocker | 要点 |
|---|---|
| 采集重复 | SNMP/Agent 负载成倍，主机 Agent 无法装两份，撞下发红线 |
| 时序断裂 | `instance_id` 标签即物理身份，全链路需注入组织维度，存储翻倍、历史断代 |
| 配置冲突 | 物理设备只有一份生效配置，复制实例只是把「谁说了算」藏起来 |
| 告警风暴 | 同一物理故障产生 N 组织 × N 条告警，收敛与响应退化 |

**注意**：这四条对任务型（拨测）对象不成立——拨测重复无害、时序按任务分开、配置本就该独立、告警各归其主，这正是简化方案成立的原因。
