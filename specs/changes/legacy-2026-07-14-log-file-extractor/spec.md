## 背景

BK-Lite 日志中心当前以 VictoriaLogs 为后端存储,链路分两层:

- **采集层(节点侧)**:6 类采集器(Filebeat / Vector / Packetbeat / Auditbeat / Snmptrapd / Winlogbeat)部署在被监控节点,采集后通过 NATS 转发
- **中转层(云区域容器节点)**:**bklite 每个云区域自带一个 NATS Vector,所有日志数据写入 VictoriaLogs 都会过这一层** —— 这是本次提取的天然执行位置

当前各采集器在采集阶段把每条日志的原始文本写入 `_msg` 字段,**不进行结构化提取**。
结果导致:用户搜索时只能基于 `_msg` 做关键字全文匹配,**无法基于字段做聚合、TopN 统计、告警条件**。

参考 Graylog 提供的 8 种提取器类型(Grok / JSON / 正则 / 正则替换 / 分割 / 子串捕获 / 复制输入 / 数据映射),
Vector 底层 VRL 全部具备等价能力。本次目标是把提取能力产品化、UI 可视化,作用于**所有经中转的采集类型**。

## 已确认事实

1. 采集链路:`agents/fusion-collector` → Vector(文件 / docker / k8s / syslog)→ NATS `vector` 主题 → VictoriaLogs。
2. 采集器配置模板目录:`server/apps/log/support-files/plugins/Vector/file/file.child.toml.j2`(文件采集唯一)。
3. 实例元数据模型:`server/apps/log/models/instance.py` 中 `CollectInstance`(实例级),与 `CollectType`(采集器类型级)区分。
4. 现有 6 类采集器:Vector 4 类 + Filebeat 9 类 + Packetbeat / Auditbeat / Snmptrapd / Winlogbeat 各 1 类(见 `server/apps/log/constants/plugin.py`)。
5. 集成页路径:`web/src/app/log/(pages)/integration/`,下含 `list` / `grouping` / `receive` 等子页面。
6. 搜索页路径:`web/src/app/log/(pages)/search/`,已支持基于字段的 `stats by` 查询。
7. **中转层**:`bklite` 每个云区域的容器节点上自带一个 NATS Vector,接收所有采集器的日志,统一写入 VictoriaLogs。本次提取在该中转层执行,覆盖所有经中转的采集类型。

## 问题定义

文件采集用户拿到原始日志后,**没有任何自助工具**把文本里的关键信息(状态码、用户 ID、请求路径、耗时等)抽成结构化字段。
下游能力全部受限:

- 告警策略(`tasks/services/policy_scan.py:22`)无法基于字段聚合,只能基于关键字匹配。
- 仪表盘(`analysis/dashBoard/widgets/`)无法展示"按接口路径的报错 TopN"等业务化视图。
- 搜索页(`search/page.tsx`)的"按字段过滤"功能没有可用的字段可选。

**根因不是 VLogs 不支持 stream field**,而是采集阶段没有把字段抽出来,自然落不进 stream field。

## 目标

1. 在「日志集成 → 文件采集实例」操作栏提供「日志提取」入口,点击后用抽屉展示该实例的提取器列表。
2. 支持 6 种提取器类型(详见 7.1),用户通过可视化表单配置,不接触底层配置。
3. 提取器按用户定义的顺序执行;支持在列表里调整顺序。
4. 配置保存后,Vector 二进制需包含全量 Grok 模式库(本版本 UI 不暴露 Grok,只做能力预备)。
5. 支持从搜索页反向新建:对 `_msg` 文本选中触发新建提取器弹窗,预填源字段与样本。
6. 用户在配置过程中能加载 VictoriaLogs 中该实例的最近样本进行实时测试,所见即所得。

## 非目标(本版本明确不做)

1. **其他采集类型**(docker / syslog / k8s / filebeat 系列 / packetbeat / auditbeat / snmptrapd / winlogbeat)的提取器支持。
2. AI 自动推断日志 schema。
3. 历史数据回填(已存在的 `_msg` 字符串不会被重新提取)。
4. 用户直接编辑底层 VRL / Vector 配置。
5. Grok 模式的可视化选择器(Grok 类型在配置 UI 中不暴露)。
6. 数据映射(Lookup Table)类型的提取器。
7. 跨日志源关联分析。
8. 单条规则的「软启停开关」(本期不提供 enable 字段,只能通过删除/编辑绕过)。
9. 提取规则的"模板市场"或社区分享。
10. 在搜索页反向新建时,允许用户**修改 sample 后再保存**以外的更复杂交互(如自动建议类型)。
11. 批量导入时,除"全量覆盖"外的合并策略(不提供"按规则名合并""追加到末尾""按序号插入"等模式)。
12. 批量导入的进度条 / 取消中途中断能力。
13. 批量导入的事务一致性(部分失败回滚)。
14. 中转 vector 的控制链实现(配置怎么推到中转、中转怎么拉 extract_rules、重载策略、HA / 多实例负载均衡)—— 由开发调研,本规格只做产品层设计。

## 关键设计决策

### 7.1 提取器配置项(用户需填的字段)

**通用字段(所有类型都填)**:

| 字段 | 必填 | 含义 | 备注 |
|---|---|---|---|
| 日志实例 | - | (上下文显示,不存为规则字段) | 弹窗顶部展示"当前实例: <name>",让用户明确操作对象 |
| 规则名 | ✓ | 在抽屉列表里显示 | 实例内唯一,用户可读;**UI 提供默认值**(如 `rule_1` / `extract_http_status` 之类按类型与序号生成),用户可改 |
| 类型 | ✓ | 6 选 1,显示顺序固定 | UI 下拉顺序:复制输入 → 分隔提取 → 键值提取 → 正则提取 → 正则替换 → Json |
| 源字段 | ✓ | 从哪个字段读 | 默认 `message`,可选 VLogs 已落库字段 |
| 新字段 | ✓ | 写入哪个 stream field | 各类型见下表;**字段名由"目标字段"统一改名为"新字段"** |
| 删除旧字段 | ✗ | 提取成功后是否删除源字段 | 默认 `否`;适用所有 6 种;**边界**:源字段为系统保留字段(如 `_msg` / `host` / `_time`)时,UI 禁用此勾选 |

**类型展示顺序与显示名(UI 固定)**:

| 顺序 | 显示名 | 英文枚举 | 典型场景 |
|---|---|---|---|
| 1 | 复制输入 | copy | 字段重命名、加标签 |
| 2 | 分隔提取 | split | CSV / 固定分隔符按位取段 |
| 3 | 键值提取 | kv | Java/Go 应用 `k=v k=v` 日志 |
| 4 | 正则提取 | regex | 自由格式日志提取多字段 |
| 5 | 正则替换 | regex_replace | 敏感信息脱敏 |
| 6 | Json | json | 整行 JSON 展平 |

#### 复制输入(copy)

| 字段 | 必填 | 含义 | 备注 |
|---|---|---|---|
| 默认值(可选) | ✗ | 源字段缺失时使用 | 留空 = 源字段为 null 时目标也为 null |

#### 分隔提取(split)

| 字段 | 必填 | 含义 | 备注 |
|---|---|---|---|
| 分隔符 | ✓ | 切割依据(如 `,` / `\|` / 空白) | |
| 索引 | ✓ | 取第 N 段,0-based | UI 给范围提示,失败兜底不丢消息 |

#### 键值提取(kv)

| 字段 | 必填 | 含义 | 备注 |
|---|---|---|---|
| 键值分隔符 | ✗ | k 与 v 之间的分隔符 | 默认 `=` |
| 字段分隔符 | ✗ | 多对 kv 之间的分隔符 | 默认 空格 |
| key 白名单 | ✗ | 只提取这些 key,逗号分隔 | 留空 = 全部 |

#### 正则提取(regex)

| 字段 | 必填 | 含义 | 备注 |
|---|---|---|---|
| 正则表达式 | ✓ | 命名捕获语法 `(?P<name>...)` | UI 给"命名捕获语法"提示链接 |

#### 正则替换(regex_replace,典型场景:脱敏)

| 字段 | 必填 | 含义 | 备注 |
|---|---|---|---|
| 正则表达式 | ✓ | 匹配要处理的内容 | |
| 替换内容 | ✓ | 替换成什么 | 支持 `{{name}}` 引用命名捕获 |
| 新字段 | ✗ | 写入哪个 stream field | **留空 = 默认行为:覆盖源字段**;填了新字段名 = 写新字段(源字段不变) |

#### Json(json,整行 JSON 展平)

| 字段 | 必填 | 含义 | 备注 |
|---|---|---|---|
| (无其他必填) | - | - | 整行解析,无 pattern 可填 |

**类型相关特别说明**:
- **新字段** 字段在所有 6 种类型中都是"写入目标",仅 regex_replace 在留空时有特殊默认行为(覆盖源字段)。
- **删除旧字段** 勾选后:copy 等价于"重命名";其他类型等价于"提取后丢弃源字段"。
- **regex_replace + 删除旧字段 + 新字段=源字段**:三连组合时禁止(会把结果也删掉),UI 实时校验阻止。
- **regex_replace + 删除旧字段 + 新字段≠源字段**:允许(脱敏后单独存到新字段,源字段被删)。
- regex 的 `(?P<name>...)` 一个正则可同时抽多个字段,**一条规则 → 多个输出字段**。
- kv / json 一条规则也是多个输出字段(对象 merge 到顶层)。
- split / copy / regex_replace 单条规则只输出一个字段(新字段一个)。

### 7.2 部署约束:Vector Grok 模式全量打包

- **约束**:节点 Vector 进程的二进制或容器镜像需包含全量 Grok 模式库(VRL `parse_grok` 函数能识别 ~120 个内置 pattern)。
- **预期实现方式**:由后端开发调研当前节点 Vector 部署形态(独立二进制 / 容器 / sidecar),确认默认行为是否已包含。若未包含,由部署侧补充打包配置。
- **本版本 UI 不暴露 Grok**,但底层能力预备好,后续要做"模式选择器"时不需要再改部署。

### 7.3 提取执行层(中转 vector)

**执行位置**:bklite 每个云区域自带的 NATS Vector(**中转层**),**不在节点侧采集器**。

**两个核心理由**:
- **性能隔离**:复杂 VRL 提取(正则、JSON 展平、grok 模式)吃 CPU,跑在节点上会拖累同机的业务服务。中转层是专门的"提取服务",抗压。
- **范围覆盖**:中转是所有日志写入 VLogs 的统一入口,一次支持所有 6 类采集器;若放节点侧,要每种单独适配采集器模板,工作量巨大。

**用户视角无感知**:用户仍按"采集实例 → 日志提取 → 配置规则"的流程操作;
**配置存哪、Vector 模板怎么改、下发链路如何同步**等控制链实现细节**由开发调研,不在本规格范围**。

### 7.4 失败兜底的期望行为

| 场景 | 期望行为 |
|---|---|
| 单条规则解析失败 | **不丢整条日志**;原 `_msg` 保留;失败信息累加到固定的错误字段(命名 `ext_parse_errors` 之类) |
| 整批规则全部失败 | 同上;不阻塞后续采集 |
| 正则表达式非法 | 配置保存前被拒绝;UI 提示具体错误位置 |
| 源字段不存在(比如依赖前一条规则的输出) | 当前规则自动跳过;错误字段记录"源字段缺失" |
| 两条规则写同一新字段 | 后写入覆盖前写入;UI 配置时实时校验提示 |
| 单实例规则数量过多(后端开发定阈值,建议 20) | 后端在保存时给出提示;不影响功能 |

## 用户旅程

### 8.1 入口

路径:`web/src/app/log/(pages)/integration/` 下的文件采集实例列表

- 每个文件采集实例的操作栏增加「日志提取」按钮(其他采集类型实例不显示此按钮)
- 点击「日志提取」→ 右侧抽屉滑出,展示该实例的提取器列表

### 8.2 抽屉:提取器列表

抽屉内容:
- 标题:`<实例名> · 日志提取器`
- 工具栏(右上角,从左到右):「导出」「导入」「+ 新建提取器」三个按钮
- 列表:每行展示一条规则
  - 序号(可拖拽 / 上下箭头调序)
  - 规则名
  - 类型(用 icon + 文字)
  - 源字段 → 新字段(简写)
  - 编辑按钮 / 删除按钮(行末 hover 出现)
- 空状态:提示「还没有提取器,点击右上角新建」+ 一个引导「先去搜索页看看有没有需要的字段」

**「导出」按钮**:把当前实例的提取器列表下载为 JSON 文件。文件名包含实例名 + 时间戳。

**「导入」按钮**:打开"批量导入"弹窗(详见 8.7),用于把一份 JSON 提取器配置批量应用到多个文件采集实例。

### 8.3 新增/编辑弹窗

点击「+ 新建」或列表行「编辑」→ 在抽屉上叠加 Modal 弹窗。

弹窗结构:
- 顶部:规则名输入框
- 中部:类型下拉(6 选 1)
- 下部:**根据所选类型动态渲染配置项表单**(见 7.1 各类型字段表)
- 底部:工具栏
  - 「加载样本」按钮(从 VLogs 拉历史)
  - 「运行测试」按钮(对当前样本 + 当前规则做解析预览)
  - 「取消」「保存」按钮

**类型切换时**:已填的字段如果对当前类型无意义,提示用户"切换类型会清空部分字段"。

**保存校验**:
- 规则名实例内唯一
- 必填项非空
- 正则类型规则:正则表达式需可编译

### 8.4 测试功能(在新增/编辑弹窗内完成)

**历史日志浏览区(弹窗内嵌)**:
- 「加载日志」按钮:后端从 VictoriaLogs 拉取该实例最近的 `_msg` 历史日志(后端定阈值,建议 50 条 + 分页),展示在弹窗底部"日志浏览区"
- 浏览区内每条日志可展开查看完整内容,并可点击"用此条做测试"→ 选中作为当前测试样本
- 日志不足时:提示「暂无历史日志,请先确保该实例有日志采集」
- **该区域既用于"查看已有日志内容",也用于"选择测试样本"**,二合一

**运行测试**:
- 用户在浏览区选中一条样本 → 点「运行测试」
- 后端对"当前规则 + 该样本"做解析模拟,返回结果:
  - 解析出的字段列表(字段名 + 值 + 类型)
  - 失败时:具体哪条规则失败 + 失败原因
  - 错误信息(若有)
- 结果在浏览区右侧"测试结果区"显示,可视、可对比

**重要**:测试功能是**纯预览**,不写入 VictoriaLogs,不影响实际采集。

### 8.5 排序

- 列表行序号列支持拖拽调序(PC 端)
- 同时提供上下箭头按钮(移动端 / 拖拽不便场景)
- 顺序变更即时保存(无需点"保存列表"按钮)
- 顺序即执行顺序:**前一条规则输出的字段,后一条规则可作为源字段引用**

### 8.6 反向新建

**单入口**:`_msg` 详情区文本选择。

- 位置:搜索结果每条日志的 `_msg` 详情区
- 交互:用户用鼠标选中一段文本 → 浮出气泡「对选中文本新建提取器」
- 点击气泡:
  - 弹窗打开
  - 类型**未预选**(让用户根据文本特征选:看着像 JSON → json;像 k=v → kv;否则 regex)
  - 源字段预填 = `message`
  - 测试区预填一条样本 = 选中那段文本(允许用户编辑修改)

**本版本不提供的入口(明确不实现)**:
- ~~VLogs 字段内容(单元格值)点击~~
- ~~字段名点击~~

### 8.7 批量导出 / 导入

**使用场景**:运维在某个文件采集实例上配好一套提取策略,需要**把同一套策略批量应用到其他 N 个文件采集实例**(比如 100 台机器采同样的应用日志)。

**导出**:
- 抽屉工具栏「导出」按钮 → 后端把当前实例的 `extract_rules` 列表(按当前顺序)生成 JSON 文件,前端触发浏览器下载
- 文件名建议:`<实例名>_extractors_<时间戳>.json`
- JSON 内容:每条规则的完整字段(规则名 / 类型 / 源字段 / 新字段 / 类型相关配置 / 启用状态等),不包含"日志实例"上下文(因为导入时可应用到任意实例)

**导入**:
- 抽屉工具栏「导入」按钮 → 打开"批量导入"弹窗
- 弹窗流程:
  1. **选文件**:用户选一个之前导出的 JSON 文件
  2. **预览**:弹窗显示 JSON 里的提取器数量 + 规则名清单(用户确认是不是自己导出的)
  3. **选目标实例**:多选下拉(列出当前用户有权限的文件采集实例,过滤掉已选中的当前实例)
  4. **应用策略**:本次 MVP 仅支持"全量覆盖" —— 把目标实例的现有 `extract_rules` **完全替换**为导入的内容
  5. **点「应用」**:后端对每个目标实例依次下发,完成后弹窗显示结果(成功 N 个 / 失败 M 个 + 失败原因)
- 导入后,各目标实例的提取器列表立即更新,Vector 配置按各实例的 `update_interval` 节奏生效

**应用结果反馈**:
- 成功:目标实例 + 应用的规则数量
- 失败:目标实例 + 失败原因(权限不足 / Vector 配置渲染失败 / NATS 推送失败 等)
- **MVP 不做事务回滚**:部分成功部分失败时,成功的保留,失败的不动,弹窗明确告诉用户结果

## 验收标准

### A. 配置与持久化

1. 在「日志集成 → 文件采集」任一实例,操作栏可见「日志提取」按钮;非文件采集实例不显示。
2. 点击「日志提取」,抽屉正常打开,展示该实例当前的提取器列表(空实例见空状态文案)。
3. 在抽屉内可成功新增一条 regex 规则的提取器,填写规则名、源字段、新字段、正则,保存后抽屉列表立即出现新规则,刷新页面后仍在。
4. 编辑、删除、调序操作即时生效,刷新后保持。
5. 6 种类型各能成功新增至少一条规则,抽屉列表显示类型 icon 与字段简写。
6. **(范围覆盖)** 在某个**非文件采集**实例(例:某个 filebeat apache 采集实例)上配置一条提取规则,该实例的日志同样能按规则提取到新字段。

### B. 测试功能

6. 新增/编辑弹窗内,「加载样本」能拉取该实例最近 5 条样本(无样本时显示明确提示)。
7. 选一条样本 + 一条 regex 规则,「运行测试」返回解析结果(JSON 字段名 / 值 / 类型),与正则命名捕获一致。
8. regex_replace、json、kv、split、copy 各类型至少一条规则能跑通测试,失败场景能看到具体错误。
9. 错误正则表达式在保存时被拒绝,UI 提示具体错误位置。

### C. 反向新建

10. 搜索详情区用鼠标选中一段 `_msg` 文本 → 浮气泡可见,点击后弹窗打开,测试区预填选中文本。
11. 搜索详情区用鼠标选中一段 `_msg` 文本 → 浮气泡可见,点击后弹窗打开,测试区预填选中文本。

### D. 失败兜底

12. 一条故意写错的 regex 规则保存后,实际采集的日志 `_msg` 仍保留,固定错误字段(命名 `ext_parse_errors`)记录失败信息。
13. 两条规则写同一新字段,后保存的覆盖先保存的,UI 实时提示冲突。

### E. Vector 部署

14. 节点 Vector 进程 / 容器 / sidecar 二进制包含全量 Grok 模式库(由部署侧验证)。

### F. 批量导出 / 导入

15. 实例配置 N 条提取器后,点「导出」能下载到 JSON 文件,内容包含全部规则字段且能再次导入。
16. 抽屉「导入」按钮:选 JSON 文件 → 预览规则列表 → 选 2 个以上目标实例 → 点「应用」后,所有目标实例的 extract_rules 被全量替换为导入内容,Vector 配置按各自节奏生效。
17. 导入弹窗结果反馈:成功实例数 + 失败实例数(及失败原因),部分失败时成功的保留。
18. 非法 JSON 文件(非导出格式、字段缺失、类型非法)被导入弹窗拒绝,UI 提示具体错误。

## 风险与未决(技术实现阶段需调研)

1. **样本拉取的边界**:
   - VictoriaLogs 查询性能(无索引字段的 `_msg` 全文扫)
   - 样本缓存策略(同一实例多次拉样本是否复用)
2. **测试模拟的执行环境**:
   - 后端是启 Vector 进程跑 VRL,还是用 Python 模拟正则/kv/json 逻辑?精度与性能取舍
   - 模拟结果与真实 Vector 行为是否完全一致
3. **保存失败的回滚**:
   - 渲染出来的 Vector 配置不合法时(罕见,但要兜底),如何不破坏已运行的 Vector 进程
4. **配置下发的实时性**:
   - 当前 Vector 配置 10s 拉一次(`sidecar.yml:update_interval: 10`),用户保存到生效的延迟
   - 是否需要"保存即推送"通道,绕过轮询
5. **多实例并发编辑**:
   - 同一文件采集实例被多个用户同时编辑提取器,后写覆盖前写的处理
6. **规则数量上限**:
   - 单实例建议 20 条阈值,具体值由后端开发实测
7. **Vector 版本约束**:
   - VRL `replace` 等函数需要的 Vector 版本基线
8. **抽取字段类型**:
   - 抽取结果统一以字符串入库(本次 MVP 不暴露 int/float/timestamp 类型转换),后续是否需要
9. **中转层的单点风险**:
    - 中转 vector 挂了 = 所有该云区域的日志采集断流
    - 中转挂了 / 卡了,提取规则全部失效,**所有下游(告警 / 仪表盘 / 搜索)都受影响**
    - HA / 多实例负载均衡由开发在控制链调研时一并考虑
10. **配置同步的边界**:
    - 用户保存 → 提取规则生效的延迟(中转 vector 怎么 reload 配置文件)
    - 大量实例同时改时,中转配置重载的并发安全
    - 中转启动时怎么拉到全量 extract_rules(冷启动期间未应用规则的日志会怎样)
11. **批量导入的边界**:
    - 同一批导入的目标实例数量上限(后端定阈值,建议单批 ≤ 50 个,超过时分批)
    - 批量下发的实时性(每个实例都走 `update_interval: 10` 轮询,100 个实例是否串行还是并行)
    - 导入过程中如果用户关闭抽屉,后端任务是否取消还是继续跑完
    - 跨组织 / 跨用户的实例权限校验(用户 A 能否把规则批量导入到用户 B 有权限的实例)
    - 导出文件的版本兼容性(老版本导出的 JSON 能否被新版本正确导入)

## 不在本规格讨论范围(开发阶段确认)

- 后端具体 API 端点设计、函数签名
- 前端具体组件复用、Ant Design 组件选型
- VRL 渲染模板的具体写法
- 单元测试与 E2E 测试的具体测试用例(验收标准是黑盒视角)
- 国际化文案的具体措辞
