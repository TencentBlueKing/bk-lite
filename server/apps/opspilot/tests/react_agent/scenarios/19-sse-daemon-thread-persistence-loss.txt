# Scenario 19: SSE Persistence Relies on Daemon Threads, May Lose Audit and State

## Issue Reference
- GitHub Issue: #2959
- Title: [opspilot] SSE 持久化依赖守护线程，可能导致对话审计与执行状态丢失

## Problem Statement

SSE (Server-Sent Events) persistence operations use daemon threads (`daemon=True`).
Daemon threads are forcibly terminated when the main process exits, which means:
- In-flight database writes are aborted
- Conversation history may be partially saved or lost
- Execution state updates may not complete
- Audit trail becomes incomplete

### Original Implementation (Buggy - Before Fix)
```python
# sse_chat.py (before fix)
def _log_and_update_tokens(final_stats, ...):
    thread = threading.Thread(
        target=_do_log_and_update,
        args=(...),
        daemon=True,  # PROBLEM: Killed on main process exit!
    )
    thread.start()

# engine.py (before fix)
def _record_conversation_history(...):
    thread = threading.Thread(
        target=self._do_record_history,
        args=(...),
        daemon=True,  # PROBLEM: Killed on main process exit!
    )
    thread.start()
```

### Fixed Implementation (Synchronous Execution)
```python
# sse_chat.py (after fix)
def _log_and_update_tokens_sync(final_stats, skill_name, skill_id, current_ip, kwargs, user_message, show_think, history_log=None):
    """
    Synchronous logging and token update.
    
    Called after stream ends, does not block streaming response.
    Failures are logged but don't raise exceptions.
    """
    try:
        # Process final content
        final_content = final_stats["content"]
        if not show_think:
            final_content = re.sub(r"<think>.*?</think>", "", final_content, flags=re.DOTALL).strip()
        
        # Save history log
        if history_log:
            history_log.conversation = final_content
            history_log.save()
        
        # Insert skill log
        if current_ip:
            insert_skill_log(current_ip, skill_id, log_data, kwargs, user_message=user_message)
    
    except Exception as e:
        logger.error(
            "SSE persistence failed: skill_name=%s, skill_id=%s, error=%s",
            skill_name, skill_id, str(e),
            exc_info=True,
        )

# engine.py (after fix)
def _record_execution_result(self, input_data, result, success, start_node_type=None):
    """Record workflow execution result - synchronous"""
    try:
        task_result = self._ensure_execution_result_started(input_data, start_node_type)
        # ... update task_result ...
        task_result.save(update_fields=[...])
    except Exception as e:
        logger.exception(f"记录工作流执行结果失败: execution_id={self.execution_id}, error={str(e)}")
        # Failure doesn't affect main flow
```

### Key Design Decision
The fix converts daemon thread persistence to **synchronous execution** because:
1. SSE persistence happens AFTER the `STATS` event (stream already ended)
2. No blocking concern - user has already received the response
3. Synchronous execution guarantees data persistence
4. Simpler error handling and debugging

### Remaining Daemon Thread (Intentional)
```python
# engine.py line 1063 - NOT a persistence issue
thread = threading.Thread(target=execute_in_background, daemon=True, name="SSE-SubsequentNodes")
```
This daemon thread is for **workflow node execution** (not persistence). The persistence calls 
(`_record_execution_result`) inside this thread are synchronous within the thread context.

## Affected Files
- `server/apps/opspilot/utils/sse_chat.py`: `_log_and_update_tokens_sync()` - now synchronous
- `server/apps/opspilot/utils/chat_flow_utils/engine/engine.py`: `_record_execution_result()`, `_record_conversation_history()` - now synchronous

## Implementation Details

### SSE Chat Persistence Flow
```
1. Stream starts → yield chunks to client
2. Stream ends → yield STATS event
3. After STATS → call _log_and_update_tokens_sync() synchronously
4. Persistence completes → response fully done
```

### Engine Persistence Flow
```
1. Node executes → result available
2. Call _record_execution_result() synchronously
3. Database updated → continue to next node
```

### Error Handling
- Persistence failures are logged with full context (skill_name, skill_id, execution_id)
- Failures don't raise exceptions (don't affect already-sent response)
- `exc_info=True` captures full stack trace for debugging

#### 1. test_sse_persistence_completes_before_response_ends
**Goal**: Verify SSE persistence completes synchronously after stream ends.
**Setup**: Mock database, mock SSE response

1. Start SSE chat response
2. Generate conversation content
3. End stream (STATS event)

**Setup**: Mock database, create ChatFlowEngine
