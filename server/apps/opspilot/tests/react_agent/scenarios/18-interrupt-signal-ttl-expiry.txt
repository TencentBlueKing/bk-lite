# Scenario 18: Interrupt Signal TTL Expiry May Cause Long Tasks to Run Uncontrolled

## Issue Reference
- GitHub Issue: #2960
- Title: [opspilot] 中断信号仅保存在带 TTL 的缓存，可能导致长任务失控继续执行

## Problem Statement

Interrupt signals are stored in Django cache with a fixed TTL (default 3600 seconds).
For long-running tasks (e.g., large knowledge base training, complex multi-agent workflows),
if the task runs longer than the TTL, the interrupt signal expires and the task continues
executing even though the user requested cancellation.

### Original Implementation (Buggy - Before Fix)
```python
# execution_interrupt.py
INTERRUPT_CACHE_TTL = int(os.getenv("WORKFLOW_INTERRUPT_CACHE_TTL", "3600"))  # 1 hour

def request_interrupt(execution_id: str, reason: str = "user_manual", ...):
    cache.set(_get_interrupt_cache_key(execution_id), payload, INTERRUPT_CACHE_TTL)
    # After 3600 seconds, this key expires!

def is_interrupt_requested(execution_id: str) -> bool:
    return get_interrupt_request(execution_id) is not None
    # Returns False after TTL expires, task continues!
```

### Fixed Implementation (Database Fallback)
```python
def _check_interrupt_in_database(execution_id: str) -> bool:
    """Database fallback: check WorkFlowTaskResult.status == INTERRUPTED"""
    from apps.opspilot.enum import WorkFlowTaskStatus
    from apps.opspilot.models import WorkFlowTaskResult
    
    return WorkFlowTaskResult.objects.filter(
        execution_id=execution_id,
        status=WorkFlowTaskStatus.INTERRUPTED,
    ).exists()

def is_interrupt_requested(execution_id: str) -> bool:
    """Dual-check mechanism: cache first, database fallback"""
    if not execution_id:
        return False
    
    # Fast path: cache hit
    cache_result = get_interrupt_request(execution_id)
    if cache_result is not None:
        return True
    
    # Slow path: database fallback
    return _check_interrupt_in_database(execution_id)
```

### Risk Scenarios (Now Mitigated)
1. **Knowledge Base Training**: Training 10,000+ documents may take hours → DB fallback ensures interrupt detected
2. **Complex Multi-Agent Workflows**: Deep research tasks with many sub-agents → DB fallback ensures interrupt detected
3. **Browser Automation**: Long-running browser-use tasks with many steps → DB fallback ensures interrupt detected
4. **Batch Processing**: Processing large datasets in workflow nodes → DB fallback ensures interrupt detected

## Implementation Details

### Configuration
```python
# Environment variable (optional)
WORKFLOW_INTERRUPT_CACHE_TTL = int(os.getenv("WORKFLOW_INTERRUPT_CACHE_TTL", "3600"))
# Default: 1 hour cache TTL
# For long-running workflows, increase this value
# Example: export WORKFLOW_INTERRUPT_CACHE_TTL=7200  # 2 hours
```

### Dual-Check Mechanism
1. **Cache (Fast Path)**: O(1) lookup, used for most cases
2. **Database (Slow Path)**: Only when cache misses, uses `exists()` for performance
3. **Automatic Fallback**: No code changes needed in callers

### Database Query Optimization
```python
# Uses exists() instead of get() for performance
WorkFlowTaskResult.objects.filter(
    execution_id=execution_id,
    status=WorkFlowTaskStatus.INTERRUPTED,
).exists()  # Returns True/False, no object instantiation
```

### Test Class: TestInterruptSignalDualCheck

#### 1. test_interrupt_detected_from_cache
**Goal**: Verify interrupt signal is detected from cache (fast path).
**Setup**: Mock cache with interrupt signal
**Action**: Call `is_interrupt_requested(execution_id="test-exec-1")`
**Assert**: Returns True, database NOT queried

#### 2. test_interrupt_detected_from_database_after_cache_expiry
**Goal**: Verify interrupt signal is detected from database when cache expires.
**Setup**: 
- Mock cache to return None (expired)
- Mock database with WorkFlowTaskResult.status == INTERRUPTED
**Action**: Call `is_interrupt_requested(execution_id="test-exec-1")`
**Assert**: 
- Returns True
- Database query executed
- Log message indicates "cache expired"

#### 3. test_no_interrupt_when_both_cache_and_db_empty
**Goal**: Verify no false positives when no interrupt exists.
**Setup**: Mock cache to return None, mock database with no matching record
**Action**: Call `is_interrupt_requested(execution_id="test-exec-1")`
**Assert**: Returns False

#### 4. test_database_fallback_uses_exists_optimization
**Goal**: Verify database query uses `exists()` for performance.
**Setup**: Mock database
**Action**: Call `_check_interrupt_in_database(execution_id="test-exec-1")`
**Assert**: Query uses `.exists()` (not `.first()` or `.get()`)
