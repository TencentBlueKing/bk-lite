## 序言

当前，IT 基础设施规模持续扩张，运维场景的复杂性与作业风险随之呈指数级增长。传统依赖人工 SSH 逐台操作、脚本口耳相传的作业模式，已无法适应现代 DevOps 与持续交付对运维效率和安全合规的双重要求。为此，BK-Lite 项目组经过长期深入的工程实践与需求沉淀，推出面向中型 IT 运维团队的统一自动化作业平台——BK-Lite 作业管理。

本白皮书系统阐述 BK-Lite 作业管理模块的技术体系、核心能力与工程实现，旨在帮助技术决策者、平台架构师及一线运维工程师深入理解其设计理念与应用价值。白皮书以真实工程规格为基础，所有技术参数均来自 PRD v1.1（2026-05-28）及对应代码实现，不做虚构夸大。

BK-Lite 作业管理的核心价值理念可以凝练为一句话：**"执行即批量、变更即安全、操作即留痕、自动即闭环"**。这不仅是系统设计的基本原则，更是 BK-Lite 对现代运维自动化应有形态的完整表达。

## 1. 运维自动化的时代背景与挑战

随着企业数字化转型深入推进，IT 基础设施规模持续扩张，运维作业的广度与深度均已超出传统手工模式的承载上限。在此背景下，运维团队普遍面临以下五类系统性挑战，每一类均有其独特的工程危害和业务影响。

### 1.1 批量操作低效：逐台手工作业已触及效率天花板

对一百台机器执行补丁安装，工程师需逐台 SSH 登录、粘贴命令、等待结果，保守估计三名工程师耗费一整天方可完成。这一模式的低效性不在于工程师的能力，而源于工具链的结构性缺陷：人工串行操作的时间复杂度随主机规模线性增长，当主机数量突破数十台时，批量操作的时间成本便已无法接受。更为严峻的是，在安全应急响应场景中，高危 CVE 漏洞的修复往往有严格的时间窗口要求，逐台操作模式使得"全量修复"在时限内无法完成，直接影响企业的安全合规状态。

> **小结**：人工逐台操作的效率瓶颈已是行业共识，批量自动化执行是运维工程效率提升的必要前提，而非可选项。

### 1.2 脚本资产散乱：知识孤岛加剧人员流动风险

多数中型运维团队的脚本资产分散于工程师个人工作机、内部 Wiki、即时通讯记录等各类载体中：版本控制缺失、命名约定随意、参数说明不完整。这一状态的直接后果是：一位高水平工程师离职，其三年积累的脚本财富随之消散，新成员被迫重新开发已有功能，形成显著的重复建设成本。更深层的问题在于，同一功能可能在团队内存在多个版本，执行哪个版本本身就构成运维决策的不确定因素，增大了操作错误的概率。

> **小结**：脚本资产的碎片化是运维知识管理的核心痛点，将隐性经验沉淀为共享资产是运维团队走向工程化的关键一步。

### 1.3 高危操作风险：批量放大效应使单次误操作后果难以估量

批量操作的风险量级是单台操作的百倍放大。`rm -rf /` 的误操作、文件分发目标路径填写错误导致 `/etc/ssh/sshd_config` 被覆盖——这类事故在缺乏技术拦截机制的环境中随时可能发生，且批量执行的特性意味着后果会在数秒内同步扩散至所有目标主机。依靠流程规范和人工审核在高压紧急处置场景中形成的约束力十分有限，工程师在凌晨应急处置时的操作正确性不能仅依赖主观谨慎来保障。

> **小结**：技术层面的安全拦截机制是高危操作防控的可靠手段，流程规范只能作为补充，不能替代系统级防护。

### 1.4 审计缺失：操作留痕能力不足构成重大合规风险

在缺乏集中审计机制的环境中，回答"昨晚谁在那台机器上执行了什么"这一问题，往往需要逐台检索 `~/.bash_history`，而该文件极易被清除或篡改。这一现状对有等级保护（等保二级及以上）、ISO 27001 或内部安全审计要求的企业构成直接的合规风险。审计准备工作的工作量，也因缺乏集中化操作记录而呈现出不可接受的增长。

> **小结**：操作审计能力的缺失不是功能短板，而是合规基线要求，任何面向受监管行业的运维平台均须将其作为核心能力设计。

### 1.5 自动化断层：告警与处置之间的人工链接制约闭环能力

当前监控告警系统已具备较成熟的检测能力，但从告警触发到问题处置之间，仍然依赖值班工程师手动介入：收到告警、判断原因、登录目标主机、执行处置脚本。这一人工链接不仅引入响应延迟，更使得"无人值守自愈"能力无法实现。对于已知原因的常见故障（如磁盘使用率超阈值触发日志清理），将处置链路完全自动化是消除人工依赖、提升系统弹性的必要条件。

> **小结**：运维自动化的最终形态是闭环自愈，告警与作业执行之间的 API 化联动是实现这一目标的关键技术基础。

## 2. BK-Lite 自动化作业技术体系

### 2.1 设计原则

BK-Lite 作业管理的整体技术体系由以下四项核心设计原则驱动：

**原则一：执行即批量。** 系统以批量并行执行为默认模式，单台执行是批量执行的特例。所有执行路径均针对多目标并发场景优化，支持最大 10 个并发工作线程，目标规模从十台到数百台均无需修改执行配置。

**原则二：变更即安全。** 安全检查在执行链路的最前端介入，优先于任何命令到达执行节点。高危命令规则与高危路径规则构成双层安全防线，对手动、定时、API 三种触发来源一视同仁，无绕过路径。

**原则三：操作即留痕。** 每次作业执行自动生成完整的元信息快照，包含发起人、触发来源、执行内容、目标主机及每台主机的输出结果。记录由系统自动生成，不依赖人工填写，历史执行上下文在事后修改脚本后依然完整可查。

**原则四：自动即闭环。** 通过 NATS + REST 双通道开放接口，作业执行能力可被告警系统、外部工单系统、CI/CD 流水线等上下游系统调用。执行完成后通过 callback_url 主动推送结果，配合指数退避重试（默认起始间隔 5s，上限 120s，最多 5 次）机制，确保结果通知可靠送达，形成无人介入的完整闭环。

## 3. 批量执行引擎（BatchExec）

批量执行引擎是 BK-Lite 作业管理对外暴露执行能力的核心入口，涵盖快速执行、文件批量分发和开放接口三个子能力模块。

### 3.1 快速执行：从想法到命令落地的最短路径

快速执行（QuickExec）专为临时命令和应急处置场景设计，是整个作业管理体系中延迟最低的执行入口。用户在单一页面内完成作业配置：填写作业名称、选择目标主机（支持弹窗选择、关键字搜索、批量勾选）、指定内容来源（作业模板引用或临时输入）、配置执行参数与超时时间，即可立即提交执行。

执行提交后，系统立即创建执行记录并返回 `task_id`，实际执行推入 Celery 异步队列。用户被自动跳转至作业详情页，实时观察每台目标主机的执行状态与输出结果。整体执行速度取决于目标主机数量、网络条件及脚本本身的执行时长。

**执行参数规格**：

- 支持脚本语言：Shell、Python、PowerShell、Batch
- 默认执行超时：600 秒（可配置）
- 最大并发工作线程：10（单批次）
- 执行状态流转：`pending → running → success / failed / timeout / cancelled`

### 3.2 文件批量分发：安全受控的大规模文件推送

文件分发功能支持将配置文件、安装包、补丁等文件一次性推送至数百台目标主机。文件暂存于 MinIO 对象存储（临时文件保留期默认 7 天，可通过 `permanent=true` 参数指定为永久文件），执行节点从 MinIO 拉取文件后按目标路径分发至各主机。

文件分发配置项包括：目标主机（多选）、上传文件（支持普通文件与压缩包）、目标路径、覆盖策略（覆盖已存在文件 / 跳过已存在文件）及超时时间。目标路径在到达执行节点前，须经过安全管控层的高危路径规则检查，匹配禁止分发策略的路径请求将被直接阻断，无法到达目标主机。

文件分发详情页提供传输进度维度的可观测能力，展示每台主机的文件路径、传输进度、传输速率、完成状态及总量统计，使大规模分发操作的执行状态一览无余。

### 3.3 NATS-Executor：开放接口的异步执行通道

BK-Lite 通过 NATS 消息总线提供面向第三方系统的开放执行接口，将作业执行能力 API 化。调用方通过 NATS 发布执行请求，系统异步处理后返回 `task_id`；调用方可通过轮询接口主动查询状态，也可在请求中携带 `callback_url`，由 BK-Lite 在任务进入终态后主动推送结果。

核心开放接口清单：

| 接口 | 通道 | 描述 |
|---|---|---|
| `bklite.job_script_execute` | NATS | 脚本执行（异步，返回 task_id） |
| `bklite.job_file_distribute` | NATS | 文件分发（异步，返回 task_id） |
| `bklite.job_status_batch_query` | NATS | 批量查询执行状态 |
| `bklite.job_detail_query` | NATS | 查询作业详情 |
| `bklite.job_target_list` | NATS | 查询目标列表 |
| `POST /api/v1/job_mgmt/api/open/upload_file` | REST | 文件上传（需 API Token） |
| `DELETE /api/v1/job_mgmt/api/open/delete_file` | REST | 文件删除（需 API Token） |

回调机制采用指数退避重试策略：任务进入终态后，BK-Lite 向 `callback_url` 发送 HTTP POST，失败后按指数退避间隔（默认起始 5s，上限 120s，含随机抖动）自动重试，最多重试 5 次，确保结果通知的可靠送达。

## 4. 双执行引擎（DualEngine）

双执行引擎是 BK-Lite 适配异构主机环境的核心机制。两套引擎在接口上对上层完全透明，系统根据目标主机的驱动类型自动路由，无需用户感知底层差异。

### 4.1 Sidecar 引擎：面向有 Agent 主机的高性能通道

Sidecar 引擎适用于已通过节点管理安装控制器的目标主机。命令下发路径为：BK-Lite Server → NATS 消息总线 → 目标云区域的 Sidecar 节点 → 目标主机。整个链路基于 NATS 消息传递，具备连接稳定、延迟低的特性，适合大规模并发批量操作场景。

Sidecar 节点部署在云区域内（可部署于目标主机本身，也可部署于同一网络分区的代理主机），充当该云区域的执行代理，接收来自 NATS 的命令并在目标主机上执行，将执行输出实时回传至 BK-Lite Server。

凭据管理方面，节点管理同步来源的 Sidecar 目标主机在本地节点执行，无需额外配置 SSH 凭据，连接凭据由节点管理统一维护，作业管理直接引用。

### 4.2 Ansible 引擎：面向无 Agent 主机与复杂编排的扩展能力

Ansible 引擎适用于以下两类场景：其一，无法或不适合安装 Agent 的目标主机（通过 SSH / WinRM 直接连接）；其二，需要执行多步骤有序编排任务的 Playbook 作业。

执行路径为：BK-Lite Server 选择目标云区域内可用的 Ansible 容器节点，构建 Ansible Inventory，通过 SSH（Linux）或 WinRM（Windows）将 Playbook 或命令下发至目标主机。Ansible 引擎对 Playbook 作业有独占性约束：当执行内容为 Playbook 类型时，系统仅允许选择 Ansible 驱动的目标主机，不兼容主机在选择列表中自动禁用并显示原因提示。

手动录入的目标主机支持 SSH 密码认证与 SSH 密钥认证两种 Linux 凭据方式，以及 WinRM 凭据用于 Windows 主机。所有凭据集中加密存储，不明文回显，安全参数在作业执行历史中同样以加密形式保留。

## 5. 安全管控（SafeGuard）

安全管控是 BK-Lite 作业管理区别于通用脚本执行工具的核心技术差异所在。SafeGuard 模块在执行链路的最前端构建双层拦截防线，确保高危操作在触达任何执行节点之前被识别和处置。

### 5.1 高危命令拦截：双重扫描机制

高危命令规则作用于所有命令执行场景（快速执行、脚本库引用、API 调用）。DangerousChecker 在命令内容到达执行引擎之前对其进行双重扫描：

- **包含匹配**：大小写不敏感的字符串包含检查，适用于精确关键字拦截（如 `rm -rf /`）；
- **正则匹配**：支持完整正则表达式，覆盖复杂命令模式的识别需求。

处理策略支持两级：
- **禁止执行**：命中规则的请求被直接阻断，执行节点不会收到任何指令；
- **二次确认**：命中规则后弹出确认对话框，用户明确确认后方可继续执行，为高压场景提供人工把关节点。

规则管理支持组织范围生效，安全管理员可在组织级别统一配置，规则状态支持启用/停用切换，满足不同业务环境的差异化安全策略需求。

### 5.2 高危路径保护：文件分发的安全边界

高危路径规则专用于文件分发目标路径的安全管控。路径匹配支持两种方式：

- **精确匹配（路径前缀语义）**：精确匹配采用路径前缀语义，即一条 `/etc` 规则同时保护 `/etc/ssh/sshd_config`、`/etc/crontab` 等所有子路径，单条规则覆盖整个目录树；
- **正则匹配**：支持正则表达式，用于表达复杂路径模式。

处理策略同样支持禁止分发与二次确认两级，规则支持组织范围生效与启停管理。需要注意，高危路径匹配当前仅支持精确匹配与正则匹配，不支持通配符模式，管理员在配置规则时应以路径前缀语义代替通配符。

### 5.3 安全管控的工程价值

安全管控机制的工程价值体现在三个层面：

**防御深度**：拦截在执行链路最前端（BK-Lite Server 侧）实施，无论是手动触发、定时触发还是 API 调用，所有触发路径均须经过同一套规则检查，不存在绕过路径。

**合规可配置性**：安全规则由平台管理员与安全管理员按业务实际灵活配置，精细控制哪些操作需要保护，而不影响日常合规操作的执行效率。

**审计可追溯性**：所有被拦截的操作请求均留存记录，为安全事件溯源提供完整的操作上下文。

## 6. 脚本编排与调度（Orchestrate）

脚本编排与调度模块承载 BK-Lite 作业管理的知识资产化与周期性自动化能力，由脚本库、Playbook 库、定时任务和异步回调四个子模块构成。

### 6.1 脚本库：运维知识的资产化沉淀

脚本库是 BK-Lite 将隐性运维经验转化为团队共享资产的核心机制。平台支持新建四种类型的脚本：Shell、Bat（Windows Batch）、Python、PowerShell，覆盖 Linux 与 Windows 主流运维场景。

脚本内容支持 Jinja2 参数模板语法，工程师可为同一脚本定义多个参数（参数名、标签、是否加密、默认值、提示信息），实现"一脚本、多场景"的复用能力。加密参数在执行时不明文回显，安全敏感信息（如密码、Token）可在参数层面得到保护。

内置脚本提供常用运维场景的开箱即用模板，支持查看与引用，但不允许修改，确保内置知识库的一致性与可信度。

脚本库的团队价值体现在全生命周期：
- **知识录入**：日常运维产生的临时脚本可随时沉淀入库，附加描述和参数说明；
- **复用引用**：执行时从脚本库选择模板，参数以表单形式填写，无需重复编写；
- **人员传承**：工程师离职，其积累的脚本留存于平台，成为组织共享财富；新成员通过浏览脚本库了解团队运维规范，Onboarding 成本显著降低。

### 6.2 Playbook 库：复杂流程的标准化编排

Playbook 库面向多步骤有序执行的复杂运维场景，如数据库安装、环境初始化、批量配置变更等。平台支持上传标准 Ansible YAML 格式的 Playbook 文件，并支持版本更新管理。

Playbook 详情页展示参数定义列表（参数名、默认值、说明），执行时用户只需填写参数表单，无需手动编辑 YAML 文件，降低 Ansible 使用门槛。执行过程中，平台按 Playbook 步骤维度展示每台主机的执行状态与日志，哪台机器哪个步骤失败一目了然，精准定位问题无需逐台排查。

### 6.3 定时任务：例行运维的无人值守自动化

定时任务模块将运维工程师从高重复性的例行作业中解放出来，基于 Celery 调度引擎实现周期性作业的无人值守执行。

**调度策略**支持简单策略与复杂策略两类：
- **简单策略**：每天一次、每 N 小时一次、只执行一次，适用于常规例行任务；
- **复杂策略**：标准 Cron 表达式（分/时/日/月/周），支持表达式描述与下次执行时间预览，适用于精细化调度需求（如 `0 2 * * *` 表示每天凌晨 2 点执行）。

**并发策略**是定时任务区别于通用调度器的重要设计，应对上次执行未完成而新触发时间已到的边界场景，支持三种处理方式：
- **跳过**：本次触发直接跳过，等待下次调度；
- **运行**：不等待上次完成，立即开始新的执行实例；
- **排队**：将本次触发加入队列，等待上次执行完成后依序执行。

定时任务支持作业类型：脚本执行与文件分发；任务状态支持启用/停用切换；列表页展示下次执行时间预览，确保调度配置的直观可验证性。

### 6.4 参数加密与执行快照

执行安全与可追溯性是 Orchestrate 模块的基础性质量属性。加密参数（标记 `is_encrypted=true` 的脚本参数）在执行时不明文回显，在执行历史快照中同样以加密形式保留，防止敏感信息泄露。

每次作业执行生成完整的执行快照，包含：脚本来源、脚本语言、执行用户、代码行数与完整脚本内容。即使在作业执行完成后修改了原始脚本，历史执行的快照依然完整保留，确保每次执行的上下文可追溯、不可篡改。

## 8. 部署与演进路径

### 8.1 部署架构

BK-Lite 作业管理采用 Docker Compose 单机部署模式，无需 Kubernetes 或复杂容器编排平台，显著降低部署门槛。完整部署包含以下服务组件：

| 组件 | 说明 | 部署形式 |
|---|---|---|
| BK-Lite Server | Django 应用，承载 REST API 与 NATS API | Docker 容器 |
| NATS Server | 消息总线，推荐部署于内网 | Docker 容器 |
| Celery + Redis | 异步任务队列，用于定时调度和执行回调 | Docker 容器 |
| PostgreSQL | 存储执行记录、脚本库、安全规则等持久化数据 | Docker 容器 |
| MinIO | 文件分发场景下的临时文件对象存储 | Docker 容器 |

**最小部署规格**：单台 8 核 16GB 服务器，适用于 100 台以内主机规模的运维场景。

### 8.2 分阶段演进路径

BK-Lite 作业管理的价值实现是渐进式的，推荐分五个阶段落地：

**第一阶段（Day 1）：完成首次批量执行验证**
完成 Docker Compose 部署后，在节点管理中确认至少一台主机在线，进入「快速执行」，输入 `uptime && free -m`，选中目标主机，点击执行。执行提交后可在详情页实时观察每台主机的运行状态与内存信息，验证整个执行链路的连通性。

**第二阶段（Week 1）：沉淀核心脚本资产**
将团队最常用的 5-10 个运维脚本录入「脚本库」，定义 Jinja2 参数、补充描述与默认值。复杂多步骤流程封装为 Playbook 上传至「Playbook 库」。从此刻起，每次执行均可复用资产，知识沉淀正式开始。

**第三阶段（Week 2-4）：配置例行自动化**
将高重复性例行任务（磁盘检查、日志清理、服务健康探测）创建为定时任务，配置 Cron 表达式与并发策略。验证一次自动触发后，该类工作进入无人值守状态。

**第四阶段（Month 1）：建立安全防线**
安全管理员在「高危命令配置」中添加禁止执行规则，在「高危路径配置」中保护 `/etc`、`/boot`、`/usr` 等关键系统路径。安全防线建立后，即使在高压应急处置场景中，系统也会在技术层面完成最后一道把关。

**第五阶段（Month 2+）：接入企业工具链**
通过 NATS 开放接口将作业管理与补丁管理系统、外部工单系统、CI/CD 流水线集成，或通过 OpsPilot 实现 AI 驱动的自动化执行。作业管理从团队工具演进为企业自动化工具链中的执行引擎，价值随整合深度持续放大。

## 9. 客户价值

### 9.1 效率跃升：批量操作替代逐台执行

**批量执行效率**：对大批量主机执行批量命令，传统人工 SSH 逐台操作时间随主机数线性增长；使用 BK-Lite 作业管理，以最大 10 个并发工作线程并行下发，执行期间工程师无需逐台等待，可并行处理其他工作。批量替代串行操作本身带来的效率收益与主机规模正相关。

**例行任务自动化**：将高重复性例行巡检（磁盘检查、日志清理）配置为定时任务后，相关操作进入无人值守状态，运维工程师可将精力投入到架构优化和更高价值的平台建设中。

**应急响应提速**：凌晨告警场景下，使用快速执行可对多台目标主机并发下发信息采集或处置命令，相比逐台 SSH 操作，响应速度随并发数量级提升，整体 MTTR（平均恢复时间）可得到有效改善。

### 9.2 风险量化降低：一次生产事故的防范价值远超工具投入

高危命令与路径拦截机制的价值难以用数字精确衡量，但一次因批量误操作导致的生产级事故，其直接损失（服务中断、数据恢复、紧急处置人力）往往远超任何工具的全年投入。对于 SLA 要求严格的生产服务，高危拦截机制的存在本身就是一道价值极高的技术保险。

此外，有了技术层面的安全兜底，工程师在紧急处置时的心理压力减小，操作更加从容，慌乱下的人为误操作概率随之降低，形成安全文化与技术防控的正向循环。

### 9.3 知识资产化：人员流动不再意味着能力流失

脚本库与 Playbook 库的积累是组织层面的知识资产建设投资。将散落在个人工作机和聊天记录中的隐性运维知识，转化为团队共享的显性资产，从根本上消除人员流动带来的知识流失风险。新成员入职后可直接浏览脚本库了解团队运维规范，Onboarding 效率大幅提升。

### 9.4 合规基础：集中审计记录满足操作留痕要求

完整的作业执行记录天然满足等级保护（等保二级及以上）和内部审计对操作留痕的合规要求。"谁在什么时候对哪些主机执行了什么"由系统自动记录，不依赖人工填写，执行历史快照在作业完成后即使修改原始脚本也完整保留。相比分散翻查各台机器操作历史，集中化记录可通过作业记录页面按时间范围、操作类型等维度筛选查询，显著降低审计准备的工作量。

## 10. 技术演进展望

BK-Lite 作业管理的技术演进方向由当前用户场景的深入洞察与行业技术趋势共同驱动，重点聚焦以下四个领域：

**执行引擎扩展**：在现有 Sidecar 与 Ansible 双引擎基础上，探索对更多执行协议的原生支持，进一步扩展无 Agent 场景的覆盖范围与执行性能，同时优化大规模并发场景下的执行吞吐量与资源利用率。

**安全管控深化**：在当前命令级与路径级拦截的基础上，探索基于行为分析的动态风险评估机制，将静态规则匹配与上下文感知的智能审计能力相结合，提升高危操作识别的精准性与自适应能力。

**AI 驱动自动化**：与 OpsPilot AI 智能体的深度集成是重要演进方向，包括：自然语言作业意图理解、基于历史执行数据的智能脚本推荐、以及异常执行模式的自动识别与告警联动，推动运维自动化向 AI 智能体驱动的方向演进。

**可观测性增强**：在现有执行记录与详情展示的基础上，扩展执行趋势分析、资源消耗统计、高危拦截事件汇总等运营视图能力，为平台管理员提供更完整的运维自动化健康度洞察。

| 参数项 | 规格值 |
|---|---|
| 接口协议 | NATS（异步消息）+ REST HTTP（文件操作） |
| 认证方式 | API Token（REST 接口） |
| 回调协议 | HTTP POST |
| 回调重试策略 | 指数退避，起始间隔约 5s，上限 120s，含随机抖动，最多 5 次 |
| 状态查询 | 支持批量查询（`bklite.job_status_batch_query`） |

### A.7 基础设施规格

| 组件 | 说明 |
|---|---|
| BK-Lite Server | Django REST Framework 应用 |
| 消息总线 | NATS Server，推荐内网部署 |
| 异步队列 | Celery + Redis |
| 持久化存储 | PostgreSQL |
| 对象存储 | MinIO（文件分发临时存储） |
| 部署方式 | Docker Compose，无需 Kubernetes |
| 最小规格 | 单台 8 核 16GB 服务器，适用 100 台以内主机规模 |

## 附录 B：术语表

| 术语 | 释义 |
|---|---|
| 作业（Job） | BK-Lite 中一次具体的自动化执行操作，包括脚本执行、文件分发和 Playbook 执行三种类型 |
| 目标（Target） | 可接收作业下发的主机对象，包含 IP、操作系统、驱动类型、连接凭据等信息 |
| 执行驱动（Driver） | 命令或文件到达目标主机所使用的传输和执行机制，支持 Sidecar（有 Agent，节点管理同步目标）和 Ansible（无 Agent，手动录入目标）两种驱动；Ansible 驱动在 Linux 主机上使用 SSH 连接，在 Windows 主机上使用 WinRM 连接 |
| Sidecar | 安装在目标主机（或云区域内代理主机）上的轻量级控制器，通过 NATS 接收并执行命令 |
| 云区域（Cloud Region） | 网络互通的主机集合，BK-Lite 以云区域为单位管理执行节点和主机分组 |
| Playbook | Ansible 标准 YAML 编排文件，用于描述多步骤、有序的复杂运维流程 |
| 脚本库 | 存放可复用运维脚本的资产库，支持 Jinja2 参数化和引用，是团队运维知识的沉淀中心 |
| DangerousChecker | BK-Lite 安全管控模块中负责高危命令检查的核心组件，在执行链路最前端实施拦截 |
| 高危命令规则 | 对脚本内容进行正则或包含匹配的安全规则，命中后触发禁止执行或二次确认 |
| 高危路径规则 | 对文件分发目标路径进行精确或正则匹配的安全规则，保护关键系统路径不被覆写 |
| 定时任务 | 与 Cron 表达式绑定的周期性作业，按设定时间自动执行，无需人工触发 |
| 快速执行（QuickExec） | 为临时命令和紧急处置场景设计的即时执行入口，无需预先配置，打开即用 |
| NATS | 高性能轻量级消息总线，BK-Lite 用于模块间通信和命令下发的核心基础设施 |
| Celery | Python 分布式异步任务队列，BK-Lite 用于处理定时调度和异步作业执行 |
| Jinja2 | Python 模板引擎，BK-Lite 脚本库用其实现脚本内容的参数化替换 |
| WinRM | Windows Remote Management，微软提供的 Windows 远程管理协议，用于管理 Windows 目标主机 |
| task_id | 每次作业执行的全局唯一标识符，用于状态查询、详情获取和回调匹配 |
| callback_url | 第三方系统调用作业管理 API 时传入的回调地址，任务进入终态时 BK-Lite 主动向该地址推送结果 |
| 指数退避重试 | 回调失败后按指数退避间隔自动重试（默认起始约 5s，上限 120s，含随机抖动），最多 5 次，由 Celery 任务机制持久化保证，确保结果通知可靠送达 |
| 并发策略 | 定时任务中处理上次执行未完成而新触发时间已到的行为策略，支持跳过、运行、排队三种 |
| 执行终态 | 作业不再变化的最终状态，包括 success、failed、timeout、cancelled 四种 |
| 二次确认 | 高危规则处理策略之一，命中规则后弹出确认对话框，用户主动确认后方可继续执行 |
| BatchExec | BK-Lite 批量执行引擎，涵盖快速执行、文件批量分发和 NATS 开放接口 |
| DualEngine | BK-Lite 双执行引擎体系，包含 Sidecar 引擎（有 Agent）与 Ansible 引擎（无 Agent） |
| SafeGuard | BK-Lite 安全管控模块，提供高危命令拦截与高危路径保护双层防线 |
| Orchestrate | BK-Lite 脚本编排与调度模块，涵盖脚本库、Playbook 库、定时任务和异步回调 |
| MTTR | Mean Time To Recovery，平均恢复时间，衡量从故障发生到恢复正常所需的平均时长 |
| 等级保护（等保） | 中国国家信息安全等级保护制度，对信息系统按安全级别提出差异化保护要求 |
