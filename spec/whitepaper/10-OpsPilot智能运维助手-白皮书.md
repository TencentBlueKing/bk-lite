## 序言

当前，IT 运维领域正在经历一场以人工智能为核心驱动力的结构性转型。大型语言模型的工程化落地，使"自然语言与运维系统的无缝交互"从技术愿景走向生产实践。然而，模型能力的碎片化、企业私有知识的孤岛化、AI 工具链的工程门槛，以及将敏感运维数据外送至公有云的合规顾虑，共同构成了 AI 运维规模化落地的四重障碍。BK-Lite OpsPilot 正是针对上述四重障碍而设计的 AI First 旗舰模块。

OpsPilot 的核心设计主张可以概括为十六字纲领：**"模型即接入、知识即可问、智能即编排、私有即可信"**。模型接入层屏蔽供应商差异，支持超过 100 个预置模型 ID，兼容任意 OpenAI 接口规范的自托管服务；知识管理层将散落在文档、页面、系统中的运维经验通过语义化向量与知识图谱双引擎转化为可检索、可推理的私有知识资产；智能体编排层以可视化画布将 ReAct、LATS、PlanAndExecute 等多种推理范式与 ChatFlow 工作流无缝融合，零代码完成自动化运维流程构建；而贯穿全链路的私有化部署架构，则确保知识库内容、向量索引、图谱数据乃至 Embedding 推理过程全程在企业网络边界之内运行，数据主权始终归属企业自身。

OpsPilot 是 BK-Lite 平台 AI 能力的最高集成形态。它不仅是一个对话界面，更是连接模型能力、企业知识、运维工具与多渠道交付的完整智能体平台。本白皮书将系统阐述 OpsPilot 的技术体系、架构设计与四大技术支柱，为技术架构师、平台管理员及决策层提供完整的技术参考。

## 1. 智能运维的时代背景与挑战

### 1.1 知识沉淀难：隐性经验无法规模化复用

企业运维团队在多年实践中积累的排障经验、应急预案、变更规范，以非结构化形式散落于 Wiki 页面、即时通信记录、个人文档与电子邮件之中。这些知识的获取依赖人际网络而非系统检索，当关键工程师离职或轮岗时，其所掌握的隐性知识随之流失。

传统关键词检索系统的固有局限在于语义鸿沟：用户以"数据库连接数过高排查"为查询词，无法匹配文档中以"DB 连接池异常处理"描述的同一内容。知识资产客观存在，但因检索机制的语义缺失而无法被有效利用，实际利用率通常仅为 10%–20%。

### 1.2 新人上手慢：重复性问题消耗高价值人力

IT 支持团队日常接收的工单与咨询中，60%–80% 属于可从现有文档直接获得答案的重复性问题，涵盖 VPN 配置、软件安装、权限申请等标准化事务。这类问题占用了高级工程师 30% 以上的有效工时，形成典型的"人力错配"——有经验的工程师被低复杂度任务牵绊，无暇处理真正需要专业判断的事项。新入职工程师面对数百页操作手册，缺乏结构化学习路径，知识传递周期通常长达数周乃至数月。

### 1.3 AI 工程门槛高：全栈集成挑战制约落地速度

将 AI 能力引入运维场景需要解决一系列相互依赖的工程问题：选择并接入大语言模型、搭建向量检索管道、部署 Embedding 与 Rerank 服务、集成现有运维系统工具、设计多轮对话的上下文管理机制，以及将最终应用发布到企业现有的沟通渠道。上述每个环节均需专项技术能力，整合形成可运营的 AI 应用则需要数月的平台工程投入，这使大量运维团队在试点阶段即宣告放弃。

### 1.4 数据外发顾虑：合规压力阻碍 AI 规模化应用

运维数据天然具有高敏感性：服务器 IP 地址与拓扑、账号凭证、业务性能指标、故障现场日志等，均在不同程度上受到企业内部安全策略及行业监管法规的约束。将上述数据随 API 请求发送至公有云模型服务，将直接引发数据主权、隐私合规及信息安全等多维度风险。金融、医疗、政务、工业制造等强监管行业客户尤为关注这一问题，数据外发顾虑已成为制约企业级 AI 运维规模化落地的核心障碍之一。

> **小结**：运维 AI 化的核心挑战并非模型能力不足，而是知识管理体系的缺位、AI 工程集成的高门槛，以及数据安全边界的不确定性。OpsPilot 的设计目标正是系统性地消除这四重障碍，为企业提供开箱即用、私有可信的 AI 运维能力平台。

## 2. BK-Lite OpsPilot 智能体技术体系

### 2.1 设计原则

**原则一：AI First，智能能力是一等公民**

OpsPilot 以 AI 能力为核心设计出发点，而非将 AI 作为传统运维工具的附加功能。模型管理、知识库、智能体编排与工作流引擎均以 AI 推理为中心进行架构设计，平台的每一个功能模块均服务于 AI 能力的高效交付。

**原则二：私有化优先，数据主权归属企业**

OpsPilot 遵循"数据可用不可见"的安全理念。知识库文档存储于企业私有对象存储，向量索引运行于企业自有 PGVector 实例，知识图谱托管于企业内部 FalkorDB 实例。在智能版部署模式下，Embedding 向量化、Rerank 排序乃至 OCR 文字提取全部由本地 vLLM 实例承担，知识库内容在 AI 推理全过程中不离开企业网络边界。这一设计直接满足金融、医疗、政务等强监管行业的数据本地化合规要求。

**原则三：模型中立，不绑定任何单一供应商**

OpsPilot 不内置通用大语言模型，不与任何模型厂商形成依赖绑定。平台以 OpenAI Chat Completions 接口规范为统一适配层，通过 ProviderHub 机制支持超过 100 个预置模型 ID，兼容 DeepSeek、通义千问、GLM、LLaMA、Mistral 等国内外主流模型，以及任意通过 OpenAI 兼容接口接入的企业自托管服务。企业可根据任务特性、数据敏感级别和成本目标灵活分配模型资源，实现精细化的 Token 成本管控。

**原则四：零代码交付，运维人员自主掌控**

OpsPilot 遵循 PLG（产品驱动增长）理念，产品界面即操作手册。运维工程师无需 Python 编程背景，无需了解向量数据库原理，通过可视化配置界面即可完成从模型接入、知识库构建、智能体创建到渠道发布的完整闭环，最快半天即可完成第一个 AI 助手的部署上线。

**原则五：全链路可审计，AI 行为透明可溯**

每次智能体执行的完整链路均记录于执行日志：引用了哪些知识文档片段、调用了哪些工具及其参数、大模型的原始推理输出，以及各节点的 Token 消耗与耗时。这不仅满足企业合规审计要求，也为 AI 行为的持续优化提供数据基础。

## 3. 模型接入支柱：ProviderHub

### 3.1 设计目标与架构定位

ProviderHub 是 OpsPilot 的模型统一管理层，其核心职责是屏蔽不同模型供应商、不同部署形态（公有云 API、私有化推理服务、企业内网模型）之间的接口差异，为上层知识库处理管道、智能体执行框架和 ChatFlow 工作流提供统一的模型调用抽象。

ProviderHub 遵循供应商中立原则：平台不绑定任何模型厂商，不对外部模型服务产生路由依赖。企业已有的任何 DeepSeek、通义千问（Qwen）、智谱 GLM、OpenAI、Meta LLaMA、Mistral 等模型服务，无论部署在公有云还是企业内网，只要实现 OpenAI Chat Completions 兼容接口，即可直接接入 ProviderHub，无需任何适配开发。

### 3.2 模型类型体系

ProviderHub 统一管理四类基础模型能力，各类型服务于不同的平台功能层：

**LLM（大语言模型）**：承担智能体推理、ChatFlow 节点执行、知识问答生成、QA 对自动生成等核心推理任务。预置超过 100 个模型 ID，覆盖 OpenAI GPT 系列、DeepSeek 系列（含 DeepSeek-R1 推理模型）、阿里 Qwen 系列（含 Qwen3）、智谱 GLM 系列、Meta LLaMA 系列、Mistral/Mixtral、Baichuan、Yi、InternLM 等国内外主流选择，以及任意通过 OpenAI 兼容接口接入的自托管模型。

**Embedding（向量化模型）**：承担知识库文档向量化、查询向量化，是 RAG 检索管道的基础组件。预置超过 30 个模型 ID，包含 BGE 系列（bge-small/base/large-zh）、BCE Embedding Base v1、阿里 GTE/Qwen3-Embedding 系列、Jina AI 系列、M3E、Text2Vec 等。智能版部署模式下，BCE Embedding Base v1 作为默认本地 Embedding 服务由 vLLM 实例提供，知识库文档在向量化过程中不离开企业网络。

**Rerank（重排序模型）**：承担 RAG 两阶段检索的第二阶段精排，对向量检索召回的候选文档集进行精确相关性评分，显著提升最终答案的准确率。支持 BCE Reranker Base v1、BGE Reranker 系列（含 v2-m3、v2-gemma 高性能版本）、Jina Reranker v2、MiniCPM Reranker、阿里 Qwen3-Reranker 系列。智能版提供本地 BCE Reranker 服务，Rerank 推理过程不依赖外部 API。

**OCR（文档识别模型）**：承担扫描件、图片格式文档的文字提取，扩展知识库支持的文档来源范围。支持 OlmOCR（本地，支持复杂排版、手写体、数学公式）、PaddleOCR（本地，轻量开源）、DotsOCR（本地服务）、AzureOCR（云服务）。智能版内置 OlmOCR 本地服务，扫描件处理不依赖外部云服务。

### 3.3 安全机制

模型密钥（API Key）采用加密存储机制，在系统内部流转时不以明文形式出现于任何日志或接口响应中。模型资源以团队分组为可见边界，非授权团队成员无法访问或调用其他团队配置的模型实例，实现模型资源的细粒度访问控制。

### 3.4 智能版本地推理服务（vLLM）

智能版在 ProviderHub 之外额外部署 vLLM 推理服务实例，提供以下本地 AI 基础能力：

| 服务 | 默认模型 | 用途 | 显存需求 |
|------|---------|------|---------|
| bce-embedding | BCE Embedding Base v1 | 知识库文档本地向量化 | 含于总量 |
| bge-embedding | BGE 系列 | 备选向量化方案 | 含于总量 |
| bce-rerank | BCE Reranker Base v1 | 检索结果本地精排 | 含于总量 |
| olmocr | OlmOCR | 文档本地 OCR 提取 | 含于总量 |

智能版硬件要求：可用 GPU 显存大于 8 GB。满足该条件后，知识库全链路（文档解析、向量化、检索精排）在企业内网完整闭环，符合最高级别的数据本地化要求。

## 4. 知识库与 RAG 支柱：KnowledgeRAG

### 4.1 知识管理体系概述

运维知识的有效管理是 AI 运维助手发挥实际价值的基础。KnowledgeRAG 提供从文档摄取、结构化处理、向量索引构建到多模式语义检索的完整知识管理管道，同时通过 Graphiti 图谱引擎提供超越向量检索能力的关系推理支持。

所有知识资产（原始文档文件、向量索引、知识图谱）均存储于企业私有基础设施之上：文档文件持久化于企业私有对象存储（MinIO），向量索引存储于私有部署的 PGVector 实例，知识图谱存储于私有部署的 FalkorDB 实例。无论基础版还是智能版，知识库数据均不由 OpsPilot 平台持有或代理，数据主权完整归属企业。

### 4.2 文档处理管道

KnowledgeRAG 支持多来源文档摄取：本地文件上传（支持 PDF、DOCX/DOC、PPTX/PPT、XLSX/XLS/CSV、TXT、MD、JPG/PNG/JPEG 格式）、网页链接抓取（支持多层级递归爬取）及自定义文本录入。

文档进入系统后经历完整的状态机处理流程：**待处理 → 分块中 → 训练中 → 就绪**（异常时转为**错误**状态）。各阶段全程可视化跟踪，处理失败时提供明确的错误定位。

分块策略支持四种模式，适应不同文档结构特征：

- **fixed_size**（固定大小）：按字符数均匀切分，适合格式规整的文本类文档；
- **recursive**（递归语义边界）：按段落、句子等自然语义边界递归分割，保持语义完整性；
- **semantic**（语义分块）：利用 Embedding 模型的语义感知能力在话题转换处切分，适合长篇技术文档；
- **full**（全文）：不分块，整文档作为单一知识单元，适合短小精炼的操作规范文档。

对于含有扫描图片的 PDF 或图片格式文档，OCR 模型在分块前完成文字提取，确保非结构化文档内容可被纳入语义检索范围。

### 4.3 两阶段检索机制

KnowledgeRAG 采用业界成熟的两阶段检索架构，在召回覆盖度与精确相关性之间实现最优平衡：

**第一阶段：向量相似度检索（召回）**

基于 PGVector 扩展，对用户查询向量与知识库文档向量执行大规模近似最近邻搜索，召回候选文档集。支持两种检索算法：

- **similarity**（余弦相似度）：标准向量距离计算，适合精确语义匹配场景；
- **MMR**（最大边际相关性，Maximum Marginal Relevance）：在保证相关性的同时最大化结果多样性，避免召回语义高度重复的文档片段，适合问题涉及多个维度的综合查询场景。

召回模式支持三种粒度：chunk（原始分块）、origin（原始完整文档）、segment（语义段落），可根据不同智能体的上下文窗口策略灵活配置。

**第二阶段：Rerank 精排（过滤）**

对第一阶段召回的候选文档集，使用专业 Rerank 模型进行交叉编码（Cross-Encoder）精确相关性评分，将最相关的文档片段排列在前，过滤低相关性干扰内容。两阶段检索相较于纯向量检索，可显著提升最终答案的准确率。

每个知识库的相似度阈值和召回数量均可独立配置，智能体可同时绑定多个知识库并对各库设置独立阈值。开启严格模式（Strict Mode）时，智能体仅基于知识库内容回答，对超出知识范围的问题明确告知用户，而非生成无依据的推断性回答，确保知识问答的可靠性边界。

### 4.4 Graph RAG：知识图谱推理

在标准 RAG 向量检索能力之上，KnowledgeRAG 集成 Graphiti 图谱引擎与 FalkorDB 图数据库，提供基于知识图谱的关系推理能力（Graph RAG）。

Graphiti 从文档内容中自动抽取实体（服务、主机、组件、人员）与关系（依赖、调用、部署于、变更了），构建运维领域知识图谱。FalkorDB 基于 Redis 的图存储架构提供低延迟的图查询性能，支持多跳推理查询。

Graph RAG 特别适合以下运维场景：

- **服务依赖分析**："服务 A 的上下游依赖链路是什么？"——图谱多跳遍历直接给出完整依赖路径；
- **故障影响范围评估**："数据库节点 X 故障会影响哪些业务系统？"——图谱反向追踪影响域，远比向量检索的语义匹配更精确；
- **变更关联溯源**："最近哪些变更操作涉及服务 B？"——图谱按时间维度检索关联变更节点。

知识图谱支持手动重建社区以反映最新的系统拓扑变化，与文档知识库保持同步更新。

### 4.5 QA 对管理

除文档级知识摄取外，KnowledgeRAG 支持标准化问答对（QA 对）管理。QA 对作为高精度知识单元直接参与检索，适合高频标准问题的精准匹配场景（如"VPN 客户端配置步骤"、"生产环境发布流程"等）。QA 对支持手工创建、批量导入及 LLM 从文档内容自动生成三种方式，覆盖不同知识录入效率需求。

## 5. 智能体与编排支柱：AgentStudio

### 5.1 智能体执行框架

AgentStudio 是 OpsPilot 的智能体创建与编排核心，基于 LangChain/LangGraph 构建，提供从单智能体执行到复杂多步骤工作流的完整编排能力。

OpsPilot 支持三种智能体推理执行范式，覆盖从日常运维问答到深度规划的全场景需求：

**ReAct（推理-行动-观察循环）**：标准执行模式，适合大多数运维场景。每次工具调用经历"推理意图 → 选择工具 → 执行动作 → 观察结果 → 继续推理"的完整循环，AI 的每一步工具调用均有明确的推理依据，杜绝盲目调用。ReAct 模式透明度高，执行链路清晰可审计。

**LATS（语言智能体树搜索，Language Agent Tree Search）**：进阶执行模式，采用树形搜索策略对多条可能的行动路径进行评估，选择最优路径执行。适合需要在多个工具调用策略之间权衡、步骤较多且路径选择影响最终结果质量的复杂任务。

**PlanAndExecute（计划-执行）**：规划执行模式，先对复杂任务进行整体规划分解，再逐步执行各子任务，适用于需要多阶段规划与顺序执行的运维任务，如跨系统的根因定位与修复方案生成。

### 5.2 智能体配置体系

每个智能体实例包含基础配置与增强配置两个维度：

**基础配置**：绑定 LLM 模型（从 ProviderHub 注册的模型中选择）、系统提示词（定义智能体角色、能力边界与回答风格）、温度参数（控制推理输出的随机性）、智能体简介与分组标签。

**增强配置**：
- **RAG 配置**：绑定一个或多个知识库，各知识库可设置独立的相似度阈值；支持查询改写（Query Rewriting）——开启后 AI 自动扩展检索词，提升召回率；支持严格模式以限定回答范围；
- **工具配置**：挂载已在工具管理中注册的 MCP 工具，工具变量中的密码类敏感字段加密保存；
- **聊天历史**：可配置是否携带多轮对话历史，控制上下文窗口的长度策略；
- **响应方式**：支持流式输出（SSE，Server-Sent Events）与同步响应两种模式。

智能体支持从内置模板快速创建，内置模板覆盖：Web 应用巡检助手、Postgres DB 助手、Kubernetes 集群巡检助手等常用运维场景。选择模板后仅需填写目标系统连接参数，即可完成智能体配置，大幅降低场景化 AI 应用的构建成本。

### 5.3 记忆管理

AgentStudio 提供结构化记忆管理能力，支持 AI 在跨对话场景中保持有效的上下文记忆，实现长程个性化服务。

记忆管理分为两个维度：

**个人记忆空间**：记忆条目仅对创建者本人可见，用于存储用户个人的环境偏好、团队特有配置、个性化操作习惯等信息。例如，工程师在一次对话中告知 AI "我们的 Kubernetes 集群使用自定义 CNI 网络方案"，该信息写入个人记忆后，后续对话中 AI 无需再被提醒即可基于该上下文作答。

**团队记忆空间**：记忆条目在团队范围内共享，用于沉淀团队级的运维规范、系统架构说明、常用参数配置等团队共识性知识，作为知识库的动态补充层。

记忆空间支持配置写入规则与关联 LLM 模型，控制哪些对话内容会被自动提炼写入记忆，以及以何种结构存储。

### 5.4 ChatFlow 可视化工作流引擎

ChatFlow 是 AgentStudio 的核心差异化能力，提供无代码可视化工作流画布，将多个 AI 推理步骤、工具调用、条件逻辑与外部系统动作编排为完整的自动化运维流程。

ChatFlow 支持六类节点，覆盖运维自动化的完整流程控制需求：

| 节点类别 | 节点类型 | 说明与典型用途 |
|----------|---------|--------------|
| **触发** | RESTful | 外部系统通过 HTTP 接口主动触发工作流，适合告警推送场景 |
| **触发** | OpenAI | 以 OpenAI Chat Completions 接口形式触发，兼容 OpenAI 生态工具 |
| **触发** | AG-UI | 支持人机交互步骤的流程触发，含审批等待节点 |
| **触发** | 企业渠道 | 接收来自企业微信、钉钉、微信公众号的消息触发 |
| **触发** | 定时（Cron） | 按 Cron 表达式定时触发，用于自动化巡检、定期报告生成 |
| **应用** | Bot | 调用已上线的 AI 应用机器人实例 |
| **智能体** | Skill | 在工作流节点中调用智能体执行 AI 推理任务 |
| **记忆** | Memory Read/Write | 从记忆空间读取上下文或将关键信息写入记忆 |
| **逻辑判断** | Condition | 条件分支路由，按告警级别、消息类型等条件将流程路由到不同处理路径 |
| **动作** | Notification | 向企业微信、钉钉、邮件发送通知 |
| **动作** | API Call | 调用外部 API，触发作业执行、工单创建等外部系统动作 |

ChatFlow 提供节点级执行测试能力，支持单独验证每个节点的输入输出，缩短工作流调试周期。执行过程中的每个节点输入、输出、耗时均记录于执行日志，支持执行链路的逐节点追踪与问题定位。

### 5.5 AG-UI 人机协同审批

高风险运维操作（作业执行、配置变更、资源删除）不应由 AI 自主决策执行。OpsPilot 通过 AG-UI 协议实现人机协同审批机制：当 ChatFlow 执行至审批节点时，流程自动暂停并向工程师推送审批请求，支持提交（确认执行）、选择提交（从多个预置策略中选择）两种审批交互模式，工程师确认后流程继续执行。这一机制确保 AI 的自动化能力被限定在安全边界之内，关键变更操作的人工审批卡点始终由人来把控。

## 6. 工具 MCP 与渠道支柱：ToolChannel

### 6.1 MCP 工具生态

OpsPilot 采用 MCP（Model Context Protocol，模型上下文协议）作为工具能力的标准接入规范。MCP 是 AI 领域新兴的工具调用标准协议，定义了 AI 智能体与外部工具、数据源之间的标准化交互接口。通过 MCP 协议，OpsPilot 可以接入任何支持该协议的工具服务，无需针对每个工具进行定制化适配开发。

工具管理支持以下操作：工具列表管理（含名称/标签检索）、工具新增与配置（MCP 链接、变量配置）、MCP 子工具能力自动拉取。工具变量支持文本类型与密码类型区分，密码类变量强制加密存储，在配置界面以掩码形式显示，防止凭证明文泄露。

内置运维工具覆盖高频场景：Kubernetes 集群状态查询与管理、Jenkins CI/CD 构建触发、数据库健康检查与指标采集，以及通过 BK-Lite 平台作业接口触发的自动化作业执行。通过工具挂载，智能体可从"仅能对话回答"跨越到"能够执行实际运维操作"，是 AI 运维助手从建议型升级为执行型的关键能力。

### 6.2 多渠道接入体系

OpsPilot 支持一次配置、多渠道发布。AI 应用构建完成后，无需针对不同渠道重新适配，通过渠道配置即可在企业现有的沟通协作工具中发布 AI 助手。

| 渠道 | 接入方式 | 典型用途 |
|------|---------|---------|
| 企业微信 | 企业应用 API | 员工内部 AI 助手，7×24 自动应答 |
| 企业微信机器人 | Webhook | 群内 AI 应答、巡检报告推送 |
| 钉钉 | 钉钉机器人 API | 钉钉群 AI 助手 |
| 微信公众号 | 公众号消息接口 | 对外服务自动应答 |
| Web | 内置 Web Chat UI | 平台内嵌对话界面，PC/移动浏览器访问 |
| GitLab | Webhook 触发 | 代码事件驱动的 AI 响应，DevOps 场景 |

渠道配置中的所有密钥字段（企业微信 Secret、钉钉 Token 等）强制加密存储，通过接口返回时以掩码处理，防止渠道凭证泄露。

### 6.3 会话审计与运营统计

全量会话日志记录每次对话的完整链路：用户输入内容、AI 引用的知识文档片段（含文档名称与分块位置）、工具调用的参数与返回结果、大模型的原始推理输出、各节点 Token 消耗与耗时。会话日志按用户维度组织，支持按会话删除、按时间范围检索，满足合规审计的可溯源要求。

统计视图提供运营层面的量化指标：会话量趋势、活跃用户数、Bot 级 Token 消耗分布。Token 消耗的 Bot 级可视化追踪使平台管理员可以精确识别高消耗 AI 应用，据此优化提示词长度与知识库召回数量，实现 AI 应用使用成本的精细化管控。

## 8. 部署与演进路径

### 8.1 部署模式对比

OpsPilot 提供基础版与智能版两种部署模式，适应不同的数据安全要求与基础设施条件：

| 能力维度 | 基础版 | 智能版 |
|----------|--------|--------|
| OpsPilot 管理平台 | 包含 | 包含 |
| LLM 对话能力 | 对接外部或内网模型服务 | 对接外部或内网模型服务 |
| 知识库向量化（Embedding） | 调用外部 Embedding 模型 API | **本地 vLLM 提供，知识库内容不出企业网络** |
| 检索精排（Rerank） | 调用外部 Rerank 模型 API | **本地 vLLM 提供** |
| OCR 文档解析 | 需配置外部 OCR 服务 | **内置本地 OlmOCR 服务** |
| 知识库数据安全级别 | Embedding 数据经外部 API 处理 | **Embedding/Rerank/OCR 全程在企业内网，数据不外发** |
| 额外硬件要求 | 无 | 可用 GPU 显存 > 8 GB |
| 适用场景 | 快速试点、对数据安全要求一般的场景 | 金融、医疗、政务、工业制造等强监管行业 |

**选择建议**：企业若已有内网大模型服务，且对知识库数据出境无严格合规约束，基础版即可满足需求，部署更简便，上手更快。若所在行业存在数据本地化监管要求，或企业安全策略要求知识库内容全程在内网处理，应选择智能版——Embedding、Rerank、OCR 三类模型均由本地 vLLM 实例提供，知识库数据在 AI 推理全链路中不离开企业网络边界。

### 8.2 私有化部署架构

OpsPilot 的存储与计算全链路均支持私有化部署：

- **文档存储**：MinIO 私有对象存储，原始文档文件存储在企业自有基础设施；
- **向量索引**：PGVector 扩展运行于企业私有 PostgreSQL 实例；
- **知识图谱**：FalkorDB 图数据库部署于企业内网；
- **AI 推理**（智能版）：vLLM 推理服务运行于企业自有 GPU 服务器；
- **数据库**：PostgreSQL 主库存储所有配置、会话、日志数据，部署于企业内网。

上述架构确保 OpsPilot 在完全隔离的企业内网环境中运行，满足"数据不出域"的最高级别安全要求。模型密钥、工具密码、渠道密钥全程加密存储，关键敏感字段在所有接口响应中以掩码处理，不以明文形式暴露。

### 8.3 演进路径建议

建议企业按以下阶段推进 OpsPilot 的落地与深化：

**第一阶段（基础落地，1–4 周）**：完成模型接入与知识库建设。接入已有 LLM 服务，整理并上传核心运维文档（操作手册、FAQ、故障案例），在企业微信或 Web 渠道发布首个 IT 知识问答助手，验证 AI 知识问答的基础价值，建立团队使用习惯。

**第二阶段（场景深化，1–3 月）**：基于 ChatFlow 构建自动化场景。接入监控告警 Webhook，配置告警辅助分析工作流；部署定时巡检 ChatFlow，实现每日自动巡检报告生成与推送；评估智能版升级需求，根据数据安全策略决定是否切换至全内网 Embedding 处理。

**第三阶段（平台协同，3–6 月）**：完成与 BK-Lite 各模块的深度联动。接入 CMDB 工具能力，实现告警诊断的资产上下文自动获取；集成作业平台，构建"AI 分析 + 人工审批 + 自动执行"的闭环运维工作流；基于记忆管理沉淀团队运维知识，持续优化智能体能力边界。

## 9. 客户价值

### 9.1 运维效率改善

**告警响应加速**：AI 辅助告警分析可在告警触发时自动聚合资产上下文、历史故障案例检索结论和初步根因判断，L1 工程师在第一时间即可获得结构化诊断报告，有效缩短信息收集阶段耗时，辅助降低 MTTR（平均恢复时间）。实际改善幅度因环境配置、告警复杂度及知识库完备程度不同而存在差异。

**例行工作自动化**：ChatFlow 定时巡检工作流可将原本需要人工逐一执行的健康检查任务完全自动化，工程师的精力得以集中于需要专业判断的复杂事项。自动化收益与所管理的基础设施规模及巡检覆盖范围正相关。

**知识复用率提升**：语义检索配合 AI 问答可有效弥补传统关键词检索的语义鸿沟，提升知识资产的实际利用率，降低重复性问题的人工处理比例，IT 支持团队的高价值人力得以从低复杂度重复工单中解放。

**新人上手周期缩短**：新入职工程师可通过向 AI 知识助手提问快速获取操作手册、处理规范及历史故障案例，无需依赖老员工口传心授，知识传递周期显著缩短。

### 9.2 成本精细化管控

**Token 消耗透明追踪**：Studio 统计视图提供 Bot 级 Token 消耗可视化，平台管理员可精确识别高消耗来源，据此优化提示词长度与知识库召回数量，避免 AI 应用成本失控。

**模型弹性配置**：ProviderHub 的模型中立架构支持按任务特性分配模型资源——高频低复杂度问答任务使用小参数模型（成本低），深度诊断推理任务使用高性能模型（效果优）。精细化的模型资源分配在规模化使用后可有效控制 Token 成本，具体节约幅度因使用场景和模型选型而异。

### 9.3 行业适配优势

OpsPilot 的私有化部署架构与数据不外发的安全设计，使其天然适配对数据安全有严格要求的行业场景：

**金融行业**：交易系统告警分析、数据库健康巡检、变更合规审计，知识库数据（含业务指标、账户系统拓扑）全程不离开金融机构内网，满足金融数据本地化监管要求。

**医疗行业**：医疗信息系统运维支持，医疗设备网络巡检，患者数据与系统配置信息私有化存储处理，符合医疗数据保护相关法规要求。

**政务行业**：政务系统运维知识库、涉密等级系统的 AI 辅助运维，数据主权明确，不存在数据出境风险。

**工业制造**：工业控制网络独立运行，生产数据与工艺参数不与公有云交互，满足工业信息安全隔离要求。

## 10. 技术演进展望

OpsPilot 的技术演进将沿以下方向持续推进：

**Multi-Agent 协作架构**：当前 OpsPilot 支持单智能体执行与 ChatFlow 多节点编排，未来将引入 Multi-Agent 协作框架，支持多个专业领域智能体（网络智能体、数据库智能体、应用性能智能体）在统一协调器的调度下协同处理复杂的跨领域运维问题，实现更高层次的运维智能化。

**推理模型深度集成**：随着 DeepSeek-R1 等推理导向模型的成熟，OpsPilot 将在深度诊断场景中深度集成 Chain-of-Thought 推理能力，为根因分析提供可解释的逐步推理过程，而非仅输出结论，进一步提升 AI 运维建议的可信度。

**知识图谱与 CMDB 融合**：将 Graphiti 知识图谱能力与 BK-Lite CMDB 的拓扑数据深度融合，实现基于实时资产关系的动态图谱推理，使告警影响范围评估从静态文档检索升级为基于当前生产拓扑的实时推断。

**AI 应用全生命周期运营**：在现有 Token 消耗统计基础上，增加 AI 回答质量评估（用户反馈采集、知识命中率统计）、知识库健康度监控（文档更新及时性、低命中率知识识别），为 AI 应用持续优化提供数据驱动的运营体系。

**更广泛的 MCP 工具生态**：随着 MCP 协议在 AI 工具生态中的普及，OpsPilot 将持续扩展内置工具库，覆盖更多主流工单与服务管理系统（ServiceNow、JIRA）、云平台管理接口（AWS、Azure、华为云）及可观测性平台（Prometheus、Datadog），进一步降低 AI 运维落地的系统集成成本。

## 附录 B：术语表

| 术语 | 定义 |
|------|------|
| RAG（检索增强生成） | Retrieval-Augmented Generation。AI 在生成回答前先检索知识库中的相关内容，将检索到的文档片段作为上下文附加给大模型，使回答有据可查并降低大模型幻觉风险 |
| PGVector | PostgreSQL 数据库的向量存储扩展，OpsPilot 的知识库向量索引存储于私有部署的 PGVector 实例 |
| FalkorDB | 基于 Redis 的图数据库，用于存储 Graphiti 知识图谱，具有低延迟图查询特性 |
| Graphiti | 开源知识图谱框架，OpsPilot 基于此构建运维领域实体关系图谱，支持多跳推理查询 |
| Graph RAG | 知识图谱检索增强生成。在向量检索基础上增加图谱关系推理，适合服务依赖分析、故障影响范围评估等需要理解拓扑关系的运维场景 |
| Embedding（向量化） | 将文本转换为高维数值向量的过程。语义相近的文本在向量空间中距离近，是语义检索的基础技术 |
| Rerank（重排序） | 对向量检索召回的候选文档集进行交叉编码精确评分，将最相关内容排列在前，显著提升最终答案准确率 |
| MMR（最大边际相关性） | Maximum Marginal Relevance。向量检索算法，在保证相关性的同时最大化结果多样性，避免召回语义重复的文档片段 |
| vLLM | 高性能大模型推理引擎，智能版使用 vLLM 在企业内网 GPU 服务器上运行本地 Embedding、Rerank 和 OCR 服务 |
| ReAct | Reasoning + Acting。智能体工具调用执行范式：推理意图 → 选择工具 → 执行动作 → 观察结果 → 继续推理，每步工具调用均有明确推理依据 |
| LATS | Language Agent Tree Search，语言智能体树搜索。进阶智能体执行范式，通过树形搜索评估多条行动路径并选择最优路径执行 |
| PlanAndExecute（计划-执行） | OpsPilot 提供的规划执行智能体模式，先对复杂任务进行整体规划分解，再逐步执行各子任务，适用于需要多阶段规划与顺序执行的运维任务 |
| ChatFlow | OpsPilot 的可视化工作流引擎，支持无代码拖拽画布编排多智能体 AI 流程，覆盖触发、推理、条件路由、外部动作、通知等六类节点 |
| MCP（模型上下文协议） | Model Context Protocol。AI 领域工具调用标准协议，定义 AI 智能体与外部工具、数据源的标准化交互接口，OpsPilot 通过 MCP 接入外部运维工具能力 |
| AG-UI | 支持人机交互步骤的智能体执行协议。在高风险操作节点处，AI 暂停等待工程师确认后再继续执行，实现人工审批卡点 |
| SSE（服务器推送事件） | Server-Sent Events。大模型流式输出的传输技术，使 AI 回答逐字实时呈现而非等待全部生成完毕 |
| PLG（产品驱动增长） | Product-Led Growth。产品设计理念，用户无需销售支持即可自助完成从零上手的全过程，产品本身即教程 |
| MTTR（平均恢复时间） | Mean Time To Repair/Recovery。从故障发现到恢复正常服务所需的平均时间，是衡量运维响应效率的核心指标 |
| ProviderHub | OpsPilot 模型统一管理层，屏蔽不同模型供应商和部署形态的接口差异，为平台提供统一的模型调用抽象 |
| KnowledgeRAG | OpsPilot 知识库与 RAG 能力支柱，提供文档处理管道、两阶段向量检索、知识图谱推理的完整知识管理能力 |
| AgentStudio | OpsPilot 智能体与编排能力支柱，包含智能体执行框架、ChatFlow 工作流引擎、记忆管理与人机协同审批能力 |
| ToolChannel | OpsPilot 工具 MCP 与渠道能力支柱，包含 MCP 工具管理、多渠道接入配置与会话审计能力 |
| Token | 大模型处理文本的计量单位，约对应 0.75 个英文单词或 1–2 个中文字符，是 LLM API 计费的基础单位 |
| QA 对（问答对） | 管理员预置的标准化问答单元，作为高精度知识条目存储于知识库中，适合高频标准问题的精准匹配 |
