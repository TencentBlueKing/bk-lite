## 序言

人工智能正以前所未有的速度重塑全球产业格局，IT 运维亦处在一场深刻变革的临界点。过去十年，企业基础设施完成了从物理机到虚拟化、再到容器与多云混合的跨越式演进，基础设施的复杂度呈指数级上升；而与之形成尖锐对照的，是大量中小与成长型企业的运维能力建设严重滞后——工具碎片化、资产失真、告警泛滥、AI 难以落地、平台建设成本高企，成为横亘在数字化进程中的共性障碍。

BK-Lite 始终秉持"以更低的门槛，把统一、智能的运维能力真正交付到每一个团队手中"这一核心理念。我们认为，运维平台的价值不在于堆砌功能的广度，而在于让企业能够"今天部署、明天就用"，并在真实使用中持续获得回报。为此，BK-Lite 以容器化一键部署为交付形态，以统一数据底座为技术内核，以渐进式启用为建设路径，将资产、监控、日志、告警、自动化、分析与 AI 能力收敛于同一个可私有化部署的平台。

在 AI 重塑运维的时代，BK-Lite 坚持"AI First"与"私有化可信"的双重立场：让 AI 成为贯穿运维链路的内生能力，而非孤立的演示功能；让企业既能享受智能化红利，又能将数据牢牢守在自己的边界之内。我们相信，一个轻量、全栈、可信、智能的运维平台，将成为企业数字化运营不可替代的基石。

## 1. IT 运维的时代背景与挑战

### 1.1 多云混合重塑基础设施边界

云计算与容器化已成为企业 IT 的主流形态。一个典型的成长型企业，其资产可能同时分布在多个公有云、私有虚拟化集群与 Kubernetes 集群之上，每个平台各有控制台、各有模型、各有规范。基础设施的物理边界被打破，管理的逻辑边界却迟迟未能重建，资产底数、运行状态与依赖关系的不确定性由此累积。

### 1.2 工具碎片化推高协同成本

监控、日志、告警、资产、自动化往往分属不同工具，各自部署、各自学习、各自维护，彼此之间数据无法打通。运维人员每天在多个控制台之间切换，信息靠人工从一个系统搬运到另一个系统。工具越买越多，整体效率却未见提升，反而沉淀为持续的协同成本与集成负担。

### 1.3 告警泛滥与处置失序

缺乏收敛机制的监控告警，会在故障时产生成百上千条重复信息，真正关键的信号淹没在噪声中；而处置流程缺乏闭环，告警发出后无人认领、无法追溯、难以复盘。被动响应取代了主动治理，运维团队疲于奔命。

### 1.4 AI 落地门槛高与数据合规顾虑

大模型与 AIOps 概念火热，但从模型选型、向量化、知识库构建到智能体编排，工程门槛极高；同时，运维数据高度敏感，企业对将内部日志、配置与知识外发到公有云存在合规顾虑。智能化的愿景与可落地、可信赖的现实之间，存在明显的鸿沟。

### 1.5 平台建设成本高企

自建一套覆盖上述能力的运维平台，往往意味着数月的集成周期、专职团队的持续投入与高昂的软硬件成本，这对预算与人力有限的中小团队难以承受。

> **小结**　上述挑战的本质，是"复杂多变的基础设施"与"滞后且割裂的运维能力"之间的结构性错配。破解之道，不在于继续叠加更多孤立工具，而在于用一个轻量、一体、以数据为底座、可渐进建设、AI 内生的平台，从根本上重建运维的统一性与智能性。

## 2. BK-Lite 总体技术架构

### 2.1 设计原则

BK-Lite 的总体架构围绕五条设计原则构建，与本白皮书核心原则一一对应：

- **一体即平台。** 资产、可观测、告警、自动化、分析与 AI 收敛于一个平台，消除工具孤岛。
- **部署即可用。** 容器化一键部署，内置模板与自动发现，接入即出数据、出数据即出价值。
- **数据即底座。** 以统一的资产与可观测数据作为全平台协同的事实基础，模块间数据自然流转。
- **渐进即建设。** 从核心运维能力起步，按需启用智能告警、自动化与 AI，每一步均产生独立价值。
- **智能即内生。** AI 贯穿运维链路、私有化可信，而非孤立附加。

### 2.2 总体参考架构

BK-Lite 采用容器化微服务架构，默认通过 Docker Compose 编排，自下而上分为采集接入、数据存储、平台服务、业务模块与统一门户五层，层层支撑、协同贯通（如图 1 所示）。

│  网关层       Traefik（HTTPS 入口 · 路由分发 · TLS 终结）          │
├──────────────────────────────────────────────────────────────┤
│  业务模块层                                                       │
│   CMDB │ 监控 │ 日志 │ 告警中心 │ 作业 │ 运营分析 │ MLOps          │

├──────────────────────────────────────────────────────────────┤

├──────────────────────────────────────────────────────────────┤
│  数据存储层                                                       │

**采集接入层**通过多种采集代理对接异构环境；**数据存储层**以专用数据库分别承载时序、日志、图关系、向量与对象数据；**平台服务层**提供 API、异步任务与 NATS 消息总线；**业务模块层**承载十二大模块，并以系统管理为安全底座贯穿全平台；**统一门户层**则通过控制台与网关提供统一入口与安全接入。

## 3. 统一数据底座与模块协同体系

BK-Lite 的核心竞争力不在于单个模块的功能，而在于以统一数据底座驱动的模块协同。运维数据沿一条清晰的主线流动，并在协同中层层放大价值（如图 2 所示）。

```
图 2　统一数据底座驱动的模块协同

  节点管理 ──铺设采集器──▶ 监控/日志 ──指标·日志──▶ 告警中心
     │                         │                      │
     │                         ▼                      ▼
     │                  CMDB 资产与关系图谱 ◀──影响评估──┘
     │                         │（事实底座）
     ▼                         ▼
  作业自动化 ◀──修复动作──  运营分析（全局可视化）
     ▲                         │
     └──────── OpsPilot AI（调用各模块为工具，知识+实时数据推理）
```

**节点管理**将采集器铺设到各环境，是采集底座；**监控**与**日志**把指标与日志汇入平台；异常事件流向**告警中心**完成收敛与处置；**CMDB**提供资产与关系的事实上下文，支撑告警的影响评估与责任归属；**作业自动化**执行修复动作；**运营分析**把全局数据可视化；**系统管理**作为安全底座为全平台提供统一身份、权限与审计；而**OpsPilot**则把各模块封装为工具，结合知识库 RAG，为用户提供基于真实数据的智能问答与自动化编排；**MLOps**利用监控指标作为训练数据，产出异常检测、容量预测等模型并回馈监控与告警链路。

## 4. 平台覆盖能力与规格

BK-Lite 的"开箱即用"承诺，建立在丰富的内置覆盖能力之上。

**监控覆盖。** 内置 35 个采集插件，覆盖约 40 余类监控对象与子对象、约 700 项指标，涵盖主流操作系统、数据库（MySQL、Oracle、MongoDB、Redis 等）、中间件（Kafka、RabbitMQ、Etcd、Nginx、Zookeeper 等）、容器与 K8s、VMware 虚拟化、网络设备（SNMP）、硬件（IPMI/存储）与公有云。

**日志覆盖。** 内置 18 种日志采集类型，覆盖中间件、数据库、容器、网络、安全与系统日志，由 Filebeat、Vector、Packetbeat、Auditbeat、Winlogbeat、Snmptrapd 等采集器承载。

**资产自动发现。** CMDB 通过 Protocol/Job 驱动自动发现并建模 Kubernetes、VMware、网络设备、主机、物理服务器、主流数据库与中间件，以及阿里云、腾讯云的云资源（部分中间件与数据库采集对象为 Beta 状态），自动映射为模型实例、属性与关系。

| 能力维度 | 规格 |
|---|---|
| 内置监控采集插件 | 35 个（约 40 余类对象与子对象、约 700 项指标） |
| 内置日志采集类型 | 18 种 |
| CMDB 自动发现采集对象 | 约 40 类（含部分 Beta） |
| 部署方式 | Docker Compose 一键部署，支持在线/离线/幂等安装 |
| 部署版本 | 基础版（无 GPU 要求）、智能版（本地 vLLM，含 OCR/Embedding/Rerank） |
| 私有化 | 支持离线包部署，AI 检索链路可全程内网闭环 |

## 5. 部署形态与渐进建设路径

### 5.1 两种部署版本

BK-Lite 提供基础版与智能版两种安装口径，对应渐进式启用的不同阶段。

**基础版**自带 OpsPilot 管理能力，但不部署本地 vLLM 模型服务，可对接已有私有或公有云大模型，且无 GPU 要求：

```bash
curl -sSL https://bklite.ai/install.run | bash -s -
```

**智能版**在基础版之上部署 vLLM，内置 OCR、Embedding、Rerank 基础模型服务（不含通用 LLM 大模型本身），适合具备 GPU 且对数据私有化要求高的客户：

```bash
curl -sSL https://bklite.ai/install.run | bash -s - --opspilot --vllm
```

两种版本均支持离线包部署，满足强监管与内网环境的私有化诉求。

### 5.2 渐进建设路径

| 阶段 | 行动重点 | 阶段性收益 |
|---|---|---|
| 第一周 | 部署平台，统一入口，纳管核心资产，接入监控与日志 | 资产看清、状态摸明、问题查得到 |
| 第一个月 | 启用告警智能治理与自动化作业 | 噪声收敛、动作自动化 |
| 第三个月 | 打通模块协同，建立运营分析大屏与服务流程 | 全局可视、流程标准化 |
| 持续演进 | 启用 OpsPilot AI 与 MLOps 自训练模型 | 智能问答、根因分析、预测性运维 |

## 6. 客户价值

**效率：从工具切换到一体协同。** 一体化平台消除跨系统的人工搬运，统一门户缩短信息到达路径，内置模板与自动发现使单台主机/中间件的接入从数小时压缩到数分钟。告警收敛、日志秒查与影响分析的组合，显著缩短平均故障恢复时间（MTTR）。

**风险：从依赖个人到依赖机制。** 告警三层治理避免漏报与噪声，作业高危拦截与审计控制变更风险，统一权限、加密与审计为合规提供完整证据链，AI 检索链路私有化满足数据合规。

**成本：从大额自建到轻量渐进。** 一条命令的轻量部署与按需启用，避免大额一次性投入与漫长实施；基础版无 GPU 要求降低起步门槛；私有化与离线能力免去为合规额外改造。BK-Lite 以一个平台替代了过去需多套工具加大量集成才能拼出的能力组合。

## 7. 技术演进展望

BK-Lite 将持续围绕"AI First 的轻量运维平台"演进，重点方向包括：**采集生态持续扩展**，覆盖更多云厂商、信创设备与新型中间件；**AI 能力纵深**，深化 OpsPilot 对各模块的工具化封装与自动化编排，使配置与可观测数据成为 AI 推理的统一事实底座；**运维算法丰富**，在 MLOps 中提供更多开箱即用的异常检测、容量预测与日志聚类模板。所有演进均遵循同一原则：让企业以更低门槛、更短路径，把更强的运维与 AI 能力真正用起来。
