## 序言

当前，企业 IT 运维正处于从"经验驱动"向"数据驱动"的历史性转折节点。海量的监控指标、日志流水与业务遥测数据，作为数字化基础设施的伴生产物，长期以来仅被用于事后查询与人工比对，尚未真正转化为主动预测与智能决策的核心资产。这一现状，折射出 AIOps 落地的结构性困境：技术理念领先而工程路径缺失，算法能力孤立而平台生态断层。

BK-Lite MLOps 模块以运维场景为第一性原则，构建了一套完整的运维场景化机器学习运营体系。其核心主张可以概括为四句话：**"数据即资产、训练即编排、模型即服务、运维即智能"**——运维数据不再是静态历史档案，而是驱动模型迭代的活性资产；训练过程不再是实验室的临时脚本，而是可编排、可审计、可重现的工程化流程；模型不再是算法工程师的私有产出，而是平台级可调用、可治理的推理服务；运维本身，也将从被动响应走向主动预判与自适应智能。

本白皮书系统阐述 BK-Lite MLOps 的技术体系、架构设计、核心算法场景及平台协同机制，旨在为运维架构师、AIOps 实施工程师及平台决策者提供完整的技术参考依据。

## 1. AIOps 落地的时代背景与挑战

### 1.1 阈值告警机制的结构性缺陷

传统运维监控体系以固定阈值为核心判断机制。以 CPU 利用率超过 80% 为例，这一规则在业务高峰期每日可产生数十条触发记录，其中绝大多数属于预期内的正常波动，运维工程师被迫逐条核查、逐条关闭，人力资源消耗巨大。与此同时，真正具有风险信号价值的慢性异常——内存利用率在两周内以每日 0.3% 的速度持续爬升、业务低峰时段出现的非周期性流量抖动——往往在告警阈值触发之前已造成实质性影响，而此类异常恰恰因未超过静态阈值而被系统性忽略。

固定阈值告警存在两个根本性矛盾：一是单一规则无法兼容同类指标在不同服务、不同时段的正常范围差异；二是规则制定依赖人工经验，缺乏从历史模式中自动学习"正常基线"的能力。这两个矛盾共同导致告警系统同时面临误报率过高与漏报率不可控的双重困境。

> **小结：** 阈值告警的局限不在于规则本身，而在于其缺乏对运行环境历史规律的学习能力。解决路径是用自适应模型替代静态规则，而非无止境地调整阈值参数。

### 1.2 容量规划的经验依赖与滞后性

在缺乏量化预测模型的运维环境中，容量规划决策高度依赖工程师的个人经验判断。这种决策模式在两个方向上均存在系统性偏差：保守策略导致资源预留比例过高，计算与存储资源长期处于低利用率状态，产生持续的隐性成本；激进策略则使容量预留不足，当业务增长速度超出预估时，临时扩容流程滞后于实际需求，触发服务可用性风险。

更深层的问题在于，容量规划所需的多维时序数据（QPS、并发数、磁盘增长率等）早已被监控系统持续采集并存储，但在缺乏时序预测工程化能力的前提下，这些数据仅用于事后回溯，无法转化为面向未来的决策依据。"数据已经在那里，但我们没有工具用它"——这是当前容量管理的真实写照。

> **小结：** 容量规划的核心矛盾不是数据不足，而是缺乏将历史时序数据转化为未来趋势预测的工程化工具链。

### 1.3 AI 算法的落地鸿沟

据行业调研，超过七成的 AIOps 项目停留在概念验证阶段，无法进入生产环境。这一现象背后，存在一条清晰可识别的工程断层：算法工程师在研究环境中训练出的模型，需要完成容器化打包、服务注册、端口分配、接口标准化、监控接入等一系列工程化步骤，才能成为运维平台可以调用的生产级 API。这些步骤涉及多个团队的协作分工，任何一个环节的交接延迟或标准不统一，都会导致整体交付周期拉长。

更重要的是，算法工程师通常不具备运维工程的深度背景，而运维工程师通常不具备 ML 工程的完整知识栈。这种能力结构的不对称，使得"最后一公里"的工程化鸿沟成为 AIOps 落地的核心瓶颈，而非算法精度或数据质量问题。

> **小结：** AIOps 的瓶颈不在算法，而在工程化交付链路的完整性。弥合这一鸿沟需要平台级的工程化封装，而非依赖个体的跨域能力补偿。

### 1.4 从训练到上线的协作断层

在传统工作模式下，模型训练完成只是交付流程的起点，而非终点。从"模型文件准备好"到"生产 API 上线"，通常需要经历需求沟通、技术评审、容器化改造、集成测试、发布审批等多个串行环节。根据实践经验，这一周期在跨部门协作场景下普遍需要数天至数周；任何中间环节出现标准不对齐或资源调配问题，均会导致整体进度回退。

这种断层协作模式还带来另一个隐性成本：由于每次迭代成本极高，团队往往倾向于延长单次训练的优化周期，而非快速试错迭代。这与现代机器学习工程"小步快跑、快速验证"的最佳实践背道而驰，进一步降低了模型持续优化的效率。

> **小结：** 高协作成本抑制了模型迭代频率，是 AIOps 能力在组织内自发扩散的核心阻力。解决方案是将多方协作收敛为单一工作台上的自助化全流程。

## 2. BK-Lite 运维场景化 MLOps 技术体系

### 2.1 设计原则

BK-Lite MLOps 的架构设计遵循以下五项原则：

**原则一：场景优先，而非平台优先。** 系统以运维算法场景（异常检测、时序预测、日志聚类等）为第一入口，而非以算法框架或数据格式为组织核心。数据格式约束、算法候选列表、参数配置表单均按场景预置，消除通用平台的适配成本。

**原则二：自助闭环，而非跨团队接力。** 从数据集管理到训练编排，再到模型发布与在线推理，全流程收敛在单一工作台内，目标用户可在不依赖外部团队支持的前提下独立完成全链路操作。

**原则三：可观测性，而非黑盒训练。** 每次训练的参数组合、评估指标曲线、模型版本元数据均通过 MLflow 完整记录，历史任何版本的数据基线和模型制品均可下载与复现，为合规审计和决策溯源提供保障。

**原则四：基础设施复用，而非独立采购。** MLOps 复用平台现有的 PostgreSQL、MinIO、Celery+Redis 基础设施，不引入独立的存储和计算资源，有效控制总拥有成本（TCO）。

**原则五：热插拔扩展，而非硬编码封闭。** 算法配置通过管理界面动态维护，新增算法类型或下线过时算法无需修改业务代码；运行时环境通过环境变量切换 Docker 或 Kubernetes，无需应用层改造。

## 3. 数据集与标注（DataLabel）

### 3.1 样本管理体系

数据集是 BK-Lite MLOps 模型生产流程的起点。系统以"算法场景"为顶层组织单元，每个数据集归属于特定场景，场景类型决定数据文件的格式约束与合法性校验规则。支持的样本格式包括：结构化时序数据（CSV，含时间戳与数值列）、文本日志文件、图像文件（ZIP 压缩包，ImageFolder 格式或 YOLO 格式）及标注配置，对应异常检测/时序预测、日志聚类/文本分类、图片分类/目标检测等不同场景。

样本维护支持逐条上传、批量导入、元信息编辑与删除操作。每条样本支持用途标记：**训练集（train）/ 验证集（validation）/ 测试集（test）**，用途划分在数据集版本发布时固化为训练配置的组成部分，确保实验可重现性。

### 3.2 数据集版本快照机制

数据集版本快照是保障训练可重现性的核心机制。版本发布操作触发后，系统通过 Celery 异步任务完成以下工作：将当前数据集中所有已标记样本打包为不可变 ZIP 压缩包，写入 MinIO 对象存储，并在 PostgreSQL 中记录版本元数据（版本号、创建时间、文件路径、样本数量统计等）。

版本发布后，原始数据集可继续新增或修改样本，但已发布版本的制品不受影响，保持内容不变性。这一机制确保"V1.0 数据集训练了哪些样本"这一问题在任何时间节点均可精确回溯。

版本全生命周期状态如下：

| 状态 | 语义说明 | 可执行操作 |
|---|---|---|
| 待发布 | 已创建，未触发发布动作 | 触发发布、删除 |
| 发布中 | Celery 任务正在执行打包与上传 | 等待（不可操作） |
| 已发布 | 制品就绪，可绑定训练任务 | 归档、下载、启动训练 |
| 发布失败 | 打包或上传过程发生异常 | 重新发布、删除 |
| 归档 | 制品保留但标记为非活跃基线 | 恢复发布、下载 |

版本在"已发布"状态时，支持直接下载完整 ZIP 制品，用于离线分析或迁移至外部训练环境。版本唯一性约束为"数据集 + 版本号"复合主键，同一数据集不允许存在重复版本号。

### 3.3 数据质量保障

系统在样本上传阶段执行场景级格式校验：异常检测与时序预测场景要求 CSV 文件包含有效的时间戳列与数值特征；日志聚类与文本分类场景校验文本文件的编码与格式合法性；图片分类场景校验图像压缩包（ImageFolder 格式）的目录结构与标签映射完整性；目标检测场景校验 YOLO 格式压缩包（images/ + labels/ + classes.txt）的结构合法性。不符合格式约束的样本在入库阶段即被拦截，避免将数据质量问题延迟至训练执行阶段暴露。

## 4. 训练编排（TrainOrchestrate）

### 4.1 训练任务模型

训练任务是训练配置与执行状态的统一实体，包含以下核心属性：绑定的数据集版本、选用的算法配置标识、算法参数填写（由动态表单驱动）、当前执行状态及关联的 MLflow 运行记录集合。

训练任务的状态机如下：

```
    [待训练] ──启动训练──▶ [训练中] ──成功──▶ [已完成]
                                │
                                └──异常──▶ [训练失败]
```

"训练中"状态下，受权限约束可执行停止操作；停止操作通过 Webhookd 向运行时环境发送终止指令，强制回收训练容器资源。防并发机制确保同一训练任务在同一时刻最多存在一个活跃训练进程。

### 4.2 训练链路执行过程

训练任务启动时，Django 应用执行以下有序步骤：

1. 从 PostgreSQL 读取训练任务配置与绑定的数据集版本元数据；
2. 从 MinIO 获取数据集版本 ZIP 制品的访问路径；
3. 将训练配置（数据路径、算法参数、MinIO 连接信息、MLflow 追踪服务地址）序列化为 JSON 并写入 MinIO；
4. 通过 Webhookd 向 Docker 或 Kubernetes 运行时发送容器启动指令，传递配置文件路径及必要的环境参数；
5. Webhookd 完成容器拉起后，训练容器在完全隔离的环境中执行训练逻辑，通过 MLflow 协议实时上报参数与指标；
6. 训练完成后，容器自动将模型制品注册到 MLflow Model Registry，Django 应用轮询状态更新并将任务标记为"已完成"。

### 4.3 算法框架与工程技术

**scikit-learn Pipeline：** 机器学习工作流基于 scikit-learn Pipeline 构建，将数据预处理、特征变换与模型训练封装为串行执行的有序步骤。Pipeline 序列化（Pickle/Joblib）产出的模型制品可通过 MLflow 进行版本化管理与加载复现。

**时序特征工程：** 时序预测场景中，系统通过滑动窗口方法（lag features）将时序数据转换为监督学习格式，提取历史多步特征作为输入，支持多步预测输出。Prophet 模型直接接受时序格式（ds/y 列），由 Prophet 框架内部处理趋势与季节性分解。

**hyperopt 贝叶斯超参数优化：** 超参数搜索通过 hyperopt 库的 TPE（Tree-structured Parzen Estimator）算法实现。与传统网格搜索相比，TPE 算法基于贝叶斯推断，在已有评估结果的基础上自适应地集中采样高收益参数区域，以更少的评估次数获得更优的参数组合。搜索深度由 `max_evals` 参数控制，默认值由各算法配置独立指定。

**MLflow 实验追踪：** 每次训练运行自动创建一条 MLflow Run 记录，包含完整的参数快照（`mlflow.log_params`）、指标历史（`mlflow.log_metric`，支持 step 级别的时序记录）及模型制品（通过 `mlflow.pyfunc.log_model` 以自定义 wrapper 方式保存）。MLflow 后端数据库指向平台共享 PostgreSQL 实例，制品存储指向平台共享 MinIO 实例，不引入独立存储资源。

### 4.4 训练观测能力

训练观测界面提供以下可视化能力：

- **指标曲线追踪：** 实时展示各场景对应评估指标的 step 级别历史曲线，包括准确率（Accuracy）、F1 Score、Precision、Recall（分类/异常检测）、MAE、RMSE、MAPE（时序预测）、模板数量与覆盖率（日志聚类）等；
- **多运行对比：** 同一训练任务下的多次运行记录可并排对比参数组合与最终指标，支持最优版本的快速识别；
- **模型版本列表：** 每次成功训练自动在 MLflow Model Registry 中注册新版本，版本列表展示版本号、注册时间及关联运行记录；
- **制品下载：** 支持按运行记录下载训练产出的模型文件，用于离线验证或迁移部署。

## 5. 模型服务化（Serving）

### 5.1 一键发布机制

能力发布（Serving）模块将训练完成的模型版本转化为具备真实 REST API 接口的在线推理服务。服务创建时，用户指定关联的训练任务与目标模型版本，系统从 MLflow Model Registry 解析对应模型的 URI（格式：`models://{model_name}/{version}`）。

服务启动操作触发后，系统向 Webhookd 发送 serve 请求，携带模型 URI 及必要的运行时参数。Webhookd 执行以下操作：

1. 从指定镜像仓库拉取推理服务镜像；
2. 启动推理容器，容器内进程从 MLflow/MinIO 加载模型制品并初始化 HTTP 服务；
3. 动态分配宿主机端口，将容器推理接口映射至可访问的端口；
4. 通过回调机制将实际分配的端口号返回给 Django 应用，前端据此构建在线推理请求地址。

整个过程无需人工干预，从点击"启动"到服务就绪通常在数分钟内完成。

### 5.2 服务状态模型

服务状态以容器实际运行状态（container_info）为准，系统通过 Webhookd 实时同步容器状态，并在列表查询与详情接口中批量刷新。服务记录中保留一个意图状态字段（active/inactive），但实际推理能力以容器是否处于 running 状态为准：

| 容器运行状态 | 语义 |
|---|---|
| running | 容器实例正在运行，可接受推理请求 |
| stopped | 容器实例已停止 |
| error | 容器启动过程发生异常 |

服务推理能力以容器状态为唯一依据。容器意外停止时，系统保留服务记录并允许用户手动重新启动。

### 5.3 在线推理与验证

每个已发布服务均提供在线推理面板，用户可在接入外部系统之前，直接在浏览器中输入测试数据并即时查看推理结果。在线推理调用通过平台后端代理转发至推理容器，无需用户直接访问容器端口，保障网络隔离与安全性。

推理服务对外暴露标准 REST API 接口，支持以下调用方式：直接 HTTP 调用（用于告警引擎对接）、通过 BK-Lite 平台内部调用（用于其他模块集成）、注册为 OpsPilot 工具（用于智能体工具调用）。

### 5.4 模型版本管理与回滚

每次成功训练自动在 MLflow Model Registry 中注册新版本，版本号单调递增，不可覆盖。当生产环境中的现役模型出现效果退化或推理异常时，运维人员可在服务配置界面切换目标模型版本，触发服务重启并加载指定历史版本的模型制品，整个操作在分钟级内完成，对外接口地址保持不变，上游调用方无感知。

## 6. 运维算法场景（AlgoScene）

### 6.1 场景化算法工程的设计哲学

通用机器学习平台的核心设计目标是算法的普适性与可扩展性，这与运维场景的需求存在根本性错位：运维工程师需要的不是"如何配置一个机器学习实验"，而是"如何让这条指标序列的异常点被检测出来"。AlgoScene 的设计哲学是将算法工程知识封装在场景配置层，将业务问题的描述语言作为用户界面，将算法复杂度作为系统内部关注点。

每个算法场景的数据格式约束、典型算法候选列表、评估指标体系均经过运维场景化裁剪，做到"用什么数据就配什么场景，关心什么结果就看什么指标"。

### 6.2 异常检测场景（anomaly_detection）

**场景定义：** 输入为 CSV 格式的时序指标数据（时间戳 + 数值列），输出为每个数据点的异常评分及二元判断标签。

**核心算法：**

- **ECOD（Empirical Cumulative Distribution Functions based Outlier Detection）：** 基于经验累积分布函数的无监督算法，通过估计每个特征的经验分布来计算异常得分，计算效率高，适合多维指标的批量推理场景。
- **EWMA（Exponentially Weighted Moving Average）：** 基于指数加权移动平均的单变量时序异常检测算法，通过计算 EWMA 基线与滚动标准差，当标准化偏差连续超过阈值时判定异常，适用于单指标平滑基线检测场景。
- **PELT（Pruned Exact Linear Time）：** 基于 ruptures 库的变化点检测算法，通过识别时序数据的统计特性突变点进行异常标注，适用于检测指标趋势或均值的突变事件。

**典型评估指标：** F1 Score、Precision（精确率）、Recall（召回率）。

**典型运维应用：** 将三至六个月的服务器历史指标数据（CPU、内存、网络、磁盘）上传作为训练样本，训练完成的模型以推理服务 API 形式接入告警引擎。新采集的指标批次通过 API 实时推理，当异常评分超过预设阈值时触发高优先级告警。相比固定阈值，异常检测模型通过学习历史正常基线，可有效降低因静态规则导致的重复误报。

### 6.3 时序预测场景（timeseries_predict）

**场景定义：** 输入为多维历史时序数据（如 QPS、在线用户数、资源利用率等多列时序 CSV），输出为未来指定时间窗口的预测值序列，用于容量趋势研判。

**核心技术链路：**

1. **滑动窗口特征工程：** 通过滑动窗口将时序数据转换为监督学习问题，使用历史 N 个时间步（lag features）作为输入特征，支持多步预测；
2. **预测模型：** 支持以下三类算法——**GradientBoosting（sklearn GradientBoostingRegressor）**：适合非线性、复杂模式的时序预测；**RandomForest（sklearn RandomForestRegressor）**：适合高维特征场景；**Prophet（Meta Prophet）**：适合具有季节性与节假日效应的业务时序预测；
3. **hyperopt 超参优化：** 对模型核心超参数执行 TPE 贝叶斯优化，在指定的 `max_evals` 预算内自动寻找最优参数组合。

**典型评估指标：** MAE（平均绝对误差）、RMSE（均方根误差）、MAPE（平均绝对百分比误差）。

**典型运维应用：** 基于六至十二个月的历史 QPS 数据训练预测模型，每日自动输出未来 14 天的容量趋势预测，为弹性扩缩容决策提供量化依据，避免经验驱动决策引发的资源浪费或容量不足风险。

### 6.4 日志聚类场景（log_clustering）

**场景定义：** 输入为原始文本日志文件，输出为每条日志所属的模板类别编号及日志模板摘要，用于日志降噪与异常模式识别。

**核心算法：**

- **Spell（Streaming Parser for Event Logs using LCS）：** 基于最长公共子序列（LCS）的在线日志解析算法。通过逐条处理日志并与已知模板比对，自动提取日志模板并将日志分配至对应类别；支持位置权重配置、倒排索引加速匹配及聚类合并优化；使用 hyperopt 对 tau（相似度阈值）等超参数进行贝叶斯优化。与基于向量距离的聚类算法（KMeans/DBSCAN）不同，Spell 直接提取可读的日志模板，具有较强的可解释性。

**典型评估指标：** 模板数量（num_templates）、覆盖率（coverage_rate）、模板质量得分（template_quality_score）；在提供真实标签时支持分组准确率（grouping_accuracy）、F1 Score。

**典型运维应用：** 将特定服务的历史报错日志批量导入，训练 Spell 模型后，新产生的日志可通过推理 API 实时归类至已知模板。相似的重复日志自动归并至同一模板类别，运维工程师可按模板维度快速定位高频问题；与已知模板相似度低于阈值的日志被标记为未知类型（cluster_id = -1），可进一步人工研判。

### 6.5 文本分类与图像识别场景

**文本分类（classification）：** 输入为带标签的文本样本（CSV 格式，含文本列与类别标签列），当前支持 **XGBoost** 算法（XGBoostTextClassifier），评估指标为 Accuracy 与 F1。典型应用包括告警工单类型分类、日志文本类别标注等。

**图片分类（image_classification）：** 输入为图像文件集合与类别标签配置（ImageFolder 格式），使用 **YOLO 分类模型（YOLOClassification）** 进行训练，评估指标为 Accuracy。支持 GPU 加速，通过算法配置层声明 GPU 资源需求。典型应用包括生产质检图像分类。

**目标检测（object_detection）：** 输入为图像文件与标注框配置，支持 YOLO 系列算法，评估指标为 mAP（平均精度均值）与 IoU（交并比）。

### 6.6 算法热插拔扩展机制

算法配置（AlgorithmConfig）是算法场景与训练容器镜像之间的解耦层。每条算法配置记录包含：算法标识（在场景内唯一）、显示名称、关联场景类型、训练镜像地址、场景描述文本及动态表单配置（`form_config`，JSON 格式，驱动前端参数输入界面的自动渲染）。

平台管理员可在不修改任何业务代码的前提下，通过管理界面完成以下操作：为任一场景添加新算法（提供镜像地址与表单配置即可上线）、停用过时算法（隐藏其对新任务的可见性，不影响已有任务）、更新算法镜像版本。算法配置的唯一性约束为"算法类型 + 算法标识"复合主键。

## 8. 部署与演进路径

### 8.1 部署要求

BK-Lite MLOps 以 Docker Compose 方式部署，与 BK-Lite 平台其他模块共享基础设施。核心依赖服务及端口如下：

| 服务 | 端口 | 用途 |
|---|---|---|
| PostgreSQL | 5432 | 业务元数据、MLflow 后端数据库 |
| MinIO | 9000 / 9001 | 数据集文件、模型制品对象存储 |
| MLflow Tracking Server | 15000 | 实验追踪与模型注册服务 |
| Redis | 6379 | Celery 任务队列消息代理 |
| Webhookd | 8080 | 训练与推理容器生命周期代理 |

**最低资源配置：** 4 核 CPU、16 GB 内存、200 GB SSD 存储。

**GPU 支持：** 可选，通过算法配置层声明 GPU 资源需求，适用于图片分类与目标检测等深度学习场景。

**运行时切换：** 通过环境变量 `MLOPS_RUNTIME` 控制运行时模式（`docker` 或 `kubernetes`），无需修改业务代码。Docker 模式适合单机或小规模私有化部署；Kubernetes 模式适合具备容器编排基础设施的企业级环境。

### 8.2 演进路径建议

**阶段一：异常检测先行（建议第 1-2 个月）**

从运维监控数据最丰富的核心服务指标入手，选取 3-6 个核心监控指标，导出三至六个月历史数据，训练 IForest 异常检测模型并接入告警引擎。以误报率变化为核心验收指标，建立 AIOps 价值基线。

**阶段二：容量预测扩展（建议第 3-4 个月）**

在异常检测验证成熟的基础上，扩展至时序预测场景。选取 QPS、并发数等容量核心指标，训练时序预测模型，为弹性扩缩容决策建立量化支撑。

**阶段三：日志聚类覆盖（建议第 5-6 个月）**

将日志聚类模型接入日志平台智能分析功能，完成监控指标、日志数据与模型推理的三角数据闭环，显著提升根因定位效率。

**阶段四：OpsPilot 工具注册（按需推进）**

将成熟的推理服务 API 注册为 OpsPilot 工具，实现对话式智能运维场景，推动运维能力从被动响应向主动预判演进。

## 9. 客户价值

### 9.1 告警质量提升

基于历史数据训练的动态基线异常检测模型可替代静态阈值规则，有效减少因固定规则对正常业务波动的误判。运维工程师得以从重复处理无效告警的模式中解放，专注于真正需要人工研判的事件，告警响应效率得到改善。同时，基于学习型异常检测模型，慢性异常（内存持续爬升、非周期性流量异常等）的检测能力相比纯阈值规则有所提升。

### 9.2 模型交付路径简化

传统跨部门协作模式下，从模型训练完成到生产 API 上线通常需要经历数天至数周的串行流程。BK-Lite MLOps 将全流程收敛至单一工作台，从训练完成到推理服务就绪的操作路径大幅缩短，通常在数小时内完成，具体时间因环境配置与数据规模而异。具备运维背景但不具备 ML 工程背景的工程师，按照产品引导可在较短时间内完成第一个模型的训练与发布。

### 9.3 容量成本精准优化

时序预测模型提供基于历史数据的量化容量趋势预测，为弹性扩缩容决策提供数字化依据。在预测驱动的决策模式下，资源预留的过度保守（导致长期低利用率成本）与过度激进（导致临时紧急扩容成本）均得到有效抑制，容量管理效率提升。

### 9.4 基础设施成本复用

MLOps 复用平台现有 PostgreSQL、MinIO、Celery+Redis 基础设施，不额外采购独立存储和计算资源，**有效降低总拥有成本（TCO）**。MLflow 作为开源组件无许可费用，整体方案在不增加基础设施规模的前提下获得完整的模型生产能力。

### 9.5 团队能力自主扩散

通过产品化的流程封装，不具备 MLOps 工程背景的运维团队同样可以独立完成模型生产全流程，不依赖专职 ML 工程师的持续支持。AIOps 能力以"产品驱动增长（PLG）"模式在组织内部自发扩散，而非依赖集中式的专项项目推动。

### 9.6 合规审计全程可追溯

数据集版本快照确保每次训练的数据基线精确可溯，MLflow 实验追踪确保每次训练的参数与指标有完整记录，模型版本管理确保任意历史版本可重载与验证。全链路的可追溯性满足企业内部合规审计要求，为 AI 决策的可解释性提供技术保障。

## 10. 技术演进展望

**方向一：在线学习与实时漂移检测。** 当前 BK-Lite MLOps 采用批式训练范式，模型迭代依赖人工触发。后续规划在异常检测与时序预测场景引入模型漂移检测机制，当推理结果的分布特征与训练期统计量出现显著偏差时，自动触发重训练或发出模型更新建议，提升模型在持续变化运维环境中的自适应能力。

**方向二：A/B 测试与灰度流量管理。** 面向对模型版本切换有更高安全要求的生产场景，规划引入流量分割机制，支持新旧模型版本同时在线、按比例分配推理流量，通过真实生产流量验证新版本效果后再完成切换，降低模型升级风险。

**方向三：AutoML 能力增强。** 在现有 hyperopt 超参优化基础上，规划扩展神经架构搜索（NAS）和特征选择自动化能力，进一步降低运维团队在模型配置环节的认知负担，使"选数据、点启动、看结果"成为常态操作路径。

**方向四：联邦学习支持。** 面向多租户与数据隐私合规要求较高的场景，规划支持联邦学习模式，允许多个业务单元在不共享原始数据的前提下协同训练共享的基线模型，同时各自保留本地模型的精调能力。

**方向五：大模型 + 小模型协同。** 随着 OpsPilot 大语言模型能力的深化，规划将传统 ML 模型（异常检测、时序预测）的结构化推理能力与大语言模型的自然语言推理能力深度融合，形成"大模型负责理解语义与生成建议、小模型负责精确数值推理"的协同智能架构。
