## 序言

过去十年，企业 IT 基础设施完成了从物理机到虚拟化、再到容器与多云混合的跨越式演进。算力的形态在变、资源的边界在变、系统的拓扑在变——唯一不变的，是企业对"看清自己拥有什么、它们如何相互连接、又在如何变化"这一基本诉求的执着。配置管理数据库（CMDB），正是支撑这一诉求的核心系统。

然而，传统 CMDB 长期深陷一个结构性困境：它被设计为一个"静态台账"，却被要求服务于一个"动态世界"。建设之初规划周密，三个月后数据失真，半年后无人问津——这几乎是行业的普遍宿命。其根因不在于运维团队不尽责，而在于一套依赖人工录入与维护的系统，本质上无法跟上业务变化的节奏。当 CMDB 的数据不再可信，建立在其之上的告警、变更、容量、审计乃至 AI 运维，都将失去坚实的地基。

BK-Lite 始终秉持"让配置数据从静态台账走向动态可信资产"这一核心理念。我们认为，CMDB 的数据质量不应依赖人的自觉，而应由机制来保障。围绕这一理念，BK-Lite 构建了以**自动发现、图谱建模、主动订阅、全生命周期审计**为支柱的动态配置管理技术体系，使资产数据"自动生长、关系自洽、变化可知、变更可溯"。我们相信，一个可信、实时、关系完整的资产数据底座，不仅是运维效率的来源，更是企业迈向智能运维的前提。

面向未来，BK-Lite 将持续深化图谱智能、自动关联与 AI 协同能力，推动配置管理从"记录系统"演进为"决策系统"，为企业的数字化运营注入持续动能。

## 1. 配置管理的时代背景与挑战

### 1.1 混合多云重塑资产边界

云计算与容器化已成为企业 IT 的主流形态。一个典型的成长型企业，其资产可能同时分布在阿里云、腾讯云、私有 VMware 集群与多个 Kubernetes 集群之上。每个平台都有自己的控制台、自己的资源模型、自己的命名规范。资产的物理边界被彻底打破，而管理的逻辑边界却迟迟没有重建。

这种割裂带来一个朴素却尖锐的问题：当管理者问"我们现在到底有多少台生产主机、多少个数据库实例"，运维团队往往需要跨越多个系统人工汇总，最终给出的还只是一个"大概"的数字。年度资产审计耗时一到两周，结论仍难言准确。资产底数的模糊，是一切运维不确定性的源头。

### 1.2 关系缺失放大故障代价

现代 IT 系统是一张高度耦合的依赖网络。一个 Web 服务依赖应用中间件，中间件运行于虚拟机，虚拟机归属某台物理宿主机，宿主机又连接特定的网络设备。当链路上任一环节发生故障，影响会沿着依赖关系迅速扩散。

问题在于，这张依赖网络在多数企业中并不存在于任何系统里，而只存在于少数资深工程师的记忆中。于是每一次故障的影响评估，都演变为一场依赖个人经验的"人肉排查"。当关键人员休假或离职，"只有他知道这套架构"便成为悬在团队头上的风险。缺乏可信的关系图谱，平均故障恢复时间（MTTR）便难以实质性下降。

### 1.3 人工维护难以对抗数据熵增

配置数据天然趋向失真。资产一旦上线、变更或下线，最先被遗忘更新的往往就是 CMDB。人工维护本质上是在与遗忘和惰性长期对抗，而这是一场注定艰难的消耗战。更深层的矛盾在于：即便引入了自动发现工具，其采集的数据格式也常与 CMDB 模型无法对齐，需要额外的中间层转换，反而抬高了维护成本。数据越来越旧、越来越不可信、越来越无人维护——传统 CMDB 由此陷入恶性循环。

### 1.4 合规审计要求变更全程可溯

等保 2.0（三级）、ISO 27001、金融行业安全规范均明确要求 IT 资产的变更必须可追溯。然而传统 CMDB 的变更记录往往只是粗粒度的操作日志，缺乏业务语义分类，也缺乏字段级的前后值对比。审计时只能手工整理，耗时数周仍难以形成完整证据链。合规场景真正需要回答的是：谁、在什么时间、把什么、从什么值改成了什么值、是人工还是自动、是否经过审批——这是一份结构化的、可即时检索的变更档案，而非一堆无结构的日志。

> **小结**　上述四重挑战的本质，是"静态台账架构"与"动态 IT 世界"之间的结构性错配。破解之道，不在于让人更努力地维护数据，而在于用一套以机制为核心的技术体系，让数据自动保持鲜活、关系自动保持完整、变化自动被感知、变更自动被记录。

## 2. BK-Lite 动态配置管理技术体系（Dynamic CMDB）

### 2.1 设计原则

BK-Lite 动态配置管理技术体系围绕四条设计原则构建，与本白皮书的能力主线一一对应：

- **建模即标准。** 以统一的模型定义资产标准，让跨团队、跨业务线的资产口径从源头一致，杜绝"各说各话"。
- **发现即更新。** 以自动发现持续感知环境，数据的鲜活由机制保障，而非依赖人的自律。
- **关系即洞察。** 以图谱承载依赖关系，让"谁影响谁"成为可即时查询的事实，而非需要回忆的经验。
- **变更即留痕。** 以结构化审计记录每一次变化，让合规与根因分析有据可查。

## 3. 自动发现引擎（AutoDiscovery）

### 3.1 设计目标

资产数据最大的敌人是时间。自动发现引擎的设计目标，是把"数据保鲜"这件事从人的职责转移给系统的机制——配置一次，持续巡检，自动同步。

### 3.2 双驱动采集架构

自动发现引擎采用 Protocol 与 Job 双驱动架构，覆盖"无侵入"与"深采集"两类场景：

- **Protocol 驱动**通过标准协议与 API 直连目标系统，无需在目标节点安装任何 Agent。它由 Stargazer 采集代理执行，对接 Kubernetes API、vSphere SDK、SNMP 以及各云厂商 OpenAPI，适用于容器、虚拟化、网络设备与公有云资源的采集。
- **Job 驱动**通过 NATS-Executor 将采集脚本分发到目标主机执行，适用于主机与中间件等需要深度采集的场景。

两类驱动统一由 Celery 异步任务引擎调度，支持周期巡检、全局超时与单对象超时分别控制，保障大规模采集任务的稳定执行。

### 3.3 采集覆盖范围

自动发现引擎内置 40 余个官方维护的采集对象（含 Beta 状态），覆盖六大资产领域：

| 采集域 | 覆盖对象 |
|---|---|
| 容器 | Kubernetes、Docker |
| 虚拟化 | VMware（vCenter / ESXi / 虚拟机） |
| 网络设备 | 交换机、路由器、防火墙等（基于 SNMP 与内置 SOID 特征库） |
| 数据库 | MySQL、PostgreSQL、Redis、MongoDB（Beta）、Elasticsearch（Beta）、HBase（Beta）、TiDB（Beta）、MSSQL（Beta）等 |
| 云平台 | 阿里云、腾讯云 |
| 主机与中间件 | Linux / Windows 主机、Nginx / Kafka / Zookeeper 等中间件、物理服务器（含磁盘、内存、网卡、GPU 信息） |

### 3.4 关系自动恢复

自动发现引擎的关键差异化能力，在于它不止采集资产清单，更自动推导并建立资产之间的关系。Kubernetes 的"集群→命名空间→工作负载→Pod→节点"五层层级、VMware 的"vCenter→ESXi 主机→虚拟机"三层归属（含数据存储横向关联）、网络设备接口之间的连接关系——这些关系均由系统在采集时自动恢复并持续刷新，无需人工录入与维护。这一能力将"资产清单"直接升级为"关系图谱"，是关系数据可信的根本保障。

### 3.5 采集治理与安全

为确保采集过程可控、可信、安全，自动发现引擎提供以下治理机制：

- **四维采集摘要。** 每次采集完成后清晰呈现新增、更新、删除、关联四类结果统计，影响范围一目了然，各类明细可展开追查。
- **凭据安全。** 采集凭据（账号密码、API Key）通过 AES 加密落库，接口返回统一脱敏，前端不显示明文。

## 4. 关系图谱引擎（GraphEngine）

### 4.1 为什么是图

资产之间的依赖本质上是一张网络，而非一张表格。用关系型数据库的多次 JOIN 去回答"某个 Web 服务依赖的所有基础设施资源"这类跨越四到五个层级的多跳查询，效率会随层级深度急剧劣化。BK-Lite 因此选择以图数据库承载关系，让多跳遍历回归其最自然的表达形态。

### 4.2 双库分工架构

关系图谱引擎基于 PostgreSQL 与 FalkorDB（基于 RedisGraph 演进的图数据库）双库分工：

- **PostgreSQL** 负责存储模型定义、变更记录、订阅规则、公共选项库等结构化元数据，保障强一致性。
- **FalkorDB** 负责存储资产实例节点与实例间的关联关系边，承载图谱的多跳遍历。

这一分工使得"沿依赖链向上下游展开四到五层"这类查询保持毫秒级响应，效率显著优于关系型数据库的多次 JOIN。查询引擎采用参数化 CQL（Cypher Query Language），内置注入防护机制；组织级权限隔离确保不同团队的资产数据相互隔离。

### 4.3 标准化的关系建模

关系图谱并非杂乱的连线，而是建立在标准化建模之上。BK-Lite 提供 6 种关系类型——属于、组成、运行于、安装于、包含、关联——分别承载不同的业务语义；同时支持 1:1、1:n、n:1、n:n 四种映射约束。在此之上，自动关联规则可基于字段值匹配，在满足条件的实例间自动建立或刷新关联，规则变更或数据写入后即触发同步，无需人工干预。

### 4.4 拓扑分析与影响评估

关系图谱引擎对外输出两项关键能力。其一是**轻量拓扑视图**：以任意资产为中心，分层展开上下游依赖，按需加载，大规模环境下依然流畅。其二是**影响评估**：故障发生时，沿关系图谱快速识别受影响的下游业务；变更窗口前，快速评估操作的风险半径。前者将故障影响分析从"人脑回忆"变为"图谱调用"，后者将变更风险评估从"经验拍板"变为"数据支撑"。

## 5. 主动订阅引擎（ActiveSubscribe）

### 5.1 从"人找数据"到"数据找人"

传统 CMDB 是一个被动系统——只有当您主动打开它、主动查询时，它才提供信息。但运维的现实是，关键变化往往发生在您没有盯着屏幕的时候。主动订阅引擎的设计理念，是把信息触达的范式从"人找数据"彻底转变为"数据找人"。

### 5.2 五类触发与精准范围

主动订阅引擎提供五类触发条件，覆盖配置管理的主流感知场景：

- **属性变更触发。** 当指定资产的指定属性发生变化时触发。
- **关联关系变更触发。** 一条规则可同时监听多个关联模型的关系变动。
- **到期提醒触发。** 面向证书、域名、设备保修期等带时效的资产，在临期前自动提醒。
- **实例新增触发。** 当满足条件的新实例进入订阅范围时触发。
- **实例删除触发。** 当订阅范围内的实例被删除时触发。

订阅范围既支持条件筛选模式（动态匹配满足条件的实例），也支持指定实例模式（静态锁定特定对象），兼顾灵活与精确。

### 5.3 智能降噪与多渠道送达

为避免主动通知演变为"消息轰炸"，主动订阅引擎内置智能降噪机制：同一实例在短时间内的多次变更会被合并为单条通知，并在摘要中展示全部变更字段；到期提醒则采用"实例+字段+日期"去重键，确保既不重复打扰、也不遗漏事件。通知可送达站内信、企业微信、钉钉与邮件四类渠道，接收对象可以是具体用户、用户组或角色。订阅规则按组织范围隔离，支持启用、停用、编辑与删除的完整生命周期管理。

## 6. 全生命周期审计（Lifecycle Audit）

### 6.1 结构化的变更档案

合规审计需要的不是一堆操作日志，而是一份结构化、可即时检索的变更档案。全生命周期审计自动记录每一次资产操作，并在两个维度上对其结构化：

- **变更场景**：区分设备流转、关系变更、普通属性变更、自动采集/自动化写入、模型管理变更五类，赋予每次变更明确的业务语义。
- **变更来源**：区分人工操作、自动化、批量导入，清晰回答"这次变更是人改的还是系统改的"。

在此基础上，审计记录精确保留每个字段的前后值对比，形成"谁、何时、把什么、从什么改成什么、人工还是自动"的完整证据链。

### 6.2 结构化检索替代手工整理

审计时，您可以按时间范围、操作人、变更场景等多维度精确筛选，快速检索出所需记录，而不必面对无结构的日志手工整理。配合面向敏感资产的主动订阅，关键变更可实时通知安全团队，实现从"事后审计"到"实时监控"的升级。结构化的变更档案有助于显著缩短审计准备时间。

### 6.3 配置文件版本管理

针对已采集的主机实例，全生命周期审计进一步提供配置文件的完整版本历史与版本间差异对比（Diff）。"是不是配置改动导致了这次故障"——这一运维中最常见的追问，从此有了可直接调取的证据。

## 7. 配置数据底座与平台协同

BK-Lite CMDB 并非孤立的资产台账，而是 BK-Lite 平台数据流向的源头与底座。可信的配置数据一旦建立，将沿协同链路放大整个平台的运维价值（如图 2 所示）。

```
图 2　配置数据底座与平台协同

                    ┌──────────────────────┐
                    │   BK-Lite CMDB 数据底座  │
                    │  实例 · 关系 · 归属 · 变更 │
                    └───────────┬──────────┘
        ┌───────────┬───────────┼───────────┬───────────┐
        ▼           ▼           ▼           ▼           ▼
     监控        告警        作业自动化     OpsPilot    节点管理
   携带资产      影响评估     以实例为      AI 推理的   采集任务的
   上下文与      与动态       执行目标      事实依据     底层执行
   责任人        通知对象                              通道
```

| 协同模块 | 数据流向 | 客户获得的价值 |
|---|---|---|
| 监控 | CMDB 提供资产上下文 | 告警自动携带归属、责任人、架构层级信息，无需人工查询 |
| 告警 | CMDB 支撑影响评估 | 级联影响分析，通知对象随资产归属动态更新 |
| 作业自动化 | CMDB 提供目标清单 | 无需手动维护执行目标，资产变更触发自动化联动 |
| OpsPilot | CMDB 提供知识上下文 | AI 对话基于真实资产数据，分析有据可查 |
| 节点管理 | 节点管理支撑采集通道 | 跨区域采集任务稳定执行，无需额外配置网络通道 |

尤其值得强调的是与 OpsPilot AI 智能体的协同。OpsPilot 通过标准适配器实时读取 CMDB 的资产与关系数据，作为 AI 推理的事实依据。当您询问"某服务的上下游依赖是什么""最近谁修改了这台服务器的配置""这次告警可能影响哪些业务"，OpsPilot 会结合真实数据给出有依据的回答。CMDB 的数据质量越高，AI 的分析就越准确——这正是 AI 运维助手能够真正落地的前提条件。

## 8. 部署与演进路径

### 8.1 轻量部署

BK-Lite 整体采用容器化一键部署，CMDB 模块随平台启动即可使用，无需额外安装、二次开发或专项培训。

```bash
curl -sSL https://bklite.ai/install.run | bash -s -
```

### 8.2 首轮建设闭环（约 30 分钟）

一名运维工程师即可在约 30 分钟内完成"建模型—录数据—建关系—配采集—配订阅"的首轮闭环：

| 步骤 | 内容 | 预估耗时 |
|---|---|---|
| 1. 建立资产模型 | 创建分类与模型，配置核心属性字段（可用模型复制加速） | 约 10 分钟 |
| 2. 录入/导入实例 | 下载对齐模板，批量导入存量数据，导入前预校验 | 约 5 分钟 |
| 3. 建立关系并查看拓扑 | 创建实例关联，打开拓扑视图验证结构 | 约 5 分钟 |
| 4. 配置自动发现 | 选择采集对象、配置范围与凭据，执行首次采集 | 约 5 分钟 |
| 5. 配置数据订阅 | 设置触发条件与通知渠道，启用规则 | 约 5 分钟 |

### 8.3 渐进式演进

BK-Lite CMDB 支持按企业自身节奏渐进建设，每一阶段均产生独立价值：

| 阶段 | 行动重点 | 阶段性收益 |
|---|---|---|
| 第一周 | 建立核心模型，录入存量数据，配置首批自动发现 | 统一资产底数，告别分散台账 |
| 第一个月 | 完善关系拓扑，配置数据订阅，扩展采集对象 | 拓扑图谱可用，关键变更主动通知 |
| 第三个月 | 精细化模型标准，接入监控告警与作业联动 | 与平台模块深度协同，效率系统性提升 |
| 持续演进 | 基于完整数据图谱启用 OpsPilot AI 辅助运维 | AI 驱动的故障分析与运维决策 |

## 9. 客户价值

### 9.1 效率：自动化替代人工重复劳动

自动发现取代多控制台人工巡查，有效减少资产盘点的人工投入，且后续周期盘点几乎无需额外介入。关系图谱预先就绪，故障发生时可快速沿拓扑定位影响范围，有助于缩短故障定位与恢复时间。结构化变更档案支持条件检索，有助于减少审计准备工作量。

### 9.2 风险：从依赖个人到依赖机制

资产关系与变更历史以结构化数据沉淀于系统，而非封存于个别工程师的记忆。关键人员变动不再构成架构知识的断层。变更前有拓扑支撑评估、变更后有订阅保障感知、故障时有审计支撑根因——运维决策从"依赖个人经验"转向"依赖数据机制"。

### 9.3 成本：从持续投入到趋近于零

自动发现覆盖的资产，其日常维护成本趋近于零。团队人力从重复性的数据维护中释放，转投更高价值的运维改进。BK-Lite 的产品主导增长（PLG）理念意味着无需昂贵的实施、咨询与定制开发费用，团队可自助完成从零到可用的全过程。

## 10. 技术演进展望

BK-Lite 将持续推动配置管理从"记录系统"向"决策系统"演进，重点方向包括：**图谱智能化**，引入更丰富的图分析算法，将影响评估从"识别关联"提升为"预测级联"；**关联自动化**，扩展自动关联规则的语义与覆盖面，进一步降低人工建模成本；**AI 深度协同**，深化与 OpsPilot 的联动，使配置图谱成为 AI 运维推理与自动化编排的统一事实底座；**采集生态扩展**，持续丰富采集插件，覆盖更多云厂商、信创设备与新型中间件。我们相信，一个可信、实时、关系完整、可被 AI 理解的配置数据底座，将成为企业智能运维不可替代的基石。

## 附录 B：术语表

| 术语 | 释义 |
|---|---|
| **CMDB** | 配置管理数据库，用于存储和管理 IT 资产（配置项）及其关系的数据库系统。 |
| **Dynamic CMDB** | BK-Lite 动态配置管理技术体系，以自动发现、图谱建模、主动订阅、全生命周期审计为支柱，使配置数据动态保持可信。 |
| **配置项（CI）** | CMDB 中被管理的对象，在 BK-Lite 中等同于"实例"。 |
| **模型 / 实例** | 模型是对一类配置项的结构化描述；实例是模型的具体数据记录。 |
| **关系类型** | 属于、组成、运行于、安装于、包含、关联六种业务语义关系。 |
| **自动关联规则** | 基于字段匹配，在实例间自动建立或刷新关联的规则。 |
| **自动发现** | 通过采集插件自动扫描环境、识别资产与关系并写入 CMDB 的能力。 |
| **Protocol / Job 驱动** | 前者通过 API/协议直连（无需 Agent），后者通过 SSH/Agent 执行采集脚本。 |
| **Stargazer** | 云资源与协议类型采集代理组件。 |
| **FalkorDB** | 基于 RedisGraph 演进的图数据库，承载实例与关系，支持多跳遍历。 |
| **CQL** | Cypher Query Language，图数据库查询语言，BK-Lite 采用参数化查询并内置注入防护。 |
| **变更场景 / 来源** | 对变更业务语义与触发主体的结构化分类，用于审计筛选。 |
| **审批后写入** | 采集结果先经审核再入库的写入模式，用于生产环境防污染。 |
| **MTTR** | 平均故障恢复时间，衡量运维效率的核心指标。 |
| **PLG** | 产品主导增长，强调低门槛自助使用并直接获得价值。 |
