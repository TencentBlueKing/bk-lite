## 1. 采集基础设施的时代背景与挑战

### 1.1 采集代理分散：碎片化管理吞噬运维资源

在规模化企业 IT 环境中，监控采集器、日志收集 Agent 等各类数据采集代理，通常以高度碎片化的方式散布于数百乃至数千台主机之上。各类 Agent 由不同团队在不同时期以不同方式安装，缺乏统一的生命周期管理机制与版本治理规范。当安全漏洞出现、需要紧急升级时，运维团队往往需要逐台 SSH 登录目标主机，手工完成下载、替换、重启的全部操作流程。

在缺乏统一生命周期管理机制的环境中，采集代理的日常运维会消耗运维团队的大量精力。当节点规模增至千台以上，这一问题还将进一步放大，形成显著的规模化运维负债。单次紧急版本升级在人工逐台 SSH 模式下耗时漫长，在升级完成之前，生产环境长期处于版本不一致状态，安全风险窗口无法有效收窄。

> **小结**：采集代理的碎片化部署与缺乏生命周期管理，是导致运维成本居高不下的根本性结构问题，亟需系统性的统一治理手段加以破解。

### 1.2 多云多区域隔离：跨网络边界管控的结构性障碍

现代企业的基础设施早已突破单一数据中心的边界。公有云 VPC、私有机房、灾备中心相互并存，不同网络域之间存在严格的安全隔离策略，直接的网络互通往往受到限制甚至被完全禁止。在这一现实背景下，传统的单中心管控模式面临根本性的技术挑战。

传统应对方案通常选择为每个网络区域单独部署一套独立的管控系统，虽然解决了单区域内的管理问题，却带来了新的困境：管理入口随区域数量线性增长，数据视图高度碎片化，运维协同成本持续叠加。一套配置变更需要在多个独立平台上重复执行，一次全局审计需要跨越多套系统逐一核查，这种"以部署换连通"的代价，实质上是将网络隔离的物理成本转化为了运维效率的永久损耗。

> **小结**：多云多区域的网络隔离是企业基础设施的客观现实，而非临时状态。有效的采集管控平台必须在架构层面原生支持跨域统一视图，而非依赖逐区域独立部署来回避这一挑战。

### 1.3 异构环境适配：架构多样性带来的批量操作风险

国产化战略的持续推进，使得 ARM64 架构服务器在生产环境中的比例显著上升，形成了 x86_64 与 ARM64 并存的异构化主机格局。与此同时，Windows Server 与各类 Linux 发行版的操作系统混合部署，也是绝大多数中大型企业的常态。在这种复杂的异构环境中，批量操作的风险被成倍放大——将为 ARM64 架构编译的采集器安装包误推送至 x86_64 节点，其结果不是操作失败，而是节点上的采集进程可能以异常方式崩溃，批量化的误操作随即演变为批量化的生产故障。

手工管理不同操作系统与 CPU 架构的安装包版本矩阵，极度依赖操作人员的个人经验与主观谨慎性，系统层面没有任何自动化的保护机制。这一缺失在规模化运维场景中是不可接受的安全隐患。架构兼容性校验必须在批量操作发起之前，由平台以自动化方式强制执行，将错误消除在入口处。

> **小结**：异构环境的多样性是事实，而非例外。采集管控平台须在架构层面内建架构一致性校验机制，以系统手段代替人工防护，将低级错误的防线前移至操作入口。

### 1.4 配置粗放治理：安全合规与可观测性的双重缺失

在缺乏统一配置管理平台的环境中，采集器的配置文件通常以手工逐台维护的方式存在。这种模式带来的问题是多维度的：没有版本历史意味着变更无法追溯、回滚无从实施；没有变更审计意味着合规审查时无法提供有效证据链；配置内容不一致意味着采集中断时需要逐台排查，定位"哪台机器的配置出了问题"本身就消耗大量工时。

更为严峻的安全问题是明文密码问题。在传统配置管理实践中，数据库访问密码、API 密钥等敏感凭证，往往以明文形式直接写入采集器配置文件，分散存储于数百台主机之上。CIS 基准明确要求敏感信息不得以明文形式存储于配置文件，等保三级同样对配置文件中的密码管理提出了明确要求。散落于各台主机的明文密码，是信息安全审计中的高频高风险发现项，一次集中审计扫描即可暴露系统性的安全管理漏洞。

> **小结**：配置粗放治理既是效率问题，更是安全合规问题。密文变量管理与配置模板化，是现代采集配置管理体系的基本安全要求，也是通过主流安全合规认证的前提条件。

## 2. BK-Lite 采集节点管理技术体系

### 2.1 设计原则

BK-Lite 采集节点管理技术体系的架构设计，遵循以下五项核心原则：

**统一视图原则**：无论节点分布于何种网络域或云环境，均在同一管理平面呈现统一的节点清单与状态视图，消除信息孤岛。

**安全默认原则**：敏感信息的加密存储、传输与动态注入作为系统默认行为内建于平台，而非依赖用户的主动配置。全链路 TLS 加密覆盖所有节点与平台之间的通信路径。

**架构感知原则**：平台在执行任何涉及安装包推送的批量操作之前，自动感知并校验目标节点的操作系统与 CPU 架构，架构不一致时强制拦截并给出明确提示。

**配置模板化原则**：采集配置以模板化方式统一维护，通过变量占位符实现跨区域、跨环境的差异化参数注入，以一份模板驱动多套环境，消除配置冗余。

**轻量无侵入原则**：控制器以单一可执行文件分发，目标主机无需预装任何运行时依赖（Python、Java 等），向下兼容低版本 SSH 协议与无 sudo 权限的受限环境。

## 3. 技术支柱一：统一纳管（UnifiedManage）

### 3.1 云区域：网络边界感知的逻辑分区治理

云区域（Cloud Region）是 BK-Lite 节点管理体系的顶层逻辑分组单元。其设计理念在于：承认网络隔离是企业基础设施的客观现实，并在此基础上构建能够跨越网络边界、实现统一视图的管理抽象层。

每个云区域对应一个独立的网络域，拥有专属的通信代理环境与变量集合。云区域的关键治理规则包括：区域名称全局唯一性约束，防止命名混淆；系统内置默认云区域不可编辑，保障直连环境的基础可用性；存在关联节点的云区域禁止删除，防止误操作导致的节点孤立；代理地址变更时自动联动更新所有关联环境变量，确保下游配置的一致性传播。

新增云区域后，平台自动初始化服务状态与变量模板。运维人员进入云区域"环境"页面，填写代理 IP/域名并执行平台生成的容器部署脚本，即可完成该区域通信环境的搭建，无需手工编写任何配置文件。

### 3.2 节点全生命周期管理

节点以唯一节点 ID 进行标识，支持多组织归属，查询结果依据当前用户的节点权限进行过滤，满足企业级多租户场景下的访问控制需求。

节点清单视图提供多维度的属性展示，包括节点 IP、活跃状态、操作系统类型、CPU 架构、控制器版本、已安装采集组件及其运行状态。平台支持多条件组合筛选，运维人员能够精准定位如"所有 ARM64 且控制器可升级的 Linux 节点"这类复合条件的节点集合，并直接对筛选结果发起批量操作。

节点活跃状态的判定标准为：最近 **60 秒**内有 Sidecar 心跳上报则判定为活跃，反之为非活跃。该机制确保节点状态反映的是实时运行情况，而非历史快照。当节点被手动从平台中删除后，若其 Sidecar 继续正常上报，系统将自动将该节点重新纳入管理清单，防止因误删导致的管理盲区。

### 3.3 环境管理与通信基础设施

云区域的环境管理功能负责区域通信基础设施的搭建与维护。平台依据代理地址与配置参数，自动生成标准化的容器部署脚本，运维人员将脚本复制到代理主机执行即可完成 Stargazer 与 NATS-Executor 的部署启动。对于基于 Kubernetes 的部署场景，平台提供 K8s 升级引导入口，降低容器化环境的接入复杂度。

环境状态健康卡片实时展示各服务组件（Stargazer、NATS-Executor）的运行状态，支持一键刷新，使运维人员能够快速判断区域通信链路的整体健康水平。

## 4. 技术支柱二：控制器-采集器架构（Controller-Collector）

### 4.1 控制器双组件架构：Sidecar 与 NATS-Executor

部署在每台被纳管主机上的控制器（Controller），由两个功能定位明确分离的子组件构成：

**Sidecar** 专注于本地采集组件的进程生命周期管控，负责采集器进程的启动、停止、重启，以及节点状态与采集器清单的周期性向上上报。Sidecar 是控制器与平台的"感知通道"，平台通过 Sidecar 持续获知每台节点上采集组件的真实运行状态。

**NATS-Executor** 专注于与平台的双向消息通信，接收来自平台的 NATS 消息指令（如文件下载、命令执行、Ansible 任务等），在本地执行后将结果精确回传至平台。NATS-Executor 是控制器的"执行通道"，所有平台发起的操作指令均经由这一路径到达节点。

控制器以单一可执行文件的形式分发（Linux 平台为 `bklite-controller-installer`，Windows 平台为 `bklite-controller-installer.exe`），目标主机无需预装 Python、Java 或任何运行时依赖，向下兼容低版本 SSH 协议与无 sudo 权限的受限环境，实现真正意义上的低侵入性部署。

### 4.2 控制器安装模式：覆盖全场景网络拓扑

控制器提供远程安装与手动安装两种模式，以覆盖不同网络拓扑与安全策略下的安装场景。

**远程安装模式**适用于平台能够通过 SSH 直接访问目标主机的场景。运维人员在平台界面填写目标主机的 SSH 凭据（支持密码认证与密钥认证），平台自动完成安装包下载、解压、安装的全部流程，并通过步骤状态树实时展示每个安装步骤的执行状态。安装失败时，步骤级日志直接在控制台可读，运维人员无需登录目标服务器即可定位问题，显著缩短故障排查时间。

**手动安装模式**适用于 Windows 节点、受安全策略限制无法 SSH 直连的节点，或其他受限网络场景。平台生成携带有效令牌的 PowerShell 安装脚本（Linux 场景为 Shell 脚本），运维人员通过 SCCM、SaltStack 或其他分发渠道将脚本推送至目标节点执行。控制器安装完成后自动回传状态，管理员在进度面板实时查看每台节点的安装结果，无需登录目标主机进行人工核查。手动安装节点具备独立的状态流转机制（等待安装→安装成功），支持状态精确追踪。

批量控制器操作规则要求所选节点操作系统与安装方式保持一致；远程安装/卸载面向同操作系统且安装方式一致的节点批次，防止操作模式混淆导致执行异常。

### 4.3 采集器批量部署与生命周期管控

采集器（Collector）是运行在节点上执行实际数据采集任务的组件，包括 Telegraf、Vector、Filebeat、Kafka-Exporter 等多类主流采集生态组件（完整支持矩阵见附录 A）。

采集器的批量部署流程由任务驱动架构支撑：运维人员在节点列表中通过多条件筛选选定目标节点集合，从组件库中选择目标采集器与版本，平台自动感知每台节点的操作系统与 CPU 架构，从安装包库中匹配对应架构的安装包进行推送。批量采集器操作的前提条件是所选节点操作系统与 CPU 架构一致；不满足条件时平台强制拦截并给出明确提示，将架构误配风险消除于操作入口。

对已安装的采集器，平台支持在节点层面执行启动、停止、重启操作；失败的安装任务与运行操作支持一键重试，无需重新发起整批任务，显著降低局部失败的处理成本。批量操作能力显著压缩了采集器全量升级的工时，大批量节点的升级操作从人工逐台模式的数天级工作量转变为集中发起、平台托管执行的自动化流程。

节点首次纳管时，系统依据节点操作系统为默认运行的采集器自动初始化预置主配置；若节点已存在自定义主配置，则不执行覆盖，充分尊重已有配置的个性化设置。对节点采集器执行启动或重启时，若节点尚无该采集器主配置且采集器提供默认模板，系统自动补建默认主配置后执行操作，实现"开箱即用"的初始化体验。

## 5. 技术支柱三：配置分层与变量（LayeredConfig）

### 5.1 主/子配置分层架构

BK-Lite 采集节点管理采用主配置与子配置的两级分层配置架构，在配置灵活性与管理可控性之间实现精确平衡。

**主配置**（Main Configuration）定义采集组件的全局基础参数，构成节点上该采集器运行的配置基础。核心规则是同一节点在同一采集器维度，仅保留一份生效的主配置，消除配置冲突的可能性。主配置可在节点详情抽屉内直接编辑。

**子配置**（Sub Configuration）是针对具体采集任务的模块化配置片段，同一节点可关联多个子配置，按排序序号与创建时间顺序生效，形成灵活的配置叠加机制。子配置与主配置的解耦设计，使得新增一个采集指标项无需修改全局主配置，变更影响范围精确可控，配置变更的安全性大幅提升。

采集配置与节点之间构成多对多关联关系：一份配置可关联多个节点，一个节点可关联多个配置，支持批量绑定与批量解绑操作，适应多节点统一配置管理的规模化场景。配置记录按云区域归属管理，确保跨区域配置的隔离与一致性。

### 5.2 变量管理与密文动态注入

变量管理是 BK-Lite 采集节点管理安全合规体系的核心组成部分，也是实现配置模板跨区域复用的技术基础。

云区域变量以"云区域 + 变量键"维度唯一标识，支持普通类型与密文类型两种变量类型。**密文类型变量**在 Web 界面默认脱敏展示，在传输路径上加密保护，不以任何明文形式暴露给不具备相应权限的操作人员。编辑密文变量时，若未显式输入新值，系统保留原始密文值，防止意外清空带来的服务中断。

变量注入机制的工作原理为：采集配置模板中以占位符引用变量键（如 `${MYSQL_PASSWORD}`），Sidecar 在向平台拉取配置时获得渲染后的完整配置内容，敏感凭证通过环境变量在采集组件的运行时动态注入，任何中间环节的配置文件中均不出现明文密码。这一机制直接满足 CIS 基准、等保三级等主流安全标准对"敏感信息不得明文存储于配置文件"的合规要求。

代理地址变更时，系统自动联动更新所有关联的环境变量，确保配置一致性在区域层面的完整传播，消除手工同步带来的遗漏风险。

### 5.3 ETag 增量配置同步协商

在大规模节点集群中，如果每次配置同步都执行全量数据传输，平台 API 层面的处理压力与网络带宽消耗将随节点数量线性增长，形成不可接受的性能瓶颈。BK-Lite 采集节点管理通过引入基于 HTTP ETag 的增量配置同步协商机制，从根本上解决这一问题。

ETag 协商机制的工作流程如下：Sidecar 在周期性配置同步请求中携带上次同步时服务端返回的 ETag 值；若配置内容自上次同步以来未发生变化，服务端返回 HTTP 304 Not Modified，Sidecar 直接沿用本地缓存，无需传输任何配置数据；当配置发生变更时，缓存版本随之刷新，服务端返回新的 ETag 值及完整配置内容，确保变更被及时同步到目标节点。

这一机制在配置稳定期（无变更场景）大幅降低平台 API 处理压力与网络带宽消耗；在配置活跃变更期，变更能够在 Sidecar 的下一个同步周期内被精确感知并拉取，兼顾了资源效率与变更及时性。

## 6. 技术支柱四：安装包与组件治理（PackageGov）

### 6.1 安装包版本治理：四元组唯一性约束

安装包版本管理混乱是大规模异构环境中批量操作失败与生产故障的高频根因。BK-Lite 采集节点管理引入四元组唯一性约束，从源头建立安装包版本的严格治理规范。

四元组的构成为：**操作系统（OS）+ CPU 架构（Arch）+ 对象（Object）+ 版本（Version）**。在这一约束下，同操作系统、同 CPU 架构、同对象、同版本的安装包不允许重复上传，彻底消除版本混淆的技术可能性。安装包路径遵循标准格式 `installer/<os>/<arch>/<filename>`，以目录结构在存储层强化架构隔离。

上传时，平台对安装包文件名执行命名规范合法性校验，并自动识别版本信息；不符合命名规范的安装包在入口处被前置拦截，确保进入系统的每一个安装包都满足版本管理标准。删除安装包操作同步删除对应的存储文件，防止存储层面的孤立文件积累。

### 6.2 组件库：统一的采集组件注册与分发中心

组件库（Component Library）是 BK-Lite 采集节点管理中采集器组件的统一注册与发现中心。运维人员通过组件库管理可在节点上部署的采集组件，支持按应用标签、系统标签与 CPU 架构维度进行筛选，快速定位适合特定场景的采集器。

组件库支持采集器的新增、编辑与删除，支持对已注册采集器关联安装包进行版本维护，确保从组件注册到安装包管理的完整闭环。批量安装操作从组件库中选定目标采集器与版本后，平台自动完成各目标节点的架构匹配与安装包选取，操作人员无需关注底层的架构差异细节。

### 6.3 令牌安全分发机制

自动化安装流程中，安装脚本与安装包的下载链接携带有效令牌，以防止未授权访问与凭据泄露。BK-Lite 采用双重令牌限制策略：

- **安装脚本令牌**：有效期 **30 分钟**，单令牌最多使用 **5 次**；适应批量安装场景的时间窗口，同时限制单令牌的使用范围。
- **安装包下载令牌**：有效期 **10 分钟**，单令牌最多使用 **3 次**；对安装包下载施加更严格的时间与次数双重约束。

令牌过期或超出使用次数后自动失效，即使令牌被截获，攻击者也无法利用过期令牌访问安装资源。这一机制为安全审计提供了可追溯的访问控制证据，满足金融、政务等行业对自动化操作凭据管理的合规要求。

## 8. 部署与演进路径

### 8.1 快速入门：五步完成首次节点纳管

BK-Lite 采集节点管理遵循"引导式渐进上手"设计理念，运维人员无需预先培训，参照以下五步即可完成从零到首个节点纳管并开始数据采集的完整过程。

**第一步：创建云区域**
进入 BK-Lite 控制台 Node 模块，点击"新增"创建云区域，填写区域名称与代理 IP/域名。若为直连环境，系统内置默认云区域可直接使用，跳过此步骤。

**第二步：部署区域通信环境**
进入云区域详情页"环境"标签，填写代理 IP/域名，点击"生成部署脚本"，将容器部署脚本复制到代理主机执行，等待环境状态卡片显示"正常"即可。

**第三步：安装节点控制器**
Linux 节点选择远程安装模式，填写 SSH 凭据，平台自动完成全程并实时展示步骤状态树。Windows 节点选择手动安装模式，复制 PowerShell 脚本至目标节点执行。控制器安装完成后节点自动出现在清单中。

**第四步：批量部署采集组件**（数分钟内覆盖多台）
在节点列表勾选目标节点，通过"托管组件→安装组件"选择采集器与版本，平台自动匹配架构并推送，安装进度实时可见，失败节点一键重试。

**第五步：下发采集配置**
进入节点详情抽屉，编辑主配置与子配置，点击"应用"下发至节点。Sidecar 收到配置后自动触发采集器重载，数据采集即刻生效。

### 8.2 规模化演进建议

随着被纳管节点规模的扩大，建议按以下路径逐步推进平台能力演进：

**阶段一（初期接入）**：完成基础云区域搭建与控制器部署，建立节点清单统一视图，实现主要采集组件的统一安装与配置管理。

**阶段二（规模扩展）**：引入云区域变量管理，推进配置模板化改造，将散落的明文密码迁移至密文变量机制，实现跨区域配置的统一化治理；建立安装包版本管理规范，推行四元组唯一性约束。

**阶段三（批量治理）**：全面运用批量操作能力，利用 ETag 增量同步机制降低大规模集群的配置同步压力；建立采集组件的版本治理规范，定期执行批量升级以维持版本一致性；充分发挥多组织归属与权限过滤能力，支撑多团队协同运维场景。

**生态集成扩展**：持续扩展采集器支持矩阵，引入更多云原生监控（如 OpenTelemetry Collector）与安全审计（如 Falco）生态组件；深化与主流 CMDB 系统的双向集成，强化资产数据的一致性保障机制。

### A.4 关键业务规则速查

| 规则域 | 关键规则 |
|---|---|
| 云区域 | 名称全局唯一；默认区域禁止编辑；存在节点时不可删除；代理地址变更联动更新关联环境变量 |
| 节点 | 以节点 ID 唯一标识；支持多组织归属；查询结果按用户权限过滤；活跃判定：60 秒内有上报；手动删除后若 Sidecar 继续上报则自动重新纳管 |
| 控制器批量操作 | 远程安装/卸载要求同 OS + 同安装方式；Windows 节点走手动安装 |
| 采集器批量操作 | 要求所选节点 OS 与 CPU 架构一致，否则系统拦截并给出提示 |
| 采集配置 | 同节点同采集器仅一份生效主配置；子配置按排序序号与创建时间顺序展示 |
| 配置初始化 | 节点首次纳管自动初始化预置主配置；已有自定义主配置时不覆盖 |
| 变量安全 | 密文变量默认脱敏；未输入新值时编辑操作保留原值 |
| 安装包 | 四元组维度唯一，不可重复上传；删除时同步删除存储文件 |

## 附录 B：术语表

| 术语 | 解释 |
|---|---|
| 云区域（Cloud Region） | 节点资源的顶层逻辑分组单元，对应一个独立网络域，拥有专属通信代理环境与变量集合，用于隔离不同场景的节点资源 |
| 控制器（Controller） | 部署在被纳管主机上的管理组件，由 Sidecar 与 NATS-Executor 组成，负责采集组件的生命周期管控与平台通信 |
| Sidecar | 控制器的进程管控子组件，负责节点侧采集组件的启动/停止/重启，以及节点状态与采集器清单的周期性上报 |
| NATS-Executor | 控制器的消息执行子组件，接收来自平台的 NATS 消息指令并在本地执行，将结果回传平台；同时兼任区域通信代理角色 |
| Ansible-Executor | 基于 NATS RPC 的 Ansible 执行组件，支持 ad-hoc 与 Playbook 两种执行模式，为作业模块提供节点侧执行能力 |
| 采集器（Collector） | 运行在节点上执行数据采集任务的组件，如 Telegraf、Vector、Filebeat 等 |
| 主配置（Main Configuration） | 定义采集组件全局基础参数的配置文件，同节点同采集器仅有一份生效 |
| 子配置（Sub Configuration） | 针对具体采集任务的模块化配置片段，可多个叠加，按排序序号顺序生效 |
| Stargazer | BK-Lite 平台侧通信代理服务，部署在云区域代理主机上，负责跨网络域的消息路由；不作为节点侧采集器管理，通过容器部署脚本独立部署 |
| ETag 缓存协商 | 基于 HTTP ETag 的增量配置同步机制，内容未变化时服务端返回 304，Sidecar 复用缓存，仅在内容变化时传输全量数据 |
| fusion-collectors | BK-Lite 控制器安装包的默认包名称，安装目录为 `/opt/fusion-collectors`（Linux）或 `C:\fusion-collectors`（Windows）；控制器由 Sidecar 与 NATS-Executor 组成，与同名安装包绑定分发 |
| 安装包令牌 | 带有效期的临时下载凭证：安装脚本令牌（30 分钟/最多 5 次），安装包下载令牌（10 分钟/最多 3 次） |
| 变量注入（Variable Injection） | 通过云区域环境变量在采集组件运行时动态替换配置模板占位符的机制，避免敏感信息硬编码 |
| SidecarApiToken | 节点身份鉴权令牌，与节点 ID 绑定，覆盖 Sidecar 与平台核心服务之间的所有 HTTP 与 NATS 通信 |
| 四元组唯一性约束 | 安装包版本管理规则：同"操作系统 + CPU 架构 + 对象 + 版本"组合的安装包不允许重复上传 |
| 节点活跃状态 | 最近 60 秒内有 Sidecar 心跳上报的节点判定为活跃，反之为非活跃 |
| NATS JetStream | 高性能消息总线，提供低延迟发布/订阅与请求/响应消息模式，作为平台与节点之间任务指令与状态数据的传输通道 |
