# 系统管理 · 应用

## 1. 背景与目标

应用管理平台各应用的访问控制：角色、数据权限与自定义菜单，是全平台 RBAC 的核心，决定"谁能进哪个应用、看哪些菜单、访问哪些数据"。

## 2. 范围定义

- 角色：应用维度的角色与菜单授权。
- 数据权限：组织维度的数据访问规则。
- 自定义菜单：按应用定制菜单结构。

## 3. 关键能力

### 3.1 角色

- 角色按应用划分，每个角色记录其可访问的菜单清单。
- 权限以"菜单-操作"的动作粒度表达（如新增、编辑、查看），后端按权限拦截接口。
- 超级管理员拥有全部菜单；普通用户按所属角色获得菜单。
- 平台为应用预置内置角色，初始化时按既定菜单授权自动建立，开箱即用：
  - 系统管理应用内置安全管理员角色，授予用户与用户组管理、认证源管理、安全策略管理权限；当启用敏感信息能力时，自动追加敏感信息的查看/新增/编辑/删除权限。
  - 系统管理应用内置审计角色，授予审计日志与错误日志的查看权限。
- 重复初始化时内置角色按其菜单清单幂等更新，不会重复创建。

### 3.2 数据权限

- 按应用、按组织配置数据访问规则，限定该组织用户在该应用可见的数据范围。
- 同一组织在不同应用可有不同的数据规则。

### 3.3 自定义菜单

- 支持按应用创建自定义菜单组（菜单树），每个应用同一时间仅一个生效。
- 允许在不改代码的前提下做菜单结构定制；定制仅作用于前端菜单展示，不改变基于角色的实际权限。

## 4. 关键规则

- 角色为扁平结构（无层级），权限由"菜单 + 角色"映射决定。
- 内置角色随应用初始化预置；具有用户组查看权限的角色（如安全管理员）也可查看角色树，以便在其权限范围内维护用户与角色关系。
- 数据权限由消费方应用按组织上下文过滤数据落地。
- 菜单限制仅在接口/界面层通过权限拦截生效，不在数据层做行级约束。

## 5. 关键技术架构选择

- 系统管理是全平台中央 IAM，其他模块通过其权限映射校验访问。
- 权限在前端做导航守卫、在后端做接口拦截；数据访问再按组织数据规则过滤。

## 6. 验收标准

- 管理员可配置应用角色及其菜单授权，用户按角色获得对应菜单与操作权限。
- 系统管理应用初始化后存在安全管理员、审计等内置角色，分别具备对应的管理与查看权限，且重复初始化不产生重复角色。
- 数据权限可按应用、按组织限定数据可见范围。
- 自定义菜单可按应用定制并生效，且不影响实际权限判定。
