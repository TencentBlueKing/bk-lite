# Dev Task List 编写规范

## 1. 目标
- Dev Task List 用于定义“从 Tech Plan 到开发执行前”的可落地任务与检查门禁。
- 文档应支持：开发前评审、任务分配、执行跟踪、发布前验收。

## 2. 目录结构（强制）
- 必须与 `solution_plan` / `tech_plan` 保持同层级结构：`<模块>/<日期>/<同名文档>.md`。
- 推荐示例：
  - `spec/dev_task_list/监控系统/20260228/搜索页面重构.md`

## 3. 必须包含
- 关联文档：对应 Solution Plan、Tech Plan 路径。
- 开发前检查（Gate）：上线前必须确认的前置条件。
- 开发任务清单：至少包含 P0/P1 分层（可增加 P2）。
- 联调任务：跨角色协同事项。
- 验证清单：Done 到 Verified 的验收项。
- 风险与阻塞记录：问题、影响、处理策略、状态。
- 发布批次建议：按能力拆分发布顺序。

## 4. 任务字段规范（强制）
- 每个任务至少包含：
  - `ID`
  - `任务`
  - `负责人`
  - `依赖`
  - `预估`
  - `状态`
  - `验收标准`
- 状态流转统一为：`Todo -> In Progress -> Blocked -> Done -> Verified`。
- 任务 ID 建议格式：`FE-P0-01`、`BE-P1-02`、`QA-P2-01`、`INT-01`。

## 5. 表达要求
- 以“可执行、可验收、可回滚”为优先。
- 任务描述要避免大而空，必须落到可交付结果。
- 验收标准应可直接用于测试或评审，不使用模糊词。

## 6. 与其他文档分工
- 不替代 Requirements：不重复业务背景与需求来源。
- 不替代 Solution Plan：不重复阶段目标描述。
- 不替代 Tech Plan：不展开技术方案细节，仅承接为执行任务。

## 7. 开发前检查门禁（最小集合）
- 关键兼容链路已确认（如 URL 参数、历史入口、权限口径）。
- 并发与状态一致性策略已确认（请求取消、幂等、覆盖规则）。
- 数据结构版本策略已确认（schema_version、默认值补齐、兼容策略）。
- 发布与回滚路径已确认（开关、版本回退、观测指标）。
- 测试范围已确认（前端/后端/联调/关键 E2E）。

## 8. 完整性检查（每次生成后必做）
- 是否与 `solution_plan` / `tech_plan` 同步同名、同日期、同模块。
- 是否存在未声明依赖的任务（孤立任务）。
- 是否存在无验收标准的任务。
- 是否存在缺少负责人或状态的任务。
- 是否覆盖发布前门禁与回滚演练项。

## 9. 人工判断项要求
- 对以下内容必须显式标记“需人工判断/确认”：
  - 任务优先级取舍（P0/P1 边界）。
  - 人员分配与排期冲突。
  - 发布批次拆分与变更窗口选择。
- 无法自动确定的项，必须进入“待确认项（TODO）”。

## 10. 待确认项（TODO）要求
- 若存在不确定项，文末必须有 `待确认项（TODO）` 段落。
- TODO 要逐条写清“问题 + 影响范围 + 需要谁确认”。
- 若无待确认项，明确写 `- 无。`
