# 监控系统｜集成自建 API 报数模版方案

## 1. 方案概览

本次需求不是在现有“内置采集模版”基础上再增加一种复杂配置页，而是新增一类 **自建 API 报数模版**，用于给第三方系统提供标准化报数入口。第三方按模板定义的指标结构和接入规范，通过平台开放 API 主动上报监控数据，平台完成认证、校验、异步投递。

现有模版与本次模版的差异如下：

- **现有内置模版**
  - 平台内置
  - 自带配置逻辑与 UI
  - 主要用于平台主动采集外部数据
- **本次新增 API 报数模版**
  - 用户自建
  - 协议固定
  - 接入页以展示和说明为主
  - 本质是“第三方主动向平台报数”

整体策略如下：

- **展示层统一**：在监控集成模版列表中与老模版一起展示
- **实现层拆分**：自建 API 报数模版不强行复用老模版配置逻辑，独立承载模板定义、接入说明与报数链路
- **链路标准化**：统一使用固定 API 协议、固定鉴权方式、固定异步投递流程

---

## 2. 背景与目标

当前监控集成能力以内置模版为主，适合标准化采集场景，但无法覆盖“第三方业务系统主动通过 API 上报监控数据”的诉求。对于这类接入，用户需要的不是复杂的持久化配置流，而是完整的模板定义、指标维护、接入说明页、实例浏览/创建能力以及真实报数链路，才能形成可用闭环。

### 本次目标

建设一套完整的“自建 API 报数模版”能力，覆盖：

1. 模版定义与管理
2. 模版指标结构维护
3. 接入说明页与实例管理
4. 接入文档生成
5. 开放 API 报数入口
6. 认证、校验、异步投递到 NATS
7. 基于现有实例体系的轻量接入闭环

## 4. 关键业务规则

### 4.1 模版规则

1. 模版类型为“自建 API 报数模版”
2. 模版必须绑定一个监控对象
3. `template_id` 由用户输入，作为模板唯一标识
4. `template_id` 创建后不可修改
5. 模版绑定对象创建后不可修改
6. 名称、描述允许修改

### 4.2 接入说明页规则

1. 创建模板后保留接入页
2. 接入页主要用于展示：
   - API
   - token
   - 请求体格式
   - 实例列表
3. 接入页展示内容默认不入库
4. 页面上真正执行并落库的动作只有：
   - 新建资产（单条创建实例）
   - 导入资产（批量创建实例）

### 4.3 token 规则

1. token 来源于系统管理
2. token 不在“模版定义”阶段入库
3. 接入页按当前用户启用组织查询系统管理 token
4. 若查到多条，默认取第一条
5. 若未查到，提示用户去系统管理创建
6. 第三方报数时使用该 token
7. 服务端通过系统管理校验方法完成认证

### 4.4 资产规则

1. 资产复用现有监控资产实例体系
2. 接入页实例列表仅用于浏览与展示 `instance_id`
3. 不新增模板级实例绑定表
4. 当前真正执行并落库的动作仅包含：
   - 新建资产（单条创建）
   - 导入资产（批量创建）
5. 导入字段暂定：
   - `id`
   - `instance_name`
6. 导入遇到重名资产时：
   - 跳过
   - 提示用户

### 4.5 指标规则

1. 同一模板内指标组不允许重名
2. 不同模板之间允许同名
3. 建议同一模板内指标名同样保持唯一
4. 第三方报数指标必须属于模板已定义指标

### 4.6 删除规则

删除模板时：
- 删除模板本身
- 删除模板相关指标组/指标
- 保留资产本体

## 5. 总体方案

整体采用“**模板定义 + 接入说明 + 开放报数 + 异步投递**”四层架构。

### 5.1 模板定义层

负责定义：
- 模版基础信息
- 绑定监控对象
- 指标组
- 指标结构

该层只描述“第三方应该按什么结构报数”，不负责模板级组织、token、资产绑定持久化。

### 5.2 接入说明层

负责展示：
- 当前用户启用组织下可用的系统管理 token
- 报数 endpoint
- 请求体格式与示例
- 当前对象下实例列表（供用户查看 `instance_id`）
- 新建资产与导入资产入口

该层不是持久化配置承载层，除实例创建类动作外，其余展示信息默认不入库。

### 5.3 开放报数层

对第三方提供统一 API：
- 固定 URL
- 固定请求协议
- 固定认证方式
- 固定返回风格

### 5.4 异步投递层

接口接收请求后：
1. 做同步校验
2. 投递异步任务
3. 由异步任务完成格式转换并发送到 NATS

该设计用于保障：
- 接口响应稳定
- 不阻塞第三方请求
- 复用现有 Celery/NATS 能力
- 与当前轻量接入模式保持一致

### 7.1 模版列表页

在现有监控集成模版列表中，统一展示：
- 现有内置模版
- 自建 API 报数模版

自建模版支持操作：
- 新建
- 编辑
- 删除
- 查看详情

建议在卡片增加如下标识：
- 自建
- API
- 绑定对象

### 7.2 模版详情页

自建 API 报数模版详情页建议拆成以下页面区域：

1. 基本信息
2. 接入说明
3. 指标
4. 接口文档

设计原则：
- 不复用老模板那种复杂采集配置页
- 以“定义 + 说明 + 文档”为核心

### 7.3 接入说明页

接入说明页承载展示与轻量操作，需包含：

1. 当前组织 token 展示
2. token 缺失提示
3. API endpoint
4. 请求体格式示例
5. 实例列表（浏览用）
6. 新建资产
7. 导入资产

交互原则：
- 页面以展示为主，不额外保存模板级接入配置

- 实例列表主要用于让用户查看 `instance_id`

### 7.4 接口文档页

用于对外输出标准接入文档，建议固定呈现：

1. API endpoint

5. 响应示例
6. 错误码说明（如有）

目标：
页面可直接作为第三方接入说明使用。

## 8. 报数链路设计

### 8.1 同步处理阶段

第三方请求进入后，服务端完成以下同步动作：

1. 基础结构校验
2. token 校验
3. 模版存在性校验
4. 模版状态校验
5. 实例存在性校验
6. 实例组织归属校验
7. 指标结构校验
8. 提交异步任务

返回原则：
同步接口返回“accepted / rejected”语义，不等待异步投递结果。本期采用批量 `instances[]` 上报结构，非法实例或指标按过滤处理，合法数据进入异步链路。

### 8.2 异步处理阶段

异步任务负责：

1. 读取任务参数
2. 将报数 JSON 转换为与 stargazer 对齐的数据格式
3. 发送到 NATS
4. 失败重试
5. 记录错误信息

### 8.3 异步技术方案

本期建议采用：

- API 接口同步接收
- Celery 作为异步执行器
- Celery 任务执行数据格式转换并投递到 NATS

选择原因：
1. 仓库已有 Celery 基础
2. 可做削峰
3. 可重试
4. 可避免 API 请求线程直接持有 NATS 投递压力
5. 可与 stargazer 的投递格式保持一致

推荐模式：
采用“**同步校验 + Celery 异步 + NATS**”模式，而非“API 直接发 NATS”。

## 10. 风险与应对

### 风险 1：将新模板硬塞进老模板配置体系，导致后续维护复杂
**应对：**
- 展示层统一
- 实现层拆分
- 不复用老模板采集配置逻辑

### 风险 2：token 依赖系统管理，跨系统口径可能不一致
**应对：**
- 由监控模块封装查询与校验能力
- 前端不直接承载系统管理业务逻辑

### 风险 3：开放 API 同步直发 NATS 导致性能和稳定性问题
**应对：**
- 采用“同步校验 + Celery 异步 + NATS”模式

### 风险 4：报文校验不严导致脏数据进入下游
**应对：**
- 同步阶段做模板、实例、组织、指标多层校验
- 非法数据按实例维度过滤并记录错误日志

### 风险 5：资产复用现有监控实例体系后，业务边界可能有偏差
**应对：**
- 本期按 MVP 复用
- 后续如发现边界不适配，再单独抽象资产体系

### 风险 6：NATS 消费端未就绪，影响端到端闭环
**应对：**
- 本期优先保证可靠接收与投递
- 通过 Celery 重试与错误日志支持补偿

## 11. 回滚与降级策略

### 11.1 回滚原则
- 仅回滚新增自建 API 模版能力
- 不影响现有内置模版与老采集链路

### 11.2 回滚方式
1. 隐藏前端新模板入口
2. 关闭开放报数接口
3. 停止对应 Celery 任务
4. 不删除资产数据

### 11.3 降级策略
- 若 NATS 异常：异步任务重试并输出错误日志
- 若系统管理 token 服务异常：报数接口拒绝请求并提示认证失败
- 若导入不稳定：临时关闭导入，仅保留手工新增资产
