# Solution Plan 编写规范

## 1. 目标
- Solution Plan 用于描述“如何分阶段落地需求”。

## 2. 必须包含
- 方案概览：总体思路。
- 分阶段计划：里程碑、每阶段交付。
- 验收清单：阶段性完成标准。
- 风险与应对：主要风险及缓解措施。

## 3. 表达要求
- 强调执行路径与交付节奏。
- 可包含过程性安排，但不替代更新日志。

## 4. 与其他文档分工
- 不替代 PRD（PRD写最终状态）。
- 不替代 Tech Plan（Tech Plan写技术细节）。

## 5. 完整性检查（每次生成方案后必做）
- 必须执行一次“遗漏项检查”，至少覆盖：
	- 范围边界是否清晰（In Scope / Out of Scope）。
	- 依赖方是否齐全（上下游系统、角色协同）。
	- 验收口径是否可执行（可验证、可回归）。
	- 风险与回滚是否可落地（失败路径、降级策略）。
- 若发现不确定项，必须在文末新增“待确认项（TODO）”，逐条列出，不可省略。

## 6. 人工判断项要求
- 对以下内容必须显式标记“需人工判断/确认”：
	- 业务取舍（优先级、范围裁剪、体验取舍）。
	- 规则口径冲突（跨文档或跨系统定义不一致）。
	- 发布策略选择（灰度范围、回滚阈值、影响面评估）。
- 未能自动确定的结论，不得伪确定，统一进入“待确认项（TODO）”。
