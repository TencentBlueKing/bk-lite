# 告警中心-相关性规则新增即时告警类型 Solution Plan

版本：v1.0

日期：2026-06-03

需求文档：[20260531.告警中心-相关性规则新增即时告警类型.md](../../requirements/告警中心/20260531.告警中心-相关性规则新增即时告警类型.md)

---

## 1. 方案概览

在告警中心相关性规则中新增"即时告警（Instant Alert）"策略类型，逐条事件命中即出告警，不进入聚合窗口。核心做法：在事件入库链路上新增一条 **INSTANT 旁路**，与现有"Beat → DuckDB 聚合"管线并行，通过策略类型路由互斥。

技术要点：
- 复用 `AlarmStrategy` 模型，仅扩展 `strategy_type` choices（无新增字段）
- 复用 `Alert` 表 + Alert↔Event M2M，下游状态机、分派、屏蔽、操作 0 改动
- 旁路在 `AlertSourceAdapter.main()` 内、`create_events` 之后注入；纯内存匹配，命中后 `bulk_create` 同步落库
- 分派复用 `async_auto_assignment_for_alerts` 异步触发
- `AggregationProcessor` 排除 INSTANT 策略；`beat_close_alert` 排除 INSTANT 告警
- 前端复用相关性规则表单，新增类型选项后动态显隐聚合相关字段

---

## 2. 范围定义

### 2.1 In Scope
- 后端：`AlarmStrategyType` 新增 `INSTANT` 常量；Strategy serializer 增加 INSTANT 校验分支
- 后端：新增 `InstantAlertDispatcher` + `InstantMatcher` + `InstantStrategyCache`；新增 Celery 任务 `build_instant_alerts`（兜底）
- 后端：改造 `AlertSourceAdapter.main()` 注入旁路；改造 `AggregationProcessor._get_active_strategies` 排除 INSTANT；改造 `beat_close_alert` 排除 INSTANT
- 前端：相关性规则表单/列表/详情增加"即时告警"类型分支
- 数据库:仅 `CharField.choices` 变化的 migration，无 DDL

### 2.2 Out of Scope
- 自动关闭策略（PRD 明确不引入）
- 即时告警独立统计指标、独立通知通道
- 现有 SMART_DENOISE / MISSING_DETECTION 字段调整
- 智能降噪的"告警→事故"聚合逻辑改造（即时告警因指纹唯一，天然不参与）

### Phase 2：旁路核心实现（后端核心）

**预估工时**：1.5 天

| 步骤 | 任务 | 涉及文件 |
|---|---|---|
| 2.1 | 新增 `InstantMatcher.match_in_memory(event, match_rules)`，复刻 `StrategyMatcher` 的 eq/ne/contains/regex/in/gt/lt 语义，作用于 Python 对象属性 | `server/apps/alerts/aggregation/strategy/instant_matcher.py` |
| 2.2 | 新增 `InstantStrategyCache`，60s TTL 内存缓存 active INSTANT 策略 | 同 dispatcher 文件或独立 cache 模块 |
| 2.3 | 新增 `InstantAlertDispatcher.dispatch(bulk_events)`，编排匹配 + 落库 + 分派触发 | `server/apps/alerts/aggregation/processor/instant_dispatcher.py` |
| 2.4 | 新增 `_bulk_build_instant_alerts(hits)`，按 strategy_id 分桶 `bulk_create` Alert + M2M | 同上 |
| 2.5 | 新增 Celery 任务 `build_instant_alerts(hits)`，专用队列 `instant_alerts`，仅在命中数超阈值时调用 | `server/apps/alerts/tasks/tasks.py` |
| 2.6 | `AlertSourceAdapter.main()` 在 `create_events` 之后插入 `InstantAlertDispatcher.dispatch(bulk_events)` | `server/apps/alerts/common/source_adapter/base.py:501` |
| 2.7 | `AggregationProcessor._get_active_strategies` 加 `.exclude(strategy_type=INSTANT)` | `server/apps/alerts/aggregation/processor/aggregation_processor.py:70` |
| 2.8 | `beat_close_alert` 查询 Alert 时排除 `strategy__strategy_type == INSTANT` | `server/apps/alerts/tasks/tasks.py:52` |

**指纹方案**：`fingerprint = md5(f"instant:{strategy_id}:{event_id}")`
- 与聚合指纹命名空间完全隔离，碰撞概率 ≈ 0
- 重复投递天然幂等（`bulk_create(ignore_conflicts=True)`）
- 同一事件命中 N 个 INSTANT 策略 → N 条 Alert，指纹两两不同

**同步 / 异步阈值**：`INSTANT_SYNC_THRESHOLD = 50`
- 命中数 ≤ 50：adapter 线程内同步 `bulk_create`
- 命中数 > 50：`send_task(build_instant_alerts.name, [hits])`，避免拖死 adapter

**Alert 字段映射**：
- `fingerprint`、`alert_id`（uuid）、`event_count = 1`
- `level = event.level`（PRD"告警级别继承事件级别"）
- `title / description / resource_* / service / item / location / source_id` 直接取自 event
- 跳过 `_normalize_fingerprint` 的标题改写，保留事件原始文案

## 5. 关键决策

| 决策项 | 选定方案 | 说明 |
|---|---|---|
| 旁路注入点 | `AlertSourceAdapter.main()` 内 `create_events` 之后 | 所有入口（NATS/Webhook/Restful/Zabbix/Prometheus）统一旁路，改动集中 |
| 指纹方案 | `md5("instant:<sid>:<eid>")` | 命名空间隔离 + 强幂等；与现有 md5 格式兼容 |
| 多策略命中 | 每命中一个策略产 1 条 Alert | 语义清晰，每策略独立分派/模板生效 |
| 与原管线关系 | 读法 B：旁路只针对"产 Alert"，Event 本身仍进原管线 | 实现最简；同一事件双命中是用户规则配置的预期结果 |
| 同步/异步落库 | 命中 ≤ 50 同步 bulk_create；> 50 走 Celery | 1-event-per-call 主场景下 Celery 中转是纯开销；保留异步兜底应对大批量 |
| 分派 | 异步 Celery（复用 `async_auto_assignment_for_alerts`） | 不阻塞 adapter；与现有聚合告警一致 |
| Alert 构造 | 复用 `AlertBuilder` 字段映射 + bulk_create 落库 | 字段一致性有保障；避免逐行事务 |
| 策略缓存 | 60s TTL 内存缓存 + save 钩子清空 | INSTANT 策略读多写少；省 1000 RPS 下的重复 DB 查询 |
| 自动关闭 | `beat_close_alert` 按 strategy_type 排除 | PRD"不引入任何自动关闭策略" |
| 模型字段 | 不新增字段 | `match_rules` / `params.alert_template` 复用现有结构 |

## 6. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 单次调用塞入大批事件导致 adapter 同步落库阻塞 | 中 | `INSTANT_SYNC_THRESHOLD = 50` 阈值降级到 Celery；超 5000 截断并 ERROR 日志 |
| INSTANT 策略缓存过期窗口内策略变更未生效 | 低 | TTL 60s 上限；Strategy save 钩子主动 `cache_clear` |
| Celery 异步分支任务失败 | 低 | 重试指数退避 ×3；最终失败 ERROR 日志；不引入死信表（避免过度设计） |
| 一个事件双命中产生 2 条告警让用户困惑 | 低 | 文档说明读法 B 语义；前端"即时告警"卡片增加提示文案 |
| `bulk_create` ignore_conflicts 在不同 DB 后端行为差异 | 低 | 测试覆盖 PostgreSQL（主用）+ MySQL；其他后端遵循各自语义，不阻塞主路径 |
| 高并发下 M2M 写入与现有聚合管线争用 | 低 | M2M 表无唯一约束争用，常规 INSERT 即可；监控 DB write QPS |
