# 告警中心-相关性规则新增"即时告警"类型

## 1. 背景与问题
现有接入管线强制走"指纹聚合"：相同指纹的事件归并为一条告警。该机制适合运维类重复事件，但不适合业务事件（订单失败、支付异常、风控命中等）——这类事件每一条都有独立业务含义，用户要的是"匹配即立刻知道"，任何聚合都会丢上下文、加感知延迟。

因此需要在相关性规则中新增"即时告警"类型：命中规则的事件不聚合、不收敛，逐条产出告警并立即进入分派。

## 2. 业务需求
- 相关性规则新增类型"即时告警"。
- 即时告警规则按筛选条件匹配事件，命中后**一事件一告警**，立即进入分派与屏蔽评估。
- 复用现有相关性规则的配置入口、告警实体、分派与屏蔽链路，不新增菜单、不新增实体。

## 3. 业务逻辑

### 3.1 配置项
| 分类 | 字段 | 必填 | 说明 |
|---|---|---|---|
| 基础 | 策略名称、生效组织、分派组织 | ✅ | 与现有相关性规则一致 |
| 匹配 | 匹配方式 | ✅ | 仅"按条件筛选"，**不允许"全部"** |
| 匹配 | 筛选条件 | ✅ | 复用现有条件组 DSL，未填不可保存 |
| 产出 | 告警模板 | ✅ | 复用现有告警模板组件 |

### 3.2 产出口径
- 每一条命中的事件产生一条独立告警，告警数 = 事件数。
- 告警指纹强制按事件唯一标识生成，不与任何现有告警归并。
- 告警 `event_count` 恒为 1。
- 告警级别继承事件级别，与现有逻辑一致。

### 3.3 生命周期
- 状态机与现有告警一致：未分派、待处理、处理中、已解决、已关闭。
- **本期不引入任何自动关闭策略**，由用户认领并关闭。

### 3.4 与其他规则的关系
- 即时告警规则匹配优先于现有指纹聚合管线；命中即旁路，未命中走原流程，行为零变化。
- 即时告警告警同样接受现有分派规则与屏蔽策略。
- 即时告警告警因指纹唯一，天然不参与"智能降噪"的告警→事故聚合。

## 4. 关键技术架构选择
- **聚合逻辑与现有相关性规则不同**：现有类型走"指纹聚合 + 窗口收敛"，即时告警类型不聚合、不收敛，逐条事件直接产出告警。
- **对处理实时性有强要求**：业务场景的核心诉求是"遇到问题事件能立马产生告警"，因此即时告警规则的匹配与告警产出需要在事件到达的瞬时完成，不能进入任何延迟队列或批处理窗口。

## 5. 验收口径
- 可在相关性规则中选择"即时告警"并保存；匹配方式无"全部"选项；未填筛选条件不可保存。
- 命中规则的事件逐条产生告警，告警数 = 事件数，指纹两两不同。
- 即时告警告警立即进入分派评估；命中屏蔽策略时通知被抑制，记录仍生成。
- 即时告警告警级别与事件原始级别一致。
- 即时告警告警可被认领、分派、改派、关闭、批量操作；不会自动关闭。
- 即时告警规则与告警按生效组织正确隔离。

## 6. 约束与边界
**In Scope**：相关性规则新增"即时告警"类型、配置表单、产出口径、与现有分派/屏蔽链路的复用。

**Out of Scope**：自动关闭策略、全量即时告警、独立统计指标、独立通知通道或事件实体、对现有两类规则的字段调整。
