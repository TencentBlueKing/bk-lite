# 设计 - 告警中心相关性规则新增"即时告警"类型

> 对应需求：[20260531.告警中心-相关性规则新增即时告警类型.md](./20260531.告警中心-相关性规则新增即时告警类型.md)

## 1. 架构与数据流

在事件入库链路上新增一条"INSTANT 旁路"，与现有"Beat → DuckDB 聚合"管线并行，两条管线通过策略类型路由互斥。

```
NATS / Webhook / Restful 等
      │
      ▼
AlertSourceAdapter.main()
      │
      ├─ create_events() …… 原样入库 Event（已有）
      ▼
[NEW] InstantAlertDispatcher.dispatch(bulk_events)
      │   1. 读 active 的 INSTANT 策略（内存缓存）
      │   2. 内存匹配 events × strategies
      │   3. 命中 ≤ 阈值 → 同步 bulk_create Alert + M2M
      │      命中 > 阈值 → Celery 异步 bulk_create
      │   4. 每条 Alert 触发 async_auto_assignment_for_alerts
      │
      ▼
event_operator() + handle_recovery_events()  ← 原路径不变
      │
      ▼
Celery Beat → AggregationProcessor           ← 只跑 SMART_DENOISE / MISSING_DETECTION
```

**关键不变量**：
- `AggregationProcessor._get_active_strategies()` 加 `.exclude(strategy_type=INSTANT)`，聚合器永不处理 INSTANT 策略
- 一个 Event 同时匹配 INSTANT 与 SMART_DENOISE 规则时，两类 Alert 各自独立产出（读法 B：旁路只针对"产 Alert 这件事"，Event 本身正常进原管线）
- 命中 INSTANT 多策略时，每命中一个策略产出一条独立 Alert

## 2. 模型与常量改动

### 常量
`apps/alerts/constants/constants.py`：

```python
class AlarmStrategyType:
    SMART_DENOISE = "smart_denoise"
    MISSING_DETECTION = "missing_detection"
    INSTANT = "instant"                       # NEW

    CHOICES = (
        (SMART_DENOISE, "智能降噪"),
        (MISSING_DETECTION, "缺失检测"),
        (INSTANT, "即时告警"),                 # NEW
    )
```

### 模型
`AlarmStrategy` **不加新字段**，复用：
- `match_rules` —— 复用条件组 DSL
- `params` —— 只放 `alert_template`
- `team / dispatch_team` —— 复用
- `auto_close / close_minutes` —— INSTANT 不读取（PRD 不引入自动关闭）

### 迁移
仅 `CharField.choices` 变化 → 一个 alter-choices migration，无 DDL 风险。

### Serializer 校验
`serializers/strategy.py` 中 `strategy_type == INSTANT` 时：
- `match_rules` 必填且非空（PRD"未填筛选条件不可保存"）
- 拒绝 `match_type == "all"` 等价的空规则（PRD"不允许全部"）
- `params` 只校验 `alert_template` 必填
- 静默清理 `window_size / group_by / aggregation_*` 等聚合字段，防止前端误传

## 3. 旁路注入实现

### 新增模块

`apps/alerts/aggregation/processor/instant_dispatcher.py`：

```python
INSTANT_SYNC_THRESHOLD = 50   # 命中数 ≤ 阈值走同步，> 阈值走 Celery

class InstantAlertDispatcher:
    @staticmethod
    def dispatch(bulk_events):
        strategies = InstantStrategyCache.get()      # 60s TTL 内存缓存
        if not strategies:
            return
        events = [e for batch in bulk_events for e in batch
                  if e.action == EventAction.CREATED]
        if not events:
            return

        hits = []   # [(strategy_id, event_id), ...]
        for event in events:
            for strategy in strategies:
                if InstantMatcher.match_in_memory(event, strategy.match_rules):
                    hits.append((strategy.id, event.event_id))

        if not hits:
            return

        if len(hits) <= INSTANT_SYNC_THRESHOLD:
            _bulk_build_instant_alerts(hits)         # adapter 线程内同步
        else:
            current_app.send_task(
                build_instant_alerts.name, args=[hits])
```

`apps/alerts/aggregation/strategy/instant_matcher.py`：复刻 `StrategyMatcher` 的 eq/ne/contains/regex/in/gt/lt 语义，但作用于 Python 对象属性，零 DB 查询。

`apps/alerts/aggregation/processor/instant_dispatcher.py` 内 `_bulk_build_instant_alerts(hits)`：
- 按 `strategy_id` 分桶
- 一次性 `Event.objects.in_bulk(all_eids)` 与 `AlarmStrategy.objects.in_bulk(all_sids)`
- 每桶一个事务：
  - 构造 Alert 列表（`fingerprint = md5("instant:<sid>:<eid>")`，`event_count = 1`，字段从 Event 直接映射）
  - `Alert.objects.bulk_create(alerts, ignore_conflicts=True)`
  - `Alert.events.through.objects.bulk_create([...], ignore_conflicts=True)`
- 收集所有新 alert_id → `send_task(async_auto_assignment_for_alerts.name, [alert_ids])`

新 Celery 任务 `build_instant_alerts(hits)`（`apps/alerts/tasks/tasks.py`）：直接调用 `_bulk_build_instant_alerts(hits)`，专用队列 `instant_alerts`。

### 现有代码改动点

1. `apps/alerts/common/source_adapter/base.py:501` `AlertSourceAdapter.main()` 在 `create_events` 之后、`event_operator` 之前插一行：
   ```python
   InstantAlertDispatcher.dispatch(bulk_events)
   ```
   旁路自身 catch 全部异常并记日志，不影响主流程。

2. `apps/alerts/aggregation/processor/aggregation_processor.py:70` `_get_active_strategies` 加：
   ```python
   .exclude(strategy_type=AlarmStrategyType.INSTANT)
   ```

3. `apps/alerts/serializers/strategy.py` Strategy serializer 加 INSTANT 分支校验（见 §2）。Strategy save 钩子内调 `InstantStrategyCache.cache_clear()`。

4. `apps/alerts/tasks/tasks.py:52` `beat_close_alert`（自动关闭任务）查询 Alert 时增加过滤：排除 `strategy__strategy_type == INSTANT` 的告警，保证即时告警永不被自动关闭。

### 关键设计点
- **指纹**：`md5("instant:<strategy_id>:<event_id>")`，命名空间与聚合指纹完全隔离，重复投递天然幂等。
- **多策略命中**：每命中一个策略产 1 条 Alert（PRD"事件数 = 告警数"在单策略口径下成立；多策略命中是用户规则配置的预期结果）。
- **跳过 `_normalize_fingerprint`**：旁路自己生成最终指纹与标题/描述（直接取 event 原始字段，符合"原汁原味"诉求）。

## 4. 性能与流量模型

**目标流量**：1000 RPS，每次 1 个 event（NATS/HTTP 调用粒度）。

**同步路径每次调用代价**：
- 策略缓存读取：0 DB
- 内存匹配：1 event × N 策略（典型 N ≤ 20），微秒级
- 命中 ≤ 阈值（典型 0~2 条）：1~2 行 Alert INSERT + 1~2 行 M2M INSERT
- send_task 入队：1 次（命中时）
- **新增端到端开销 ≈ 5~10ms / 请求**

**为什么不再用 Celery 中转作为主路径**：
- 1000 RPS × 1 event/call 模式下，每次推 Celery 任务只做 1 行 bulk_create，批量优化失效，反而引入 broker 往返与 worker 启停开销。
- 同步直写 + PostgreSQL 单实例轻松 ≥1 万 INSERT/s 完全可承载。

**Celery 异步分支的意义**：兜底应对单次调用塞入大批事件的边界场景（NATS 接口支持，但不是主场景）。命中数超 `INSTANT_SYNC_THRESHOLD = 50` 时自动降级，避免拖死 adapter 线程。

**分派始终异步**：复用现有 `async_auto_assignment_for_alerts`，不阻塞 adapter 主线程。

**防护栏**：
- 单次 dispatch 命中数超 5000 → 截断并 ERROR 日志，防异常事件源刷爆
- Celery 任务失败：retry 指数退避，最多 3 次；最终失败 ERROR 日志（不引入死信表，避免过度设计）

## 5. 与其他管线的关系

| 接触点 | 行为 |
|---|---|
| Event 行本身 | 旁路不改、不删、不打标 → 聚合器看到的事件集合完全不变 |
| 策略路由 | 聚合器 `exclude(strategy_type=INSTANT)` → INSTANT 策略永不被聚合器看到 |
| 指纹空间 | `instant:` 前缀命名空间与聚合指纹完全隔离 |
| `_is_existing_alert` | 基于指纹查询，即时 Alert 不会被误认为已有聚合告警 |
| DB 写入 | 旁路 1~2 行小 INSERT，与聚合器写入无锁竞争 |
| 告警实体 | 复用 Alert 表 + Alert↔Event M2M，下游状态机/操作/分派/屏蔽 0 改动 |
| 智能降噪聚合 | 即时 Alert 因指纹唯一，天然不参与告警→事故聚合（PRD §3.4） |
| 自动关闭 | INSTANT Alert 不进入 `beat_close_alert` 自动关闭队列（按策略类型过滤） |

## 6. 前端改动

**入口不变**：复用"相关性规则"列表 + 编辑表单，不新增菜单。

**类型选择**：
- 单选增加第三项 **"即时告警"**
- 选中后动态显隐：
  - 隐藏：window_size / group_by / auto_close / close_minutes 等聚合参数
  - 保留：策略名称、生效组织、分派组织、匹配方式、筛选条件、告警模板
  - 匹配方式：只渲染"按条件筛选"一个选项
  - 筛选条件：必填，未填禁用保存按钮
  - 告警模板：必填

**列表 / 详情**：
- 类型列与筛选下拉增加"即时告警"选项
- 详情页隐藏聚合相关字段
- 列表行为（启停 / 编辑 / 删除）完全复用

**API**：复用 `/api/v1/alerts/strategy/` CRUD，不新增 endpoint。

**i18n**：新增少量 key（`strategy.type.instant` 等）。
