# 告警中心-新增"告警丰富"能力（可扩展引擎，渐进式交付）

## 1. 背景与问题

告警从各源进入告警中心后，携带的字段往往只够"知道哪台机器的哪个指标出了问题"，但不够"知道该找谁、属于哪个业务、影响多大"。处理人需要离开告警中心，自己去 CMDB、监控、文档里反复查证，响应被拉长。

现状已有一个**事件级丰富**雏形（`AlertSourceAdapter.enrich_event`）：在事件入库时，按 `resource_type→model_id`、`resource_id→_id` 固定映射查一次 CMDB，把**整条实例无差别灌进 `event.labels`**。它有三个硬伤：

1. **只有一个全局开关、硬编码 CMDB**——既不能按场景配置、也无法扩展其他丰富源。
2. **固定字段、整条灌入 labels**——无法"按哪几个字段查、取哪几个字段"，且丰富字段与原生 labels 键冲突、污染收敛（group_by）维度，下游也分不清哪些是原始、哪些是丰富出来的。
3. **每条事件同步串查一次**——CMDB 做厚后单次查询更重，入库是热路径，批量事件下有压垮 CMDB、拉长入库时延的风险。

业界主流告警平台（BigPanda 的 enrichment-map、Moogsoft 的 Workflow Enrichment、ServiceNow 的 CI Binding、PagerDuty 的 Event Orchestration）的共识是：**身份/维度类丰富命中在原子信号层（即本系统的 Event），且发生在收敛之前**——因为收敛、分派、屏蔽都要按丰富出的维度来做，字段不在收敛前备好，下游就用不上。本期对齐这一共识。

本期目标两条：**① 扩展丰富数据来源**（先把 CMDB 做厚，并立起可插拔的多源框架）；**② 让丰富结果驱动下游**（界面展示、收敛、分派、通知可直接引用丰富字段）。落地方式采用 **渐进式交付**：每一轮独立可用，且后一轮是"加插件"而非"重写"。

> **关于"PLG"措辞的诚实说明**：本能力本质是一个**可扩展的声明式丰富引擎**，不是开箱即自助的产品。L1 只交付引擎 + 一个 CMDB Provider + 一条出厂预设，**没有面向用户的配置/展示界面**——此时用户是被动受益者，无法自助。真正让它"产品化、可被用户自助使用"的要素（界面、激活漏斗、产品度量）集中在 **L1.5**（见 §3.9）。请勿把 L1 单独理解为 PLG 闭环。

## 2. 业务需求

- **丰富继续命中 Event 层**，复用现有事件丰富时机（入库阶段），不为本期 CMDB 维度丰富新建告警级管线。
- **丰富结果落入独立的 `enrichment` 结构字段**（Event、Alert 各一个），按来源命名空间分区，与 `labels` 物理隔离：原生 labels 不被污染、收敛不误伤、下游能明确区分并引用丰富字段。
- **丰富不再是固定字段**，而是可定义的"声明式查找"：用户可声明用告警的**任意 1..N 个字段作为查询入参**去定位记录，并从返回结果中**挑选任意字段、重命名**作为丰富输出。本期不要求做配置界面，但引擎必须支持任意入参/出参的能力。
- **立起"规则 + 数据源（Provider）"骨架**：丰富行为由可启停的丰富规则驱动；数据源以 Provider 形式插拔。本期只交付 CMDB 一个 Provider，但规则模型与 Provider 接口在本期定形，后续新增 HTTP、脚本、指标、知识库等源只是新增 Provider，零返工。
- **CMDB 丰富做厚**：从"只查单实例基础字段"升级为可丰富责任人、所属业务系统、关联关系/拓扑等多维度（具体维度由规则的出参投影声明，不写死）。
- **出厂自带 CMDB 预设模板**：用户开箱即得常见场景（按资源定位、投影责任人/业务系统等），无需面对空白配置。降低空白态门槛；但须说明 L1 阶段该预设仍需 L1.5 的展示界面才"对用户可见"，本身不构成自助能力。
- **丰富是尽力而为**：任何丰富失败、超时、源不可用都**绝不阻塞或失败事件入库**，告警照常生成，仅丰富字段缺失。
- **丰富结果可被下游引用**：Event 上的丰富在收敛成 Alert 时按来源合并到 `Alert.enrichment`，供界面展示、收敛维度、分派规则、通知模板引用。

## 3. 业务逻辑

### 3.1 丰富引擎的本质：声明式 Lookup（查找/连接）原语

丰富引擎不是"CMDB 丰富器"，而是一个与具体数据源解耦的**声明式查找原语**，逻辑等价于数据库 join / 表格 VLOOKUP：

```
入参绑定（event 字段 → 查询键，1..N 组合）
   → 调度 Provider（黑盒 fetch，引擎不感知源语义）
      → 出参投影（结果字段 → enrichment 键，挑选 + 重命名，1..N）
```

引擎只认 `fetch(query_params) -> records`，**不认识任何数据源专有概念**。CMDB 的 `model_id` 在引擎眼里只是一个普通的入参绑定，其特殊性全部下沉到 CMDB Provider 内部。这是 L2/L3 能零返工接入 HTTP/脚本的根本前提。

### 3.2 两个 key 必须分离

规则配置中存在两个语义不同、不可合并的"键"：

| 名称 | 作用 | 类比 SQL | 基数 |
|---|---|---|---|
| **入参绑定键（join key）** | 用 event 的哪几个字段去**定位哪条记录** | `WHERE` | 任意 1..N 字段组合 |
| **出参投影名（projection）** | 从返回记录里**挑哪些字段、改成什么名字**落进 `enrichment` | `SELECT … AS` | 任意 1..N 字段 |

合并这两者，引擎会在"复合查询"和"无冲突输出"间二选一，两头都做不好；分开后同一引擎既支持组合定位、又输出干净不冲突。

### 3.3 出参基数策略（cardinality）

一次查找可能返回 0 / 1 / N 条记录，规则须显式声明多结果处理策略 `on_multiple`：

- **0 条** → 记为 miss，**负结果也进缓存**（见 3.6），避免反复空查。
- **1 条** → 直接投影。
- **N 条** → 按 `on_multiple` 取值：`first`（取首条，默认）/ `merge`（合并）/ `list`（存为列表）。

### 3.4 丰富结果的落点与命名空间

- Event、Alert 各新增 `enrichment` JSON 字段，结构按"来源/规则"命名空间分区，例如 `enrichment.cmdb.responsible_person`。
- 与 `labels` 物理隔离：丰富字段不进 labels，**不参与、不污染 group_by 收敛维度**；收敛若要用丰富维度，须显式引用 `enrichment.*`，不会误伤。
- 不同规则投影出的同名字段（如两个规则都产出 `owner`）因落在各自命名空间下而不冲突。

### 3.5 规则匹配与执行管线

- 每条丰富规则含：**匹配条件**（决定作用于哪些事件，复用告警策略既有的 `match_rules` 匹配语法以保持一致）+ **入参绑定** + **Provider 类型与配置** + **出参投影** + `on_multiple` + 启停开关。
- 执行时机：事件入库的丰富阶段（沿用 `_transform_alert_to_event` 时点），对命中规则的事件执行管线：
  匹配规则 → 计算入参绑定键 → 查 TTL 缓存 → 未命中则交 Provider 批量查询 → 出参投影写回 `event.enrichment[来源]` → 回填缓存。
- 全程尽力而为：单条规则/单次查询失败被隔离，记日志，不影响其他规则与事件入库。

### 3.6 性能与稳定性：四层兜底

入库是热路径、一批可能上百条事件、CMDB 做厚后单查更重，须保证"既不压垮丰富源、又不拉长入库时延"。采用**同步执行 + 四层叠加**，使入库增加的时延与事件条数解耦：

1. **批量查询（最大杠杆）**：Provider 接口以**批量**为一等公民——`fetch_batch(keys) -> {key: record}`。一批事件按 Provider/源维度聚合，**每个聚合维度只发 1 次批量请求**，而非每资源 1 次。CMDB 侧需新增批量查询能力（复用底层已支持的 `inst_id__in` 批量操作符），暴露批量 RPC。
2. **批内去重**：查询前按入参绑定键去重，同键只进一次批量集合。
3. **跨批 TTL 缓存**：按"来源 + 入参绑定键"缓存，短 TTL；**命中与未命中（负结果）都缓存**。告警抖动时同资源反复出现，命中率高。
4. **有界 + 降级**：单批可丰富的不同记录数设上限，超出部分记日志后跳过（入库时延有天花板）；批量查询包超时与熔断，源慢或不可用时直接跳过丰富、绝不阻塞入库。

净效果：入库热路径新增时延 ≈「本批不同源/聚合维度数 × 单次批量查询」，与事件条数无关；丰富源只收到少量批量请求而非洪水。

### 3.7 Event → Alert 传播

- 收敛阶段（`AlertBuilder`）将成员事件的 `enrichment` 按来源合并到 `Alert.enrichment`。
- 合并策略：同一来源下，成员事件取值一致时直接采用；跨资源取值不一致时按来源约定保留（取首条或列表，与 3.3 一致），供界面展示与下游引用。

### 3.8 与下游的关系

- **界面展示**：告警详情/列表可读取 `enrichment.*` 展示丰富字段，可过滤/搜索。
- **收敛 / 分派 / 通知**：可显式引用 `enrichment.*` 字段作为收敛维度、分派匹配条件或通知模板变量。本期保证字段就位与可引用，下游各自如何消费由其自身规则配置决定，不在本期强约束。

### 3.9 交付阶梯（Provider 扩展性 + 产品表面，本期定位 L1）

下表是**交付阶梯**，不是"PLG 增长漏斗"——L1/L2/L3 主线衡量的是 **Provider 可扩展性（工程维度）**，L1.5 单独衡量**产品表面与可度量性（用户维度）**。诚实标注每一档"谁能用、自助到什么程度"：

| 阶段 | 交付 | 面向对象 / 自助程度 |
|---|---|---|
| **L1（本期）** | 立"规则 + Provider 接口"骨架；只实现 CMDB Provider 并做厚；独立 `enrichment` 字段；四层性能兜底；出厂 CMDB 预设模板 | **运维（被动受益）**：告警后台自动带上责任人/业务系统，但**无界面**，无法自助查看或开关，预设改字段仍需工程改 |
| **L1.5（紧接本期，产品表面）** | **独立的「告警丰富」配置页**：丰富规则的完整 CRUD（建/改/删/启停），可视化配置匹配条件 + 入参绑定 + Provider 选择 + 出参投影 + `on_multiple`；出厂初始化一条「内置-CMDB 资源丰富」规则，在该页可见、可改、可停（非隐藏硬编码）；告警详情/列表**展示** `enrichment.*` 并可过滤；接入**产品埋点**（被丰富告警占比、命中预设告警数、用户自建/改规则数、规则启用率等） | **运维 + 管理员（真自助）**：用户能自己建/改一条丰富规则——"我配出来一条"成为激活动作；引擎产出"对用户可见、可配、可度量" |
| **L2** | 不写代码新增 HTTP/REST Provider，复用 L1.5 已落地的规则 CRUD 页（仅多一种 `provider_type` 与其配置表单） | **配置者（自助接源）**：在已有配置页里换个 Provider 即接入新源，零返工 |
| **L3** | 脚本/函数、指标/时序、知识库等 Provider，每个只是新增实现 + 注册 | **平台/生态**：丰富源生态化 |

规则模型与 Provider 接口在 L1 定形，L1.5 为"补产品表面"，L2/L3 为"加插件"，均不返工。

> **顺序说明**：L1 先交付的是"厚而隐形"的引擎，L1.5 紧接补"薄而可见"的产品表面。两档合起来才构成一个用户能看见、能开关、能被度量的完整最小价值；单独的 L1 不应被当作可独立证明价值的交付——这也是该能力上一轮实现后被搁置的直接原因（无表面、无度量、无人能验证其价值）。

## 4. 关键技术架构选择

- **丰富命中 Event 层、复用现有时机。** 不为本期维度丰富新建告警级管线；情境/诊断类丰富（爆炸半径、关联变更、相似历史、AI 根因等）若未来要做，再作为告警/事故级的独立迭代，不与本期混入同一管线。
- **引擎 = 声明式 Lookup 原语，Provider = 纯黑盒插件。** 引擎只持有"入参绑定 / 调度 / 出参投影"三段契约，不内置任何源专有词汇（含 `model_id`）。Provider 实现 `fetch_batch(keys, config) -> {key: record}`；不支持批量的源由框架以受限并发的单查回退兜底。
- **规则承重、Provider 可插拔。** 新增 `EnrichmentRule` 模型（`match_rules` + 入参绑定 + `provider_type` + `provider_config` + 出参投影 + `on_multiple` + `is_active`）。L1 仅 `cmdb` 一种 `provider_type`。
- **独立 `enrichment` 字段，与 labels 隔离。** Event、Alert 各新增 `enrichment` JSON 字段，按来源命名空间分区，不污染收敛维度、下游可区分引用。
- **同步执行 + 四层性能兜底（批量查询 / 批内去重 / 跨批 TTL 缓存 / 有界降级）。** 入库时延与事件条数解耦；CMDB 侧新增批量查询 RPC（复用底层 `inst_id__in`）。丰富尽力而为，失败不阻塞入库。
- **空白态护栏：通用引擎 + 出厂预设规则。** 引擎底层支持任意字段绑定/投影，出厂初始化一条「内置-CMDB 资源丰富」规则（作为一条真实的 `EnrichmentRule` 记录种入，而非隐藏硬编码），用户开箱即得常见丰富，且能在 L1.5 配置页里直接看到、复制、修改它——空白态门槛与"看不懂如何配"两个问题一并消除。
- **L1.5 独立配置页让引擎通用性首次可被用户行使。** 双 key 分离、`on_multiple`、任意投影等灵活性通过 L1.5 的规则 CRUD 页暴露给用户：用户自己建/改一条规则即是激活动作。L2 接 HTTP 时复用同一配置页，仅新增 `provider_type` 与其配置表单，零返工。

## 5. 验收口径

- 丰富在事件入库阶段执行，结果落入 Event 的独立 `enrichment` 字段（按来源命名空间），不写入 `labels`、不进入 group_by 收敛维度。
- 丰富规则可声明用告警任意 1..N 个字段作为查询入参、从返回结果挑选任意字段并重命名作为输出；入参绑定键与出参投影名为两套独立配置。
- 引擎不依赖任何数据源专有概念即可调度 Provider；CMDB 的 `model_id` 等通过入参绑定传入，而非引擎内置。
- 多结果按规则的 `on_multiple`（`first`/`merge`/`list`，默认 `first`）处理；查不到记为 miss 且负结果进缓存。
- 一批事件对同一丰富源仅发起批量查询（每聚合维度 1 次），批内按入参绑定键去重，跨批命中 TTL 缓存（含负结果）；单批丰富记录数有上限、超出跳过并记日志。
- CMDB 源具备批量查询能力，单批多资源不退化为逐资源查询。
- 任一规则/查询失败、源超时或不可用时，事件正常入库、告警正常生成，仅对应丰富字段缺失，并有日志可定位。
- 出厂提供至少一个开箱即用的 CMDB 预设模板，启用后无需逐字段配置即可产出常见丰富字段（如责任人、所属业务系统）。
- Event 的 `enrichment` 在收敛成 Alert 时按来源合并到 `Alert.enrichment`，可在告警详情/列表展示与过滤。
- 新增 HTTP 等 Provider 时无需改动规则模型与引擎管线（以接口契约可验证）。

**L1.5 产品表面验收口径**（紧接本期交付，证明引擎"有面、可配、可度量"）：

- 存在一个独立的「告警丰富」配置页，**挂在告警中心「设置」导航下，与「相关性规则」「屏蔽策略」并列**（前端 `src/app/alarm/(pages)/settings/` 下与 `correlationRules`、`shieldStrategy` 同级），支持丰富规则的完整 CRUD：新建、编辑、删除、逐条启停；启停/增删立即生效于后续入库丰富。
- 配置页可可视化编辑一条规则的全部要素：匹配条件、入参绑定（任意 1..N 字段）、Provider 类型与配置、出参投影（挑字段 + 重命名）、`on_multiple`；保存前对绑定/投影做基本校验。
- 出厂初始化一条「内置-CMDB 资源丰富」规则，以真实 `EnrichmentRule` 记录存在，在配置页中可见、可改、可停、可复制为新规则；用户全新建一条 CMDB 规则也能跑通。
- 告警详情/列表可读取并展示 `enrichment.*` 字段，且可按其过滤。
- 接入产品埋点并可读出至少：被丰富告警占比、命中出厂预设的告警数、用户自建/编辑规则数、丰富规则启用率——使该能力具备可观测的采纳漏斗，而非仅工程正确性。

## 6. 约束与边界

**L1 In Scope**：事件级丰富的可插拔重构（`EnrichmentRule` 模型 + `EnrichmentProvider` 批量接口 + 声明式 Lookup 引擎）、CMDB Provider 及其做厚、独立 `enrichment` 字段（Event/Alert）与命名空间隔离、入参绑定/出参投影双 key 分离、`on_multiple` 基数策略、四层性能兜底（批量查询 RPC + 批内去重 + 跨批 TTL 缓存 + 有界降级）、Event→Alert 丰富传播、出厂 CMDB 预设模板、丰富字段供下游引用的就位保证。

**L1.5 In Scope（紧接本期，产品表面与度量）**：独立的「告警丰富」配置页——丰富规则的完整 CRUD（建/改/删/启停）及匹配条件、入参绑定、Provider 选择、出参投影、`on_multiple` 的可视化配置与基本校验；出厂初始化「内置-CMDB 资源丰富」规则（真实记录，页内可见/可改/可复制）；告警详情/列表展示与过滤 `enrichment.*`；产品埋点（被丰富告警占比、命中预设告警数、用户自建/改规则数、规则启用率等采纳指标）。

**Out of Scope**：L1.5 CMDB 配置页之外的 Provider 专属配置表单（HTTP/REST、脚本/函数、指标/时序、知识库等 Provider 本体及其表单随 L2/L3 增量交付，复用同一 CRUD 框架）；规则的导入导出/版本管理/灰度编排等高级运维能力；将丰富时机迁移到 Alert/Incident 级；情境/诊断类丰富（爆炸半径、关联变更、相似历史、AI 根因摘要）；异步丰富与"丰富前窗口"；下游（收敛/分派/通知）如何具体消费丰富字段的规则改造（本期只保证字段就位与可引用，不改下游规则语义）。

## 7. 扩展契约：如何新增丰富源（Provider 开发指南）

本节把"加新源零返工"从承诺固化为**可循的标准契约**——这是本设计存在的核心目的。

> **本期范围声明**：L1/L1.5 **只实现内置 CMDB Provider**（`provider_type = cmdb`）一种，并出厂初始化「内置-CMDB 资源丰富」一条规则为准。本节中的 **Zabbix 等均为后续扩展样例**，用于固化扩展时的标准动作，**不在本期交付**。本节的价值在于：让 L1 落地的抽象边界经得起"未来真的来加一个源"的检验。

### 7.1 两条扩展轴：先分清你要加的是"数据"还是"代码"

| 你想加的东西 | 本质 | 成本 | 动作 |
|---|---|---|---|
| 同一个源、另一条规则/另一组丰富字段（如 CMDB 再丰富"机房位置"） | 加**数据**（一条 `EnrichmentRule` 记录） | **零代码** | L1.5 配置页新建规则 |
| 标准 RESTful 的新源（`GET /asset?ip=x` 即可查） | 加**配置** | **零代码**（L2 通用 HTTP Provider 落地后） | 配置页选 `http` provider_type + 填 `provider_config` |
| 调用方式特殊的新源（Zabbix JSON-RPC、需预登录/特定批量信封等） | 加**代码**（一个专用 Provider 插件） | **~1 个文件** | 实现 `XxxProvider.fetch_batch` + 注册 `provider_type` |
| 全新形态源（脚本/函数、指标/时序、知识库） | 加**代码**（一个 Provider/源） | **~1 个文件/源** | 同上 |

**绝不需要改动的部分**：声明式 Lookup 引擎、`EnrichmentRule` 模型、L1.5 配置页框架、四层性能兜底（去重/缓存/降级）、Event→Alert 传播。新增源的全部改动收敛在"一个 Provider 实现 + 一次注册"。

### 7.2 Provider 抽象契约（唯一接口）

引擎对任何源只持有一份契约，**不感知任何源专有语义**（含 `model_id`、Zabbix 的 `hostid` 等，全部下沉到各 Provider 内部）：

```python
class EnrichmentProvider:
    provider_type: str           # 全局唯一标识，规则用它选择本 Provider（如 "cmdb"/"zabbix"/"http"）

    def fetch_batch(self, keys, config) -> dict:
        """
        入: keys   —— 引擎已【去重】的一批入参绑定键，形如 [{"ip": "1.2.3.4"}, {"ip": "..."}]
            config —— 该规则的 provider_config（连接信息、源专有参数，如 CMDB 的 model_id）
        出: {key: record} —— 命中的键→原始记录；未命中的键可缺省（引擎按 miss 处理并缓存负结果）
        约束: 批量是一等公民——一批 keys 必须尽量【一次】查询完成，不得退化为逐 key 查。
              不支持批量的源，由框架以受限并发的单查回退兜底（Provider 仍只实现 fetch_batch）。
        """
```

实现一个新 Provider = **实现这一个方法 + 注册 `provider_type`**，再无其他。投影、缓存、降级都不归 Provider 管。

### 7.3 引擎 vs Provider 职责边界（谁做什么，别越界）

| 职责 | 归属 | 说明 |
|---|---|---|
| 规则匹配（决定作用于哪些事件） | **引擎** | 复用告警策略 `match_rules` 语法 |
| 计算入参绑定键、批内去重 | **引擎** | Provider 收到的已是去重后的 keys |
| 调度 Provider、超时/熔断/降级 | **引擎** | 源慢或挂了，引擎跳过，绝不阻塞入库 |
| **连源、查源、把源记录批量取回** | **Provider** | 唯一需要写的逻辑 |
| TTL 缓存（含负结果）、跨批复用 | **引擎** | 按"来源 + 入参键"缓存 |
| 出参投影（挑字段/重命名/`on_multiple`） | **引擎** | Provider 只还原始 record，投影是规则配置驱动 |
| 写回 `event.enrichment[来源]`、Event→Alert 合并 | **引擎** | Provider 不碰 enrichment 结构 |

一句话：**Provider 只回答"给我一批 key，还我一批原始 record"，其余全是引擎的事。**

### 7.4 新增 Provider 的标准步骤（以 Zabbix 为样例，后续扩展）

```python
# providers/zabbix.py —— 后续扩展示意，非本期交付
class ZabbixProvider(EnrichmentProvider):
    provider_type = "zabbix"

    def fetch_batch(self, keys, config):
        # keys   = [{"ip": "1.2.3.4"}, {"ip": "1.2.3.5"}, ...]  ← 引擎已去重
        # config = {"url": ..., "user": ..., "token": ...}      ← 规则的 provider_config
        # 1. 用 config 登录拿 auth（源专有的预登录细节封死在这里）
        # 2. 一次 host.get(filter={"ip": [所有 ip]}) 批量查回（批量是一等公民）
        # 3. 组装并返回 {key: host_record}；查不到的 key 缺省即可
        return { ... }
```

步骤固定为四步，且每一步都不触碰引擎/模型/配置页：
1. 新建 `providers/<source>.py`，实现 `fetch_batch`；
2. 注册 `provider_type`（加入 Provider 注册表 / `provider_type` 枚举）；
3. （如需）为该 `provider_type` 在 L1.5 配置页补一份 `provider_config` 表单 schema；
4. 用户在配置页选该源、填连接、绑字段、投字段——用法与 CMDB 完全一致。

### 7.5 通用 HTTP Provider vs 专用 Provider：怎么选

- **标准 RESTful、一个 GET 就能按键查** → 用 L2 通用 HTTP Provider，**零代码**配置接入。
- **调用方式特殊**（JSON-RPC、需预登录、批量信封特定、分页/限流复杂，如 **Zabbix**） → **直接写专用 Provider**。不要把通用 HTTP Provider 撑成能表达一切——一个专用 `fetch_batch` 反而更稳、更省。这条边界本身就是契约的一部分：通用 HTTP Provider 负责"好查的源"，专用 Provider 负责"难查的源"，两者共用同一规则模型与配置页。
