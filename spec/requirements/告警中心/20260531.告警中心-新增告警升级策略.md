# 告警中心-新增"告警升级"

## 1. 背景与问题
现有告警分派规则只解决"通知给谁、通知多频"两个问题，没有回答"通知了但没人响应该怎么办"。告警分派后，处理人因休假、会议、通知未到达等原因长时间不响应时，告警会一直停留在"待处理"状态直到故障扩大。

业界对标产品（PagerDuty、Opsgenie、Grafana OnCall 等）均提供升级（escalation）能力来兜底这一场景：在规定时长内无人认领，自动换一批更高级/更广的处理人。BK-Lite 缺位，需要补齐。

本期仅做**超时升级**：告警在规定时长内未被认领时，自动切换到下一层处理人。其他升级形态（严重度升级、频次升级、手动升级）不在本期。

## 2. 业务需求
- 在**告警分派规则**内新增一个命名的、可视化的"升级链"配置区块——它是分派规则的一等公民区块，与"分派人员 / 通知渠道 / 通知频率"并列展示，让从对标产品迁移来的用户一眼认得出"这就是 escalation"。
- 升级链由有序的多个**层级**组成，每层指定处理人与"未认领等待时长"，可选指定通知渠道（未指定则沿用分派规则的渠道）。
- 升级链**内生于分派规则**，随分派规则一同保存、生效、停用，不产生需要二次绑定的独立实体，也没有跨规则复用与全局默认。一条分派规则要么配了升级链、要么没配。
- 告警在某一层的等待时长内未被认领，自动进入下一层并通知该层处理人；走完最后一层仍未认领则不再升级。
- 告警被认领、已解决、已关闭后，立即终止后续升级。

## 3. 业务逻辑

### 3.1 升级链配置项
| 分类 | 字段 | 必填 | 说明 |
|---|---|---|---|
| 链级 | 是否启用升级 | — | 不启用时该分派规则行为与现状完全一致 |
| 链级 | 升级模式（累加 / 替换） | ✅ | **替换**：进入下一层只通知本层处理人；**累加**：进入下一层时，前面各层处理人继续保留、新层处理人加入。整链统一一个开关，不做逐层混搭 |
| 层级 | 升级层级列表 | ✅ | 有序，至少 1 层；每层包含：处理人 / 本层未认领等待时长（分钟）/ 通知渠道（可选，未指定则沿用分派规则的渠道） |

### 3.2 触发与终止
- **触发口径**：以告警是否被**认领**为信号。告警停留在"待处理"超过当前层的等待时长 → 进入下一层并通知该层处理人。
- **计量口径**：跨级用**时间**——以"本层开始后、告警仍停留在待处理的累计时长"是否达到本层等待时长为判据，**不按提醒次数计量**。这样升级节奏与提醒频率解耦：改提醒间隔不会意外提前或推迟升级；提醒因屏蔽/会话观察期/投递失败被跳过时，升级时钟照常推进，不会静默失效。
- **终止口径**：告警被认领（进入"处理中"）、已解决、已关闭 → 立即终止后续升级。
- **走完仍未响应**：到达最后一层后**不再升级**，但最后一层处理人继续按级内提醒节奏被催促，直到达到分派规则配置的最大提醒次数后彻底停止，告警保持"待处理"由用户后续处置。
- **改派/分派变更**：发生改派时，升级计时**重置**到第一层开始；新分派人若仍不响应，再按完整升级链走。

### 3.3 升级只换人，不改级别
- 升级仅改变"通知谁、用什么渠道"，**不改变告警自身的级别（severity）**，避免污染告警级别统计与趋势。

### 3.4 与现有分派提醒的关系（两个独立时钟）
- 分派规则原有的"通知频率（最大次数 + 间隔）"语义是**同一批人多次催**（级内催促节奏），解决"叫不醒"。
- 升级语义是**没响应就换/扩人**（跨级时间阈值），解决"叫错人/此人无法响应"。
- 两者是两个独立时钟、并存互不替代：同一条告警在等待升级的同时，仍按提醒频率被当前在岗的处理人重复通知。
- **通知去重按"当前累计在岗集合"计算**：替换模式下在岗集合即本层处理人；累加模式下为截至当前层的各层处理人并集。同一轮提醒对同一在岗集合只发一次，避免累加模式下已在岗的人被重复 ping。

### 3.5 与屏蔽策略的关系
- 升级触发的通知同样接受现有屏蔽策略评估；命中屏蔽则抑制该次通知，但**不阻断升级**（升级时钟仍按时间推进到下一层）。

### 3.6 与即时告警的关系
- 即时告警同样可被升级覆盖（前提是其命中的分派规则配置了升级链）。两类规则正交，互不干扰。

### 3.7 不挂事故
- 本期升级只服务于"告警"实体，不挂"事故"。未来若需事故升级，可在此基础上扩展，本期不展开。

## 4. 关键技术架构选择
- **升级内生于分派规则，复用单一通知管道。** 升级链作为分派规则（`AlertAssignment`）的配置区块存储，不新建独立实体、不引入引用关系与全局默认；底层复用现有的提醒轮询任务（`AlertReminderTask`）这一**单一、按间隔轮询、感知告警状态**的循环，由它在跨过本层等待时长时切换当前在岗集合。如此避免"独立升级引擎 + 分派引擎"两套驱动争抢"当前该通知谁"而误呼的风险。
- **"名词给方案2、管道给方案1"的混合形态。** 对用户呈现为一个可识别、可对标 PagerDuty/Opsgenie 的命名"升级链"区块（可发现性），对系统则是分派规则内生配置、零二次绑定（零摩擦激活、覆盖全部已分派告警，包括即时告警）。
- **时间驱动、分钟级实时性。** 以告警停留"待处理"的时长作为升级判据，绑定告警状态机（认领/已解决/已关闭即终止）。实时性要求为分钟级，与现有分派提醒轮询机制同一量级；提醒任务需记录"当前层级"与"本层开始时间"两个时间锚点以支撑跨级判断。

## 5. 验收口径
- 分派规则内可配置升级链：开启/关闭升级、选择升级模式（累加/替换）、增删排序层级，每层指定处理人、等待时长与可选渠道；升级链随分派规则一同保存生效，无独立菜单、无引用绑定、无全局默认。
- 告警在某一层等待时长内未被认领 → 自动进入下一层并通知该层处理人；超过最后一层仍未认领则不再升级，最后一层按级内提醒节奏催促至最大次数后停止。
- 升级以"待处理"时长为判据、不按提醒次数计量；提醒被屏蔽或跳过时升级时钟仍正常推进。
- 替换模式下仅通知本层处理人；累加模式下各层处理人并集持续在岗，且同一在岗集合在同一轮提醒不被重复通知。
- 告警被认领、已解决、已关闭后不再触发后续升级；改派后升级计时从第一层重置开始。
- 升级只改变通知对象与渠道，不改变告警级别。
- 升级通知与原分派提醒并存、互不替代；命中屏蔽时该次通知被抑制，但升级层级仍按时间推进。
- 即时告警在其命中的分派规则配置了升级链时同样被升级覆盖。
- 升级链随分派规则按生效组织正确隔离。

## 6. 约束与边界
**In Scope**：分派规则内的升级链配置区块、累加/替换链级开关、超时升级的时间驱动触发与终止逻辑、与级内提醒/屏蔽机制的并存与去重关系、复用现有提醒轮询任务的单一通知管道。

**Out of Scope**：独立升级策略菜单与跨分派复用、全局默认升级策略、严重度升级、频次升级、手动升级动作、事故级别的升级、跨告警的批量升级、升级过程中的热替换、逐层不同的累加/替换混搭。
