# 告警中心-新增"告警升级"

## 1. 背景与问题
现有告警分派规则只解决"通知给谁、通知多频"两个问题，没有回答"通知了但没人响应该怎么办"。告警分派后，处理人因休假、会议、通知未到达等原因长时间不响应时，告警会一直停留在"待处理"状态直到故障扩大。

业界对标产品（PagerDuty、Opsgenie、Grafana OnCall 等）均提供升级（escalation）能力来兜底这一场景：在规定时长内无人认领，自动换一批更高级/更广的处理人。BK-Lite 缺位，需要补齐。

本期仅做**超时升级**：告警在规定时长内未被认领时，自动切换/扩展到下一层处理人。其他升级形态（严重度升级、频次升级、手动升级）不在本期。

## 2. 业务需求
- 在**告警分派规则**内新增一个命名的、可视化的"升级链"配置区块——它是分派规则的一等公民区块，与"分派人员 / 通知渠道 / 通知频率"并列展示，让从对标产品迁移来的用户一眼认得出"这就是 escalation"。
- **分派人员是初始第一棒责任人**；升级链由有序的多个**升级层**组成，是分派之后逐棒升级的责任人。每层指定处理人与"未认领等待时长"（语义见 §3.1），可选指定通知渠道（未指定则沿用分派规则的渠道）。
- 升级链**内生于分派规则**，随分派规则一同保存、生效、停用，不产生需要二次绑定的独立实体，也没有跨规则复用与全局默认。一条分派规则要么配了升级链、要么没配。
- 初始分派人在其等待时长内未认领，自动升级到第一个升级层并通知该层处理人；逐棒类推；走完最后一棒仍未认领则不再升级。
- 告警被认领、已解决、已关闭后，立即终止后续升级。

## 3. 业务逻辑

### 3.1 升级链配置项

**层级语义（B 模型）**：分派规则的"分派人员"是这条告警的**初始第一棒责任人**；升级链里配置的"升级层级 1、2…"是分派之后**逐棒升级**的责任人。运行期由系统构造"有效链 = [初始分派人] + 各升级层"驱动。

| 分类 | 字段 | 必填 | 说明 |
|---|---|---|---|
| 链级 | 是否启用升级（命名为"告警升级"） | — | 不启用时该分派规则行为与现状完全一致 |
| 链级 | 升级模式 | — | 本期 UI 仅提供**累加**：升级到下一棒时，前面各棒处理人继续保留、新棒加入。替换模式为后端预留能力，本期不在 UI 暴露 |
| 层级 | 升级层级列表 | ✅ | 有序，至少 1 层；每层包含：处理人 / 未认领等待时长（分钟）/ 通知渠道（可选，未指定则沿用分派规则的渠道）。**未认领等待时长的语义**：升到本层之前、**上一棒责任人**的认领窗口——第 1 个升级层的等待时长对应**初始分派人**的窗口，第 2 个升级层的等待时长对应第 1 个升级层处理人的窗口，以此类推；末棒无后续，不再升级。UI 当前约束最多 3 层（数据模型与后端不限层数）|

### 3.2 触发与终止
- **触发口径**：以告警是否被**认领**为信号。告警停留在"待处理"超过当前层的等待时长 → 进入下一层并通知该层处理人。
- **计量口径**：跨级用**时间**——以"本层开始后、告警仍停留在待处理的累计时长"是否达到本层等待时长为判据，**不按提醒次数计量**。这样升级节奏与提醒频率解耦：改提醒间隔不会意外提前或推迟升级；提醒因屏蔽/会话观察期/投递失败被跳过时，升级时钟照常推进，不会静默失效。
- **终止口径**：告警被认领（进入"处理中"）、已解决、已关闭 → 立即终止后续升级。
- **走完仍未响应**：到达最后一层后**不再升级**，但最后一层处理人继续按级内提醒节奏被催促，直到达到本层提醒次数上限后彻底停止（见 3.4 计数口径），告警保持"待处理"由用户后续处置。
- **改派/分派变更**：发生改派时，升级计时与级内提醒计数**双重置**到第一层开始；新分派人若仍不响应，再按完整升级链走。

### 3.3 升级只换人，不改级别
- 升级仅改变"通知谁、用什么渠道"，**不改变告警自身的级别（severity）**，避免污染告警级别统计与趋势。

### 3.4 认领资格：始终累加
- **通知对象（本期累加模式）**：升级到某一棒时，通知截至当前棒的各棒处理人并集（含初始分派人）。
- **认领资格**始终累加：凡是被本条告警的初始分派/升级链通知过的人，都进入告警处理人集合（`Alert.operator`）且不再移除，认领校验认这个累加集合。
- 含义：前面棒的处理人晚些才看到通知，**仍然有权认领**——升级的本意是"多一个人兜底"，不是"剥夺前面人的处理权"。（替换模式为后端预留：仅决定催谁，不决定谁有权接；本期 UI 不暴露。）

### 3.5 与现有分派提醒的关系（两个独立时钟）
- 分派规则原有的"通知频率（最大次数 + 间隔）"语义是**同一批人多次催**（级内催促节奏），解决"叫不醒"。
- 升级语义是**没响应就换/扩人**（跨级时间阈值），解决"叫错人/此人无法响应"。
- 两者是两个独立时钟、并存互不替代：同一条告警在等待升级的同时，仍按提醒频率被当前在岗集合重复通知。
- **级内提醒计数随层重置**：通知频率的"最大次数"定义为"对**同一在岗集合**最多催 N 次"。每次升级跨层都改变了在岗集合（替换=换人、累加=扩人），已不是"同一批人"，因此**进入新层时提醒计数归零、最大次数预算重置**。这样最后一层能拿到完整的催促额度，不会被前面各层提前耗尽而哑火。改派同理重置（见 3.2）。
- **通知去重按"当前累计在岗集合"计算**：替换模式下在岗集合即本层处理人；累加模式下为截至当前层的各层处理人并集。同一轮提醒对同一在岗集合只发一次，避免累加模式下已在岗的人在一轮内被重复 ping。
  - 注：累加模式下重置提醒计数后，并集中的老成员会在新一层的提醒预算内被继续催促——这是预期行为（他们仍挂在未响应的告警上），同一轮去重只保证一轮内不重复，不跨层抑制。

### 3.6 与屏蔽策略的关系（不同阶段、不交叉）
- 屏蔽策略作用于 **Event 聚合阶段**：命中屏蔽的事件被置为屏蔽状态、不参与聚合，**根本不会生成告警**。
- 告警一旦生成，其分派/提醒/升级通知**均不再二次过屏蔽**（现状告警级通知本就不经屏蔽评估）。
- 因此升级与屏蔽在不同阶段、互不交叉：不存在"升级通知命中屏蔽被抑制"这一路径，也无需为升级新建告警级通知抑制能力。升级时钟自然按时间推进，不受屏蔽影响。

### 3.7 与即时告警的关系
- 即时告警同样可被升级覆盖（前提是其命中的分派规则配置了升级链）。两类规则正交，互不干扰。

### 3.8 不挂事故
- 本期升级只服务于"告警"实体，不挂"事故"。未来若需事故升级，可在此基础上扩展，本期不展开。

## 4. 关键技术架构选择
- **升级内生于分派规则，仅复用通知发送出口、不复用提醒轮询循环。** 升级链作为分派规则（`AlertAssignment`）的配置区块存储（落 `config`/新增 JSON 字段，无独立实体、无引用关系与全局默认）。**升级由一个独立的周期扫描任务驱动**，与现有提醒轮询任务（`AlertReminderTask` + `check_and_send_reminders`）并行运行；两者只共享同一个感知告警状态的通知发送出口（`sync_notify`）。
  - **为何不复用提醒轮询循环：** 现有提醒轮询以 `next_reminder_time <= now` 为加载门控，任务只在"该提醒了"才被取出。若把升级判定塞进该循环，升级粒度会被提醒间隔绑架（提醒间隔 30 分钟、层等待 10 分钟时，第 10 分钟根本不会被扫描到），违背"分钟级实时""两个独立时钟"的目标。独立扫描任务以 `layer_deadline <= now` 自行每分钟扫描，升级时钟不看提醒发没发、跳没跳。
- **"名词给方案2、管道给方案1"的混合形态。** 对用户呈现为一个可识别、可对标 PagerDuty/Opsgenie 的命名"升级链"区块（可发现性），对系统则是分派规则内生配置、零二次绑定（零摩擦激活、覆盖全部已分派告警，包括即时告警）。复用的是**发送出口**，不是驱动循环。
- **时间驱动、分钟级实时性。** 以告警停留"待处理"的时长作为升级判据，绑定告警状态机（认领/已解决/已关闭即终止）。升级扫描与现有提醒轮询同为每分钟级，量级一致。
- **独立的升级任务表 `AlertEscalationTask`。** 每条告警在分派且命中规则配了升级链时，创建一条与 `AlertReminderTask` 解耦的升级任务，记录：分派策略、升级模式快照、升级链层级快照、当前层级 index、本层开始时间（升级 due 计算锚点）、是否激活。解耦的原因：提醒任务在该级别无有效提醒频率时不会创建，升级不应被此耦合。级内提醒计数 `reminder_count` 在跨层时归零、`max_count` 预算按当前在岗集合重置。

## 5. 验收口径
- 分派规则内可配置升级链：开启/关闭"告警升级"、增删排序升级层（本期累加模式，最多 3 层），每层指定处理人、等待时长与可选渠道；分派人员=初始第一棒，与升级层各自独立；升级链随分派规则一同保存生效，无独立菜单、无引用绑定、无全局默认。
- 初始分派人在其等待时长内未认领 → 自动升级到第一个升级层并通知该层处理人；逐棒类推；走完最后一棒仍未认领则不再升级，最后一棒按级内提醒节奏催促至本棒最大次数后停止。
- 升级以"待处理"时长为判据、不按提醒次数计量；提醒被跳过或处于会话观察期时升级时钟仍正常推进；升级判定每分钟扫描，不被提醒间隔绑架。
- 认领资格始终累加：凡被初始分派/升级链通知过的人均可认领该告警；累加模式下各棒处理人并集持续在岗，且同一在岗集合在同一轮提醒不被重复通知。
- 升级到新一棒时级内提醒计数归零、最大次数预算重置；改派后升级计时与提醒计数从初始第一棒重置开始。
- 告警被认领、已解决、已关闭后不再触发后续升级。
- 升级只改变通知对象与渠道，不改变告警级别。
- 升级通知与原分派提醒并存、互不替代，由两个独立周期任务共享同一发送出口驱动。
- 屏蔽作用于 Event 聚合阶段，升级通知不二次过屏蔽；两者不交叉。
- 即时告警在其命中的分派规则配置了升级链时同样被升级覆盖。
- 升级链随其所属分派规则的启停而启停：规则停用即停止其下所有告警的后续升级。

## 6. 约束与边界
**In Scope**：分派规则内的升级链配置区块（B 模型：分派人=初始第一棒 + 各升级层）、累加模式、超时升级的时间驱动触发与终止逻辑、认领资格累加、与级内提醒的并存/计数重置/去重关系、独立升级扫描任务（`AlertEscalationTask`）与复用现有通知发送出口、运行期"有效链"构造（`build_effective_chain`）。替换模式为后端预留、本期 UI 不暴露。

**Out of Scope**：独立升级策略菜单与跨分派复用、全局默认升级策略、严重度升级、频次升级、手动升级动作、事故级别的升级、跨告警的批量升级、升级过程中的热替换、逐层不同的累加/替换混搭、告警级通知屏蔽抑制能力（现状不存在，本期不引入）。
