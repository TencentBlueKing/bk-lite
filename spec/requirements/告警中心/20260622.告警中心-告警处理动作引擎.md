# 告警中心-新增"告警处理"能力（通用动作引擎，渐进式交付）

## 1. 背景与问题

告警进入告警中心、经丰富与收敛后，目前的下游只有一条路：**通知人**——按分派规则把告警推送给责任人（企业微信/邮件/Webhook 等渠道）。系统能"告诉人出事了"，但不能"自己去做点什么"。

这带来两个缺口：

1. **没有自动处置闭环**。很多告警的处置动作是高度确定、可脚本化的（重启进程、清理磁盘、扩容、拉起健康检查），运维平台里也已经有现成的作业/脚本（`job_mgmt`）。但告警与作业之间没有任何连接——人收到通知后，还得自己跳到作业平台、手选目标机、手填参数、手动跑一遍。响应被这段"人肉搬运"拉长。

2. **没有对外系统的动作出口**。告警发生后需要在 ITSM 建工单、或回调其他系统的场景，目前完全没有承接，只能靠通知渠道里塞一个自定义 Webhook 勉强凑合，且无配置、无记录、无幂等。

本期要补上的是**告警的"动作"维度**：在通知人之外，让告警能**触发动作**。第一类动作落地"**触发一个已存在的作业**"（对齐夜莺 `${ibex}` 自愈模式：哪台机器告警，就在哪台机器上跑既有脚本）。同时把它建成一套**通用引擎**——动作类型可插拔，后续接 ITSM 建单、Webhook 回调等只是"加一个 handler"，而非重写。

### 现状可复用资产

- 告警中心已有一套**声明式丰富引擎（L1）**：规则模型 + OR-of-AND 匹配器 + 字段绑定 + 可插拔 Provider。本能力的设计语言与之**对称**——把"取数 Provider"换成"执行 Handler"，匹配器与字段绑定直接复用。
- `job_mgmt` 已支持程序化触发作业（按脚本/playbook + 目标列表 + 参数执行，异步运行，支持执行结果回调）。
- 告警生命周期已有创建、确认、升级、关闭/恢复等状态。

### 业界共识（调研背书）

调研了夜莺（n9e）、BigPanda、Keep（keephq）、PagerDuty 四款主流工具的告警处理设计，结论高度收敛：

- **检测与动作解耦、动作做成可插拔的"处理器/Provider/Action 对象"**：四家无一例外（夜莺 Processor 管道、Keep 统一 Provider、PagerDuty Action 对象、BigPanda AutoShare）。
- **触发时机区分生命周期事件**：夜莺自愈脚本"只在触发时跑、不在恢复时跑"；Keep 有 `only_on_change`；BigPanda 按 created/reopened/updated/closed 触发。
- **幂等防风暴**：BigPanda 用 `(incident, target, update)` 幂等键，避免同一动作反复触发；夜莺用重复间隔 + 最大次数。
- **目标/参数从告警动态解析**：夜莺自愈脚本不写死目标机，从事件的 `ident` 自动取主机建任务。
- **ITSM 建单多在 incident（关联后）层级**：BigPanda、PagerDuty 都把建单放在关联后的 incident 上，避免"告警风暴=工单风暴"，且与 ITSM 做字段双向同步。

本期对齐上述共识。

## 2. 业务需求

- **声明式规则驱动**：运维人员可配置"什么样的告警 → 在什么时机 → 触发什么动作 → 用哪些字段作为目标和参数"。规则可启停、按团队隔离。
- **自动触发**：告警**创建**、**状态流转**、**恢复**时，引擎自动评估规则并对匹配到的告警执行动作。触发事件由规则**显式声明**（与夜莺/Keep 一致），未声明的事件不触发。
- **手动触发**：可在告警详情页对单条告警手动执行某个动作，用于人工补救或失败重跑。手动触发不受自动幂等限制。
- **作业动作（本期落地）**：触发 `job_mgmt` 中**已存在的作业/脚本**；**目标主机与执行参数从告警字段动态解析**（复用丰富引擎的字段绑定），实现"哪台告警修哪台"。本期只对接作业，不新建脚本编辑能力。
- **通用可扩展**：动作类型以**可插拔 Handler** 形式存在，规则模型与 Handler 接口本期定形。后续接 ITSM 建单、Webhook 回调等只是新增 Handler，零返工。
- **幂等防风暴**：自动触发下，**同一规则对同一告警的同一触发事件只执行一次**；手动触发不受限。
- **执行可观测**：每次触发落一条执行记录（参数、目标、状态、外部引用如作业执行 ID）；动作结果通过回调回写，**呈现在告警时间线**上。
- **不阻塞告警主流程**：动作执行异步进行，任何动作的失败、超时、外部系统不可用都**绝不阻塞或失败告警本身**的产生与流转。
- **失败不自动重试**：动作失败仅记录并可见，交由人工重跑（本期不做自动重试/冷却重复触发）。

## 3. 业务逻辑（关键行为）

### 3.1 引擎的本质：声明式"匹配 → 动作"原语

动作引擎不是"作业触发器"，而是与具体动作解耦的**声明式触发原语**，与丰富引擎同构：

```
触发事件命中（告警生命周期事件 ∈ 规则声明的触发事件）
  → 条件匹配（OR-of-AND，复用丰富引擎匹配器）
    → 字段绑定（告警字段 → 目标主机 + 动作参数，复用丰富引擎绑定）
      → 调度 Handler（黑盒 execute，引擎不感知动作语义）
        → 记录执行 + 异步执行 + 结果回写时间线
```

引擎只认 `execute(resolved_params) -> result`，**不认识任何动作专有概念**。作业的"脚本 ID/目标列表"在引擎眼里只是一次普通的字段绑定与一份 Handler 配置，其特殊性全部下沉到作业 Handler 内部。这是后续零返工接入 ITSM/Webhook 的根本前提。

### 3.2 触发事件显式声明

规则必须声明它在**哪些告警生命周期事件**上触发（如：创建、状态流转、恢复）。这与夜莺"自愈只在触发时跑、不在恢复时跑"一致——避免恢复时误跑修复作业。手动触发是独立通道，不依赖规则的触发事件声明。

### 3.3 幂等键

自动触发的幂等粒度为 **(规则, 告警, 触发事件)**：同一条规则对同一条告警的"创建"触发一次、"恢复"再触发一次，互不挤占；同一触发事件重复进入则跳过。该粒度借鉴 BigPanda 的 `(incident, target, update)` 幂等键，防止告警抖动引发动作风暴。手动触发不计入幂等，每次都是新执行。

### 3.4 目标与参数的动态绑定

作业动作所需的**目标主机**与**执行参数**，默认从告警（及其已丰富的 CMDB 字段）中按声明绑定解析——对齐夜莺从 `ident` 自动取主机的模式。绑定逻辑复用丰富引擎的字段绑定原语，不重复造轮子。

### 3.5 执行与结果回写

引擎匹配命中后**同步落一条执行记录**（状态=待执行），随即**异步**调用 Handler 执行，避免阻塞告警管线。作业执行结果通过回调回写到执行记录（成功/失败/超时、外部作业执行 ID），并**追加一条告警时间线条目**，使处置过程在告警详情可追溯。失败仅记录，不自动重试。

### 3.6 作用层级：先 alert 级，incident 级预留

本期引擎作用在 **alert（单条告警）层级**——契合作业自愈"针对具体主机"的天然粒度（夜莺同此）。规则的"作用对象"设计为**可扩展的扩展点**，为后续 **incident（关联后事件）层级**预留：ITSM 建单等动作行业上多在 incident 层级触发，以避免"告警风暴=工单风暴"。本期不实现 incident 级触发，但模型不堵死这条路。

## 4. 关键技术选择

| # | 决策 | 理由 / 行业背书 |
|---|---|---|
| 1 | **独立动作引擎，镜像现有丰富引擎** | 与团队刚建成的 L1 引擎同一套设计语言，心智成本低；四家工具均"检测/动作解耦" |
| 2 | **可插拔 Handler 注册表**：本期 job，预留 itsm/webhook | Keep 的 Provider、PagerDuty 的 Action 对象、夜莺的 Processor 同构；满足"现在 job、以后 ITSM" |
| 3 | **复用丰富引擎的匹配器与字段绑定** | 二者与"取数/执行"无关，提取为告警侧共用，避免重复实现 |
| 4 | **触发时机 = 创建 + 状态流转 + 手动**，触发事件在规则上显式声明 | Keep `only_on_change`、BigPanda 生命周期触发、夜莺"自愈只在触发时跑" |
| 5 | **幂等 = (规则, 告警, 触发事件) 一次**（自动），手动不限 | BigPanda 的 `(incident, target, update)` 幂等键，防动作风暴 |
| 6 | **目标/参数从告警字段动态绑定** | 夜莺从 `ident` 自动取主机；契合"哪台告警修哪台" |
| 7 | **作用层级先 alert、incident 级预留** | 作业自愈是主机级；ITSM 建单的 incident 级留待后续阶段 |
| 8 | **执行异步 + 记录 + 回写时间线，不自动重试** | 不阻塞告警管线；失败可见、可人工重跑 |

## 5. 渐进式交付

每一轮独立可用，后一轮是"加 Handler"而非"重写"。

- **L1（本期）**：动作引擎骨架 + 规则模型 + 作业 Handler + 执行记录与时间线回写 + 自动/手动触发。交付"告警触发已有作业"的完整闭环。
- **L1.5（视需要）**：规则配置界面、执行记录列表、告警详情页的动作时间线展示与手动触发入口（面向用户的自助配置/可见性）。
- **L2**：ITSM 建单 Handler + incident 级触发 + 工单字段映射与状态双向同步（对齐 BigPanda/PagerDuty）。
- **L3**：Webhook/脚本等更多 Handler；按需的失败自动重试/冷却窗口、多动作链编排（PagerDuty/Keep 的工作流形态）。

## 6. 本期不做（YAGNI）

- ITSM / Webhook Handler 的具体实现（仅预留抽象与扩展点）。
- incident 级触发。
- 多动作链 / 条件分支编排。
- 失败自动重试、冷却窗口内的重复触发。
- 在告警中心内新建/编辑作业脚本（只触发 `job_mgmt` 中已存在的作业）。
