# 运营分析支持 YAML 导入导出

日期：2026-03-27

## 背景

当前运营分析场景下，目录结构、仪表盘、拓扑、架构图等画布内容，以及画布内组件配置、命名空间、数据源等对象分散维护，缺少统一的配置搬运能力。

这导致用户在跨环境迁移、模板复用、备份恢复、批量分发等场景中，需要逐项重建配置，效率低且容易遗漏对象间依赖关系。

运营分析需要提供基于 YAML 的统一导入导出能力，将一组相关分析资产整体导出，并在目标环境中完成校验、导入、冲突处理和依赖恢复。

## 需求描述

运营分析新增“YAML 导入导出”能力。以统一 YAML 规则承载分析资产配置，支持导入与导出。覆盖对象至少包括：目录结构、仪表盘、拓扑、架构图等画布、画布内组件配置、命名空间、数据源，以及这些对象之间的关联关系。

除产品内页面操作外，该能力还需通过开放接口对外提供，支持外部系统以接口方式触发导出、提交导入、获取校验结果与导入结果。

### 1）导出对象范围与粒度

- 导出入口分为两类：目录下任意节点支持导出；独立画布对象支持直接导出。
- 当用户在目录树任意节点发起导出时，系统按该节点范围递归导出其下全部子目录及分析资产。
- 当用户在单个画布对象上发起导出时，系统按该画布范围导出相关内容。
- 支持导出目录结构，保留目录层级、父子关系和展示顺序。
- 支持导出仪表盘。
- 支持导出拓扑、架构图等画布。
- 支持导出画布内组件配置，包括组件类型、布局信息、展示配置、查询条件、交互配置等内容。
- 支持导出命名空间。
- 支持导出数据源。
- 支持导出对象之间的引用关系，确保导入后能够恢复内容关联，而不是仅恢复孤立对象。
- 支持按单个对象导出，也支持按目录或分析空间批量导出关联内容。

### 2）YAML 规则与章节组织

- 导出结果以 YAML 作为标准载体，内容应具备良好的可读性和可审查性。
- 导入导出使用同一套 YAML 格式规则，不因导入对象类型不同而拆分为多套模板或多种结构定义。
- YAML 中需明确区分不同对象类型，便于用户理解导出内容的组成。
- YAML 应按章节组织不同类型对象内容；当用户仅需导入或导出某一类对象时，仅保留对应章节即可，不要求额外使用该对象专属模板。
- YAML 中需体现对象标识、名称、层级关系、依赖关系、版本信息和必要元数据。
- YAML 中不得直接导出数据库层面的主键 ID、外键 ID 或其他仅在当前环境内有效的内部标识，避免导入后因标识不可复用而导致迁移失败。
- YAML 中对象引用关系应基于可迁移的业务标识或逻辑标识表达，而不是基于数据库层 ID 表达。
- YAML 内容应支持完整表达运营分析对象及其关系，避免因格式限制导致配置丢失。
- 同一份 YAML 应能够表达一组彼此关联的分析资产，而不仅是单个对象快照。

### 3）导出规则

- 用户可选择导出单个仪表盘、单个画布、单个目录，或一组分析资产。
- 当用户仅导出数据源、目录、仪表盘或其他单类对象时，导出结果仍应遵循统一 YAML 格式规则，仅包含对应章节内容。
- 当导出对象存在依赖内容时，系统应仅导出该批内容实际依赖的数据源与命名空间，避免目标环境导入后内容不可用。
- 当导出内容未引用某个数据源或命名空间时，不导出该数据源或命名空间配置。
- 当从目录节点发起导出时，应包含该目录下全部子目录、仪表盘、画布及其他分析资产，并仅分析这些内容实际依赖的数据源与命名空间；不因目录树之外或目录内无引用关系的其他配置而一并导出。
- 当从单个画布发起导出时，仅导出该画布实际依赖的数据源与命名空间。
- 导出时应明确包含哪些对象，避免用户误以为已完整导出。
- 导出内容应支持作为模板复用，也支持作为备份归档。
- 导出结果应具备稳定性，相同配置在未变更情况下重复导出，内容结构应保持一致，便于版本管理与差异比对。

### 4）导入规则

- 系统支持上传 YAML 并执行导入。
- 导入前应对 YAML 内容进行校验，识别格式错误、对象缺失、依赖不完整、字段不合法等问题。
- 当 YAML 仅包含部分章节时，系统应按已提供章节执行导入，不要求用户补齐无关章节或改用其他模板。
- 导入时应能够识别并拒绝使用数据库层 ID 作为对象身份或关联关系的 YAML 内容，避免导入后形成不可迁移或错误绑定的数据。
- 导入时应识别 YAML 中包含的对象类型、数量和依赖关系，并向用户展示导入概览。
- 导入支持将目录结构、仪表盘、画布、组件配置、命名空间、数据源及其关联关系恢复到目标环境。
- 导入仅允许执行新增和更新，不得因 YAML 中未包含某些对象而删除目标环境中的原有配置。
- 导入完成后，恢复后的对象应可正常访问、展示和编辑。
- 导入过程应支持失败拦截，避免在关键依赖缺失时产生不可用的半成品内容。

### 5）开放接口规则

- 导入导出能力需提供开放接口，支持外部系统以编程方式调用，不依赖控制台页面人工操作。
- 开放接口至少覆盖导出、导入提交、导入预检、结果查询四类能力。
- 外部系统可通过接口发起单对象导出或批量导出，并获取 YAML 导出结果。
- 外部系统可通过接口提交 YAML 导入内容，并获得导入任务的受理结果。
- 当导入过程包含预检、冲突识别、失败拦截等环节时，外部系统可通过接口获取明确结果，而不是仅得到成功或失败的简单状态。
- 开放接口返回内容应能够体现对象清单、冲突信息、失败原因、处理结果摘要，便于外部系统集成自动化流程。
- 开放接口应与页面能力保持一致，避免出现页面可导入导出而接口不可用或结果不一致的情况。
- 开放接口需遵循现有平台认证与鉴权机制，确保仅授权调用方可访问相关导入导出能力。
- 开放接口应支持外部系统识别哪些对象被新增、覆盖、跳过或导入失败。

### 6）冲突与覆盖规则

- 当目标环境中已存在同名目录、同名仪表盘、同名画布、同名命名空间或同名数据源时，系统应提供明确的冲突提示。
- 对于冲突对象，系统应支持用户选择处理方式，至少包括跳过、覆盖、另存为新对象。
- 冲突处理应尽量保持对象间引用关系正确，避免仅导入部分内容后导致关联断裂。
- 导入前应让用户明确知道哪些对象会被新增、哪些对象会被覆盖、哪些对象因冲突未处理。
- 导入的覆盖范围仅限于 YAML 中明确声明且被用户确认处理的对象，不得连带删除目标环境中未出现在本次导入包内的其他对象。

### 7）命名空间与数据源规则

- 导入导出需覆盖命名空间与数据源信息，满足跨环境迁移需求。
- 当目标环境不存在 YAML 中引用的命名空间或数据源时，系统应支持一并导入或提示用户完成映射。
- 当目标环境已存在可复用的数据源或命名空间时，系统应支持用户在导入时进行绑定或替换。
- 数据源中的敏感信息不应以可直接泄露的形式导出。
- 若数据源导入后仍需补充认证信息或连接参数，系统应给出明确提示，避免用户误判为可直接使用。

### 8）关联恢复规则

- 目录与子目录关系在导入后需保持一致。
- 目录与仪表盘、目录与画布的归属关系在导入后需保持一致。
- 画布与内部组件的布局、位置、配置、查询关系在导入后需保持一致。
- 仪表盘与其依赖的数据源、命名空间、组件配置之间的关联在导入后需保持一致。
- 拓扑、架构图等画布中的节点、连线、分组、样式配置在导入后需保持一致。
- 所有对象及其关联关系在导入后应由目标环境重新生成内部数据库标识，不依赖源环境数据库 ID 复用。
- 导入后对象不应出现明显缺失、错位、引用失效或页面无法展示的问题。

### 9）风险提示与结果反馈

- 导入前应提供预检结果，帮助用户识别风险，而不是在导入完成后才暴露问题。
- 导入过程中若存在部分对象不可处理，系统应明确提示失败原因和受影响范围。
- 导入完成后应向用户反馈结果摘要，包括成功数量、失败数量、跳过数量、覆盖数量。
- 对于无法自动恢复的依赖项，系统应明确列出待用户处理的项目。
- 导入结果反馈中应明确说明本次导入不会删除目标环境原有配置，避免用户将导入误解为全量同步。
- 导入导出能力应优先服务于迁移、备份、复制、模板下发等真实业务场景，而不是仅提供格式转换功能。

## 适用范围

- 运营分析目录结构导入与导出
- 运营分析仪表盘导入与导出
- 运营分析拓扑、架构图等画布导入与导出
- 运营分析画布内部组件配置导入与导出
- 运营分析命名空间与数据源导入与导出
- 运营分析对象间依赖关系导入与导出
- 运营分析导入导出开放接口

## 非目标（本期不包含）

- 运行时数据、历史查询结果、报表快照等实例数据导出
- 图片、附件、截图、外部文件等非配置型资产导出
- 跨产品的通用配置交换标准制定
- 对现有分析对象模型的大规模重构
- 自动修复全部依赖冲突的智能编排能力
- 基于 Excel、JSON、压缩包等其他载体的多格式导入导出
- 基于导入包对目标环境进行删除式同步或清理未包含对象的能力

## 验收标准

- 用户可在目录下任意节点发起导出，也可在单个画布对象上直接发起导出。
- 当用户从目录节点发起导出时，导出结果包含该目录下全部子目录及其递归内容。
- 用户可将目录结构、仪表盘、拓扑、架构图等画布导出为 YAML。
- 导出的 YAML 中包含画布内部组件配置、命名空间、数据源及必要关联关系。
- 导入导出使用统一 YAML 格式规则；当仅处理数据源、目录或其他单类对象时，仅需保留对应章节即可，无需使用额外模板。
- 当导出内容实际依赖数据源或命名空间时，导出结果包含对应配置；当导出内容不涉及某个数据源或命名空间时，导出结果不包含其配置。
- 导出的 YAML 中不包含数据库层面的主键 ID、外键 ID 或其他不可迁移的内部数据库标识。
- 用户可通过 YAML 将上述内容导入目标环境。
- 外部系统可通过开放接口发起导出，并获取 YAML 导出结果。
- 外部系统可通过开放接口提交 YAML 导入，并获取受理结果、预检结果或执行结果。
- 导入前系统可识别 YAML 合法性，并对格式错误、对象缺失、依赖不完整给出明确提示。
- 当 YAML 依赖数据库层 ID 建立对象身份或关联关系时，系统可明确识别并拒绝导入。
- 导入前系统可展示即将导入的对象清单及对象类型。
- 当目标环境存在同名对象时，系统可给出冲突提示并支持用户选择处理方式。
- 导入执行后，仅对本次导入命中的对象执行新增或更新，不会删除目标环境中原有且未出现在 YAML 中的配置。
- 导入完成后，目录层级、对象归属关系、画布布局及组件配置保持正确。
- 导入完成后，仪表盘、拓扑、架构图等对象可正常打开、展示和编辑。
- 数据源与命名空间在导入后可被正确恢复、映射或提示补全。
- 数据源敏感信息不会以明文可直接泄露的形式出现在导出结果中。
- 导入完成后，系统可反馈成功、失败、跳过、覆盖等处理结果。
- 对于未能自动恢复的依赖项，系统会给出明确清单和后续处理提示。
- 开放接口返回结果中可明确区分新增、覆盖、跳过、失败对象及对应原因。
