# CMDB 采集任务多凭据

日期：2026-05-31

## 背景

采集任务当前只支持录入一组凭据，但实际环境下同一任务里的多个对象常常需要不同凭据，导致用户被迫拆任务、漏采或重复劳动。

## 业务场景

- 实例模式：一个任务采多套主机 / 数据库 / 中间件，账号不一致。
- IP 段模式：发现一段网络上的资产，事先并不知道每个 IP 该用哪把钥匙。

## 业务逻辑

1. **凭据池**：一个采集任务允许配置 1..3 个凭据（同鉴权族，同协议同字段结构）。凭据池有序，作为冷启动时的试探顺序。
2. **命中关系**：执行期间按"对象 ↔ 凭据"记录可用关系，状态为 未探测 / 成功 / 已知失败。作用域为任务，跨任务不共享。**该机制只存在于底层，不在前端展示。**
3. **自适应执行**：每个对象优先用"成功"的凭据；其次按顺序试"未探测"的凭据；"已知失败"的跳过。
4. **失败门槛**：同一 (对象, 凭据) 连续 2 次失败才标记"已知失败"，避免单次抖动误伤。
5. **渐进式冷却期**：失败标记由系统自动控制，用户无需配置。冷却时长按"已知失败"连续次数渐进延长：第 1 次 1 小时 → 第 2 次 4 小时 → 第 3 次及之后 24 小时封顶。任一次试探成功，冷却阶梯归零。
6. **凭据变更**：任意凭据被编辑后，该任务内与之关联的命中关系自动清空。
7. **兼容**：老任务的单组凭据等价于"1 个凭据的池"，行为不变。

## 关键技术架构选择

- 凭据池在采集任务现有凭据字段上做语义升级，老数据迁移为 1 元素列表，不引入新的"凭据分组"结构。
- 命中关系作为任务的附属状态独立存储，按任务隔离，不与采集结果耦合，也不跨任务沉淀；仅服务于派发期决策，不暴露到 API / 前端。
- 命中决策发生在派发阶段——平台侧根据命中关系为每个对象选定凭据，再下沉到 Stargazer 执行；插件侧仍按"单凭据 + 一批目标"契约，无需逐个改造。
- 周期采集的调度与新增 / 更新 / 删除合并口径不变，命中关系只影响"用哪个凭据"，不影响"采到的集合 vs CMDB 已有"的差集。

## 非目标

- 不引入凭据分组（凭据 + 目标范围 scope）。
- 不做跨任务命中共享。
- 不在前端展示命中关系，不提供用户侧的重置入口。
- 不开放冷却期配置项。
- 不改诊断工具与插件契约。

## 验收标准

- 一个任务可配置 1..3 个凭据并正常执行；首次按顺序试探、命中即停。
- 二次起命中"成功"的对象直接复用对应凭据，不再触发其他凭据的鉴权请求。
- 连续 2 次失败后标"已知失败"，冷却期内跳过；冷却时长按 1h → 4h → 24h 渐进（24h 封顶），成功后归零。
- 编辑凭据后，相关命中关系自动清空。
- 老任务无须迁移即可继续运行。
