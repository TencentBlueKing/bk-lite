# CMDB 采集任务多凭据

## 背景与问题

当前采集任务仅支持配置一组凭据。在真实环境中，同一任务下的多个对象经常需要不同账号或密钥：

- 实例模式下，一个任务可能同时采集多套主机、数据库、中间件，不同对象的凭据并不一致。
- IP 段模式下，任务会先发现一批网络目标，执行前并不知道每个目标应该使用哪一组凭据。

现有能力要求用户拆分任务或反复重试，容易带来漏采、重复劳动和配置维护成本上升。

## 本次目标

在不改变用户“一个采集任务管理一批目标”这一使用方式的前提下，为采集任务提供任务内多凭据能力，使平台能够在同一任务内为不同对象选择合适的凭据，并保留老任务的兼容行为；同时让 Stargazer 按固定周期把多凭据执行结果回推到 CMDB，由 CMDB 统一回写命中状态。

## In Scope

- 采集任务支持配置有序的多凭据池。
- 任务内记录“对象 ↔ 凭据”的命中状态。
- 执行时基于命中状态进行自适应凭据选择。
- 支持失败门槛与冷却恢复。
- 支持凭据变更后的命中状态失效。
- Stargazer 周期性将凭据执行结果推送到 CMDB，CMDB 接收后回写命中状态。
- 老任务按单凭据池兼容运行。

## Out of Scope

- 不引入“凭据分组（凭据 + 目标范围）”能力。
- 不做跨任务命中共享。
- 不在本次提供命中状态详情页展示。
- 不在本次提供任务级、对象级的手工重置入口。
- 不改变诊断工具使用方式。
- 不改变插件侧“单凭据 + 一批目标”的契约。
- 命中状态不影响采集结果与 CMDB 已有数据之间的新增、更新、删除差集口径。

## 核心术语

### 凭据池

- 首次发现的目标在尚未落库为 CMDB 实例时，仍可建立命中关系。
- 同一主机上的不同连接目标视为不同对象，例如同一 IP 的不同端口应分别判断凭据命中。
- `已知失败`：该凭据在当前任务内已被判定短期内不应继续尝试。

## 需求项

### 1. 任务多凭据配置

1. 一个采集任务必须支持配置多个凭据。
2. 老任务原有的单组凭据必须等价为“仅包含 1 个元素的凭据池”，行为保持不变。
3. 凭据池中的凭据必须保持顺序，顺序直接决定冷启动时的试探优先级。
4. 同一任务内的多凭据必须保证协议一致、鉴权方式一致、字段结构一致。

### 2. 命中状态范围与约束

1. 命中关系仅在任务内生效，不得跨任务复用。
2. 命中关系必须独立于采集结果存储，不得与单次执行结果快照混存。
3. 同一对象在同一时刻只能有 1 个当前有效的成功凭据。

### 3. 自适应凭据选择

1. 当对象已有成功命中记录时，下一次执行必须优先复用该凭据。
2. 当对象尚无成功命中记录时，平台必须按凭据池顺序进行试探。
3. 状态为“已知失败”的凭据，在冷却期内必须跳过。
4. 某对象一旦在本轮执行中命中某个凭据成功，本轮不得继续对该对象发起其他凭据的鉴权尝试。
5. 对象的凭据选择必须以任务内实际连接目标为单位，避免不同连接目标错误共享命中结果。

### 4. 失败门槛与冷却恢复

1. 同一 `(对象, 凭据)` 关系连续失败 2 次后，才能标记为“已知失败”。
2. 单次失败不得直接将 `(对象, 凭据)` 标记为“已知失败”。
3. “已知失败”关系的冷却时长必须按 1 小时、4 小时、24 小时渐进增长，24 小时封顶。
4. 冷却期结束后，该关系必须自动恢复为“未探测”，允许再次探测。
5. 任一 `(对象, 凭据)` 一旦试探成功，其失败计数与冷却阶梯必须归零。
6. 非凭据问题导致的任务级失败，不应污染对象级命中判断。

### 5. 凭据变更处理

1. 当任务中的任一凭据内容被编辑后，该任务内与该凭据相关的命中关系必须自动清空。
2. 当凭据被删除后，该凭据相关命中关系必须一并移除。
3. 当新增凭据后，平台必须将其视为新的“未探测”候选凭据。

### 6. 结果回写与交互链路

1. 平台完成派发和执行后，Stargazer 必须能够将凭据执行结果按统一格式回推到 CMDB。
2. 结果回写必须支持单条和批量两种 payload 形态。
3. CMDB 必须以接收推送为主链路更新命中状态，不再依赖按 Stargazer 实例主动拉取结果。
4. 多个 Stargazer 实例向同一个 CMDB 推送结果时，CMDB 必须能够按任务和对象维度正确处理，不要求依赖固定实例 ID。

### 7. 兼容与行为稳定性

1. 老任务无需人工迁移即可继续运行。
2. 单凭据任务在升级后不得出现额外鉴权请求或行为变化。
3. 命中状态仅影响“本轮对象使用哪个凭据”，不得改变采集对象集合本身。

## 验收口径

### 基础能力

1. 创建或编辑任务时，可成功保存多个凭据，且同一任务可正常执行。
2. 老任务在不修改配置的情况下可继续执行，结果与升级前一致。

### 冷启动与命中复用

1. 当对象尚无命中记录时，平台按凭据池顺序进行试探，某个凭据成功后立即停止该对象的后续试探。
2. 当对象已有成功命中记录时，下一次执行直接复用该凭据，不再对该对象发起其他凭据的鉴权请求。

### 失败与冷却

1. 同一 `(对象, 凭据)` 第一次失败后，不会立即进入“已知失败”。
2. 同一 `(对象, 凭据)` 连续第二次失败后，进入“已知失败”。
3. 冷却时长按 1 小时、4 小时、24 小时渐进增长，24 小时封顶。
4. 冷却期内该凭据不会再次用于该对象。
5. 冷却期结束后，该凭据可重新参与探测。
6. 一旦同一 `(对象, 凭据)` 再次成功，失败计数与冷却阶梯归零。

### 变更处理

1. 编辑凭据后，相关命中关系会自动清空。
2. 删除凭据后，相关命中关系会自动移除。
3. 新增凭据后，新凭据会作为新的候选凭据参与后续探测。

### 结果回写

1. Stargazer 周期推送后，CMDB 能接收批量结果并正确回写命中状态。
2. 多个 Stargazer 实例并行上报时，CMDB 不依赖固定实例 ID 也能完成结果处理。
3. CMDB 旧的主动拉取链路退出主流程后，不影响新的结果回写闭环。

## 约束与边界

- 本次需求强调“一个任务内多凭据”，不引入按目标范围绑定凭据的静态编排能力。
- 命中关系属于任务运行态，不属于 CMDB 资源事实数据。
- 命中状态只解决凭据复用与试探效率问题，不改变现有周期采集、增量更新、删除识别的业务口径。
