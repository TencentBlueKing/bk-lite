# CMDB 数据治理

日期：2026-06-07

## 背景

CMDB 数据长期靠人工录入与自动采集维护，但缺乏统一的数据质量度量与改进抓手：关键属性是否填全、数据是否仍然新鲜，无人量化；脏数据、僵尸数据沉积，口径逐渐漂移。

本需求引入产品驱动（PLG）式的数据治理：由产品自身度量健康度、暴露具体问题、引导用户自助修复，让用户"边用边治"。渐进式落地，本期先做最直观、可量化的两个维度——**完整性**与**新鲜度**，关系完整性等维度后续扩展。

本需求依赖《模型字段支持治理标记》的 `关键属性` 与 `时效性` 标记；健康度的对外展示见《运营分析-数据治理健康度展示》。

## 需求描述

### 1）健康度判定

- **完整性达标**：实例所属模型的全部关键属性字段均已填写。
- **新鲜度达标**：实例所有带时效性标记的字段都在其档位时窗内被核实过；任一字段超期则不达标（`基本不变` 及未标记字段不参与）。
- **完全健康**：完整性达标且新鲜度达标。
- 只统计存在治理标记字段的模型；无标记的模型不纳入治理。

### 2）健康度评分

- **总健康度** = 完全健康资产占比。
- 下钻为**完整性健康度**与**新鲜度健康度**两维，定位问题来自哪一维。
- 三项评分均可按模型、按组织聚合查看。

### 3）新鲜度的"核实"语义

- 新鲜度衡量的是"数据多久没被核实仍属实"，而非单纯"值多久没变"。
- 字段被变更、被成功重采、或被人工"确认仍有效"，任一都视为一次核实并刷新其新鲜度。

### 4）数据治理页

- 新增「数据治理」页，承载"看问题 → 修问题"闭环。
- **概览**：当前总健康度与两维健康度，可按模型/组织下钻。
- **问题清单**：按模型、组织、问题类型（缺关键属性 / 新鲜度过期）筛出问题实例，展示其所属模型与组织、具体缺了哪些关键属性或哪些字段过期。
- **批量修复**，三种动作：
  - **触发重新采集**：对自动发现覆盖的问题实例，批量重采刷新。
  - **人工编辑**：补全关键属性或更新过期字段。
  - **确认仍有效**：值不变，仅记录一次核实，用于确属真实但长期不变的数据。
- 修复沿用资产的操作权限：批量操作需对全部目标有权限，否则整批失败。

### 5）健康度趋势

- 持续记录健康度的历史变化，使用户能看到治理是否在改善，而非只看当前值。

### 6）面向运营分析的查询能力

- 对外提供健康度（总/两维/按模型/按组织）、历史趋势、问题清单的查询能力，供运营分析展示。

## 关键技术架构选择

- 完整性以关键属性标记为准、新鲜度以时效性档位的系统默认时窗为准，**口径完全沿用《模型字段支持治理标记》，治理页不另设配置**。
- 新鲜度的"核实时间"复用 CMDB 既有的变更记录与采集成功记录，叠加人工确认事件，不引入独立的数据采集。
- 健康度由**周期性后台任务**计算并留存历史快照，趋势与运营分析展示均基于快照，避免每次实时重算。
- 健康度计算、查询、修复均按用户组织权限范围过滤，权限模型复用资产侧既有规则。

## 适用范围

- CMDB 数据治理页（概览、问题清单、批量修复）
- 健康度的周期计算与趋势
- 面向运营分析的治理查询能力

## 非目标（本期不包含）

- 关系完整性维度（孤儿节点、悬挂关系、基数异常）：后续阶段。
- 嵌入式资产页轻提示：后续阶段。
- 归档 / 僵尸资产退役动作：后续阶段。
- 一致性/重复、规范性等其他治理维度：不在本期。
- 治理口径与阈值自定义：沿用系统预定义口径，不在治理页另设配置。
- 健康度的看板展示：见《运营分析-数据治理健康度展示》。

## 验收标准

- 按"完整性达标且新鲜度达标=完全健康"算出总健康度（完全健康资产占比）及两维健康度，并可按模型、组织聚合。
- 仅含治理标记字段的模型纳入统计。
- 新鲜度按"核实"语义判定：变更、成功重采、人工确认任一均刷新新鲜度；超过档位时窗判为过期。
- 数据治理页可按模型/组织/问题类型筛出问题实例并展示具体问题。
- 批量修复支持重采、编辑、确认仍有效三种动作；缺任一目标权限时整批被拦截。
- "确认仍有效"刷新新鲜度且不改变字段值。
- 健康度可查看历史趋势。
- 计算、查询、修复均按用户组织权限范围过滤，并对运营分析提供查询能力。
