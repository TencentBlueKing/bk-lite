# PRODUCT_SENSE.md

> 做产品决策时的「默认价值观」。当需求模糊、取舍冲突时,按本文判断;细化需求见 [openspec/](openspec/) 与 [spec/](spec/)。

## 1. 产品定位

BK-Lite = **AI-first 的轻量运维平台**。两个关键词决定取舍:
- **AI-first**:能力优先以智能化方式呈现(opspilot 助手、算法服务、异常检测),而非堆传统表单。
- **轻量(Lite)**:对标重型运维平台做减法 —— 默认开箱即用,少配置、少前置依赖。功能加法前先问「这会不会让它变重」。

## 2. 默认取舍

| 冲突 | 默认倾向 |
|------|---------|
| 功能完整 vs 上手成本 | 偏上手成本:合理默认 > 暴露全部开关 |
| 灵活可配 vs 开箱即用 | 偏开箱即用,高级项收进二级 |
| 新交互范式 vs 一致性 | 偏一致性:复用 `web/DESIGN.md` 组件,不造轮子 |
| 即时反馈 vs 后台正确 | 用户可见操作优先即时反馈 + 后台补偿(参考告警三层闭环) |

## 3. 运维用户的真实场景

目标用户是运维/SRE,在「出事时」使用。因此:
- **告警/监控**类功能,可读性与可操作性 > 信息密度炫技。最近的工作方向印证这点(仪表盘卡片 tooltip 去术语化 `3bf05c39e`、每行帮助 tooltip `886d0d787`)。
- 错误状态要给「下一步怎么办」,不只报错。
- 危险操作(删除、批量、权限变更)要可回滚、可追踪。

## 4. 何时该停下来问

按 [core-beliefs §2](docs/design-docs/core-beliefs.md),不确定先问、不盲猜。产品层面,以下情况必须先确认而非自行发挥:
- 改变默认行为 / 默认权限范围。
- 新增需要用户额外配置的步骤。
- 影响告警是否触发、数据是否采集等「静默则出事」的链路。
